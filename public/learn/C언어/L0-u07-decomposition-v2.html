Chapter 07

분해(Decomposition)

복잡한 문제를 단순한 부분 문제로 나누는 분해 사고력을 익히고, 일상과 프로그래밍에서 분해가 어떻게 활용되는지 이해합니다.

이 유닛에서 배우는 것

  • 분해(Decomposition)의 정의를 자신의 말로 설명할 수 있습니다.
  • 일상의 복잡한 문제를 3단계 이상으로 분해할 수 있습니다.
  • 프로그래밍에서 큰 프로그램을 작은 기능 단위로 나눌 수 있습니다.
  • 트리 구조를 사용하여 분해 결과를 시각적으로 표현할 수 있습니다.

UNIT 07 분해(Decomposition)
실습 1

"아침에 학교 갈 준비하기"를 떠올려 보십시오. 이 과정을 최소 5가지 단계로 나누어 적으십시오.

실습 2

라면을 끓이는 과정을 순서대로 적어 보십시오. 가능한 한 세부적으로 나누십시오.

실습 3

"수학 시험 100점 받기"라는 목표가 있습니다. 이 목표를 달성하기 위해 해야 할 일을 여러 단계로 나누어 적으십시오.

예측

생각해 볼 점 만약 한 사람이 급식의 모든 과정을 혼자 처리해야 한다면 어떻게 될까요? 여러 사람이 나누어 담당한다면 어떤 점이 달라질까요? 이 질문의 답이 이번 유닛의 핵심입니다.

이야기 — 큰 문제, 작은 조각

1000조각짜리 직소 퍼즐을 받았다고 상상해 보십시오. 1000개의 조각을 한꺼번에 보면 어디서부터 시작해야 할지 막막합니다. 하지만 경험 많은 사람은 이렇게 합니다.

1
1단계: 가장자리 조각 찾기
한쪽 면이 직선인 조각만 골라낸다. 약 100~120개.
2
2단계: 테두리 완성하기
가장자리 조각을 연결하여 퍼즐의 외곽 틀을 만든다.
3
3단계: 색상별 분류
나머지 조각을 하늘(파랑), 나무(초록), 건물(갈색) 등 색상으로 분류한다.
4
4단계: 영역별 조립
분류된 조각들을 영역별로 조립한다.
5
5단계: 영역 연결
완성된 영역들을 서로 연결하여 전체 퍼즐을 완성한다.

"1000조각 퍼즐 완성하기"라는 거대한 문제가, 5개의 작은 문제로 바뀌었습니다. 각각의 작은 문제는 원래 문제보다 훨씬 다루기 쉽습니다. 이것이 바로 분해(Decomposition)입니다.

핵심 원리 큰 문제를 작은 문제로 나누면 해결이 쉬워집니다. 작은 문제 하나하나는 단순하고, 해결 방법도 명확합니다. 모든 작은 문제를 해결하면 큰 문제도 자연스럽게 해결됩니다.

핵심 개념 — 분해란 무엇인가

분해(Decomposition)란 복잡한 문제를 더 단순하고 관리 가능한 부분 문제(sub-problem)로 나누는 사고 과정입니다. 컴퓨팅 사고력의 4대 요소 중 하나이며, 문제 해결의 첫 번째 단계입니다.

분해의 정의 분해(Decomposition)
복잡한 문제나 시스템을 더 작고 관리하기 쉬운 부분으로 나누는 과정입니다.

나눈 각 부분은 독립적으로 이해하고 해결할 수 있어야 합니다.

분해는 특별한 기술이 아닙니다. 우리는 이미 일상에서 분해를 사용하고 있습니다. 다만 의식하지 못할 뿐입니다.

일상 속 분해 — 이사하기

"이사하기"는 매우 복잡한 문제입니다. 하지만 누구도 이사를 한 번의 행동으로 끝내지 않습니다. 자연스럽게 작은 단계로 나누어 진행합니다.

1
단계 1: 짐 싸기
방별로 물건을 분류하고 상자에 담는다. 깨지기 쉬운 물건은 따로 포장한다.
2
단계 2: 운반하기
포장된 상자를 트럭에 싣고 새 집으로 이동한다.
3
단계 3: 짐 풀기
상자를 방별로 배치하고, 물건을 꺼내어 정리한다.

"짐 싸기"도 더 작게 나눌 수 있습니다. 주방 물건 싸기, 거실 물건 싸기, 침실 물건 싸기 등으로 나누면 한 번에 한 방만 집중하면 됩니다. 분해는 필요에 따라 여러 단계로 반복할 수 있습니다.

일상 속 분해 — 요리하기

김치찌개를 만든다고 합시다. 이것도 분해할 수 있습니다.

1
단계 1: 재료 준비
김치 썰기, 두부 깍둑 썰기, 돼지고기 먹기 좋은 크기로 자르기, 대파 송송 썰기
2
단계 2: 조리
냄비에 기름 두르기, 고기 볶기, 김치 넣고 볶기, 물 붓고 끓이기, 두부 넣기
3
단계 3: 담기
그릇에 담고 대파 올리기, 밥과 함께 상에 내기

각 단계가 명확하면 요리를 처음 하는 사람도 따라할 수 있습니다. 분해가 잘 되면 전혀 모르는 일도 실행할 수 있게 됩니다.

분해의 핵심 — 더 이상 나눌 수 없을 때까지

좋은 분해는 각 부분이 충분히 단순해질 때까지 나누는 것입니다. "충분히 단순하다"는 것은 바로 실행할 수 있을 정도로 구체적이라는 뜻입니다.

프로그래밍에서의 분해

프로그래밍에서도 분해는 동일하게 적용됩니다. 큰 프로그램은 작은 기능들의 조합으로 이루어집니다. 한 번에 전체 프로그램을 만드는 것이 아니라, 작은 기능을 하나씩 만들고 조합합니다.

프로그래밍의 분해 원칙 큰 프로그램 = 작은 기능들의 조합

하나의 기능은 하나의 역할만 담당합니다. 각 기능이 독립적이면 수정, 재사용, 테스트가 모두 쉬워집니다.

분해 실습 — 학교 성적 관리 프로그램

"학교 성적 관리 프로그램"을 만들어야 한다고 합시다. 이 프로그램이 해야 할 일은 무엇일까요? 한꺼번에 생각하면 막막하지만, 분해하면 명확해집니다.

1
1단계: 큰 기능 파악
성적 관리 프로그램이 해야 할 일을 나열한다: 성적 입력, 성적 조회, 평균 계산, 석차 산출, 결과 출력
2
2단계: 각 기능을 세분화
"성적 입력"을 더 나눈다: 학생 이름 입력, 과목별 점수 입력, 입력값 검증(0~100 범위 확인)
3
3단계: 가장 작은 단위 확인
"입력값 검증"은 더 나눌 필요 없다. 숫자가 0 이상 100 이하인지 확인하면 끝이다.
4
4단계: 작은 기능부터 구현
가장 단순한 기능(점수 1개 입력받기)부터 만들고, 점차 조합하여 전체를 완성한다.

이 과정을 트리(나무) 구조로 시각화하면 더 명확하게 볼 수 있습니다.

트리 구조로 분해 표현하기

트리의 맨 위(루트)에는 전체 문제가 있고, 아래로 내려갈수록 더 작고 구체적인 부분 문제가 됩니다. 맨 아래의 잎(leaf)은 바로 구현할 수 있는 가장 작은 단위입니다.

트리 구조의 특징 - 위에서 아래로 갈수록 더 구체적입니다.
- 같은 층의 항목들은 서로 독립적입니다.
- 잎(leaf) 노드는 더 이상 나눌 필요가 없는 최소 단위입니다.
- 모든 잎을 해결하면 부모 노드가 해결되고, 결국 루트 전체가 해결됩니다.

분해의 장점

분해의 3가지 핵심 장점 1. 이해가 쉽다
복잡한 전체를 한 번에 이해할 필요가 없습니다. 작은 부분 하나씩 이해하면 됩니다. 1000조각 퍼즐도 한 영역씩 보면 간단합니다.

2. 재사용이 가능하다
잘 만든 작은 기능은 다른 프로그램에서도 쓸 수 있습니다. "평균 계산" 기능은 성적 프로그램뿐 아니라 통계 프로그램에서도 그대로 사용할 수 있습니다.

3. 팀 작업이 가능하다
문제를 나누면 여러 사람이 동시에 작업할 수 있습니다. 한 사람이 입력 기능을 만드는 동안 다른 사람이 계산 기능을 만들 수 있습니다.

분해의 추가 장점

장점설명예시
디버깅 용이문제가 생기면 어느 부분에서 발생했는지 빠르게 찾을 수 있다평균이 틀리면 "평균 계산" 부분만 확인
테스트 용이작은 부분을 독립적으로 테스트할 수 있다입력 기능만 따로 실행하여 확인
유지보수 용이한 부분을 수정해도 다른 부분에 영향이 적다출력 형식을 바꿔도 계산 로직은 그대로
진행률 파악전체 중 얼마나 완성했는지 정확히 알 수 있다5개 기능 중 3개 완성 = 60%
복잡도 감소한 번에 생각해야 할 양이 줄어든다전체 1000줄 대신 한 기능 50줄에 집중

더 많은 분해 예시

분해는 프로그래밍에만 사용되는 것이 아닙니다. 모든 분야에서 복잡한 문제를 해결할 때 분해를 사용합니다.

큰 문제분해 결과
책 한 권 쓰기주제 선정 → 목차 작성 → 챕터별 초안 → 수정 → 교정 → 출판
영화 한 편 만들기시나리오 → 캐스팅 → 촬영 → 편집 → 음향/음악 → 배급
웹사이트 만들기디자인 → 화면 구성(HTML/CSS) → 기능 개발 → 데이터베이스 → 배포
자동차 만들기엔진 → 차체 → 내장 → 전자장치 → 조립 → 검사
학교 축제 준비기획 → 장소 준비 → 무대 설치 → 프로그램 운영 → 뒷정리

잘못된 분해 vs 올바른 분해

분해를 할 때 흔히 하는 실수가 있습니다. 나누기는 했지만 나눈 조각이 서로 겹치거나, 빠진 부분이 있는 경우입니다.

10축 매핑

이번 유닛에서 배운 내용을 10개의 학습 축으로 연결합니다.

축 1: C언어 [필수] 이 유닛의 C 연결: 아직 C 코드를 직접 작성하지 않습니다. 그러나 분해는 C 프로그래밍의 핵심 설계 원칙입니다.
함수(function): C에서 분해의 결과물이 함수입니다. 큰 프로그램을 작은 기능으로 나누면, 각 기능이 하나의 함수가 됩니다.
main 함수: C 프로그램의 main 함수는 전체 프로그램의 루트(root)입니다. main에서 다른 함수를 호출하는 구조가 곧 분해 트리입니다.
축 2: 컴퓨팅 사고력 [필수] 사고 패턴: 분해(Decomposition)
컴퓨팅 사고력 4대 요소: 분해, 패턴 인식, 추상화, 알고리즘

분해는 문제 해결의 첫 번째 단계입니다. 문제를 나누지 않으면 패턴을 찾을 수도, 추상화할 수도, 알고리즘을 설계할 수도 없습니다. 분해가 모든 것의 출발점입니다.
축 3: 디버깅 [필수] 분해와 디버깅의 관계:
프로그램을 잘 분해하면 디버깅이 쉬워집니다. 문제가 발생했을 때 전체 프로그램을 뒤질 필요 없이, 어느 기능에서 문제가 생겼는지 빠르게 좁혀 나갈 수 있습니다.

디버깅도 분해다: "프로그램이 안 된다" → "어느 기능이 안 되는가?" → "그 기능의 어느 단계가 안 되는가?" → 원인 발견
축 4: KOI — 정보올림피아드 [필수] 대회 문제 풀이 = 분해
대회 문제를 받으면 가장 먼저 할 일은 분해입니다.
1. 입력을 읽는다
2. 핵심 계산을 한다
3. 결과를 출력한다

핵심 계산 부분을 더 나누면 풀이 전략이 보입니다. 이번 유닛의 대회 문제 10개를 분해하는 연습을 통해 이 감각을 익히십시오.
축 5: 코딩 습관 [선택] 코드를 작성하기 전에 분해하는 습관:
- 바로 코드부터 치지 않는다
- 종이에 트리 구조를 그린다
- 각 기능을 한 문장으로 설명할 수 있는지 확인한다
- 가장 작은 기능부터 구현한다

이 습관이 있으면 어떤 복잡한 프로그램도 두렵지 않습니다.
축 6: 실생활 연결 [선택] 분해는 프로그래밍만의 기술이 아닙니다. 일상의 모든 복잡한 문제에 적용됩니다.

- 시험 공부: 과목별 → 단원별 → 개념별로 나누어 학습
- 대청소: 방별로 나누고, 각 방을 정리/청소/정돈으로 나눔
- 여행 계획: 교통, 숙소, 일정, 예산으로 나누어 준비
- 프로젝트 관리: 기업에서도 WBS(Work Breakdown Structure)라는 이름으로 동일한 분해 기법을 사용합니다
