에러 헌터란?

에러 헌터 프로젝트는 버그가 숨어있는 코드를 받아서 모든 에러를 찾고 수정하는 활동입니다. 실제 프로그래밍에서 디버깅은 코드 작성만큼 중요한 능력입니다.

에러 헌터 절차 Step 1: 코드를 읽고 눈에 보이는 문제를 표시합니다.
Step 2: 컴파일합니다. 에러 메시지를 읽고 첫 번째 에러부터 수정합니다.
Step 3: 컴파일 에러가 모두 해결되면 실행합니다.
Step 4: 실행 결과를 기대 출력과 비교합니다.
Step 5: 결과가 다르면 논리 에러를 찾아 수정합니다.
Step 6: gcc -Wall로 경고까지 모두 해결합니다.

미션 1 — 컴파일 에러 사냥

아래 코드에는 컴파일 에러가 5개 있습니다. 모두 찾아 수정하십시오.

기대 출력 Hello, World!
I love C programming!
Let's go!
Cmission1.c
1#include <stdio.h>;
2
3int main(void) {
4 printf("Hello, World!\n")
5 printf("I love C programming!\n");
6 printf("Let's go!\n")
7 retrun 0;
8}
1
에러 1: 1번 줄
#include ; → 전처리 지시자 뒤에 세미콜론이 있습니다. 세미콜론을 제거합니다.
2
에러 2: 4번 줄
printf("Hello, World!\n") → 세미콜론이 없습니다. 끝에 ; 을 추가합니다.
3
에러 3: 6번 줄
printf("Let's go!\n") → 세미콜론이 없습니다. 끝에 ; 을 추가합니다.
4
에러 4: 7번 줄
retrun → return 철자 오류입니다.
5
에러 5: 닫는 }
닫는 중괄호 }가 없습니다. return 0; 다음 줄에 }를 추가합니다.

기대 출력 ====================
  자기소개
====================
이름: 김코딩
학교: 코딩중학교
취미: 프로그래밍

Cmission2.c
1#include <stdio.h>
2
3int main(void) {
4 printf("====================\n");
5 printf(" 자기소개\n")
6 printf("====================\n");
7 printf("이름: 김코딩\n");
8 printf("취미: 프로그래밍\n");
9 printf("학교: 코딩중학교\n");
10 printf("====================\n");
11 return 0;
12}
1
컴파일 에러: 5번 줄

2
논리 에러: 8-9번 줄
기대 출력은 학교 → 취미 순서이지만 코드는 취미 → 학교 순서입니다. 8번과 9번 줄의 순서를 바꿉니다.

미션 3 — 에러 메시지 해독

에러 메시지만 보고 원인을 추론하는 연습입니다.

Ccmd
1quiz.c:3:5: error: expected '{' before 'printf'
2quiz.c:4:5: error: expected ';' before 'return'
3quiz.c:5:1: error: expected identifier or '(' before '}' token

이 에러 메시지로부터 원인을 추론합니다.

1
에러 1 해독
3번 줄의 printf 앞에 { 가 필요합니다 → int main(void) 뒤에 여는 중괄호 {가 없습니다.
2
에러 2 해독
4번 줄의 return 앞에 ; 가 필요합니다 → 3번 줄의 printf 끝에 세미콜론이 없습니다.
3
에러 3 해독
에러 1({누락)을 수정하면 이 에러도 사라집니다. 연쇄 에러입니다.
핵심 교훈 에러 3개가 표시되었지만 실제 원인은 2개(여는 { 누락 + 세미콜론 누락)입니다. 항상 첫 번째 에러부터 수정하고 재컴파일하십시오.

미션 4 — 주석 처리 디버깅

코드가 길어지면 에러 위치를 찾기 어렵습니다. 주석 처리로 범위를 좁히는 기법을 연습합니다.

Cmission4.c
1#include <stdio.h>
2
3int main(void) {
4 printf("Step 1\n");
5 printf("Step 2\n");
6 printf("Step 3\n")
7 printf("Step 4\n");
8 printf("Step 5\n");
9 return 0;
10}

이 코드는 컴파일 에러가 발생합니다. 하지만 어디서 에러가 나는지 모르겠다면 주석 처리로 범위를 좁힙니다.

1
Step 1: 후반부 주석 처리
Step 3~5를 /* */로 감쌉니다. 컴파일합니다. → 성공. 문제는 후반부에 있습니다.
2
Step 2: 범위 좁히기
Step 3만 살리고 Step 4~5를 주석 처리합니다. 컴파일합니다. → 에러 발생. Step 3에 문제가 있습니다.
3
Step 3: 원인 확인
6번 줄 printf("Step 3\n")에 세미콜론이 없습니다. 추가합니다.
4
Step 4: 주석 해제
모든 주석을 해제하고 다시 컴파일합니다. → 성공.
TIP: 이진 탐색 디버깅 코드를 절반으로 나누어 주석 처리하고, 에러가 어느 절반에 있는지 확인합니다. 다시 그 절반을 반으로 나눕니다. 이 방법을 반복하면 에러 위치를 빠르게 좁힐 수 있습니다.

이것은 이진 탐색(binary search) 알고리즘의 원리와 같습니다.

용어 정리

용어설명
디버깅(debugging)프로그램의 버그(오류)를 찾아 수정하는 과정입니다. 프로그래밍에서 가장 중요한 능력 중 하나입니다.
버그(bug)프로그램의 오류를 뜻합니다. 1947년 하버드 컴퓨터에서 발견된 나방(bug)에서 유래했습니다.
이진 탐색 디버깅코드를 절반씩 나누어 에러가 있는 범위를 좁히는 디버깅 기법입니다. 이진 탐색 알고리즘과 같은 원리입니다.
연쇄 에러(cascade error)하나의 실수가 여러 에러 메시지를 발생시키는 현상입니다. 첫 번째 에러를 수정하면 나머지가 사라집니다.
기대 출력(expected output)프로그램이 올바르게 동작할 때 나와야 하는 출력입니다. 실제 출력과 비교하여 논리 에러를 찾습니다.

10축 매핑

이번 유닛에서 배운 내용을 10개의 학습 축으로 연결합니다.

축 1: C언어 [필수] 이 유닛의 C 지식: 에러 유형 종합, gcc -Wall, 에러 메시지 해독 실전
깊이: LAYER 1 — U01~U10의 모든 문법과 에러를 종합하여 실전 적용
핵심: 코드를 작성하는 것만큼 디버깅하는 것이 중요합니다
축 2: 컴퓨팅 사고력 [필수] 사고 패턴: 알고리즘적 사고(Algorithmic Thinking)
에러 헌터 절차(읽기→컴파일→수정→실행→비교→경고해결)는 하나의 알고리즘입니다. 정해진 절차를 따르면 누구나 체계적으로 디버깅할 수 있습니다.

주석 처리 디버깅은 이진 탐색 알고리즘의 원리를 디버깅에 적용한 것입니다.
축 3: 디버깅 [필수] 이 유닛의 디버깅 종합:
1. 에러 메시지 읽기 (파일:줄:설명)
2. 첫 번째 에러부터 수정
3. 주석 처리로 범위 좁히기 (이진 탐색)
4. 기대 출력과 실제 출력 비교 (논리 에러)
5. gcc -Wall로 경고까지 해결

이 다섯 가지 기법을 모든 디버깅에 적용하십시오.
축 4: KOI — 정보올림피아드 [필수] 난이도: TIER 1
대회에서 디버깅:
- CE(컴파일 에러) → 에러 메시지 읽기
- WA(틀렸습니다) → 출력 비교, 예외 케이스 확인
- RE(런타임 에러) → 0으로 나누기, 범위 초과 확인
- 디버깅 속도가 대회 성적을 좌우합니다
축 7: 프로젝트 [선택] 이 유닛 자체가 프로젝트입니다.
에러 헌터는 기획→설계→구현이 아닌, 분석→진단→수정의 프로젝트입니다.

실무에서 버그 리포트를 받으면 이와 동일한 절차로 문제를 해결합니다.
