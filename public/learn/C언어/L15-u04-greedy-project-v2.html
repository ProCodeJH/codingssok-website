동전 거스름돈 그리디의 정당성
그리디가 최적해를 보장하는 조건: 큰 동전이 작은 동전의 배수여야 합니다.

한국 화폐 {500, 100, 50, 10}:
500 = 5 x 100, 100 = 2 x 50, 50 = 5 x 10
-> 배수 관계 성립 -> 그리디 최적해 보장

반례 {400, 300, 100}으로 600원을 거슬러 줄 때:
그리디: 400 x 1 + 100 x 2 = 3개
최적해: 300 x 2 = 2개
-> 배수 관계 불성립 -> 그리디 실패

배수 관계가 아닐 때는 동적 프로그래밍(DP)이 필요합니다.

27/* 그리디 동전 교환 */
28CoinResult greedyCoinChange(const CoinSet *cs, int amount) {
29 CoinResult r;
30 memset(&r, 0, sizeof(r));
31 r.valid = 1;
32
33 int remain = amount;
34 for (int i = 0; i < cs->count; i++) {
35 r.used[i] = remain / cs->coins[i];
36 remain -= r.used[i] * cs->coins[i];
37 r.totalCoins += r.used[i];
38 }
39
40 if (remain != 0) {
41 r.valid = 0; /* 거슬러 줄 수 없음 */
42 }
43 return r;
44}

46/* 완전 탐색으로 최적해 찾기 (금액이 작을 때만 사용) */
47int bruteMinCoins;
48int bruteUsed[MAX_COINS];
49int bestUsed[MAX_COINS];
50
51void bruteForceCoin(const CoinSet *cs, int idx, int remain,
52 int coinCount) {
53 if (remain == 0) {
54 if (coinCount < bruteMinCoins) {
55 bruteMinCoins = coinCount;
56 memcpy(bestUsed, bruteUsed, sizeof(int) * cs->count);
57 }
58 return;
59 }
60 if (remain < 0 || idx >= cs->count) return;
61 if (coinCount >= bruteMinCoins) return; /* 가지치기 */
62
63 int maxUse = remain / cs->coins[idx];
64 for (int k = maxUse; k >= 0; k--) {
65 bruteUsed[idx] = k;
66 bruteForceCoin(cs, idx + 1,
67 remain - k * cs->coins[idx],
68 coinCount + k);
69 }
70 bruteUsed[idx] = 0;
71}

대회 기출

요약

핵심 정리
1. 동전 거스름돈: 큰 단위부터 최대한 사용하는 그리디 전략
2. 그리디 정당성 조건: 동전이 배수 관계일 때 최적해 보장
3. 반례: {400, 300, 100}으로 600원 -- 그리디 3개 vs 최적 2개
4. 활동 선택: 종료 시간 기준 오름차순 정렬 후 그리디 선택
5. 정렬 기준이 핵심: 시작 시간 정렬은 최적해를 보장하지 않음
6. 겹침 판정: start >= lastEnd (등호 포함, 종료와 시작이 같으면 가능)
7. 그리디 시간 복잡도: O(n log n) -- 정렬이 지배적
8. 완전 탐색: O(2^n) -- 정확하지만 n이 크면 사용 불가
9. 그리디 적용 조건: 탐욕 선택 속성 + 최적 부분 구조
10. 그리디 한계를 이해하면 DP로의 전환 시점을 판단할 수 있음

다음 유닛으로

다음: DP 기초
동전 교환과 회의실 배정을 통해 그리디 알고리즘의 강력함과 한계를 모두 경험했습니다. 동전 종류가 배수 관계가 아닐 때 그리디는 실패하며, 이때 필요한 것이 동적 프로그래밍(DP)입니다. 다음 유닛에서는 DP의 기본 원리인 메모이제이션과 상향식 테이블 채우기를 학습하고, 그리디가 실패하는 동전 교환 문제를 DP로 해결합니다.
