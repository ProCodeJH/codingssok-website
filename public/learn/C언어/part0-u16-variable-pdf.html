수학 시간의 x

수학 시간에 'x + 3 = 7에서 x는?' 이런 문제 본 적 있지? 여기서 x는 미지수야. '아직 모르는 수'라는 뜻이야.

수학에서 x + 3 = 7이라고 쓰면, x가 뭔지 찾아야 해요. x는 4죠? 여기서 x는 '아직 모르는 수', 즉 미지수예요. 등호(=)는 '왼쪽과 오른쪽이 같다'는 뜻이에요.

[핵심] 수학의 변수
수학에서 변수 = 미지수 (모르는 수를 찾는 것)
수학에서 = 기호 = '같다' (왼쪽과 오른쪽이 같은 값)

코딩 세계의 x

코딩에서 x = 7이라고 쓰면, 뜻이 완전히 달라요. 'x라는 이름의 상자에 7을 넣어라'라는 뜻이에요. x라는 이름표가 붙은 상자를 하나 꺼내서, 그 안에 7이라는 숫자를 넣는 거예요.

변수 = 이름표가 붙은 상자. x = 7은 'x 상자에 7을 넣어라'!
[핵심] 코딩의 변수
코딩에서 변수 = 이름표가 붙은 저장 상자 (값을 넣어두는 곳)
코딩에서 = 기호 = '넣어라' (오른쪽 값을 왼쪽 상자에 넣어라)
수학에서 x = 7은 'x가 7이다'이고, 코딩에서 x = 7은 'x에 7을 넣어라'야. 같은 글자인데 뜻이 완전 달라!

등호(=)의 두 얼굴

같은 '=' 기호인데, 수학과 코딩에서 의미가 완전히 달라요!

x = x + 1이 가능하다고?!

수학에서 x = x + 1을 쓰면? 말도 안 돼요! 어떤 수가 자기 자신에 1을 더한 것과 같을 수는 없으니까요. 하지만 코딩에서는 완전히 자연스러운 명령이에요!

코딩에서 x = x + 1은 '상자에서 꺼내고 → 1 더하고 → 다시 넣어라'라는 뜻이에요!
x = x + 1 추적
단계명령x의 값
1int x = 5;5
2x = x + 1;6
3x = x + 1;7
4x = x + 1;8
수학 선생님이 보면 'x = x + 1은 불가능!' 이라고 하겠지만, 코딩 선생님은 '아주 자주 쓰는 표현이야!'라고 할 거야.

C언어에서 변수 사용하기

C언어에서 변수를 쓰려면 먼저 선언해야 해요. '이런 이름의 상자를 만들겠다'라고 컴퓨터에게 알려주는 거예요.

variable_demo.c
1#include <stdio.h>
2
3int main(void) {
4 int x; // 정수를 담을 상자 x를 만들어라
5 x = 7; // x 상자에 7을 넣어라
6 printf("x = %d\n", x); // x = 7
7
8 x = x + 1; // x를 꺼내서 1 더하고 다시 넣어라
9 printf("x = %d\n", x); // x = 8
10
11 x++; // x = x + 1의 줄임말!
12 printf("x = %d\n", x); // x = 9
13
14 // 선언과 동시에 초기화도 가능
15 int y = 100;
16 printf("y = %d\n", y); // y = 100
17
18 return 0;
19}
실행 결과
x = 7 x = 8 x = 9 y = 100

변수 이름 규칙

[핵심] C 변수 이름 규칙
가능: 영문자, 숫자, _ (밑줄)

[심화] 변수의 정체 — 메모리 주소

변수라는 '상자'는 사실 컴퓨터 메모리(RAM)의 특정 위치예요. 모든 메모리 위치에는 주소(address)가 있어요.

address_demo.c
1#include <stdio.h>
2
3int main(void) {
4 int x = 42;
5
6 printf("x의 값: %d\n", x); // 42
7 printf("x의 주소: %p\n", &x); // 0x7ffd1234abcd (매번 다름)
8 printf("x의 크기: %zu바이트\n", sizeof(x)); // 4
9
10 // & = 주소 연산자 ("어디에 있니?")
11 // sizeof = 크기 연산자 ("얼마나 크니?")
12
13 return 0;
14}
실행 결과
x의 값: 42 x의 주소: 0x7ffd1234abcd x의 크기: 4바이트
[심화] 변수 = 이름 + 주소 + 크기 + 값
변수는 4가지 속성이 있어요:

1. 이름: 프로그래머가 부르는 이름 (예: score)
2. 주소: 메모리에서의 위치 (예: 0x7ffd...)
3. 크기: 차지하는 바이트 수 (int = 4바이트)
4. 값: 현재 저장된 데이터 (예: 42)

컴파일러는 이름을 주소로 바꿔요. 실행 파일에는 변수 이름이 없어요!
편지에 비유하면, 변수 이름은 '받는 사람', 주소는 '우편 주소', 크기는 '우체통 크기', 값은 '편지 내용'이야!

[심화] 변수의 스코프 (범위)

변수가 살아있는 범위를 스코프(scope)라고 해요. 중괄호 {} 안에서 선언한 변수는 그 안에서만 사용 가능!

scope_demo.c
1#include <stdio.h>
2
3int g = 100; // 전역 변수: 프로그램 전체에서 사용 가능
4
5int main(void) {
6 int a = 10; // 지역 변수: main 함수 안에서만
7
8 {
9 int b = 20; // 블록 변수: 이 중괄호 안에서만
10 printf("a=%d, b=%d, g=%d\n", a, b, g); // OK
11 }
12
13 // printf("%d", b); // ❌ 에러! b는 위 블록에서만 존재
14 printf("a=%d, g=%d\n", a, g); // OK
15
16 return 0;
17}

[심화] const — 변하지 않는 변수

값이 절대 바뀌면 안 되는 변수는 const를 붙여요. 실수로 바꾸려 하면 컴파일러가 에러를 내요.

const_demo.c
1#include <stdio.h>
2
3int main(void) {
4 const int MAX_SCORE = 100; // 상수: 바꿀 수 없음
5 const double PI = 3.14159265358979;
6
7 printf("최고 점수: %d\n", MAX_SCORE);
8 printf("원주율: %.2f\n", PI);
9
10 // MAX_SCORE = 200; // ❌ 컴파일 에러! const는 못 바꿔요
11
12 return 0;
13}
[팁] #define vs const
#define MAX 100 — 전처리기 매크로 (타입 없음, 디버깅 어려움)
const int MAX = 100; — 상수 변수 (타입 있음, 디버깅 쉬움)

현대 C에서는 const를 추천해요!

컴파일러가 int x = 42;를 mov eax, 42로 변환해요!

[심화 내용] 어셈블리로 보는 변수

var_asm.asm
1; int x = 42; x = x + 1;
2; x86-64 어셈블리
3
4 mov DWORD PTR [rbp-4], 42 ; x = 42 (스택에 저장)
5 mov eax, DWORD PTR [rbp-4] ; eax = x(메모리 → 레지스터)
6 add eax, 1 ; eax = eax + 1
7 mov DWORD PTR [rbp-4], eax ; x = eax(레지스터 → 메모리)
8
9; 최적화하면:
10 mov DWORD PTR [rbp-4], 43 ; 컴파일러가 미리 계산!
[팁] 컴파일러 최적화
컴파일러는 x = 42; x = x + 1;을 보면 x = 43으로 미리 계산해요!
이걸 상수 접기(constant folding)라고 해요.

[심화 내용] 변수 바인딩 — 이름과 값의 연결

프로그래밍 언어 이론에서 변수는 이름(Name)과 속성(Attribute)의 바인딩(Binding)이에요.

[심화 내용] 변수 바인딩의 6가지 속성
변수 = 다음 6가지의 바인딩:

1. 이름(Name): 식별자 (x, count, total)
2. 타입(Type): int, float, char 등
3. 주소(Address): 메모리 위치 (l-value)
4. 값(Value): 저장된 데이터 (r-value)
5. 수명(Lifetime): 메모리가 할당/해제되는 시점
6. 스코프(Scope): 이름이 유효한 코드 범위

바인딩 시점:
- 컴파일 타임: C의 int (타입), static 변수 (주소)
- 링크 타임: extern 변수 (주소)
- 로드 타임: 전역 변수 (주소)
- 런타임: 지역 변수 (주소, 값)
[심화 내용] l-value vs r-value
l-value = 주소가 있는 것 (대입의 왼쪽에 올 수 있는 것)
r-value = 값 (대입의 오른쪽에 올 수 있는 것)

int x = 5;
x = x + 1;

왼쪽 x = l-value (x의 주소, "어디에 넣을까?")
오른쪽 x = r-value (x의 값 5, "뭘 읽을까?")
x + 1 = r-value (계산 결과 6)

5 = x; // 에러! 5는 l-value가 아님 (주소 없음)
x + 1 = 10; // 에러! 수식은 l-value가 아님

[심화 내용] SSA — 컴파일러의 변수 관리

현대 컴파일러는 내부적으로 변수를 SSA(Static Single Assignment) 형태로 변환해요.

[심화 내용] SSA 형태
SSA 규칙: 모든 변수는 딱 한 번만 대입된다

원래 코드:
x = 1;
x = x + 2;
x = x * 3;

SSA 변환:
x1 = 1;
x2 = x1 + 2; // x1을 읽어서 x2에 저장
x3 = x2 * 3; // x2를 읽어서 x3에 저장

왜 SSA?
- 각 "버전"이 독립적 -> 최적화가 쉬움
- 불필요한 변수 제거, 상수 전파, 루프 최적화
- LLVM, GCC 등 모든 현대 컴파일러가 사용

x = x + 1이 수학적으로 불가능해 보이지만,
SSA에서는 x2 = x1 + 1이라 자연스러워요!

✓ 올바른 코드

[심화 내용] Rust의 소유권 시스템
Rust는 제3의 방법: 소유권(Ownership)

- 각 값은 딱 하나의 소유자(변수)만 가짐
- 소유자가 스코프를 벗어나면 자동 해제
- GC 없이도 메모리 안전!
- 컴파일 타임에 메모리 오류 검출

C의 성능 + GC의 안전성 = Rust의 소유권

2024 StackOverflow 설문: "가장 사랑받는 언어" 8년 연속 1위!

[심화 내용] 변수의 진화
1940s — 기계어: 메모리 주소를 직접 사용 (변수 이름 없음)
MOV [0x1000], 42 (주소 0x1000에 42 저장)

1950s — FORTRAN: 최초의 변수 이름 도입
X = 42 (컴파일러가 주소를 자동 배정)

1970s — C: 타입 시스템 + 포인터
int x = 42; (타입 + 이름 + 값)

1990s — Java: 가비지 컬렉션 + 참조 타입
Integer x = 42; (자동 메모리 관리)

2010s — Rust: 소유권 시스템
let x: i32 = 42; (컴파일 타임 메모리 안전)

2020s — AI 코드 생성: 변수 이름도 AI가 추천!
6가지
변수의 바인딩 속성 수: 이름, 타입, 주소, 값, 수명, 스코프. 이 6가지가 변수의 모든 것!
~300배
CPU 레지스터가 RAM보다 빠른 정도. 변수를 레지스터에 넣으면 극적인 성능 향상!
변수(Variable)
값을 저장할 수 있는 이름 붙은 상자. 수학에서는 미지수, 코딩에서는 저장 공간.
대입(Assignment)
변수에 값을 넣는 것. C에서 = 기호가 하는 일.
미지수
수학에서 아직 모르는 수. 방정식을 풀어서 값을 찾아요.
선언(Declaration)
변수를 처음 만드는 것. int x; 처럼 타입과 이름을 정해요.
초기화(Initialization)
변수를 만들면서 동시에 값을 넣는 것. int x = 0;
스코프(Scope)
변수가 사용 가능한 범위. 중괄호 {} 안에서 선언하면 그 안에서만 사용 가능.
전역 변수
함수 바깥에 선언. 프로그램 전체에서 사용 가능. 자동 0 초기화.
지역 변수
함수/블록 안에 선언. 그 안에서만 사용 가능. 초기화 안 하면 쓰레기 값.
const
값을 바꿀 수 없는 상수 변수. 실수로 수정하면 컴파일 에러.
레지스터
CPU 내부의 초고속 저장소. RAM보다 약 300배 빠름.
메모리 주소
RAM에서 변수가 위치한 곳의 번호. & 연산자로 확인.
++ 연산자
x = x + 1의 줄임말. x의 값을 1 증가시켜요.
== (비교 연산자)
'같은가?'를 확인하는 기호. = (대입)과 헷갈리면 안 돼요!
바인딩
이름과 속성(타입, 주소, 값 등)의 연결. 컴파일/런타임에 결정.
l-value
주소가 있는 표현식. 대입문의 왼쪽에 올 수 있는 것.
r-value
값을 나타내는 표현식. 대입문의 오른쪽에 올 수 있는 것.
SSA
Static Single Assignment. 각 변수가 딱 한 번만 대입되는 중간 표현. 컴파일러 최적화의 핵심.
가비지 컬렉션(GC)
사용하지 않는 메모리를 자동으로 해제하는 시스템. Java, Python 등에서 사용.
소유권(Ownership)
Rust의 메모리 관리 방식. 각 값에 하나의 소유자만 허용.
메모리 누수
할당한 메모리를 해제하지 않아 계속 쌓이는 버그. C에서 free 누락 시 발생.
