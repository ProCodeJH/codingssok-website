알고리즘이 뭘까?

요리할 때 레시피 따라하지? 1번 재료 준비, 2번 볶기, 3번 간하기... 이 레시피가 바로 알고리즘이야! 컴퓨터한테 주는 레시피라고 생각하면 돼!
[핵심] 알고리즘이란?
알고리즘(Algorithm) = 문제를 해결하기 위한 단계별 절차

알고리즘의 5가지 조건:
1. 입력이 있어요 (0개 이상)
2. 출력이 있어요 (1개 이상)
3. 유한성: 유한한 단계로 끝나야 해요
4. 명확성: 각 단계가 애매하지 않아야 해요
5. 실행가능성: 각 단계를 실제로 수행할 수 있어야 해요
레시피 = 요리의 알고리즘, 알고리즘 = 컴퓨터의 레시피!
알콰리즈미
'알고리즘'이라는 말은 9세기 페르시아 수학자 알콰리즈미(Al-Khwarizmi)의 이름에서 왔어요. 그는 대수학(Algebra)의 창시자이기도 해요!

좋은 알고리즘의 3가지 조건

정확한 답 + 빠른 속도 + 읽기 쉬운 코드 = 좋은 알고리즘!
좋은 알고리즘의 3조건
조건의미예시
정확성모든 입력에 올바른 답1~100 합 = 반드시 5050
효율성적은 시간·공간으로 해결O(n) vs O(n²) 차이
명확성읽고 이해하기 쉬움좋은 변수명, 주석, 구조

같은 문제, 다른 알고리즘

같은 문제라도 풀 수 있는 방법이 여러 가지예요. '가장 큰 수 찾기'를 3가지 방법으로 비교해요!

방법마다 비교 횟수가 달라요!
find_max_linear.c
1#include <stdio.h>
2
3// 방법 3: 순차 탐색 — 가장 효율적! O(n)
4int findMax(int arr[], int n) {
5 int max = arr[0]; // 첫 번째를 일단 최대로
6 for (int i = 1; i < n; i++) {
7 if (arr[i] > max)
8 max = arr[i]; // 더 큰 수 발견!
9 }
10 return max;
11}
12
13int main(void) {
14 int data[] = {5, 2, 8, 1, 9, 3, 7};
15 int n = 7;
16 printf("최댓값: %d\n", findMax(data, n)); // 9
17 return 0;
18}
실행 결과
최댓값: 9
[핵심] 효율의 차이가 왜 중요할까?
데이터 7개일 때는 차이가 작지만...
데이터 1,000,000개면?

• 순차 탐색(O(n)): 약 100만 번 → 0.001초
• 모든 쌍 비교(O(n²)): 약 5천억 번 → 수 분~수 시간

데이터가 커질수록 효율의 차이가 폭발적으로 벌어져요!

알고리즘 → C 코드 변환 3단계

말(사람 언어) → 의사코드(반쯤 코드) → C 코드(실행 가능)!
sum_pseudocode.c
1// ===== 1단계: 말로 쓰기 =====
2// "1부터 N까지 하나씩 더해서 합을 구한다"

20}

[심화] 1부터 N까지 합: 반복 vs 공식

sum_formula.c
1#include <stdio.h>
2
3int main(void) {
4 int n = 100;
5
6 // 방법 1: 반복문 — O(n) 시간
7 int sum1 = 0;
8 for (int i = 1; i <= n; i++)
9 sum1 += i;
10
11 // 방법 2: 가우스 공식 — O(1) 시간!
12 int sum2 = n * (n + 1) / 2;
13
14 printf("반복문: %d\n", sum1); // 5050
15 printf("공식: %d\n", sum2); // 5050
16 // 둘 다 같은 답이지만 공식은 한 번에 끝!
17 return 0;
18}
[팁] 가우스의 전설
10살 가우스에게 선생님이 "1부터 100까지 더해!"라고 시켰을 때
가우스: "1+100=101, 2+99=101, ... 50쌍이니까 101×50=5050!"

반 아이들이 100번 더하는 동안 1번의 곱셈으로 끝냈어요!
이게 바로 알고리즘의 힘이에요.

[심화] 시간 복잡도 — Big-O

알고리즘의 효율을 비교할 때 Big-O 표기법을 사용해요.

[심화] Big-O 표기법
입력 크기 N에 따라 연산 횟수가 어떻게 증가하는지 나타내는 표기법

빠른 순서:
• O(1) — 상수: 입력 크기와 무관 (공식 계산)
• O(log n) — 로그: 이진 탐색
• O(n) — 선형: 한 번 순회
• O(n log n) — 선형 로그: 병합/퀵 정렬
• O(n²) — 이차: 2중 반복문
• O(2ⁿ) — 지수: 부분집합 탐색
• O(n!) — 팩토리얼: 순열 탐색
N에 따른 연산 횟수 비교
Big-ON=10N=100N=1000N=10⁶
O(1)1111
O(log n)371020
O(n)101001,000100만
O(n log n)336649,9662,000만
O(n²)10010,000100만1조!
O(2ⁿ)1,02410³⁰∞∞
O(n²)이 '느리다'고 느껴지면 O(2ⁿ)을 보세요... N=100이면 우주의 나이보다 오래 걸려요! 알고리즘 선택이 얼마나 중요한지 알겠죠?

[KOI 필수] 패러다임별 키워드
문제를 보면 어떤 패러다임을 쓸지 판단해야 해요!

• "모든 경우" "N ≤ 20" → 완전 탐색
• "최대/최소" "정렬 후" → 그리디
• "반으로 나눠" "재귀" → 분할정복
• "최대/최소" "점화식" "N ≤ 10000" → DP
• "조합" "가능한 배치" → 백트래킹

[심화] 알고리즘 설계 실전: 동전 거슬러주기

coin_greedy.c
1#include <stdio.h>
2
3int main(void) {
4 int coins[] = {500, 100, 50, 10};
5 int n = 4;
6 int amount = 1260;
7 int count = 0;
8
9 printf("%d원 거슬러주기:\n", amount);
10 for (int i = 0; i < n; i++) {
11 int use = amount / coins[i];
12 if (use > 0) {
13 printf(" %d원 × %d개\n", coins[i], use);
14 count += use;
15 amount -= coins[i] * use;
16 }
17 }
18 printf("총 동전 %d개\n", count);
19 return 0;
20}
실행 결과
1260원 거슬러주기: 500원 × 2개 100원 × 2개 50원 × 1개 10원 × 1개 총 동전 6개
[주의] 그리디가 항상 최적은 아니야!
동전이 {500, 400, 100, 10}이면?
800원: 그리디 → 500+100+100+100 = 4개
최적: 400+400 = 2개!

그리디가 최적인지 증명이 필요해요!

[심화 내용] 알고리즘 정당성 증명

[심화 내용] 정당성 증명 방법
알고리즘이 정확하다는 것을 어떻게 증명할까?

1. 수학적 귀납법: n=1일 때 성립 → n=k 성립 가정 → n=k+1 증명
2. 루프 불변식(Loop Invariant): 반복문 시작 전·중·후에 항상 성립하는 조건
3. 모순법: 최적이 아니라고 가정 → 모순 발견 → 최적 증명

예: 순차 탐색의 정당성
불변식: 'i번째까지 검사한 원소 중 최대값이 max에 저장되어 있다'

[심화 내용] P vs NP 문제

[심화 내용] P vs NP — 밀레니엄 문제
P: 다항 시간(O(nᵏ))에 풀 수 있는 문제
→ 정렬, 최단 경로, 최대 흐름

NP: 답이 주어지면 다항 시간에 검증할 수 있는 문제
→ 스도쿠, 외판원, 해밀턴 경로

P = NP? — 아직 아무도 모름!
100만 달러 상금이 걸린 밀레니엄 7대 난제 중 하나!

NP-완전(NP-Complete) 문제를 만나면?
→ 완벽한 해 대신 근사 알고리즘 사용

[심화 내용] 알고리즘 설계 패러다임

[심화 내용] 5대 알고리즘 설계 기법
1. 브루트포스 (완전 탐색)
모든 경우를 다 시도. 확실하지만 느림.
O(n!), O(2^n) 등

2. 분할 정복 (Divide & Conquer)
나누고 → 각각 풀고 → 합치기
Merge Sort O(n log n), Binary Search O(log n)

3. 동적 프로그래밍 (DP)
중복 부분 문제를 메모이제이션으로 한 번만 계산
피보나치, 배낭 문제, 최단 경로

4. 탐욕법 (Greedy)
매 순간 최선의 선택. 최적 보장은 아님.
거스름돈, Kruskal, Dijkstra

5. 백트래킹 (Backtracking)
시도 → 실패하면 되돌리기
N-Queens, 스도쿠, 미로 탈출

[심화 내용] P vs NP 문제

[심화 내용] 컴퓨터 과학 최대의 미해결 문제
P: 다항 시간에 풀 수 있는 문제
정렬 O(n log n), 최단 경로 O(V+E), ...

NP: 다항 시간에 검증할 수 있는 문제
답이 맞는지 확인은 빠르지만, 찾기는 어려울 수 있음

NP-완전: NP 중 가장 어려운 문제들
SAT, 해밀턴 경로, 부분합, 그래프 색칠, ...
하나라도 다항 시간에 풀면 → 전부 풀림!

P = NP?
지금까지 아무도 증명/반증 못 함
클레이 수학 연구소 밀레니엄 상금 100만 달러

대부분의 전문가: P ≠ NP라고 믿지만 증명은 못 함

[심화 내용] 정지 문제 — 계산의 한계

[심화 내용] 정지 문제 (Halting Problem)
질문: "주어진 프로그램이 유한 시간에 끝나는가?"
를 판단하는 프로그램을 만들 수 있는가?

답: 불가능! (튜링, 1936)

증명 (대각선 논법):
만약 Halt(P, I) 함수가 존재한다면...
D(P) = if Halt(P,P) then 무한루프 else 멈춤
D(D)는? Halt(D,D)가 참이면 무한루프... 모순!

의미:
1. 완벽한 디버거/바이러스 탐지기는 불가능
2. 모든 프로그램의 정확성을 자동 검증하는 건 불가능
3. 수학에서 증명 불가능한 참인 명제가 존재 (괴델)

컴퓨터에도 근본적인 한계가 있다!
[C 코드] 설계 패러다임 비교 - 거스름돈
1#include <stdio.h>
2
3// 그리디: 큰 동전부터
4void greedy_change(int amount) {
5 int coins[] = {500, 100, 50, 10};
6 int count = 0;
7 printf("%d원 거스름돈 (그리디): ", amount);
8 for (int i = 0; i < 4; i++) {
9 while (amount >= coins[i]) {
10 amount -= coins[i];
11 count++;
12 printf("%d ", coins[i]);
13 }
14 }
15 printf("(%d개)\n", count);
16}
17
18int main() {
19 greedy_change(1260);
20 greedy_change(730);
21 greedy_change(500);
22 return 0;
23}

[활동] 알고리즘 설계 연습
다음 문제를 어떤 설계 기법으로 풀지 골라보세요:

1. 사전에서 단어 찾기 → 분할 정복 (이진 탐색)
2. 거스름돈 최소 동전 수 → 탐욕법
3. 계단 오르기 경우의 수 → DP
4. 비밀번호 맞추기 → 완전 탐색
5. 미로 탈출 → 백트래킹

문제를 보고 어떤 패러다임이 맞는지
판단하는 능력 = 알고리즘 설계 실력!
알고리즘
문제를 해결하기 위한 유한한 단계별 절차. 입력→처리→출력.
의사코드
프로그래밍 언어는 아니지만 코드처럼 쓴 알고리즘 표현.
정확성
모든 입력에 대해 올바른 답을 내는 성질.
효율성
적은 시간·공간으로 문제를 해결하는 성질.
Big-O
알고리즘의 시간/공간 복잡도를 나타내는 표기법. O(n), O(n²) 등.
시간 복잡도
입력 크기에 따른 연산 횟수의 증가 패턴.
그리디
매 순간 최선의 선택을 하는 알고리즘 전략.
분할정복
문제를 나누고, 각각 풀고, 합치는 전략.
DP
Dynamic Programming. 중복 계산을 저장하여 재사용하는 전략.
백트래킹
유망하지 않은 경로를 가지치기하여 탐색을 줄이는 전략.
NP
답의 검증이 다항 시간인 문제 클래스. P=NP? 미해결.
루프 불변식
반복문 실행 전·중·후에 항상 성립하는 조건. 정당성 증명에 사용.
Big-O 표기법
알고리즘의 최악 시간 복잡도를 나타내는 표기. O(n), O(n log n) 등.
동적 프로그래밍(DP)
중복 부분 문제를 메모이제이션으로 한 번만 계산하는 기법.
NP-완전
NP 중 가장 어려운 문제 클래스. 하나 풀면 전부 풀림.
정지 문제
프로그램이 유한 시간에 끝나는지 판단하는 문제. 불가능함이 증명됨(튜링, 1936).
근사 알고리즘
최적해는 못 구하지만 "충분히 좋은" 해를 빠르게 구하는 알고리즘.
메모이제이션
계산 결과를 저장해두고 같은 입력이 오면 재사용하는 기법.
