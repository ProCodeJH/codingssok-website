컴파일이 뭐야?

우리가 쓴 C 코드는 사람 언어에요. 컴퓨터는 이걸 못 읽어요. 그래서 기계어로 번역하는 과정이 필요해요. 이걸 컴파일이라고 해요!

컴파일(compile)은 사람이 쓴 소스 코드(.c 파일)를 컴퓨터가 이해하는 기계어로 한꺼번에 번역하는 과정이에요. 이 번역을 해주는 프로그램이 컴파일러이고, 우리는 gcc라는 컴파일러를 사용해요.

소스코드(.c) → 컴파일러(gcc) → 실행파일(.exe)
[핵심] 컴파일 실습 명령어
터미널에서 다음 명령어를 입력해요:

gcc hello.c -o hello

• gcc — 컴파일러 이름
• hello.c — 번역할 소스 코드 파일
• -o hello — 결과 파일 이름을 hello로 지정

실행하려면:
./hello (Mac/Linux) 또는 hello.exe (Windows)

컴파일 v 인터프리터

프로그래밍 언어를 기계어로 바꾸는 방법은 두 가지가 있어요.

컴파일은 책 전체를 미리 번역하는 것이고, 인터프리터는 한 줄씩 동시통역하는 거예요. C는 미리 번역하니까 빠르고, Python은 한 줄씩 하니까 편하지만 느려요!
컴파일(C) = 책 전체 번역 출판 / 인터프리터(Python) = 동시통역
[핵심] 컴파일 vs 인터프리터 비교
컴파일 방식 (C, C++, Java)
• 코드 전체를 한꺼번에 번역 → 실행 파일 생성
• 실행 속도가 빠름
• 번역 중 에러가 나면 실행 파일이 안 만들어짐

인터프리터 방식 (Python, JavaScript)
• 코드를 한 줄씩 번역하며 실행
• 실행 속도가 느림
• 에러가 난 줄까지는 실행됨

에러의 3가지 종류

코드를 쓰다 보면 반드시 에러를 만나게 돼요. 에러는 3종류가 있어요. 각각 원인과 해결법이 달라요.

에러의 3종류 — 언제 발생하고, 어떤 느낌인지 비교!
[핵심] 에러 3종류
1. 컴파일 에러 = 번역 실패
• 문법이 틀렸을 때 발생 (세미콜론 빠짐, 오타 등)
• 실행 파일이 아예 안 만들어져요
• 예: printf("hello") ← 세미콜론 없음!

2. 런타임 에러 = 실행 중 폭발
• 컴파일은 됐지만 실행 중에 문제 발생
• 프로그램이 갑자기 멈춰요
• 예: 0으로 나누기, 없는 파일 열기

3. 논리 에러 = 결과가 틀림
• 에러 메시지 없이 프로그램은 잘 돌아가는데 답이 틀림
• 가장 찾기 어려운 에러!
• 예: 더하기를 해야 하는데 빼기를 한 경우
에러는 무서운 게 아니에요! 에러 메시지는 컴퓨터가 '여기가 이상해요!'라고 알려주는 친절한 도움말이에요. 에러를 읽는 습관을 들이세요!
[활동] 일부러 에러 3종류 만들기
1단계 — 컴파일 에러 만들기
세미콜론을 빼고 컴파일해 보세요. 에러 메시지를 읽어보세요.

2단계 — 런타임 에러 만들기
printf("%d", 10/0); 을 써보세요. 컴파일은 되지만 실행하면?

3단계 — 논리 에러 만들기
'나이 + 1'을 출력해야 하는데 '나이 - 1'을 출력하는 코드를 써보세요.

용어 정리

[심화] 컴파일 과정 완전 분석

[심화] 컴파일 4단계 상세
소스 코드 → 실행 파일까지:

1단계: 전처리 (Preprocessing)
gcc -E main.c -o main.i
- #include → 헤더 파일 내용 삽입
- #define → 매크로 치환
- 주석 제거
- #ifdef 조건부 컴파일
결과: 순수 C 코드 (수천~수만 줄!)

2단계: 컴파일 (Compilation)
gcc -S main.i -o main.s
- 어휘 분석 (토큰화)
- 구문 분석 (파싱, AST 생성)
- 의미 분석 (타입 체크)
- 중간 코드 생성
- 최적화
- 어셈블리 코드 생성

3단계: 어셈블 (Assembly)
gcc -c main.s -o main.o
- 어셈블리 → 기계어 (바이너리)
- 오브젝트 파일 생성
- 아직 주소 미결정 (재배치 정보 포함)

4단계: 링킹 (Linking)
gcc main.o -o main
- 여러 .o 파일 결합
- 라이브러리 함수 연결 (printf 등)
- 최종 주소 결정
- 실행 파일 생성!
[C 코드] 컴파일 각 단계 체험
1// step1.c - 각 단계를 직접 보세요!
2#include <stdio.h>
3#define GREETING "Hello from step1!"
4
5int add(int a, int b) {
6 return a + b;
7}
8
9int main() {
10 printf("%s\n", GREETING);
11 printf("3 + 5 = %d\n", add(3, 5));
12 return 0;
13}
14
15// 터미널에서 실행:
16// gcc -E step1.c -o step1.i (전처리 결과 보기)
17// gcc -S step1.c -o step1.s (어셈블리 보기)
18// gcc -c step1.c -o step1.o (오브젝트 파일)
19// gcc step1.o -o step1 (링킹 → 실행 파일)
20// ./step1 (실행!)
21
22// step1.i: stdio.h 내용이 수천 줄 삽입됨!
23// step1.s: mov, call printf 등 어셈블리
24// step1.o: 바이너리 (읽기 불가)
25// step1: 실행 가능!
실행 결과
Hello from step1! 3 + 5 = 8

[심화 내용] 컴파일러 내부 구조

[심화 내용] 컴파일러의 6단계
Front-end (언어 의존):

1. 어휘 분석 (Lexical Analysis)
소스 코드 → 토큰 스트림
int main() → [INT] [IDENT:main] [LPAREN] [RPAREN]
도구: Lex, Flex

2. 구문 분석 (Syntax Analysis)
토큰 → 추상 구문 트리 (AST)
a + b * c → +(a, *(b, c))
도구: Yacc, Bison

3. 의미 분석 (Semantic Analysis)
타입 체크, 선언 확인
int x = "hello"; → 에러!

Middle-end (최적화):

4. 중간 코드 생성 + 최적화
상수 접기: 3+5 → 8
죽은 코드 제거
루프 최적화

Back-end (기계 의존):

5. 코드 생성
중간 코드 → 타겟 기계어
x86, ARM, RISC-V 등

6. 레지스터 할당
변수를 CPU 레지스터에 매핑
NP-완전 문제! 휴리스틱 사용

[심화 내용] 컴파일러 최적화 예제

[심화] 컴파일 vs 인터프리트 상세 비교

[심화] 같은 코드, 다른 실행 방식
printf("Hello"); 실행 경로:

C (컴파일):
1. gcc: C → 어셈블리 → 기계어 (.exe)
2. OS가 .exe 로드
3. CPU가 기계어 직접 실행
4. 시스템 콜로 화면에 출력
속도: ★★★★★ (가장 빠름)

Java (바이트코드+JIT):
1. javac: Java → 바이트코드 (.class)
2. JVM 로드
3. 인터프리트 시작, 자주 쓰는 코드는 JIT 컴파일
4. 점점 빨라짐 (워밍업)
속도: ★★★★ (JIT 후 C에 근접)

Python (인터프리트):
1. CPython: 소스 → 바이트코드 (.pyc)
2. PVM(Python VM)이 바이트코드 해석
3. 한 명령씩 실행
속도: ★★ (C의 ~1/50)

[심화 내용] 크로스 컴파일과 타겟 아키텍처

[심화 내용] 크로스 컴파일
크로스 컴파일: 다른 플랫폼용 실행 파일 생성

예: PC(x86)에서 ARM용 코드 컴파일
arm-linux-gnueabihf-gcc main.c -o main_arm

왜 필요한가:
1. 라즈베리파이(ARM)에서 직접 컴파일 → 느림
2. PC에서 크로스 컴파일 → 빠름!
3. 임베디드 장치는 컴파일러 설치 불가

대표 타겟:
- x86_64: PC, 서버
- ARM: 스마트폰, 라즈베리파이
- RISC-V: 차세대 오픈소스 CPU
- MIPS: 라우터, 게임기(PSP)
- AVR: 아두이노

같은 C 코드가 다른 CPU에서 돌아가는 이유!
= C의 이식성 (Portability)
[C 코드] 조건부 컴파일 — 플랫폼 대응
1#include <stdio.h>
2

13 #endif

15 printf("OS: macOS\n");

[심화 내용] 빌드 최적화 기법

[심화 내용] 빌드 시간 줄이기
1. 증분 빌드 (Incremental Build)
변경된 파일만 재컴파일 → Make가 자동 처리

2. 프리컴파일드 헤더 (PCH)
자주 쓰는 헤더를 미리 컴파일
#include 매번 파싱 → 한 번만!

3. 병렬 빌드
make -j8 → 8개 코어로 동시 컴파일

4. ccache

[심화 내용] 컴파일러 만들기 — 미니 프로젝트

[심화 내용] 간단한 수식 계산기 컴파일러
목표: "3+5*2" → 13 계산하는 미니 컴파일러

1단계: 토크나이저
"3+5*2" → [3] [+] [5] [*] [2]

2단계: 파서 (재귀 하강)
expr = term ((+|-) term)*
term = factor ((*|/) factor)*
factor = number | (expr)

AST: +(3, *(5, 2))

3단계: 평가
*(5,2) = 10
+(3, 10) = 13

이게 컴파일러의 핵심 원리!
GCC도 이것의 수만 배 복잡한 버전일 뿐

대학 컴파일러 수업 첫 과제가 바로 이것!
[C 코드] 미니 수식 파서
1#include <stdio.h>
2#include <ctype.h>
3
4const char *p;
5
6int number() {
7 int n = 0;
8 while (isdigit(*p)) n = n*10 + (*p++ - '0');
9 return n;
10}
11
12int factor() {
13 if (*p == '(') { p++; int r = expr(); p++; return r; }
14 return number();
15}
16
17int term() {
18 int r = factor();
19 while (*p=='*'||*p=='/') {
20 char op = *p++;
21 int f = factor();
22 r = (op=='*') ? r*f : r/f;
23 }
24 return r;
25}
26
27int expr() {
28 int r = term();
29 while (*p=='+'||*p=='-') {
30 char op = *p++;
31 int t = term();
32 r = (op=='+') ? r+t : r-t;
33 }
34 return r;
35}
36
37int main() {
38 const char *tests[] = {"3+5*2", "10-2*3", "(3+5)*2", "100/4+3*2"};
39 for (int i = 0; i < 4; i++) {
40 p = tests[i];
41 printf("%s = %d\n", tests[i], expr());
42 }
43 return 0;
44}

[심화 내용] 면접 문제
Q1. 전처리, 컴파일, 어셈블, 링킹의 차이?
A: 전처리=#매크로 처리, 컴파일=C→어셈블리, 어셈블=어셈블리→기계어, 링킹=여러 .o 결합+라이브러리 연결

Q2. -O2와 -O0의 차이?
A: -O0=최적화 없음(디버깅용), -O2=상수접기/인라이닝/루프최적화 등 적용(릴리스용)

Q3. 정적 링킹 vs 동적 링킹?
A: 정적=라이브러리 코드 복사(독립 실행, 큼), 동적=실행 시 .so/.dll 로드(작음, 의존성)

Q4. "undefined reference" 에러 원인?
A: 함수를 선언했지만 정의가 없거나, 해당 .c 파일 / 라이브러리를 링킹하지 않음
