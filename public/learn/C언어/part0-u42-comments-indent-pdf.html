주석과 들여쓰기

코드에 메모(주석)를 남기고, 깔끔하게 정리(들여쓰기)하는 법을 배워요.

주석이란?

코드를 쓸 때, 미래의 나에게 메모를 남길 수 있어요. 이 메모를 주석이라고 해요. 컴퓨터는 주석을 완전히 무시하고, 사람만 읽어요!

주석(comment)은 코드 속에 쓰는 메모예요. 컴퓨터는 주석을 무시하고 건너뛰어요. 주석은 오직 사람을 위한 것이에요. 1주일 전에 내가 쓴 코드도 까먹을 수 있으니까, 주석을 남겨두면 나중에 '아, 이게 이런 뜻이었구나!' 하고 바로 이해할 수 있어요.

주석의 두 가지 종류 — // 한 줄 주석과 /* */ 여러 줄 주석
[핵심] 주석 쓰는 법
1. 한 줄 주석 — // 뒤에 쓰기
// 이것은 한 줄 주석이에요
printf("안녕!"); // 화면에 출력

2. 여러 줄 주석 — /* */ 사이에 쓰기
/* 이것은
여러 줄에 걸친
주석이에요 */
주석 = 미래의 나에게 보내는 편지예요! 3개월 후에 코드를 다시 열어보면, 주석 덕분에 '아, 이게 이거구나!' 바로 이해할 수 있어요.

들여쓰기란?

들여쓰기(indentation)는 중괄호 { } 안의 코드를 오른쪽으로 밀어서 쓰는 것이에요. 보통 Tab 키 또는 Space 4칸으로 해요. 들여쓰기를 하면 '이 코드가 어디에 속하는지' 한눈에 보여요.

왼쪽은 읽기 힘든 나쁜 코드, 오른쪽은 깔끔한 좋은 코드
[주의] 나쁜 코드 예시
#include <stdio.h>
int main(void){
int a=10;int b=20;int c;
c=a+b;printf("%d",c);return 0;}

이 코드는 돌아가긴 하지만, 읽기가 너무 힘들어요! 변수 이름도 a, b, c라서 뭘 하는 코드인지 모르겠어요.
[핵심] 좋은 코드 예시
#include <stdio.h>

int main(void) {
int score1 = 10; // 첫 번째 점수
int score2 = 20; // 두 번째 점수
int total; // 합계

total = score1 + score2;
printf("%d", total);

return 0;
}

들여쓰기도 되어있고, 변수 이름도 의미가 있고, 주석도 있어요!

좋은 변수 이름 짓기

변수 이름을 a, b, c, x, y처럼 짓지 마세요! 나중에 보면 뭐가 뭔지 몰라요. 의미가 담긴 이름을 지어야 해요.

[팁] 좋은 변수 이름 vs 나쁜 변수 이름
나쁜 이름: a, b, c, x, y, temp1, temp2
좋은 이름: score, name, age, height, total, count

변수 이름만 보고 '이 변수가 무슨 역할인지' 알 수 있으면 좋은 이름이에요!
코드는 쓰는 시간보다 읽는 시간이 훨씬 많아요. 주석, 들여쓰기, 좋은 이름 — 이 세 가지가 코드를 읽기 쉽게 만들어줘요!
들여쓰기를 하면 코드의 구조가 한눈에 보여요
[활동] 나쁜 코드 vs 좋은 코드 퀴즈
아래 코드를 주석, 들여쓰기, 좋은 변수 이름을 넣어서 다시 써 보세요:

int main(void){int a=5;int b=3;int c;c=a*b;printf("%d",c);return 0;}

힌트: a는 가로, b는 세로, c는 넓이예요.

[심화] 들여쓰기 스타일

[심화] 주요 들여쓰기 스타일
1. K&R 스타일 (Kernighan & Ritchie)
int main() {
if (x > 0) {
printf("양수");
}
}
→ 가장 널리 사용. Linux 커널 스타일.

2. Allman 스타일 (BSD)
int main()
{
if (x > 0)
{
printf("양수");
}
}
→ 중괄호가 새 줄. Visual Studio 기본.

3. GNU 스타일
int main ()
{
if (x > 0)
{
printf ("양수");
}
}
→ 2칸 들여쓰기. 중괄호도 들여쓰기.

[심화 내용] 코딩 스타일 가이드

[심화 내용] 유명 코딩 스타일 가이드
1. Linux Kernel Coding Style
- 탭 = 8칸 (넓은 들여쓰기)
- K&R 스타일 중괄호
- 함수는 80줄 이하
- 줄 길이 80자 이하 (모니터 작던 시절 유산)
- "깊은 중첩 대신 조기 반환"

2. Google C++ Style Guide
- 2칸 들여쓰기
- CamelCase 함수, snake_case 변수
- 클래스 멤버 뒤 밑줄: member_

3. MISRA C (자동차/의료)
- goto 금지, 동적 할당 금지
- switch에 default 필수
- 모든 분기에 중괄호 필수

[심화 내용] 클린 코드 원칙

[심화 내용] Robert C. Martin의 클린 코드
"Clean Code" (2008)의 핵심 원칙:

1. 의미 있는 이름
int d; → int elapsed_days;
int t; → int total_score;
이름만 보고 용도를 알 수 있게!

2. 함수는 한 가지만
하나의 함수 = 하나의 역할
20줄 이하가 이상적

3. 주석 대신 코드로 표현
// 학생이 합격인지 확인
if (score >= 60) → if (is_passing(score))

4. 중복 제거 (DRY)
Don't Repeat Yourself
같은 코드 2번 쓰면 → 함수로!

5. 일관된 형식
들여쓰기, 중괄호, 네이밍 규칙 통일
팀 전체가 같은 스타일

6. 에러 처리
해피 패스만 코딩하지 말 것

[활동] 코드 리뷰 연습
다음 코드의 문제점을 찾아보세요:

int f(int* a,int n){
int s=0;int i;
for(i=0;is=s+a[i];}return s;}

문제점:
1. 함수명 f → sum_array
2. 변수명 s, a, n → sum, array, count
3. 들여쓰기 없음
4. 중괄호 일관성 없음
5. 공백 부족 (가독성 저하)
6. 주석 없음 (무슨 함수인지 모름)

같은 기능이지만 읽기 좋게 다시 작성해 보세요!

Don't Repeat Yourself. 같은 코드를 두 번 쓰지 마라. 중복은 버그의 씨앗!
코드는 쓰는 시간보다 읽는 시간이 10배 많아. 그래서 "미래의 나"를 위해 깔끔하게 쓰는 거야. 6개월 후의 나는 남이나 다름없으니까!

[심화 내용] 코드 리뷰 문화

[심화 내용] 코드 리뷰란?
코드 리뷰 = 다른 사람이 내 코드를 검토

왜 하는가:
1. 버그 조기 발견 (테스트 전 발견!)
2. 코드 품질 향상 (가독성, 설계)
3. 지식 공유 (팀 전체가 코드를 이해)
4. 일관된 스타일 유지

리뷰 체크리스트:
□ 이름이 의미 있는가?
□ 함수가 하나의 역할만 하는가?
□ 에러 처리가 되어 있는가?
□ 중복 코드가 없는가?
□ 테스트가 있는가?
□ 들여쓰기/스타일이 일관적인가?

Google에서는:
모든 코드가 리뷰 없이 머지 불가!
코드 리뷰 = 소프트웨어 품질의 핵심

1/**
2 * @file calculator.c
3 * @brief 사칙연산 계산기
4 * @author 자현
5 * @date 2026-03-03
6 */
7
8#include <stdio.h>
9#include <stdlib.h> // exit()
10
11// 상수 정의
12#define VERSION "1.0"
13
14// 함수 프로토타입
15double calculate(double a, char op, double b);
16void print_usage(void);
17
18/**
19 * @brief 두 수의 사칙연산을 수행한다.
20 * @param a 첫 번째 피연산자
21 * @param op 연산자 (+, -, *, /)
22 * @param b 두 번째 피연산자
23 * @return 연산 결과
24 * @warning 0으로 나누면 에러 메시지 출력 후 0 반환
25 */
26double calculate(double a, char op, double b) {
27 switch (op) {
28 case '+': return a + b;
29 case '-': return a - b;
30 case '*': return a * b;
31 case '/':
32 if (b == 0) {
33 // FIXME: 부동소수점 0 비교는 정확하지 않을 수 있음
34 fprintf(stderr, "에러: 0으로 나눌 수 없습니다\n");
35 return 0;
36 }
37 return a / b;
38 default:
39 fprintf(stderr, "에러: 알 수 없는 연산자 '%c'\n", op);
40 return 0;
41 }
42}
43
44int main(void) {
45 printf("계산기 v%s\n", VERSION);
46 printf("3 + 5 = %.1f\n", calculate(3, '+', 5));

48 return 0;
49}

[활동] 코드 정리 연습
다음 코드를 "클린 코드"로 리팩토링해 보세요:

// 원본 (읽기 어려운 코드)
int f(int x){if(x==1||x==2)return 1;int a=1,b=1,c;for(int i=3;i<=x;i++){c=a+b;a=b;b=c;}return b;}

개선 포인트:
1. 함수명: f → fibonacci
2. 매개변수명: x → n
3. 들여쓰기 추가
4. 주석 추가 (알고리즘 설명)
5. 변수명 개선: a,b,c → prev, curr, next
6. 가드 절: n<=0 처리

리팩토링 전후의 가독성을 비교해 보세요!

[심화 내용] 코드 품질 자동 검사 도구
1. cppcheck
무료 오픈소스 정적 분석기
메모리 누수, 버퍼 오버플로우, 스타일 검사
cppcheck --enable=all main.c

2. clang-tidy
LLVM 기반 린터
코딩 스타일 + 버그 패턴 + 모던 C 제안
clang-tidy main.c -- -Wall

3. Valgrind
런타임 메모리 분석 (Linux)
valgrind --leak-check=full ./main
메모리 누수, 초기화 안 된 변수 검출

4. AddressSanitizer (ASan)
gcc -fsanitize=address main.c
버퍼 오버플로우, use-after-free 검출

[활동] 주석/들여쓰기 마스터 체크리스트
□ // 한 줄 주석과 /* */ 여러 줄 주석을 안다
□ 좋은 주석 = "왜(why)" 설명임을 안다
□ 나쁜 주석 (당연한 것 설명) 예를 안다
□ K&R vs Allman 스타일 차이를 안다
□ 4칸 들여쓰기를 일관되게 사용한다
□ 의미 있는 변수/함수 이름을 짓는다
□ DRY 원칙(중복 제거)을 이해한다
□ Doxygen 스타일 문서 주석을 작성할 수 있다
□ clang-format으로 자동 포맷팅을 사용해 봤다

주석 (comment)
코드 속에 쓰는 메모. 컴퓨터는 무시하고 사람만 읽어요.
한 줄 주석
// 뒤에 쓰는 주석. 그 줄 끝까지만 주석이에요.
여러 줄 주석
/* */로 감싸는 주석. 여러 줄에 걸쳐 쓸 수 있어요.
들여쓰기 (indentation)
코드를 오른쪽으로 밀어 쓰는 것. 코드의 구조를 눈으로 보이게 해줘요.
가독성
코드를 읽기 쉬운 정도. 주석, 들여쓰기, 좋은 이름이 가독성을 높여요.
K&R 스타일
여는 중괄호를 같은 줄에 쓰는 스타일. C의 원조 Kernighan & Ritchie가 사용.
Allman 스타일
여는 중괄호를 새 줄에 쓰는 스타일. Visual Studio 기본.
clang-format
C/C++ 코드 자동 포맷팅 도구. .clang-format으로 스타일 설정.
Doxygen
소스 코드 주석에서 API 문서를 자동 생성하는 도구.
TODO/FIXME
코드 내 작업 표시. TODO=나중에 할 것, FIXME=버그 수정 필요.
EditorConfig
다양한 에디터에서 코딩 스타일을 통일하는 설정 파일.
