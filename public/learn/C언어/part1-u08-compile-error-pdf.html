에러가 나면 겁먹지 마세요!

코드를 쓰다 보면 빨간 글씨가 잔뜩 나올 때가 있어요. 무섭죠? 하지만 이건 컴퓨터가 도와주려고 하는 거예요!

프로그래밍에서 에러는 실수가 아니라 과정이에요. 전문 프로그래머도 매일 에러를 만나요. 중요한 건 에러를 안 만드는 것이 아니라, 에러 메시지를 읽고 고치는 능력이에요.

90%
프로그래머가 코딩하는 시간 중 에러를 찾고 고치는 데 쓰는 비율이에요.

컴파일 에러가 뭘까?

우리가 쓴 C 코드는 사람이 읽는 언어예요. 컴퓨터가 이해하려면 기계어로 번역해야 해요. 이 번역 과정을 컴파일(compile)이라고 했죠?

컴파일러가 번역하다가 문법이 틀린 부분을 발견하면 멈춰 버려요. 이때 나오는 메시지가 바로 컴파일 에러(compile error)예요.

한국어로 비유하면, 맞춤법 검사기가 이 문장은 문법이 틀렸어요라고 알려주는 거랑 비슷해요!
컴파일 에러 vs 실행 에러
컴파일 에러는 번역 단계에서 잡히는 에러예요. 프로그램이 아예 만들어지지 않아요. 실행 에러는 프로그램이 돌아가다가 멈추는 건데, 이건 다음 유닛에서 배워요!

자주 나오는 컴파일 에러 TOP 5

초보 때 가장 많이 만나는 에러 5개를 모아봤어요. 이것만 알면 에러의 절반은 해결할 수 있어요!

1. 세미콜론 빠뜨리기

✕ 잘못된 코드
#include int main() { printf("Hello") return 0; }
expected ';' before 'return'
✓ 올바른 코드
#include int main() { printf("Hello"); return 0; }
컴파일 성공!

4번 줄 printf 뒤에 세미콜론이 없어요. 에러 메시지가 return 앞에서 세미콜론을 기대했다고 하는데, 컴파일러가 다음 줄에서야 알아차리기 때문이에요.

에러가 가리키는 줄이 멀쩡하면, 그 윗줄을 확인해 보세요! 세미콜론 에러는 보통 이전 줄의 문제거든요.

2. 중괄호 짝 안 맞추기

✕ 잘못된 코드
#include int main() { printf("Hello"); return 0;
expected '}' at end of input
✓ 올바른 코드
#include int main() { printf("Hello"); return 0; }
컴파일 성공!

여는 중괄호 {를 썼으면 닫는 중괄호 }가 반드시 있어야 해요. 코드가 길어지면 짝을 놓치기 쉬워요.

코딩 기초 습관
여는 중괄호 {를 쓰면 바로 닫는 중괄호 }를 쓰고, 그 사이에 코드를 채우세요. 이러면 짝이 안 맞을 일이 없어요!

3. 철자 틀리기

✕ 잘못된 코드
#include int main() { prntf("Hello"); return 0; }
undefined reference to 'prntf'
✓ 올바른 코드
#include int main() { printf("Hello"); return 0; }
컴파일 성공!

printf를 prntf로 잘못 썼어요. 컴퓨터는 한 글자만 달라도 완전히 다른 단어로 인식해요. 대소문자도 구별하니까 Printf도 에러가 나요!

4. include 빠뜨리기

✕ 잘못된 코드
int main() { printf("Hello"); return 0; }
implicit declaration of function 'printf'
✓ 올바른 코드
#include int main() { printf("Hello"); return 0; }
컴파일 성공!

#include <stdio.h>를 안 쓰면 컴파일러가 printf가 뭔지 몰라요.

5. 큰따옴표 안 닫기

✕ 잘못된 코드
#include int main() { printf("Hello); return 0; }
missing terminating '"' character
✓ 올바른 코드
#include int main() { printf("Hello"); return 0; }
컴파일 성공!

문자열은 반드시 큰따옴표 "로 열고 닫아야 해요. 하나만 빠뜨려도 컴파일러가 혼란에 빠져요.

에러가 여러 개일 때

에러 메시지가 10줄, 20줄 쏟아지면 당황스럽죠? 하지만 비밀이 있어요.

에러 메시지 읽는 순서
맨 첫 번째 에러만 고치세요! 에러 하나가 다른 에러를 연쇄적으로 만드는 경우가 많아요. 첫 번째 에러를 고치면 나머지가 사라지는 경우가 절반 이상이에요.
에러가 20개라고 겁먹지 마세요. 진짜 에러는 보통 1~2개이고, 나머지는 그 에러 때문에 꼬인 거예요!

warning은 에러가 아니에요

warning(경고)은 에러와 달라요. 경고가 나와도 프로그램은 만들어져요. 하지만 되도록 경고도 없애는 게 좋아요.

✕ 잘못된 코드
error: expected ';' 프로그램 안 만들어짐!
에러 (error)
✓ 올바른 코드
warning: unused variable 프로그램은 만들어짐
경고 (warning)

[심화 내용] 링커 에러 완전 정복

[심화 내용] 링커 에러 TOP 5
1. undefined reference to 'func'
→ 함수를 선언만 하고 정의 안 함
→ 또는 .c 파일을 컴파일에 포함 안 함
gcc main.c utils.c -o main ← utils.c 빼먹지 마!

2. multiple definition of 'var'
→ 전역 변수/함수를 여러 파일에 정의
→ 헤더에 int x = 5; 넣으면 발생!
→ 헤더: extern int x; / 소스: int x = 5;

3. undefined reference to 'sqrt'
→ math.h 함수는 -lm 필요!
gcc main.c -lm -o main

4. cannot find -lfoo
→ 라이브러리가 설치 안 됨
→ apt install libfoo-dev

5. relocation truncated
→ 32비트/64비트 불일치
→ -fPIC 옵션 추가

[심화 내용] 면접 문제: 에러 처리
Q1. segfault의 원인 3가지?
A: NULL 역참조, 배열 범위 초과, 해제된 메모리 접근

Q2. -Wall과 -Wextra의 차이?
A: Wall=주요 경고, Wextra=추가(미사용 매개변수, 부호 비교 등)

Q3. 컴파일 에러 vs 링커 에러?
A: 컴파일=문법 에러, 링커=정의 누락(undefined reference)

Q4. errno는 무엇인가?
A: 시스템 호출 실패 시 에러 코드 저장하는 전역 변수

Q5. Valgrind의 용도?
A: 메모리 누수, 초기화 안 된 변수, 버퍼 오버플로우 검출
segfault
Segmentation Fault. NULL 포인터 접근, 배열 범위 초과 등 잘못된 메모리 접근 시 OS가 프로세스 종료.

□ == vs = 혼동을 방지할 수 있다
□ errno와 strerror를 사용할 수 있다
□ GDB 기본 명령어(break, run, next, print)를 안다
