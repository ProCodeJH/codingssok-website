Part 1 진행률: 14%

컴파일은 됐는데 왜 멈추죠?

지난 유닛에서 컴파일 에러를 배웠죠? 이번에는 컴파일은 성공했는데 실행하다가 터지는 에러를 배울 거예요!

컴파일 에러는 문법 검사에서 걸리는 거예요. 마치 맞춤법 검사처럼요. 그런데 문법은 맞지만 말이 안 되는 경우가 있죠?

예를 들어 "나는 하늘을 걸었다"는 맞춤법은 맞지만, 실제로는 불가능하잖아요? 실행 에러도 비슷해요. 코드 문법은 맞는데, 실행하면 컴퓨터가 처리할 수 없는 상황이 발생하는 거예요.

실행 에러(Runtime Error)란?

실행 에러는 프로그램이 돌아가는 중에 발생하는 에러예요. 영어로 Runtime Error라고 해요. 컴파일은 멀쩡하게 되지만, 실행하면 프로그램이 갑자기 멈추거나 이상한 결과를 내요.

✕ 잘못된 코드
컴파일 에러 (Compile Error) - 번역(컴파일) 단계에서 발생 - 프로그램 자체가 안 만들어짐 - 문법이 틀렸을 때
프로그램 생성 실패
✓ 올바른 코드
실행 에러 (Runtime Error) - 실행 단계에서 발생 - 프로그램은 만들어지지만 실행 중 멈춤 - 논리가 잘못되었을 때
실행 중 비정상 종료

0으로 나누기 — 가장 유명한 실행 에러

수학 시간에 선생님이 0으로 나누면 안 된다고 했던 거 기억나요? 컴퓨터도 마찬가지예요!

1#include <stdio.h>
2
3int main() {
4 int a = 10;
5 int b = 0;
6 printf("%d\n", a / b); // 0으로 나누기!
7 return 0;
8}

이 코드는 컴파일이 잘 돼요. 문법에는 아무 문제가 없으니까요. 하지만 실행하면 프로그램이 그냥 죽어버려요.

컴파일러는 a와 b의 값이 뭔지 신경 안 써요. 문법만 맞으면 통과시켜요. 실행할 때서야 b가 0인 걸 알게 되는 거죠.

무한 루프 — 끝나지 않는 프로그램

아직 반복문을 안 배웠지만, 미리 맛보기로 볼게요. 어떤 코드는 영원히 끝나지 않아요.

1#include <stdio.h>
2
3int main() {
4 while (1) { // 조건이 항상 참
5 printf("Hello!\n");
6 }
7 return 0; // 여기까지 절대 도달 못함
8}

이 프로그램은 "Hello!"를 무한히 출력해요. 멈추려면 Ctrl + C를 눌러야 해요. 이것도 실행 에러의 일종이에요.

코딩 기초 습관
프로그램이 멈추지 않으면 Ctrl + C를 누르세요! 이건 프로그래머의 기본 탈출 키예요. 창을 닫아도 되지만, Ctrl + C가 더 깔끔해요.

논리 에러 — 조용한 버그

가장 무서운 에러는 에러 메시지가 안 나오는 에러예요. 프로그램이 잘 돌아가는 것 같은데, 결과가 틀린 거죠.

1#include <stdio.h>
2
3int main() {
4 int price = 1000;
5 int count = 3;
6 int total = price + count; // 곱하기를 더하기로!
7 printf("총 가격: %d원\n", total);
8 return 0;
9}
✕ 잘못된 코드
int total = price + count; printf("총 가격: %d원\n", total);
결과: 1003원 (틀림!)
✓ 올바른 코드
int total = price * count; printf("총 가격: %d원\n", total);
결과: 3000원 (정답!)

컴파일도 되고, 실행도 되고, 에러 메시지도 없어요. 하지만 +를 써야 할 곳에 *를 써서 결과가 완전히 달라졌어요. 이런 걸 논리 에러(logic error)라고 해요.

논리 에러는 컴퓨터가 못 잡아줘요. 사람이 직접 결과를 확인해야 해요. 그래서 프로그래머는 항상 결과를 확인하는 습관을 들여야 해요!

2

논리가 불가능 → 실행 중 멈춤 → 0으로 나누기, 무한 루프 등
3

결과가 틀림 → 프로그램은 정상 작동 → 직접 확인해야 발견

printf로 디버깅하기

실행 에러와 논리 에러를 찾을 때 가장 쉬운 방법이 있어요. 바로 printf로 중간값을 찍어보는 거예요!

1#include <stdio.h>
2
3int main() {
4 int price = 1000;
5 int count = 3;
6 int total = price + count;
7
8 // 디버깅용 출력!
9 printf("price=%d, count=%d\n", price, count);
10 printf("total=%d\n", total);
11
12 printf("총 가격: %d원\n", total);
13 return 0;
14}

이렇게 중간에 변수 값을 출력하면, 어디서 값이 이상해졌는지 바로 찾을 수 있어요. 이 방법을 printf 디버깅이라고 해요.

코딩 기초 습관
결과가 이상하면 printf로 중간 변수를 찍어보세요! 전문 프로그래머도 가장 먼저 하는 디버깅 방법이에요. 문제를 찾으면 디버깅용 printf는 지우거나 주석 처리하면 돼요.

[심화] 런타임 에러 TOP 10
1. Segmentation Fault (세그폴트)
NULL 포인터 역참조, 배열 범위 초과

2. Stack Overflow
무한 재귀, 거대 지역 배열

3. Division by Zero
정수 0 나누기 → 프로그램 종료

4. Buffer Overflow
배열/버퍼 범위 초과 쓰기

5. Memory Leak
malloc 후 free 안 함
장시간 실행 시 메모리 소진

6. Double Free
이미 해제된 메모리 다시 해제

7. Use After Free
해제된 메모리 접근
free(p); *p = 5; // UB!

8. Uninitialized Variable
초기화 안 된 변수 사용

9. Integer Overflow
INT_MAX + 1 = ? (UB!)

10. Infinite Loop

[심화 내용] Valgrind로 메모리 버그 잡기

[심화 내용] Valgrind 사용법
설치:
sudo apt install valgrind (Linux)
macOS: brew install valgrind (제한적)

사용:
gcc -g main.c -o main
valgrind --leak-check=full ./main

출력 해석:
"definitely lost: 40 bytes" → 메모리 누수!
"Invalid read of size 4" → 범위 초과 읽기!
"Use of uninitialised value" → 미초기화!
"Invalid free()" → 이중 해제!

완벽한 결과:
All heap blocks were freed
ERROR SUMMARY: 0 errors

ASan 대안 (더 빠름):
gcc -fsanitize=address -g main.c
./a.out → 에러 시 상세 보고서!
[심화 내용] 면접 문제: 런타임 에러
Q1. segfault와 bus error의 차이?
A: segfault=잘못된 주소, bus error=정렬 안 맞는 접근

Q2. 메모리 누수가 없는데도 OOM?
A: 프래그먼테이션, 또는 할당 해제 없는 계속 성장

Q3. free(NULL)은 안전한가?
A: 안전! C 표준에서 보장. 아무 동작 안 함.

Q4. Valgrind vs ASan?
A: Valgrind=느리지만 상세, ASan=빠르지만 컴파일 필요

Q5. strncpy의 함정?
A: 소스가 dst_size 이상이면 null 종료 안 함!
반드시 dst[size-1]=0; 추가!
Valgrind
메모리 디버깅 도구. 메모리 누수, 초기화 안 된 변수, 버퍼 오버플로우 검출. Linux에서 필수!
ASan
AddressSanitizer. gcc -fsanitize=address. Valgrind보다 빠른 메모리 버그 검출 도구.
컴파일 에러는 친구, 런타임 에러는 적이야! 컴파일러가 못 잡는 버그는 Valgrind/ASan으로 잡자!
방어적 프로그래밍의 핵심: NULL 체크, 범위 체크, 반환값 체크. 이 세 가지만 지켜도 크래시 80% 방지!

[심화 내용] 정의되지 않은 동작 (UB) 심층

[심화 내용] UB가 최적화에 미치는 영향
UB = 컴파일러가 뭘 해도 됨!

int x = INT_MAX;
if (x + 1 > x) { ... }
→ 컴파일러: "부호 있는 오버플로우는 UB"
→ "x+1은 항상 x보다 크다"로 가정
→ if문 제거! (최적화)

왜 위험한가:
1. 디버그 모드에서는 동작하고
릴리즈(-O2)에서는 안 동작!

2. GCC 버전마다 결과 다름!

3. 오늘 동작해도 내일은 모름!

UB 검출 도구:
-fsanitize=undefined (UBSan)
→ 런타임에 UB 발생 시 보고!

gcc -fsanitize=undefined main.c
./a.out → UB 있으면 경고 출력
[C 코드] 방어적 프로그래밍 패턴
1#include <stdio.h>
2#include <stdlib.h>
3#include <string.h>
4#include <limits.h>
5
6// 안전한 정수 덧셈 (오버플로우 체크)
7int safe_add(int a, int b, int *result) {
8 if ((b > 0 && a > INT_MAX - b) ||
9 (b < 0 && a < INT_MIN - b)) {
10 return -1; // 오버플로우!
11 }
12 *result = a + b;
13 return 0;
14}
15
16// 안전한 배열 접근
17int safe_get(const int *arr, int size, int idx, int *out) {
18 if (!arr || idx < 0 || idx >= size) return -1;
19 *out = arr[idx];
20 return 0;
21}
22
23// 안전한 문자열 복사
24int safe_strcpy(char *dst, size_t dst_sz, const char *src) {
25 if (!dst || !src || dst_sz == 0) return -1;
26 size_t len = strlen(src);
27 if (len >= dst_sz) return -1; // 공간 부족
28 strcpy(dst, src);
29 return 0;
30}
31

38 printf("100 + 200 = %d\n", result);

[심화] 흔한 런타임 에러 디버깅 가이드
증상 → 원인 → 해결:

Segfault:
→ NULL 포인터? → 포인터 초기화 + NULL 체크
→ 배열 범위? → 인덱스 범위 체크
→ 스택 오버플로우? → 재귀 깊이 제한

이상한 값 출력:
→ 미초기화 변수? → 선언 시 = 0
→ 포맷 불일치? → %d에 float 넣지 마!
→ 오버플로우? → long long 사용

무한 루프:
→ 종료 조건 확인
→ 카운터 증가 확인
→ Ctrl+C로 중단 후 printf 디버깅

메모리 누수:
→ malloc마다 free 짝 확인
→ 에러 경로에서도 free 확인
→ Valgrind로 검증
UBSan
UndefinedBehaviorSanitizer. -fsanitize=undefined로 활성화. 정수 오버플로우, NULL 역참조 등 UB 검출.
런타임 에러 방지의 3원칙: 1)입력 검증 2)범위 검사 3)반환값 체크. 이 세 가지로 안전한 코드!

[심화 내용] 에러 코드 vs 예외 (C vs C++/Java)
C: 에러 코드 반환
int result = do_something();
if (result != 0) handle_error(result);
→ 모든 호출마다 체크 필요!

C++/Java: 예외(Exception)
try { do_something(); }
catch (Error e) { handle_error(e); }
→ 정상 경로와 에러 경로 분리

Go: 에러 값 (C와 유사)
result, err := doSomething()
if err != nil { ... }

Rust: Result 타입
match do_something() {
Ok(v) => use(v),
Err(e) => handle(e),
}
→ 컴파일러가 에러 처리 강제!

C의 에러 코드 방식은 원시적이지만
오버헤드 제로! 시스템 프로그래밍에 최적
goto cleanup
Linux 커널의 에러 처리 패턴. goto는 보통 나쁘지만, cleanup에서는 가장 깔끔한 방법!
goto는 대부분 나쁘지만, "goto cleanup"은 예외! Linux 커널에서도 사용하는 정석 패턴이야!
