변수 이름 짓기

변수 이름에도 규칙이 있어요. 좋은 이름과 나쁜 이름을 구별해 봐요.

📚 이번에 배울 것

  • 변수 이름 규칙(명명 규칙)을 알 수 있어요.
  • 예약어가 무엇인지 알고, 변수 이름으로 쓸 수 없는 단어를 구별해요.
  • 좋은 변수 이름과 나쁜 변수 이름을 구별할 수 있어요.

✅ 시작 전 체크

  • 변수를 선언하고 값을 넣을 수 있다
  • int 타입과 %d를 안다
Part 1 진행률: 7%

이름에도 규칙이 있어요

변수 이름을 아무렇게나 지으면 에러가 나요! 오늘은 이름 짓는 규칙을 배울 거예요.

우리 이름에도 규칙이 있듯이, 변수 이름에도 반드시 지켜야 하는 규칙이 있어요. 규칙을 어기면 컴파일 에러가 나요.

규칙 1: 쓸 수 있는 문자

변수 이름에 쓸 수 있는 문자는 딱 세 종류예요.

1
영어 대소문자
a~z, A~Z (대문자와 소문자는 다른 글자!)
2
숫자
0~9 (단, 첫 글자로 쓸 수 없어요!)
3
밑줄(_)
underscore라고 해요. 단어 사이를 잇는 데 주로 써요.
✕ 잘못된 코드
int 1st_score = 90;
숫자로 시작! 에러
✓ 올바른 코드
int first_score = 90;
영어로 시작하면 OK
✕ 잘못된 코드
int my-age = 12;
하이픈(-) 사용! 에러
✓ 올바른 코드
int my_age = 12;
밑줄(_) 사용 OK
✕ 잘못된 코드
int my score = 90;
공백 사용! 에러
✓ 올바른 코드
int myScore = 90;
공백 없이 붙여 쓰기

규칙 2: 대소문자 구별

C언어는 대문자와 소문자를 완전히 다른 글자로 취급해요.

case.c
1#include <stdio.h>
2
3int main(void) {
4 int age = 12;
5 int Age = 25;
6 int AGE = 60;
7
8 printf("%d %d %d\n", age, Age, AGE);
9 return 0;
10}
실행 결과
12 25 60

age, Age, AGE는 전부 다른 변수예요! 하지만 이렇게 비슷한 이름을 여러 개 만들면 헷갈리니까, 보통은 소문자로 통일해요.

규칙 3: 예약어는 쓸 수 없어요

예약어는 C언어가 미리 점찍어 둔 단어예요. int, return, if, for, while 같은 단어는 변수 이름으로 쓸 수 없어요.

✕ 잘못된 코드
int int = 5;
int는 예약어! 에러
✓ 올바른 코드
int number = 5;
예약어가 아닌 이름 OK
✕ 잘못된 코드
int return = 0;
return은 예약어! 에러
✓ 올바른 코드
int result = 0;
비슷하지만 다른 단어 OK
[핵심] 자주 쓰는 예약어 목록
int char float double void return if else for while do break continue switch case const sizeof struct
이 단어들은 전부 C언어가 특별한 용도로 쓰고 있어서, 변수 이름으로 사용할 수 없어요.
예약어를 전부 외울 필요는 없어요. 에러가 나면 '혹시 예약어인가?' 의심해 보면 돼요!

좋은 이름 vs 나쁜 이름

[코딩 습관] 이름만 보고 뭔지 알 수 있게!
변수 이름은 짧지만 의미가 분명한 게 좋아요. a보다 age가 훨씬 알아보기 쉬워요. 6개월 뒤에 코드를 다시 보면 a가 뭐였는지 기억 안 나거든요.
✕ 잘못된 코드
int a = 90; int b = 3; int c = a * b;
a, b, c가 뭔지 모르겠어요!
✓ 올바른 코드
int price = 90; int qty = 3; int total = price * qty;
가격, 수량, 합계 — 한눈에 보여요!

여러 단어를 합칠 때는 두 가지 스타일이 있어요.

✕ 잘못된 코드
int totalScore = 100;
카멜 케이스 (낙타 등 모양)
✓ 올바른 코드
int total_score = 100;
스네이크 케이스 (밑줄로 연결)

둘 다 맞는 방법이에요! C언어에서는 스네이크 케이스(밑줄)를 더 많이 써요. 중요한 건 한 프로그램 안에서 한 가지 스타일로 통일하는 거예요.

[심화] 명명 규칙 심화

[심화] 주요 명명 규칙 비교
1. snake_case (C, Python, Rust):
user_name, max_value, get_count
→ C 표준 라이브러리 스타일

2. camelCase (Java, JavaScript):
userName, maxValue, getCount
→ C에서는 비주류

3. PascalCase (C#, Go export):
UserName, MaxValue, GetCount
→ C에서 타입/구조체에 사용

4. UPPER_CASE (상수, 매크로):
MAX_SIZE, PI, NULL
→ C 전처리기 매크로 관례

5. Hungarian Notation (옛날):
iCount, szName, pBuffer
→ Windows API에서 사용
→ 현대에서는 안 쓰는 추세

[심화 내용] 스코프와 네임스페이스

[심화 내용] C의 스코프 규칙
블록 스코프:
{ int x = 1; } // x는 여기서만!

함수 스코프:
goto 레이블만 해당

파일 스코프:
함수 밖 선언 = 전역 변수
static 붙이면 해당 파일에서만!

프로토타입 스코프:
void f(int x); // x는 여기서만

섀도잉 (Shadowing):
int x = 10; // 전역
void f() {
int x = 20; // 지역이 전역을 가림!
printf("%d", x); // 20
}
→ -Wshadow 경고로 방지!

수명 vs 가시성:
static 지역: 수명=전역, 가시성=지역
snake_case
단어를 밑줄로 연결. C 표준 라이브러리의 관례. strncpy, fopen, printf 등.
섀도잉
내부 스코프 변수가 외부 동명 변수를 가림. 버그 원인! -Wshadow 컴파일 옵션으로 방지.
변수 이름은 미래의 나에게 보내는 편지야! 6개월 후에 읽어도 이해할 수 있는 이름을 붙이자!
약어 사용 규칙: cnt, idx, len은 OK. 하지만 자기만 아는 약어(stn=student?)는 금지!

5. 함수명: lowercase_with_underscores
6. typedef 사용 자제
7. 전역 함수: 접두어 (subsys_action)

[심화] 나쁜 변수명 패턴
1. 한 글자 (범위 넓을 때):

(루프 변수 i, j, k는 OK)

2. 약어 남용:

3. 타입 중복:

char name_string[]; → char name[];

4. 부정 이름:

5. 일관성 없음:
getUserName(), fetch_email(), retrieveAddr
→ get_user_name(), get_email(), get_addr()
[C 코드] 이름으로 읽히는 코드
1#include <stdio.h>
2
3// 이름만으로 코드가 읽히는 예
4#define CELSIUS_TO_FAHRENHEIT(c) ((c) * 9.0 / 5.0 + 32)
5#define IS_LEAP_YEAR(y) ((y)%4==0 && ((y)%100!=0 || (y)%400==0))
6
7typedef enum {
8 GRADE_A = 0,
9 GRADE_B,
10 GRADE_C,
11 GRADE_D,
12 GRADE_F
13} Grade;
14
15Grade score_to_grade(int score) {

17 if (score >= 80) return GRADE_B;
18 if (score >= 70) return GRADE_C;
19 if (score >= 60) return GRADE_D;
20 return GRADE_F;
21}
22

24 const char *names[] = {"A","B","C","D","F"};
25 return names[grade];
26}
27
28int main() {
29 int exam_score = 85;
30 Grade student_grade = score_to_grade(exam_score);
31 printf("Score: %d -> Grade: %s\n",
32 exam_score, grade_to_string(student_grade));
33
34 double celsius = 36.5;
35 printf("%.1f C = %.1f F\n",
36 celsius, CELSIUS_TO_FAHRENHEIT(celsius));
37
38 int year = 2026;

[심화 내용] 네임스페이스 충돌 방지
C에는 네임스페이스가 없다!

해결법 1: 접두어
mylib_init(), mylib_open()
json_parse(), json_free()

해결법 2: static (파일 스코프 제한)
static void helper() { ... }
→ 다른 파일에서 접근 불가

해결법 3: 구조체 + 함수 포인터
struct MyLib {
int (*init)(void);
void (*close)(void);
};
→ 객체지향 흉내

C++ namespace와 비교:
C++: namespace mylib { void init(); }
C: void mylib_init(); // 수동 접두어
static 함수
파일 스코프 제한 함수. 다른 파일에서 접근 불가. 이름 충돌 방지 + 캡슐화.
C에는 namespace가 없어서 접두어를 써! json_parse(), xml_parse() 이런 식. 이게 C의 "네임스페이스"!
[C 코드] enum으로 가독성 높이기
1#include <stdio.h>
2
3// BAD: 매직 넘버
4// if (status == 1) ...
5// if (status == 2) ...
6
7// GOOD: enum 사용
8typedef enum {
9 STATUS_OK = 0,
10 STATUS_ERROR = -1,
11 STATUS_TIMEOUT = -2,
12 STATUS_NOT_FOUND = -3
13} StatusCode;
14
15typedef enum {
16 COLOR_RED,
17 COLOR_GREEN,
18 COLOR_BLUE,
19 COLOR_COUNT // 총 개수 트릭!
20} Color;
21
22const char* status_message(StatusCode code) {
23 switch (code) {
24 case STATUS_OK: return "Success";
25 case STATUS_ERROR: return "General error";
26 case STATUS_TIMEOUT: return "Timed out";
27 case STATUS_NOT_FOUND: return "Not found";
28 default: return "Unknown";
29 }
30}
31
32int main() {
33 StatusCode result = STATUS_TIMEOUT;
34 printf("Result: %s (%d)\n", status_message(result), result);
35 printf("Total colors: %d\n", COLOR_COUNT);
36 return 0;
37}
실행 결과
Result: Timed out (-2) Total colors: 3
enum
열거형. 정수 상수에 이름 부여. 매직 넘버 제거의 핵심 도구. COLOR_COUNT 트릭 = 총 개수 자동 계산.
if(status==1) 대신 if(status==STATUS_ERROR) — 이름이 의미를 담으면 주석이 필요 없어져!

clang-format
C/C++ 코드 자동 포맷터. .clang-format 파일로 프로젝트별 스타일 통일. 저장 시 자동 실행 설정 가능.
cppcheck
C/C++ 정적 분석 도구. 메모리 누수, 버퍼 오버플로우, 미초기화 변수 등 탐지. 무료 오픈소스.
코드 스타일 논쟁은 시간 낭비야! clang-format 설정하면 끝. 팀 전체가 같은 스타일 자동 적용!
[활동] 네이밍 마스터 체크리스트
snake_case, camelCase, PascalCase를 안다
상수는 UPPER_CASE로 작성한다
불리언은 is_, has_, can_ 접두어를 사용한다
함수명은 동사+명사로 작성한다
매직 넘버 대신 #define이나 enum을 사용한다
섀도잉(같은 이름 변수)을 피한다
약어는 널리 알려진 것만 사용한다
clang-format 등 도구를 활용한다

6개 이상이면 네이밍 마스터!
리눅스 커널 스타일
탭 8칸, 80열 제한, K&R 중괄호, snake_case. Documentation/process/coding-style.rst 참고.
접두어 네이밍
C에서 네임스페이스 부재를 보완. json_parse(), xml_load() 등 모듈명을 접두어로 사용.
네이밍의 최종 테스트: 6개월 후의 내가 이 이름만 보고 이해할 수 있는가? YES면 좋은 이름!
bool 접두어
불리언 변수는 is_, has_, can_, should_ 접두어. is_valid, has_data, can_write 등.
typedef
타입에 새 이름 부여. typedef unsigned long size_t; → size_t로 사용. 가독성과 이식성 향상.
Clean Code의 핵심: 이름에 의도를 담아라! int d; 대신 int elapsed_time_in_days; — 길어도 명확한 게 승리!
[C 코드] 좋은 네이밍 종합 예제
1#include <stdio.h>
2#include <string.h>
3
4// 상수: UPPER_CASE
5#define MAX_LINE_LENGTH 256
6#define TEMPERATURE_UNIT_CELSIUS 'C'
7
8// 타입: PascalCase
9typedef struct {
10 double min_temperature;
11 double max_temperature;
12 double avg_temperature;
13 int measurement_count;
14} TemperatureStats;
15
16// 함수: snake_case, 동사+명사
17void calculate_stats(const double temps[], int count,
18 TemperatureStats *out) {
19 if (count <= 0) return;
20 out->min_temperature = temps[0];
21 out->max_temperature = temps[0];
22 double sum = 0;
23 for (int i = 0; i < count; i++) {
24 if (temps[i] < out->min_temperature)
25 out->min_temperature = temps[i];
26 if (temps[i] > out->max_temperature)
27 out->max_temperature = temps[i];
28 sum += temps[i];
29 }
30 out->avg_temperature = sum / count;
31 out->measurement_count = count;
32}
33
34void print_stats(const TemperatureStats *stats) {
35 printf("Min: %.1f%c\n", stats->min_temperature, TEMPERATURE_UNIT_CELSIUS);
36 printf("Max: %.1f%c\n", stats->max_temperature, TEMPERATURE_UNIT_CELSIUS);
37 printf("Avg: %.1f%c\n", stats->avg_temperature, TEMPERATURE_UNIT_CELSIUS);
38 printf("Count: %d\n", stats->measurement_count);
39}
40
41int main() {
42 double daily_temps[] = {22.5, 25.3, 19.8, 28.1, 23.7};
43 int num_days = sizeof(daily_temps) / sizeof(daily_temps[0]);
44
45 TemperatureStats weekly_stats;
46 calculate_stats(daily_temps, num_days, &weekly_stats);
47 print_stats(&weekly_stats);
48 return 0;
49}

splint
C 전용 정적 분석 도구. 메모리 관리, 널 포인터, 타입 안전성 검사. lint의 현대 버전.
Clean Code
Robert C. Martin의 저서. 좋은 코드 작성법의 바이블. "이름에 의도를 담아라"가 첫 번째 원칙.
MISRA C
자동차/의료 업계 C 코딩 표준. 143개 규칙. 안전 필수 시스템의 코딩 규칙. 네이밍 규칙도 포함.
scope
변수가 보이는 범위. 블록{}, 함수, 파일 수준. 최소 스코프 원칙: 변수는 필요한 곳에서만 보이게!
명명 규칙
변수 이름을 짓는 규칙. 영어/숫자/밑줄만 사용, 숫자로 시작 불가.
예약어
C언어가 특별한 용도로 쓰는 단어. 변수 이름으로 사용할 수 없어요.
카멜 케이스
단어 시작마다 대문자: myAge, totalScore
스네이크 케이스
단어 사이를 밑줄로 연결: my_age, total_score
이번 유닛 핵심 정리
1
변수 이름: 영어/숫자/밑줄만 OK, 숫자로 시작 불가, 공백/하이픈 불가
2
예약어(int, return 등)는 변수 이름으로 못 써요
3
의미 있는 이름(age, score)을 쓰고, 한 프로그램 안에서 스타일을 통일해요
이 유닛에서 연결된 축

[컴퓨팅 사고력] 추상화 — 데이터에 의미 있는 이름 붙이기
[디버깅] 변수명 에러 (숫자 시작, 예약어 충돌)
[KOI] 코드 읽기 — 변수명으로 의도 파악
[코딩 기초] 스네이크 케이스 습관
[실생활] 이름이 중요한 이유
파일 이름을 '새 폴더', '새 폴더 (2)', '새 폴더 (3)'으로 지으면 찾기 어렵죠? 변수도 마찬가지예요. 좋은 이름은 읽기 쉬운 코드의 시작이에요.

[이미지] 좋은 이름 vs 나쁜 이름

변수 이름만 봐도 무슨 뜻인지 알아야 해요!
✕ 잘못된 코드
int a = 17; int b = 85; int c = b * 100 / a;
a, b, c가 뭔지 모름
✓ 올바른 코드
int totalQuestions = 17; int correctAnswers = 85; int scorePercent = correctAnswers * 100 / totalQuestions;
이름만 봐도 이해!
