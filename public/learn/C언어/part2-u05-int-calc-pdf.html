계산하는 프로그래밍

컴퓨터는 계산기에서 시작했어요! 오늘은 C언어로 사칙연산을 해 볼 거예요.

C언어의 산술 연산자는 5개예요. 수학 시간에 배운 것과 거의 같지만, 기호가 조금 다른 것도 있어요.

1
+ (더하기)
a + b → 두 수의 합
2
- (빼기)
a - b → 두 수의 차
3
* (곱하기)
a * b → × 대신 별표(*) 사용
4
/ (나누기)
a / b → ÷ 대신 슬래시(/) 사용
5
% (나머지)
a % b → 나눈 나머지만 구해요

기본 코드

calc.c
1#include <stdio.h>
2
3int main(void) {
4 int a = 17, b = 5;
5
6 printf("%d + %d = %d\n", a, b, a + b);
7 printf("%d - %d = %d\n", a, b, a - b);
8 printf("%d * %d = %d\n", a, b, a * b);
9 printf("%d / %d = %d\n", a, b, a / b);

12}

나머지(%) 연산자 활용

나머지 연산자 %는 생각보다 많이 써요. 대표적인 활용법을 알아봐요.

mod.c
1#include <stdio.h>
2
3int main(void) {
4 int num;
5 printf("숫자를 입력하세요: ");
6 scanf("%d", &num);
7
8 // 짝수/홀수 판별
9 printf("%d을(를) 2로 나눈 나머지: %d\n", num, num % 2);
10 // 나머지가 0이면 짝수, 1이면 홀수!
11
12 // 일의 자리 추출
13 printf("%d의 일의 자리: %d\n", num, num % 10);
14
15 // 시간 변환
16 int total_min = 200;
17 printf("%d분 = %d시간 %d분\n", total_min, total_min / 60, total_min % 60);
18 return 0;
19}
실행 결과
숫자를 입력하세요: 37 37을(를) 2로 나눈 나머지: 1 37의 일의 자리: 7 200분 = 3시간 20분
[핵심] % 연산자 3대 활용법
1. 짝수/홀수 판별: n % 2 → 0이면 짝수, 1이면 홀수
2. 자릿수 추출: n % 10 → 일의 자리, n % 100 → 두 자리
3. 단위 변환: 초 % 60 → 남은 초, 초 / 60 → 분
[코딩 습관] 에러 메시지 읽기
0으로 나누면 'Floating point exception' 같은 에러가 나요. 에러 메시지를 보면 뭐가 잘못됐는지 힌트를 얻을 수 있어요. 무서워하지 마세요!

printf에서 %를 출력하려면?

%는 서식 지정자의 시작 기호이기 때문에, 화면에 % 문자를 그대로 출력하려면 %%로 두 번 써야 해요.

✕ 잘못된 코드
printf("%d%", 50);
% 하나만 → 에러 또는 이상한 출력
✓ 올바른 코드
printf("%d%%", 50);
%% → 화면에 % 출력 (50%)

연산자 우선순위

수학과 마찬가지로, C에서도 곱하기/나누기가 더하기/빼기보다 먼저 계산돼요. 괄호가 가장 먼저예요!

연산자 우선순위 (높은 것이 먼저)
우선순위연산자예시
1 (가장 높음)( )(2 + 3) * 4 = 20
2*, /, %2 + 3 * 4 = 14
3 (가장 낮음)+, -왼쪽에서 오른쪽으로
priority.c
1#include <stdio.h>
2
3int main(void) {
4 printf("%d\n", 2 + 3 * 4); // 14 (곱하기 먼저)
5 printf("%d\n", (2 + 3) * 4); // 20 (괄호 먼저)
6 printf("%d\n", 10 - 2 * 3 + 1); // 5 (10 - 6 + 1)
7 return 0;
8}
실행 결과
14 20 5
[코딩 습관] 괄호를 적극 사용하세요!
우선순위가 헷갈릴 때는 괄호를 써 주세요. 가독성도 좋아지고 실수도 줄어들어요. a + b * c보다 a + (b * c)가 의도가 명확해요.

단항 마이너스와 양수/음수

음수를 나누면 결과는 어떻게 될까요? C99부터는 0 방향으로 자르기(truncation toward zero)가 보장돼요.

음수 나눗셈 결과 (C99 이후)
수식수학 결과C 결과설명
-7 / 2-3.5-30 방향으로 자름 (올림)

[핵심] 나눗셈과 나머지의 관계

예: (-7/2)*2 + (-7%2) = (-3)*2 + (-1) = -6 + (-1) = -7 ✓

[심화] C89 vs C99 나눗셈 차이

C89에서는 음수 나눗셈의 방향이 구현 정의(implementation-defined)였어요. 컴파일러마다 결과가 달랐다는 뜻이에요!

✕ 잘못된 코드
// C89: -7 / 2 = ? // 어떤 컴파일러: -3 // 어떤 컴파일러: -4
C89: 결과가 불확실!
✓ 올바른 코드
// C99 이후: -7 / 2 = -3 // 항상 0 방향으로 자름
C99: 결과가 확정!
[역사] C 표준의 변천
C89(ANSI C)에서는 음수 나눗셈 방향을 정해두지 않았어요. 수학에서는 '내림(floor division)'이 자연스러운데, 하드웨어(CPU)는 '0 방향 자르기'가 효율적이었거든요. C99에서 하드웨어 동작에 맞춰 '0 방향 자르기'로 확정했어요. Python은 반대로 '내림(floor division)'을 선택했답니다.

정수 오버플로우 — 숫자가 넘치면?

정수형 변수에는 저장할 수 있는 한계가 있어요. 그 한계를 넘으면 어떻게 될까요?

int는 보통 32비트(4바이트)로, -2,147,483,648 ~ 2,147,483,647 범위의 숫자를 저장할 수 있어요. 이 범위를 넘는 것을 오버플로우(overflow)라고 해요.

overflow.c
1#include <stdio.h>
2#include <limits.h>
3
4int main(void) {
5 printf("int 최대값: %d\n", INT_MAX); // 2147483647
6 printf("int 최소값: %d\n", INT_MIN); // -2147483648
7
8 int big = INT_MAX;
9 printf("최대값: %d\n", big);
10 printf("최대값 + 1: %d\n", big + 1); // 오버플로우!
11 return 0;
12}
실행 결과
int 최대값: 2147483647 int 최소값: -2147483648 최대값: 2147483647 최대값 + 1: -2147483648
최대값에 1을 더하면 최소값이 돼요! 자동차 계기판이 999,999 다음에 000,000으로 돌아가는 것처럼요!
오버플로우 시각화 — 원형으로 돌아가요
... → 2,147,483,646 → 2,147,483,647 → -2,147,483,648 → -2,147,483,647 → ...
[주의] signed 오버플로우는 정의되지 않은 동작!
C 표준에서 signed int의 오버플로우는 undefined behavior(정의되지 않은 동작)예요. 위 예시처럼 보통은 wrap-around 되지만, 컴파일러가 최적화하면서 완전히 다른 결과가 나올 수도 있어요. 오버플로우가 발생하지 않도록 미리 검사하는 게 정석이에요.

unsigned 정수의 오버플로우

unsigned int는 음수 없이 0부터 4,294,967,295까지 저장해요. unsigned의 오버플로우는 잘 정의된 동작(modular arithmetic)이에요!

unsigned_wrap.c
1#include <stdio.h>
2#include <limits.h>
3
4int main(void) {
5 unsigned int u = UINT_MAX; // 4294967295
6 printf("UINT_MAX: %u\n", u);
7 printf("UINT_MAX + 1: %u\n", u + 1); // 0으로 돌아감!
8
9 unsigned int z = 0;
10 printf("0 - 1: %u\n", z - 1); // UINT_MAX가 됨!
11 return 0;
12}
실행 결과
UINT_MAX: 4294967295 UINT_MAX + 1: 0 0 - 1: 4294967295
[핵심] signed vs unsigned 오버플로우
signed 오버플로우: 정의되지 않은 동작(UB) — 결과를 예측할 수 없어요
unsigned 오버플로우: 잘 정의됨 — 항상 값 % (UINT_MAX+1)로 계산돼요
undefined
1996년 아리안5 로켓이 발사 37초 만에 폭발했어요. 원인은 64비트 실수를 16비트 정수로 변환할 때 오버플로우가 발생한 것이었어요! 오버플로우는 로켓도 터뜨린답니다. 💥

곱셈 오버플로우 주의

큰 수끼리 곱하면 쉽게 int 범위를 넘어요. 이걸 방지하는 방법을 알아봐요.

mul_overflow.c
1#include <stdio.h>
2
3int main(void) {
4 int a = 100000;
5 int b = 100000;
6
7 // a * b = 10,000,000,000 → int 범위 초과!
8 printf("int 곱: %d\n", a * b); // 이상한 값
9
10 // 해결: long long 사용
11 long long result = (long long)a * b;
12 printf("long long 곱: %lld\n", result); // 10000000000
13 return 0;
14}
[팁] 큰 곱셈은 long long으로!
두 int를 곱할 때 결과가 int 범위를 넘을 수 있으면, 곱하기 전에 (long long)으로 형변환하세요. (long long)(a * b)는 소용없어요 — 이미 int끼리 곱해서 오버플로우가 난 뒤에 변환하는 거니까요!

[심화] 정수 승격(Integer Promotion)

char이나 short끼리 연산하면 결과가 int가 돼요. 이것을 정수 승격이라고 해요. C 표준에서 정한 규칙이에요.

promotion.c
1#include <stdio.h>
2
3int main(void) {
4 char a = 100;
5 char b = 100;
6 // char + char → int + int → int
7 // 결과: 200 (char 범위인 127을 넘지만 OK!)
8 int result = a + b;
9 printf("%d\n", result); // 200
10
11 // 주의: 다시 char에 넣으면 잘릴 수 있어요
12 char c = a + b; // 200 → char 범위 초과 → 잘림!
13 printf("%d\n", c); // 구현에 따라 -56 등
14 return 0;
15}
[핵심] 왜 정수 승격이 필요할까?
CPU는 보통 int 크기(32비트)로 연산해요. char(8비트)끼리 바로 더하는 명령어가 없는 CPU도 많아요. 그래서 C 표준은 '작은 타입은 일단 int로 키운 다음 계산하자'고 정했어요. 이걸 정수 승격(Integer Promotion)이라 해요.

[심화] 암묵적 형변환 — signed와 unsigned 혼합의 위험

signed와 unsigned를 같이 연산하면 signed가 unsigned로 변환돼요. 이때 음수가 아주 큰 양수로 바뀌는 무서운 일이 생겨요!

[심화] 암묵적 형변환 규칙 정리

두 가지 다른 타입을 연산할 때, C는 '더 넓은 타입'으로 자동 변환해요. 이것을 usual arithmetic conversions라고 해요.

1
1단계: 정수 승격
char, short → int로 승격
2
2단계: 같은 타입이면?
변환 없이 바로 연산
3
3단계: 부호 다르면?
unsigned 쪽으로 변환 (위험!)
4
4단계: 크기 다르면?
더 큰 타입으로 변환 (int → long → long long)

[심화 내용] Undefined Behavior vs Implementation-Defined

C 표준은 모든 동작을 세 가지로 분류해요. 이 구분을 이해하면 C 프로그래머로서 한 단계 성장한 거예요.

C 표준의 동작 분류
분류의미예시
잘 정의됨 (Well-defined)결과가 항상 같음7 / 2 = 3 (C99)
구현 정의 (Implementation-defined)컴파일러마다 다르지만, 문서화 필수int의 크기 (16/32/64비트)
정의되지 않음 (Undefined Behavior)아무 일이나 일어날 수 있음!signed 오버플로우, 0으로 나누기
[심화 내용] UB(Undefined Behavior)의 공포
UB가 발생하면 컴파일러는 '이 코드는 실행되지 않는다'고 가정하고 최적화할 수 있어요. 예를 들어, if (x + 1 > x)에서 signed 오버플로우는 UB이므로 컴파일러가 이 조건을 '항상 참'으로 최적화해버려요. UB는 '이상한 결과'가 아니라 '프로그램 전체의 동작이 보장되지 않는 것'이에요.

[심화 내용] 컴퓨터 산술 vs 수학 산술

수학에서는 당연한 법칙들이 컴퓨터 산술에서는 깨질 수 있어요. 오버플로우 때문이에요.

깨지는 수학 법칙들
수학 법칙수학에서컴퓨터에서이유
결합법칙(a+b)+c = a+(b+c)항상 성립하지 않음중간에 오버플로우 가능
교환법칙a+b = b+a항상 성립덧셈은 비트 레벨에서 순서 무관
분배법칙a*(b+c) = a*b+a*c항상 성립하지 않음각 곱에서 오버플로우 가능
취소 법칙a+b-b = a보통 성립최적화로 보장 가능
math_vs_computer.c
1#include <stdio.h>
2#include <limits.h>
3
4int main(void) {
5 int a = INT_MAX;
6 int b = 1;
7 int c = -1;
8
9 // 수학: (a + b) + c = a + (b + c)
10 // 컴퓨터: (INT_MAX + 1) + (-1) vs INT_MAX + (1 + (-1))
11 // 왼쪽: 오버플로우! 오른쪽: INT_MAX
12 printf("(a+b)+c = %d\n", (a + b) + c); // UB!
13 printf("a+(b+c) = %d\n", a + (b + c)); // INT_MAX

15}

C의 산술 연산자는 CPU 명령어 하나로 변환돼요. x86 아키텍처의 주요 산술 명령어를 알아봐요.

C 연산 → x86 어셈블리 명령어

CPU에서 ADD는 1 클럭, IMUL은 3~5 클럭, IDIV는 20~90 클럭이 걸려요! 나눗셈은 곱셈보다 10배 이상 느리답니다. 그래서 컴파일러는 나눗셈을 최대한 피하려고 노력해요.

[심화 내용] 컴파일러의 곱셈/나눗셈 최적화

컴파일러는 2의 거듭제곱으로 곱하기/나누기를 시프트 연산으로 바꿔요. 훨씬 빠르거든요!

1// 컴파일러가 자동으로 이렇게 바꿔요!
2
3// 원래 코드 → 컴파일러 최적화
4// x * 2 → x << 1 (왼쪽 시프트 1비트)
5// x * 4 → x << 2 (왼쪽 시프트 2비트)
6// x * 8 → x << 3 (왼쪽 시프트 3비트)
7// x / 2 → x >> 1 (오른쪽 시프트 1비트)
8// x / 4 → x >> 2 (오른쪽 시프트 2비트)
9
10// 심지어 x * 10도 최적화!
11// x * 10 → (x << 3) + (x << 1) = x*8 + x*2
[심화 내용] 상수 나눗셈의 마법
상수로 나누는 것을 컴파일러는 곱셈 + 시프트로 변환해요. 예: x / 3 → (x * 0xAAAAAAAB) >> 33. 이 기법을 'magic number division'이라 불러요. 느린 DIV 명령어 대신 빠른 MUL + SHR을 쓰는 거예요.

[심화 내용] 모듈러 연산의 수학

나머지 연산(%)은 수학에서 모듈러 산술(Modular Arithmetic)이라는 중요한 분야와 연결돼요.

합동식: a와 b를 n으로 나눈 나머지가 같으면, 'a ≡ b (mod n)'이라고 써요.
예: 17 ≡ 2 (mod 5) — 왜냐하면 17 % 5 = 2이고, 2 % 5 = 2이니까요.

모듈러 연산의 성질
성질공식예시 (mod 7)
덧셈(a+b) % n = ((a%n)+(b%n)) % n(10+12) % 7 = ((3)+(5)) % 7 = 1
곱셈(a*b) % n = ((a%n)*(b%n)) % n(10*12) % 7 = ((3)*(5)) % 7 = 1
뺄셈(a-b) % n = ((a%n)-(b%n)+n) % n(10-12) % 7 = ((3)-(5)+7) % 7 = 5
[심화 내용] 큰 수의 나머지를 구하는 트릭
10^18 같은 거대한 수의 나머지를 구할 때, 중간에 곱할 때마다 나머지를 취하면 오버플로우를 피할 수 있어요!
(a * b) % m = ((a % m) * (b % m)) % m
이 성질은 정보올림피아드, 암호학에서 핵심이에요.

[심화 내용] 모듈러 역원과 암호학

나눗셈의 모듈러 버전은 모듈러 역원이에요. a의 mod n 역원은 a * a_inv ≡ 1 (mod n)을 만족하는 수예요.

모듈러 역원은 RSA 암호에서 핵심 역할을 해요. 공개키와 개인키를 만들 때 모듈러 역원을 사용하거든요. 여러분이 인터넷 쇼핑할 때 쓰는 HTTPS 암호화의 기반이에요!

[역사] 모듈러 산술의 아버지
카를 프리드리히 가우스(1777-1855)가 그의 걸작 '산술 연구(Disquisitiones Arithmeticae)'에서 합동식 이론을 체계화했어요. 21살에 쓴 이 책이 현대 정수론과 암호학의 기반이 됐답니다!

[실무] 금융 계산은 왜 정수로 할까?

[실무] 게임의 고정소수점 연산

옛날 게임기(슈퍼 패미컴 등)에는 실수 연산 장치(FPU)가 없었어요. 그래서 정수로 소수점을 흉내 내는 고정소수점(Fixed-Point) 연산을 사용했어요.

fixed_point.c
1#include <stdio.h>
2
3// 8비트 고정소수점: 실제값 = 저장값 / 256
4#define FIXED_SHIFT 8
5#define TO_FIXED(x) ((x) << FIXED_SHIFT) // 정수 → 고정소수점
6#define FROM_FIXED(x) ((x) >> FIXED_SHIFT) // 고정소수점 → 정수
7#define FIXED_MUL(a, b) (((a) * (b)) >> FIXED_SHIFT) // 고정소수점 곱셈
8
9int main(void) {
10 int player_x = TO_FIXED(100); // 100.0
11 int speed = TO_FIXED(1) + 128; // 1.5 (256 + 128 = 384/256 = 1.5)
12
13 player_x += speed; // 100.0 + 1.5 = 101.5
14 player_x += speed; // 101.5 + 1.5 = 103.0
15
16 printf("위치(고정소수점): %d\n", player_x);
17 printf("위치(정수 변환): %d\n", FROM_FIXED(player_x));
18 return 0;
19}
[역사] 게임과 고정소수점
슈퍼 마리오, 소닉 등 클래식 게임들은 전부 고정소수점으로 물리 엔진을 구현했어요. 현대에도 임베디드 시스템(드론, 로봇)이나 오디오 처리에서 고정소수점을 사용해요. 정수 연산이 실수보다 빠르고 예측 가능하기 때문이에요.

[심화 내용] 어셈블리로 보는 사칙연산

C 코드가 어셈블리로 어떻게 변환되는지 살짝 엿봐요. GCC의 -S 옵션으로 확인할 수 있어요.

asm_example.c
1// C 코드: int c = a + b;
2// x86-64 어셈블리:
3mov eax, [rbp-4] ; a를 eax 레지스터에 로드
4add eax, [rbp-8] ; eax = eax + b
5mov [rbp-12], eax ; 결과를 c에 저장
6
7// C 코드: int c = a / b;
8// x86-64 어셈블리:
9mov eax, [rbp-4] ; a를 eax에 로드
10cdq ; eax를 edx:eax로 부호 확장
11idiv dword [rbp-8] ; edx:eax ÷ b → 몫=eax, 나머지=edx
12mov [rbp-12], eax ; 몫을 c에 저장
13// a % b의 결과는 이미 edx에 있어요!
[심화 내용] IDIV의 특별한 점
x86의 IDIV 명령어는 한 번에 몫과 나머지를 동시에 계산해요. 몫은 EAX, 나머지는 EDX에 저장돼요. 그래서 C에서 a / b와 a % b를 같이 쓰면 컴파일러가 IDIV를 한 번만 실행하고 결과를 재사용해요!

[심화 내용] 곱셈 최적화 — LEA 명령어

x86에는 LEA(Load Effective Address)라는 주소 계산 명령어가 있는데, 컴파일러가 이걸 곱셈에 활용해요!

lea_trick.asm
1; x * 3 → LEA eax, [rdi + rdi*2] (x + x*2 = x*3)
2; x * 5 → LEA eax, [rdi + rdi*4] (x + x*4 = x*5)
3; x * 9 → LEA eax, [rdi + rdi*8] (x + x*8 = x*9)
4
5; x * 6 → LEA eax, [rdi + rdi*2] ; x*3
6; ADD eax, eax ; x*3*2 = x*6
7
8; x * 7 → 여러 가지 방법:
9; 방법1: LEA eax, [rdi*8] / SUB eax, rdi(x*8 - x = x*7)
10; 방법2: IMUL eax, rdi, 7 (그냥 곱셈)
[심화 내용] 왜 이런 최적화를 할까?
LEA는 1 클럭, IMUL은 3 클럭이에요. 상수 곱셈을 LEA + 시프트 조합으로 바꾸면 2~3배 빨라질 수 있어요. 컴파일러가 알아서 최적의 조합을 찾아줘요!

[심화] 2의 보수와 정수의 내부 표현

컴퓨터는 음수를 2의 보수(Two's Complement) 방식으로 저장해요. 이 방식 덕분에 덧셈 회로 하나로 뺄셈도 할 수 있어요.

2

[실무] 비트마스크와 나머지

2

[실무] 링 버퍼(Ring Buffer)에서의 활용

용어 정리

산술 연산자
계산에 쓰는 기호. +, -, *, /, % 다섯 가지.
정수 나눗셈
int끼리 나누면 소수점 아래를 버리고 정수 결과만 남는 것.
% (나머지)
modulo 연산자. 나눗셈의 나머지를 구해요.
%%
printf에서 % 문자 자체를 출력할 때 쓰는 표기법.
Division by zero
0으로 나누는 것. 실행 에러가 발생해요.
오버플로우(Overflow)
정수가 저장 가능한 범위를 넘는 것. signed는 UB, unsigned는 wrap-around.
정수 승격(Integer Promotion)
char, short 등 작은 타입이 연산 시 자동으로 int로 변환되는 것.
암묵적 형변환(Implicit Conversion)
서로 다른 타입끼리 연산할 때 컴파일러가 자동으로 타입을 맞추는 것.
Undefined Behavior (UB)
C 표준이 결과를 보장하지 않는 동작. 프로그램이 어떤 일이든 할 수 있어요.
Implementation-Defined
컴파일러마다 다르지만, 문서화가 의무인 동작.
2의 보수(Two's Complement)
음수를 표현하는 방식. 비트를 반전하고 1을 더해요.
고정소수점(Fixed-Point)
정수를 이용해 소수를 표현하는 기법. FPU 없는 환경에서 사용.
모듈러 산술
나머지 연산을 기반으로 한 수학 체계. 시계, 요일, 암호학에 사용.

[C언어] +, -, *, /, % 산술 연산자, 정수 승격, 암묵적 형변환
[컴퓨팅 사고력] 분해 — 복잡한 계산을 단계별로 쪼개기
[디버깅] Division by zero, 정수 나눗셈(소수점 버림), 오버플로우, signed/unsigned 혼합
[KOI] 나머지 연산 (FizzBuzz, 윤년, 자릿수 분리), 큰 수 모듈러
[타과목] 수학 — 사칙연산, 나머지(mod), 합동식, 2의 보수
[컴퓨터 구조] x86 어셈블리, 2의 보수, 시프트 최적화
[실생활] 나머지 연산이 쓰이는 곳

[역사] 컴퓨터와 산술 연산의 역사
찰스 배비지(1791-1871)의 해석 기관(Analytical Engine)이 최초의 범용 계산기 설계였어요. 에이다 러브레이스가 이 기계를 위한 최초의 알고리즘을 작성했고요. 현대 CPU의 ALU(Arithmetic Logic Unit)는 덧셈, 뺄셈, 곱셈, 나눗셈을 나노초 단위로 처리해요!
