= 는 '같다'가 아니라 '넣어라'!

수학에서 =는 '같다'인데, C에서는 '오른쪽 값을 왼쪽에 넣어라'예요! 완전히 다른 의미랍니다.

대입 연산자 =는 오른쪽의 값을 계산한 다음, 왼쪽 변수에 저장하는 연산자예요. 수학의 등호(=)와는 완전히 다른 개념이에요.

✕ 잘못된 코드
// 수학에서 x = x + 1은 '불가능' // 어떤 수도 자기 자신+1과 같을 수 없어요
수학: x = x + 1 → 해 없음!
✓ 올바른 코드
// C에서 x = x + 1은 '가능' // x에 1을 더한 값을 다시 x에 넣으라는 뜻
C: x = x + 1 → x를 1 증가!
assignment_basic.c
1#include <stdio.h>
2
3int main(void) {
4 int x = 10; // 초기화: x에 10을 넣어요
5 printf("x = %d\n", x); // 10
6
7 x = x + 5; // 대입: x(10) + 5 = 15를 x에 넣어요
8 printf("x = %d\n", x); // 15
9
10 x = 100; // x에 100을 넣어요 (이전 값 15는 사라짐)
11 printf("x = %d\n", x); // 100
12 return 0;
13}
실행 결과
x = 10 x = 15 x = 100
[핵심] 대입의 방향
대입은 항상 오른쪽 → 왼쪽이에요!
x = 10; : 10이라는 값을 → x에 넣어요
y = x + 3; : x+3을 계산하고 → 결과를 y에 넣어요

짧게 쓰기

score = score + 10;을 더 짧게 쓸 수 있다면? 오늘은 복합 대입 연산자를 배워요!

같은 변수에 값을 더해서 다시 넣는 건 정말 자주 해요. 그래서 C언어는 축약형을 만들어뒀어요.

복합 대입 연산자 5종

복합 대입 연산자 정리
복합 형태풀어 쓴 형태의미
a += 5a = a + 5a에 5를 더해서 저장
a -= 3a = a - 3a에서 3을 빼서 저장
a *= 2a = a * 2a에 2를 곱해서 저장
a /= 4a = a / 4a를 4로 나눠서 저장
a %= 10a = a % 10a를 10으로 나눈 나머지를 저장

대입 연산자의 반환값

C에서 =도 연산자이기 때문에 값을 반환해요! 대입 후의 값(왼쪽 변수의 새 값)을 돌려줘요.

assign_return.c
1#include <stdio.h>
2
3int main(void) {
4 int a, b;
5
6 // 대입 연산자도 값을 반환해요!
7 printf("%d\n", a = 42); // 42 출력! (a에 42를 넣고, 그 값을 반환)
8
9 // 체이닝: 오른쪽부터 실행!
10 a = b = 0; // b = 0 → b의 값(0)을 a에 대입
11 printf("a = %d, b = %d\n", a, b); // 둘 다 0
12 return 0;
13}
실행 결과
42 a = 0, b = 0

대입 체이닝의 원리

a = b = c = 0;은 어떻게 동작할까요? 대입 연산자는 오른쪽에서 왼쪽으로(right-to-left) 결합해요.

1
1단계: c = 0
c에 0을 넣고, 0을 반환
2
2단계: b = (c = 0) → b = 0
반환된 0을 b에 넣고, 0을 반환
3
3단계: a = (b = (c = 0)) → a = 0
반환된 0을 a에 넣음
chain.c
1#include <stdio.h>
2
3int main(void) {
4 int a, b, c;
5
6 a = b = c = 100; // 세 변수 모두 100!
7 printf("a=%d, b=%d, c=%d\n", a, b, c);
8
9 // 복합 대입도 값을 반환해요 (하지만 가독성이 나빠서 권장 X)
10 int x = 5;
11 int y = (x += 3); // x = 8이 되고, y에도 8이 들어감
12 printf("x=%d, y=%d\n", x, y);
13 return 0;
14}
실행 결과
a=100, b=100, c=100 x=8, y=8
[코딩 습관] 체이닝은 초기화에만!
a = b = c = 0;처럼 같은 값으로 초기화할 때는 체이닝이 깔끔해요. 하지만 복잡한 수식에서 체이닝을 쓰면 읽기 어렵고 버그가 생기기 쉬워요. 한 줄에 대입 하나가 기본 원칙이에요.

대입 vs 초기화 — 비슷하지만 달라요!

변수를 선언하면서 값을 넣는 것(초기화)과, 나중에 값을 바꾸는 것(대입)은 비슷해 보이지만 중요한 차이가 있어요.

✕ 잘못된 코드
const int MAX = 100; MAX = 200; // 에러! const는 대입 불가
const 변수는 대입할 수 없어요!
✓ 올바른 코드
const int MAX = 100; // 초기화는 OK, 대입은 불가
const는 초기화만 가능해요
초기화 vs 대입 비교
구분초기화대입
시점선언할 때선언 이후
문법int x = 10;x = 10;
const가능불가능!
배열int a[] = {1,2,3};a = {1,2,3}; → 에러!
횟수딱 한 번여러 번 가능
init_vs_assign.c
1#include <stdio.h>
2
3int main(void) {
4 // 초기화 (선언과 동시에 값 부여)
5 int x = 10; // OK
6 const int MAX = 100; // OK: 초기화
7
8 // 대입 (나중에 값 변경)
9 x = 20; // OK: 일반 변수는 대입 가능
10 // MAX = 200; // 에러! const는 대입 불가
11
12 // 초기화하지 않은 변수
13 int y; // y의 값은 쓰레기값(unpredictable)!
14 // printf("%d\n", y); // 위험! 정의되지 않은 동작
15 y = 0; // 이렇게 대입해서 안전하게 사용
16 printf("%d\n", y); // 0
17 return 0;
18}
[주의] 초기화하지 않은 변수 사용 금지!
C에서 지역 변수를 초기화하지 않으면 쓰레기값(이전에 메모리에 남아 있던 임의의 값)이 들어 있어요. 이 값을 읽는 것은 정의되지 않은 동작(UB)이에요! 항상 초기화하는 습관을 들이세요.

lvalue와 rvalue — 왜 3 = x는 안 돼요?

x = 3;은 되는데, 3 = x;는 왜 안 될까요? '어디에 넣을 수 있는가'가 핵심이에요!

대입 연산자의 왼쪽에는 '값을 저장할 수 있는 것'만 올 수 있어요. 이것을 lvalue(left value)라고 해요.

lvalue vs rvalue
구분의미예시대입 왼쪽?
lvalue메모리 위치가 있는 것변수 x, 배열 요소 a[0]✅ 가능
rvalue값만 있는 것 (위치 없음)숫자 3, 수식 x+1❌ 불가
lvalue.c
1#include <stdio.h>
2
3int main(void) {
4 int x = 10;
5 int arr[3] = {1, 2, 3};
6
7 x = 20; // OK: x는 lvalue (메모리 위치가 있어요)
8 arr[0] = 99; // OK: arr[0]도 lvalue
9
10 // 3 = x; // 에러! 3은 rvalue (값일 뿐, 어디에 저장?)
11 // x + 1 = 5; // 에러! x+1은 rvalue (계산 결과일 뿐)
12
13 // 포인터 역참조도 lvalue!
14 int *p = &x;
15 *p = 30; // OK: *p는 lvalue (x의 메모리 위치를 가리킴)
16 printf("x = %d\n", x); // 30
17 return 0;
18}
[핵심] lvalue 판별법
'주소를 구할 수 있는가?'로 판별해요.
&x — 가능 → lvalue ✅
&3 — 불가능 → rvalue ❌
&(x+1) — 불가능 → rvalue ❌
&arr[0] — 가능 → lvalue ✅

[심화] = 와 == 혼동 — 세계에서 가장 유명한 버그

대입(=)과 비교(==)를 혼동하는 것은 C 역사상 가장 흔한 버그예요. 대입도 값을 반환하기 때문에 컴파일이 돼 버려요!

eq_bug.c
1#include <stdio.h>
2
3int main(void) {
4 int x = 5;
5
6 // 버그! = 은 대입이지 비교가 아니에요!
7 if (x = 0) { // x에 0을 넣고, 0은 거짓이므로
8 printf("x는 0이에요\n");
9 } else {
10 printf("x는 0이 아니에요\n"); // 이게 출력됨!

20 }

22}
[코딩 습관] '요다 조건문'
상수를 왼쪽에 쓰면 실수를 방지할 수 있어요:
if (0 == x) — OK
if (0 = x) — 컴파일 에러! (0은 lvalue가 아니니까)
이것을 '요다 조건문(Yoda Condition)'이라고 해요. 요다가 말하듯 거꾸로 쓰는 거예요 😄

비트 복합 대입 연산자

산술 복합 대입(+=, -= 등) 말고도, 비트 연산에 대한 복합 대입도 있어요!

비트 복합 대입 연산자
복합 형태풀어 쓴 형태의미
a &= ba = a & b비트 AND 후 저장
a |= ba = a | b비트 OR 후 저장
a ^= ba = a ^ b비트 XOR 후 저장
a <<= na = a << n왼쪽 시프트 후 저장
a >>= na = a >> n오른쪽 시프트 후 저장
bit_compound.c
1#include <stdio.h>
2
3int main(void) {
4 unsigned int flags = 0;
5
6 // 비트 설정 (OR)
7 flags |= (1 << 0); // 0번 비트 켜기: 0001
8 flags |= (1 << 2); // 2번 비트 켜기: 0101
9 printf("flags = %u\n", flags); // 5
10
11 // 비트 해제 (AND + NOT)
12 flags &= ~(1 << 0); // 0번 비트 끄기: 0100
13 printf("flags = %u\n", flags); // 4
14
15 // 비트 토글 (XOR)
16 flags ^= (1 << 2); // 2번 비트 뒤집기: 0000
17 printf("flags = %u\n", flags); // 0
18
19 // 시프트
20 int x = 1;
21 x <<= 3; // 1 → 8 (2^3)
22 printf("x = %d\n", x); // 8
23 x >>= 1; // 8 → 4 (2^2)
24 printf("x = %d\n", x); // 4
25 return 0;
26}
실행 결과
flags = 5 flags = 4 flags = 0 x = 8 x = 4
[실무] 비트 플래그 관리
게임에서 캐릭터 상태(독, 화상, 빙결 등)를 비트로 관리할 때 비트 복합 대입을 많이 써요.
status |= POISON; // 독 상태 추가
status &= ~POISON; // 독 상태 해제
status ^= INVISIBLE; // 투명 토글

[심화] 복합 대입 vs 일반 대입 — 진짜 같을까?

a += b와 a = a + b는 '거의' 같지만, 미묘한 차이가 있어요.

[심화] 핵심 차이
a += b에서 a는 한 번만 평가돼요.
a = a + b에서 a는 두 번 평가돼요.

보통은 차이가 없지만, a가 부작용(side effect)이 있는 수식이면 달라질 수 있어요!
예: arr[i++] += 5 vs arr[i++] = arr[i++] + 5
전자는 i++가 한 번, 후자는 두 번 실행돼서 결과가 달라요!

2

6

[심화 내용] 컴파일러 최적화와 복합 대입

최적화 관점에서 복합 대입은 어떤 이점이 있을까요?

현대 컴파일러는 a = a + b와 a += b를 동일한 코드로 최적화해요. 하지만 복잡한 주소 계산이 들어가면 차이가 날 수 있어요.

opt_example.c
1// 복합 대입이 유리한 경우:
2struct { int data[1000]; } *ptr = get_struct();
3
4// 일반 대입: get_struct()를 두 번 호출할 수도 있음
5get_struct()->data[calc_index()] = get_struct()->data[calc_index()] + 5;
6
7// 복합 대입: 주소 계산이 한 번만
8get_struct()->data[calc_index()] += 5;
9
10// 어셈블리 레벨에서:
11// 일반: 주소 계산 → 로드 → 주소 재계산 → 더하기 → 저장
12// 복합: 주소 계산 → 로드 → 더하기 → 저장 (주소 재사용!)

swap 패턴 — 두 변수의 값 교환

두 변수의 값을 바꾸려면 어떻게 해야 할까요? 컵 두 개의 물을 교환하는 것처럼 생각해 봐요!

두 변수의 값을 교환하는 건 프로그래밍의 기본 패턴이에요. 세 가지 방법을 비교해 봐요.

방법 1: temp 변수 사용 (정석)

swap_temp.c
1#include <stdio.h>
2
3int main(void) {
4 int a = 10, b = 20;
5 printf("교환 전: a=%d, b=%d\n", a, b);
6
7 int temp = a; // temp에 a를 백업
8 a = b; // a에 b를 넣고
9 b = temp; // b에 백업한 값을 넣어요
10
11 printf("교환 후: a=%d, b=%d\n", a, b);
12 return 0;
13}
실행 결과
교환 전: a=10, b=20 교환 후: a=20, b=10
temp를 이용한 swap 과정
처음: a=10, b=20, temp=? 1단계: a=10, b=20, temp=10 (temp = a) 2단계: a=20, b=20, temp=10 (a = b) 3단계: a=20, b=10, temp=10 (b = temp)

[심화] 방법 2: XOR swap (temp 없이!)

XOR 연산의 특성을 이용하면 추가 변수 없이 교환할 수 있어요. 하지만 실전에서는 권장하지 않아요!

swap_xor.c
1#include <stdio.h>
2
3int main(void) {
4 int a = 10, b = 20;
5 printf("교환 전: a=%d, b=%d\n", a, b);
6
7 a ^= b; // a = a ^ b
8 b ^= a; // b = b ^ (a ^ b) = 원래 a
9 a ^= b; // a = (a ^ b) ^ 원래 a = 원래 b
10
11 printf("교환 후: a=%d, b=%d\n", a, b);
12 return 0;
13}
[주의] XOR swap의 함정
1. a와 b가 같은 변수를 가리키면 0이 돼 버려요!
2. 가독성이 매우 나빠요
3. 현대 CPU에서 temp 방식보다 느릴 수 있어요 (데이터 의존성 때문)
면접 질문으로만 알아두세요. 실전에서는 temp를 쓰세요!

[심화] 방법 3: 덧셈/뺄셈 swap

2

[결론] 어떤 swap을 쓸까?

[심화 내용] volatile과 대입

volatile 키워드가 붙은 변수는 컴파일러가 대입을 최적화하지 않아요. 하드웨어 레지스터 접근에 중요해요.

volatile_assign.c
1#include <stdio.h>
2
3// 임베디드 시스템 예시
4volatile int *hardware_reg = (volatile int *)0x40021000;
5
6int main(void) {
7 // 일반 변수: 컴파일러가 최적화 가능
8 int normal = 0;
9 normal = 1; // 컴파일러가 '어차피 다음에 또 바꾸니까' 하고
10 normal = 2; // 첫 번째 대입을 제거할 수 있어요!
11 normal = 3;
12 // 최적화 후: normal = 3; (한 줄로 축소)
13
14 // volatile 변수: 모든 대입이 실제로 실행돼야 해요!
15 volatile int device = 0;
16 device = 1; // 반드시 하드웨어에 1을 씀
17 device = 2; // 반드시 하드웨어에 2를 씀
18 device = 3; // 반드시 하드웨어에 3을 씀
19 // 모든 대입이 유지됨! (하드웨어가 각 값에 반응할 수 있으니까)
20 return 0;
21}
[심화 내용] volatile이 필요한 곳
1. 하드웨어 레지스터: LED, 센서 등 제어
2. 인터럽트 핸들러: ISR에서 바꾸는 플래그 변수
3. 멀티스레드: 다른 스레드가 바꿀 수 있는 변수 (하지만 원자성은 보장 X!)
4. 메모리 매핑 I/O: 특정 메모리 주소에 쓰면 하드웨어가 동작하는 경우

[심화 내용] 원자적 대입 — 멀티스레드의 대입

여러 스레드가 동시에 같은 변수에 대입하면 경쟁 조건(race condition)이 발생해요. 이를 방지하는 방법을 알아봐요.

atomic_assign.c
1#include <stdio.h>
2#include <stdatomic.h> // C11 원자적 연산
3
4// 전역 카운터 (여러 스레드가 동시에 접근)
5atomic_int counter = 0; // 원자적 정수
6
7// 일반 대입은 위험해요!
8// counter += 1; → 읽기 + 더하기 + 쓰기 3단계
9// 중간에 다른 스레드가 끼어들 수 있어요!
10
11// 원자적 대입은 안전해요!
12void safe_increment(void) {
13 atomic_fetch_add(&counter, 1); // 한 번에 읽기+더하기+쓰기
14}
15
16// 원자적 대입
17void safe_assign(void) {
18 atomic_store(&counter, 100); // 원자적으로 100을 저장
19 int val = atomic_load(&counter); // 원자적으로 읽기
20}
[심화 내용] 원자적 연산 요약
문제: counter++는 (읽기→더하기→쓰기) 3단계라서 중간에 끊길 수 있어요
해결 1: atomic_fetch_add() — C11 표준
해결 2: mutex로 잠금 — 더 무거운 보호
해결 3: __sync_fetch_and_add() — GCC 내장 함수

[심화 내용] 대입의 어셈블리

대입 연산자는 CPU의 MOV 명령어로 변환돼요.

assign_asm.asm
1; C: x = 42;
2mov dword [rbp-4], 42 ; 상수 42를 x의 메모리에 직접 저장
3
4; C: y = x;
5mov eax, [rbp-4] ; x를 레지스터에 로드
6mov [rbp-8], eax ; 레지스터에서 y로 저장
7
8; C: x += 5;
9mov eax, [rbp-4] ; x를 로드
10add eax, 5 ; 5를 더하기
11mov [rbp-4], eax ; 결과를 x에 저장
12
13; 최적화 후: x += 5;
14add dword [rbp-4], 5 ; 메모리에 직접 더하기! (한 줄로!)
[심화 내용] 메모리 직접 연산
x86은 메모리에 직접 연산하는 명령어가 있어요: add dword [rbp-4], 5. 복합 대입이 이런 최적화와 잘 맞기 때문에, 컴파일러가 더 효율적인 코드를 생성할 수 있어요.

[심화] 대입에서의 형변환

오른쪽 값의 타입이 왼쪽 변수의 타입과 다르면, 자동으로 변환돼요. 이때 정보가 손실될 수 있어요!

assign_conv.c
1#include <stdio.h>
2
3int main(void) {
4 // 넓은 타입 → 좁은 타입: 정보 손실!
5 int big = 300;
6 char small = big; // 300은 char 범위를 넘음!
7 printf("big=%d, small=%d\n", big, small); // 300, 44
8
9 // 실수 → 정수: 소수점 버림
10 double pi = 3.14159;
11 int n = pi; // 소수점 버림!
12 printf("pi=%f, n=%d\n", pi, n); // 3.14159, 3
13
14 // 좁은 타입 → 넓은 타입: 안전!
15 char c = 'A';
16 int wide = c; // char(65) → int(65) 안전!
17 printf("c=%c, wide=%d\n", c, wide); // A, 65
18 return 0;
19}
[주의] 좁히는 변환(Narrowing Conversion)
큰 타입의 값을 작은 타입에 넣으면 잘릴 수 있어요!
int → char: 하위 8비트만 남음
double → int: 소수점 버림
long long → int: 하위 32비트만 남음
컴파일러 경고를 확인하세요!

용어 정리

대입 연산자(=)
오른쪽 값을 왼쪽 변수에 넣는 연산자.
복합 대입 연산자
계산과 대입을 합친 축약형. +=, -=, *=, /=, %= 등.
축약형
같은 의미를 더 짧게 쓰는 형태.
lvalue
대입의 왼쪽에 올 수 있는 것. 메모리 위치를 가진 표현식.
rvalue
대입의 오른쪽에만 올 수 있는 것. 값만 있고 위치가 없는 표현식.
초기화
변수를 선언하면서 동시에 값을 부여하는 것. const 변수도 초기화 가능.
체이닝
a = b = c = 0처럼 대입을 연쇄적으로 연결하는 것.
비트 복합 대입
&=, |=, ^=, <<=, >>= — 비트 연산과 대입을 합친 연산자.
volatile
컴파일러에게 '이 변수의 접근을 최적화하지 마라'고 알려주는 키워드.
원자적 연산(Atomic Operation)
중간에 끊기지 않는 연산. 멀티스레드에서 안전한 대입.
경쟁 조건(Race Condition)
여러 스레드가 동시에 같은 데이터를 바꿀 때 결과가 예측 불가능해지는 상황.
swap
두 변수의 값을 교환하는 것. temp 변수 사용이 정석.
이번 유닛 핵심 정리
1
= 는 '같다'가 아니라 '넣어라'! 오른쪽 → 왼쪽 방향
2
a += b는 a = a + b의 축약형이에요
3
+=, -=, *=, /=, %= 다섯 가지 산술 복합 대입 연산자
4
&=, |=, ^=, <<=, >>= 다섯 가지 비트 복합 대입 연산자
5
대입 연산자도 값을 반환해요 → 체이닝 가능
6
lvalue(대입 가능한 것) vs rvalue(값만 있는 것)
7
초기화(선언 시)와 대입(이후)은 다른 개념!
8
swap은 temp 변수 방식이 정석
이 유닛에서 연결된 축
[C언어] =, +=, -=, *=, /=, %=, &=, |=, ^=, <<=, >>=
[컴퓨팅 사고력] 패턴 인식 — 반복되는 대입 패턴 축약
[디버깅] = 와 == 혼동, 초기화하지 않은 변수, 좁히는 변환
[KOI] 변수 갱신, swap 패턴, 누적합
[코딩 기초] 코드 간결함, lvalue/rvalue
[실생활] 복합 대입이 쓰이는 곳
