[심화] 감소 루프가 빠를 수 있는 이유
일부 아키텍처에서는 감소 후 0과 비교하는 것이 증가 후 n과 비교하는 것보다 빨라요:
dec ecx / jnz .loop (2명령, 비교 없음!)
inc ecx / cmp ecx, n / jl .loop (3명령)
현대 컴파일러는 최적화로 이 차이를 없애지만, 임베디드 환경에서는 여전히 유효해요.

[심화] 시퀀스 포인트와 정의되지 않은 동작

C 언어의 가장 무서운 함정! 같은 변수를 한 수식에서 여러 번 변경하면 결과를 아무도 보장 못 해요.

[심화] 시퀀스 포인트(sequence point)란 '이 지점까지의 부수 효과(side effect)가 모두 완료된다'는 보장이에요. 세미콜론 ;, 쉼표 연산자 ,, &&, ||, ?:의 특정 위치가 시퀀스 포인트예요.

ub_example.c
1#include <stdio.h>
2
3int main(void) {
4 int i = 5;
5
6 // ⚠️ 정의되지 않은 동작 (UB)!
7 int result = i++ + i++; // 결과를 예측할 수 없음
8 printf("%d\n", result); // 컴파일러마다 다른 값!
9
10 return 0;
11}
[경고] i++ + i++는 왜 정의되지 않은 동작인가?
C 표준(C99 §6.5/2): 두 시퀀스 포인트 사이에서 같은 변수를 두 번 이상 변경하면 정의되지 않은 동작(Undefined Behavior)이에요.

i++ + i++에서:
• 왼쪽 i++: i를 변경 (부수 효과)
• 오른쪽 i++: i를 또 변경 (부수 효과)
• 두 변경 사이에 시퀀스 포인트가 없음!
• → UB: 컴파일러가 아무 결과나 내놓을 수 있음
UB 예시 — 컴파일러마다 다른 결과
표현식GCC 결과 (예)Clang 결과 (예)MSVC 결과 (예)
i++ + i++ (i=5)111010
++i + ++i (i=5)141314
i = i++ + 1 (i=5)676

[심화] 위 표를 보세요 — 같은 코드인데 컴파일러마다 결과가 다 달라요! 이것이 바로 정의되지 않은 동작(UB)의 무서움이에요. 이런 코드는 절대 쓰면 안 돼요.

[심화 내용] C11의 sequenced-before 개념

[심화 내용] C11부터는 '시퀀스 포인트' 대신 sequenced-before라는 더 정밀한 개념을 사용해요. 두 평가(evaluation) A, B 사이의 관계를 세 가지로 나눠요.

C11 평가 순서 관계
관계의미예시
A sequenced-before BA가 반드시 B 이전에 완료f(); g(); — f()가 g()보다 먼저
B sequenced-before AB가 반드시 A 이전에 완료-
unsequenced순서 미정, 겹칠 수도 있음f() + g() — 어느 쪽이 먼저인지 모름

[심화 내용] i++ + i++가 UB인 이유를 C11 용어로 다시 설명하면: 두 i++의 부수 효과가 unsequenced이면서 같은 객체(i)를 변경하기 때문이에요.

C11 §6.5/2: '스칼라 객체에 대한 부수 효과가 동일 객체에 대한 다른 부수 효과 또는 값 계산과 unsequenced이면 동작은 정의되지 않는다.'

[심화 내용] 안전한 것과 위험한 것
✅ 안전: i++; j = i; — 세미콜론이 시퀀스 포인트
✅ 안전: i++ && i++ — &&의 왼쪽이 먼저 (short-circuit)
❌ 위험: i++ + i++ — +는 시퀀스 포인트 아님
❌ 위험: a[i] = i++ — 읽기와 쓰기가 unsequenced
❌ 위험: i = i++ + 1 — 대입과 후위 증가가 충돌

[심화 내용] 함수 인자에서의 증감 — UB 케이스

함수 인자의 평가 순서는 C 표준에서 정해져 있지 않아요. 여기에 증감을 섞으면 큰일 나요!
func_arg_ub.c
1#include <stdio.h>
2
3void show(int a, int b) {
4 printf("a=%d, b=%d\n", a, b);
5}
6
7int main(void) {
8 int i = 1;
9
10 // ⚠️ 정의되지 않은 동작!
11 show(i++, i++); // 어떤 i++가 먼저 평가될지 모름
12
13 // ⚠️ 이것도 위험!

[심화 내용] f(i++, i++)에서 두 인자의 평가 순서는 unspecified(미정)이에요. 게다가 같은 변수를 두 번 변경하므로 undefined behavior까지 됩니다. 결과를 예측하려는 시도 자체가 무의미해요.

✕ 잘못된 코드
printf("%d %d\n", i++, i++);
UB! 순서 미정 + 이중 변경
✓ 올바른 코드
int a = i++; int b = i++; printf("%d %d\n", a, b);
분리하면 안전! 순서 보장

[심화 내용] side effect와 evaluation order — C17 변경사항

[심화 내용] C17(C18이라고도 불림)은 C11의 평가 순서 규칙을 대부분 그대로 유지했어요. 하지만 C++17에서는 중요한 변경이 있었죠. 비교해 봅시다.

C17 vs C++17 평가 순서 비교
표현식C17C++17
f(a, b, c)인자 순서 미정인자 순서 미정 (변경 없음)
a = b우변→좌변 미정우변이 좌변보다 먼저 (sequenced)
a[b] = c순서 미정b, c가 a[b] 대입보다 먼저

[심화 내용] 실전 조언
C에서는 여전히 한 수식에서 같은 변수를 두 번 변경하지 마세요.
C++17에서도 f(i++, i++)는 여전히 UB예요.
언어 표준을 외우는 것보다 '한 줄에 하나의 부수 효과' 규칙을 지키는 게 훨씬 안전해요.

[심화 내용] C vs C++ — 증감 연산자의 차이

C와 C++에서 ++i의 동작이 미묘하게 달라요. C++에서는 ++i가 lvalue인데, C에서는 아니에요!

[심화 내용] lvalue란 '메모리에 위치를 가진 값'이에요. 대입의 왼쪽(left)에 올 수 있다는 뜻이죠.

++(++i);❌ 컴파일 에러✅ 가능 (두 번 증가)

[심화 내용] 왜 C++에서 ++i를 lvalue로 만들었나?
C++의 설계 철학: '사용자 정의 타입도 내장 타입처럼' 동작해야 한다.
클래스의 operator++()가 참조를 반환할 수 있으려면, 내장 타입의 ++i도 lvalue여야 일관성이 있어요.

C에서는 이런 필요가 없었기 때문에 rvalue로 충분했어요.

[심화 내용] i++ vs ++i — 성능 차이

[심화 내용] C에서 기본 타입(int, double 등)에 대해 i++와 ++i는 성능이 완전히 같아요. 컴파일러가 동일한 코드로 최적화하니까요.

[심화 내용] 하지만 C++에서 이터레이터(iterator)나 클래스 타입을 증감할 때는 얘기가 달라요.

[심화 내용] 성능 정리
int, double 등 기본 타입: i++와 ++i 성능 동일 (컴파일러 최적화)
C++ 이터레이터/클래스: ++it이 it++보다 빠를 수 있음 (복사 비용)
결론: C++에서는 특별한 이유가 없으면 ++it 습관을 들이세요.
C에서는 아무거나 써도 됩니다.

[역사] ++ 연산자의 탄생 — PDP-11

[역사] 왜 C에 ++가 생겼을까?
1970년대, 켄 톰프슨(Ken Thompson)이 B 언어에 ++와 --를 처음 도입했어요. 데니스 리치(Dennis Ritchie)가 C를 만들 때 이를 그대로 가져왔죠.

당시 사용하던 DEC PDP-11 컴퓨터에는 autoincrement addressing mode와 autodecrement addressing mode가 있었어요. 하드웨어가 *p++ 같은 동작을 한 명령어로 처리할 수 있었던 거예요!

[역사] C++라는 이름의 유래
C++라는 프로그래밍 언어 이름이 바로 'C에 1을 더한 것'이라는 뜻이에요! C보다 한 단계 발전했다는 의미로 ++를 붙인 거예요.

비야네 스트롭스트룹(Bjarne Stroustrup)이 1983년에 명명했어요. 재미있는 건, 'C++'가 후위 증가이므로 '먼저 C를 사용하고 나서 증가한다'는 뜻으로도 읽을 수 있어요. C의 영향을 받았다는 의미기도 하죠!

만약 '먼저 발전하고 사용한다'는 뜻이었다면 '++C'가 더 맞았을 거예요.

    용어 정리

    증감 연산자
    변수의 값을 1 증가(++) 또는 1 감소(--)시키는 연산자.
    전위(++a)
    먼저 값을 증가시킨 후, 그 값을 사용해요.
    후위(a++)
    현재 값을 먼저 사용한 후, 그 다음에 증가시켜요.
    정의되지 않은 동작
    C 표준에서 결과를 보장하지 않는 코드. 컴파일러마다 결과가 다를 수 있어요.
    시퀀스 포인트
    [심화] 이 지점까지의 모든 부수 효과(side effect)가 완료되었음을 보장하는 지점. 세미콜론, &&, || 등이 해당.
    부수 효과 (side effect)
    [심화] 수식의 평가 과정에서 변수의 값이 변경되는 것. i++에서 i가 변하는 것이 부수 효과.
    lvalue
    [심화 내용] 메모리에 위치를 가진 값. 대입의 왼쪽에 올 수 있음. C++에서 ++i의 결과는 lvalue.
    rvalue
    [심화 내용] 메모리에 고정된 위치가 없는 임시 값. 대입의 오른쪽에만 올 수 있음. C에서 ++i의 결과는 rvalue.
    sequenced-before
    [심화 내용] C11에서 도입된 개념. 평가 A가 평가 B 이전에 반드시 완료됨을 의미.
    autoincrement addressing mode
    [역사] PDP-11 CPU의 주소 지정 모드. 메모리 접근 후 레지스터를 자동으로 증가시킴. C의 *p++ 패턴의 기원.
    이터레이터 (iterator)
    [심화 내용] C++에서 컨테이너의 원소를 순회하기 위한 객체. 포인터와 비슷하게 ++, *, -> 등의 연산자를 지원.
