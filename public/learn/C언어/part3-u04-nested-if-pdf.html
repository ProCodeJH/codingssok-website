Lesson 3-4

if 안에 if — 중첩 조건문

조건 안에 또 다른 조건을 넣어서 복잡한 판단을 하는 법을 배워요.

📚 이번에 배울 것

  • if 안에 if를 넣는 중첩 구조를 이해해요.
  • 언제 중첩이 필요하고 언제 &&로 대체 가능한지 알아요.
  • 중첩이 깊어지면 왜 위험한지 알아요.

✅ 시작 전 체크

  • if-else문을 사용할 수 있다
  • else if 체인을 안다
Part 1 진행률: 17%

조건 안에 또 다른 조건

놀이기구 타려면 두 가지를 만족해야 해요: 키 130cm 이상이면서 나이 10세 이상! 이럴 때 if 안에 if를 넣어요.

하나의 조건으로 부족할 때, 첫 번째 조건이 참인 경우에만 두 번째 조건을 추가로 확인하는 구조가 중첩 if예요.

중첩 if의 흐름
조건1 참? → 예 → 조건2 참? → 예 → 실행!
조건1 참? 조건2 참? ↓ 아니오
조건1 참? 다른 처리

중첩 if의 생김새

nested_shape.c
1if (조건1) {
2 // 조건1이 참일 때
3 if (조건2) {
4 // 조건1도 참이고 조건2도 참일 때
5 } else {
6 // 조건1은 참이지만 조건2는 거짓일 때
7 }
8} else {
9 // 조건1이 거짓일 때
10}
바깥쪽 if가 '1차 관문', 안쪽 if가 '2차 관문'이에요. 1차를 통과해야 2차를 검사해요!
[코딩 습관] 들여쓰기로 레벨 표시
중첩할 때마다 스페이스 4칸씩 더 들여 써요. 바깥 if는 4칸, 안쪽 if는 8칸. 들여쓰기가 없으면 어디가 안쪽인지 구분 불가!

중첩 if vs && — 언제 뭘 쓸까?

중첩 if는 &&로 바꿀 수 있는 경우가 많아요. 하지만 항상 그런 건 아니에요!

✕ 잘못된 코드
if (height >= 130) { if (age >= 10) { printf("탑승 가능"); } }
중첩: '키 통과 → 나이 통과' 2단계
✓ 올바른 코드
if (height >= 130 && age >= 10) { printf("탑승 가능"); }
&&: 한 줄로! 단순한 경우에 더 깔끔
[핵심] 중첩이 필요한 경우 vs && 가능한 경우
&&로 충분한 경우:
두 조건이 모두 참일 때 같은 처리만 하면 됨
예: if (a>0 && b>0) printf("둘 다 양수");

중첩이 필요한 경우:
첫 번째 조건 결과에 따라 다른 처리가 필요할 때
예: 키 부족이면 '키 부족' 메시지, 키 OK + 나이 부족이면 '나이 부족' 메시지
→ else 블록이 달라야 하므로 중첩 필수!

규칙: else가 필요 없으면 && 쓰고, else가 다르면 중첩!

해결책 3가지:
1. 가드 절: if(나이<18) {printf("미성년"); return;} 로 먼저 걸러내기

3. 함수 분리: can_drive(age, license, drunk) 함수 만들기

[컴퓨팅 사고력] 의사결정 트리와 계층적 분류

[사고력] 중첩 if = 의사결정 트리
의사결정 트리(Decision Tree):
질문을 통해 가지를 쳐서 결론에 도달하는 구조

동물 맞추기:
날 수 있나? → 예 → 깃털? → 예 → 새
                    → 아니오 → 박쥐
          → 아니오 → 다리 4개? → 예 → 개
                             → 아니오 → 뱀

이 트리의 각 '질문'이 if, 각 '분기'가 if-else예요!
머신러닝의 Decision Tree 알고리즘도 같은 원리!
[사고력] 4요소 심화 연결
1. 분해(Decomposition):
'놀이기구 탑승 가능?'을 → (1) 키 확인 (2) 나이 확인 두 단계로 분해
각 단계가 독립적 판단! 1단계 실패하면 2단계는 불필요.

2. 패턴 인식(Pattern Recognition):
로그인(ID→PW), 놀이기구(키→나이), 운전(나이→면허→음주) — 전부 '순차 검증' 패턴!
앞 조건 통과 → 다음 조건 검사. 이 패턴이 반복돼요.

3. 추상화(Abstraction):
로그인의 핵심은 'ID 일치 + PW 일치' 두 가지뿐.
서버 주소, 암호화 방식, UI 디자인은 핵심이 아니에요.

4. 알고리즘 설계(Algorithm Design):
의사결정 트리를 먼저 그리고 코드로 옮기기!
트리에서 '가장 많이 거르는 조건'을 가장 먼저 검사하면 효율적.
의사결정 트리를 먼저 그리고 코드로 옮겨! 트리가 깊으면(3단+) 가드 절로 평탄화!

[디버깅] 중첩 if 버그 시나리오

흔한 버그 — dangling else 재발
버그: else가 의도한 if가 아닌 다른 if에 연결됨
if (x > 0)
  if (y > 0)
    printf("1사분면");
else
  printf("x 음수");

원인: C에서 else는 가장 가까운 if에 연결! x 음수가 아니라 y<=0일 때 실행됨.
해결: 중괄호로 명확하게!
교훈: 중첩 if에서는 모든 if-else에 중괄호 필수!
흔한 버그 — 들여쓰기 함정
버그: 들여쓰기가 구조와 안 맞아서 착각
if (a > 0) {
if (b > 0) { // 안쪽인데 들여쓰기 안 함!
printf("OK");
}
}

원인: C 컴파일러는 들여쓰기를 무시! 중괄호만 인식.
하지만 사람은 들여쓰기로 구조를 파악하므로 혼란!
해결: 중첩 레벨마다 스페이스 4칸 일관 적용
교훈: '코드 포매터'를 사용하면 자동 들여쓰기!
흔한 버그 — 중첩 깊이 과다
버그: 4~5단 중첩으로 논리를 파악할 수 없음
if (a) {
  if (b) {
    if (c) {
      if (d) {
        // 뭐하는 코드?
}}}}

원인: 조건을 추가할 때마다 안쪽에 넣어서 깊어짐
해결: (1) 가드 절로 평탄화 (2) &&로 합치기 (3) 함수 분리
교훈: McCabe 복잡도 10 이상이면 반드시 리팩토링!

10축 심화 연결
[컴퓨팅사고력 — 분해+계층] 중첩 if는 '계층적 분류'예요. 먼저 대분류(키 OK?), 그 다음 소분류(나이 OK?). 도서관의 분류 체계처럼 큰 범주에서 점점 좁히는 거예요. 이 사고 패턴은 데이터베이스 인덱스, 파일 시스템 폴더 구조에도 같은 원리예요.

[디버깅 — dangling else] C에서 가장 유명한 함정! else는 가장 가까운 if에 연결되므로, 중첩에서는 반드시 중괄호를 써야 해요. 이 규칙을 모르면 논리가 완전히 다른 프로그램이 되어요.
