함수에 값을 넘겨주기

지금까지 만든 함수는 매번 같은 일만 했어. 하지만 함수에 값을 전달하면, 전달받은 값에 따라 다르게 동작할 수 있어!

1#include <stdio.h>
2
3void printNumber(int n) {
4 printf("Number: %d\n", n);
5}
6
7int main(void) {
8 printNumber(7);

10 printNumber(100);
11 return 0;
12}
실행 결과

3줄void printNumber(int n) — 괄호 안의 int n이 매개변수(parameter)예요. 함수가 외부에서 값을 받을 수 있는 '창구'예요.
8줄printNumber(7); — 호출할 때 넣어주는 7이 인수(argument)예요. 이 값이 매개변수 n에 복사돼요.

parameter vs argument

parameter: 함수 정의의 변수 / argument: 호출할 때 넘기는 값
parameter vs argument 구분
Parameter(매개변수): 함수를 정의할 때 괄호 안에 쓰는 변수. 값을 받을 준비를 하는 빈 그릇.
Argument(인수): 함수를 호출할 때 괄호 안에 넣는 실제 값. 그릇에 담기는 내용물.

쉽게 기억하기: Parameter = Placeholder(자리), Argument = Actual(실제 값)
헷갈리면 이렇게 기억해: 매개변수는 함수를 '만들 때'의 이름표, 인수는 함수를 '부를 때' 건네주는 값.

매개변수는 쉼표로 구분하고, 각각 자료형을 써줘야 해요
매개변수 규칙
1. 매개변수는 쉼표(,)로 구분해요.
2. 각 매개변수마다 자료형을 따로 써줘야 해요.
3. void printSum(int a, b)는 틀린 문법! void printSum(int a, int b)가 맞아요.

값에 의한 전달 (Call by Value)

C언어는 값을 복사해서 전달한다 (원본은 안 바뀜!)
call_by_value.c
1#include <stdio.h>
2
3void tryChange(int n) {
4 n = 99;
5 printf("Inside: n = %d\n", n);
6}
7
8int main(void) {
9 int x = 10;
10 tryChange(x);
11 printf("Outside: x = %d\n", x);

실행 결과
Inside: n = 99 Outside: x = 10
C언어에서 함수에 변수를 넘기면 값이 복사돼. 그래서 함수 안에서 아무리 바꿔도 원래 변수는 그대로야. 이걸 값에 의한 전달(call by value)이라고 해.

자주 하는 실수

✕ 잘못된 코드

b의 자료형이 빠졌어요!
✓ 올바른 코드
void printSum(int a, int b) { printf("%d", a + b); }
매개변수마다 자료형을 따로 써야 해요
✕ 잘못된 코드
printSum(int 3, int 5);
호출할 때는 자료형을 안 써요!
✓ 올바른 코드
printSum(3, 5);
호출할 때는 값만 넘겨요

용어 정리

[심화] 매개변수 전달 심화

[심화] 값 전달 vs 참조 전달
값 전달 (Call by Value):
C의 기본 방식. 복사본 전달
void f(int x) → x는 원본의 복사
원본 변경 불가!

참조 전달 (Call by Reference):
C에서는 포인터로 구현
void f(int *x) → *x로 원본 접근
원본 변경 가능!

배열 전달:
배열은 항상 포인터로 전달 (값 복사 아님!)
void f(int arr[], int n) == void f(int *arr, int n)
sizeof(arr) == sizeof(int*) (함정!)

구조체 전달:
값 전달 시 전체 복사 (비용!)
큰 구조체 → 포인터 전달 권장
const struct S *p → 읽기 전용

const correctness
const 정확성. 변경하지 않는 포인터에 const 선언. 버그 방지 + 의도 명확화. C++ 핵심 원칙!
출력 매개변수
Output Parameter. 함수가 여러 값을 반환할 때 포인터 인수 사용. divmod(a,b,&q,&r).
C는 무조건 값 전달! 원본을 바꾸려면 주소(&)를 넘겨야 해. 포인터가 필수인 이유!
배열은 예외! 배열 이름 = 포인터이므로 자동으로 참조 전달. sizeof 함정 조심!
[C 코드] 매개변수 — 값 전달 vs 포인터 전달
1#include <stdio.h>
2
3// 값 전달 (Call by Value) — 원본 변경 불가!
4void swap_fail(int a, int b) {
5 int temp = a; a = b; b = temp;
6 printf(" swap_fail 내부: a=%d, b=%d\n", a, b);
7}
8
9// 포인터 전달 (Call by Reference) — 원본 변경!
10void swap_ok(int *a, int *b) {
11 int temp = *a; *a = *b; *b = temp;
12}
13
14// 배열은 항상 포인터로 전달!
15void array_sum(const int *arr, int n, int *result) {
16 *result = 0;
17 for (int i = 0; i < n; i++)
18 *result += arr[i];
19}
20
21// 구조체 전달 — 값 복사 (주의: 큰 구조체는 비용!)
22typedef struct { double x, y; } Point;
23
24double distance(Point p1, Point p2) {
25 double dx = p1.x - p2.x;
26 double dy = p1.y - p2.y;
27 return dx*dx + dy*dy; // 제곱 거리
28}
29
30// 구조체 포인터 — 복사 비용 없음
31void point_move(Point *p, double dx, double dy) {
32 p->x += dx;
33 p->y += dy;
34}
35
36int main() {
37 int x = 10, y = 20;
38 printf("Before swap_fail: x=%d, y=%d\n", x, y);
39 swap_fail(x, y);
40 printf("After swap_fail: x=%d, y=%d (변경 안 됨!)\n\n", x, y);
41
42 swap_ok(&x, &y);
43 printf("After swap_ok: x=%d, y=%d (변경됨!)\n\n", x, y);
44

46 int sum;

[심화 내용] 인자 전달 메커니즘
C는 항상 값 전달!
포인터를 넘기는 것도 "포인터 값"의 복사

C++의 참조(Reference):
void swap(int &a, int &b) — 포인터 문법 없이 원본 수정
내부적으로는 포인터와 동일

Java:
기본형 = 값 전달, 객체 = 참조 전달
(정확히는 참조의 값 전달)

Python:
"객체 참조 전달" — 재할당은 불가, 변경은 가능

Rust:
&T (불변 빌려주기) / &mut T (가변 빌려주기)
소유권 시스템으로 안전 보장!
const 포인터
const int *p: 값 변경 불가. int *const p: 주소 변경 불가. const int *const p: 둘 다 불가!
restrict
C99. 포인터가 유일한 접근 경로임을 선언. 컴파일러 최적화 힌트. memcpy vs memmove!
swap_fail이 왜 실패? 값이 복사되니까! 원본을 바꾸려면 주소(&)를 넘겨야 해!
C는 항상 값 전달! 포인터도 "포인터 값"의 복사. 이걸 이해하면 C의 철학이 보여!
pass by name
이름 전달. Algol 60. 인자가 호출될 때마다 재평가! 매크로와 유사. "Jensen의 장치" 트릭!
default argument
기본 인자. C++/Python에서 지원. C에서는 매크로로 시뮬레이션. #define foo(x,...) foo_impl(x, ##__VA_ARGS__)
out parameter
출력 매개변수. 포인터로 결과 반환. C의 "다중 반환" 패턴. COM/Windows API에서 많이 사용!
매개변수 = 함수의 입력 슬롯! 값 전달은 안전, 포인터 전달은 강력. 용도에 맞게!

[C 코드] 매개변수 — 에러 처리 패턴
1#include <stdio.h>
2#include <string.h>
3#include <errno.h>
4
5typedef enum {
6 ERR_OK = 0,
7 ERR_NULL_PTR = -1,
8 ERR_OUT_OF_RANGE = -2,
9 ERR_INVALID_ARG = -3,
10 ERR_OVERFLOW = -4
11} ErrorCode;
12
13const char* error_string(ErrorCode ec) {
14 switch(ec) {
15 case ERR_OK: return "OK";
16 case ERR_NULL_PTR: return "Null pointer";
17 case ERR_OUT_OF_RANGE: return "Out of range";
18 case ERR_INVALID_ARG: return "Invalid argument";
19 case ERR_OVERFLOW: return "Overflow";
20 default: return "Unknown error";
21 }
22}
23
24ErrorCode safe_array_get(int *arr, int size, int index, int *out) {
25 if (!arr || !out) return ERR_NULL_PTR;
26 if (index < 0 || index >= size) return ERR_OUT_OF_RANGE;
27 *out = arr[index];
28 return ERR_OK;
29}
30
31ErrorCode safe_divide(int a, int b, int *quot, int *rem) {
32 if (!quot || !rem) return ERR_NULL_PTR;
33 if (b == 0) return ERR_INVALID_ARG;
34 if (a == -2147483647-1 && b == -1) return ERR_OVERFLOW;
35 *quot = a / b;
36 *rem = a % b;
37 return ERR_OK;
38}
39
40#define CHECK(expr) do { \
41 ErrorCode _ec = (expr); \
42 if (_ec != ERR_OK) { \
43 printf("Error: %s at %s:%d\n", error_string(_ec), __FILE__, __LINE__); \
44 return 1; \
45 } \
46} while(0)
47
48int main() {
49 int arr[] = {10, 20, 30, 40, 50};
50 int val, q, r;
51
52 CHECK(safe_array_get(arr, 5, 2, &val));
53 printf("arr[2] = %d\n", val);
54
55 ErrorCode ec = safe_array_get(arr, 5, 10, &val);
56 printf("arr[10] = %s\n", error_string(ec));
57
58 CHECK(safe_divide(17, 5, &q, &r));
59 printf("17/5 = %d rem %d\n", q, r);
60
61 ec = safe_divide(10, 0, &q, &r);

CHECK 매크로
에러 체크 자동화. 실패 시 파일명+줄번호 출력. Linux 커널, 게임 엔진에서 필수 패턴!
에러 코드 + CHECK 매크로 = C의 에러 처리 정석! 모든 함수 호출을 한 줄로 검증!
[활동] 매개변수 마스터 체크리스트
값 전달 vs 포인터 전달 차이를 안다
swap 함수를 올바르게 구현한다
const 포인터를 적절히 사용한다
void*로 제네릭 함수를 만든다
에러 코드+out 파라미터 패턴을 안다
구조체를 값/포인터로 전달하는 차이를 안다
CHECK 매크로로 에러 처리한다

strict aliasing
C99 규칙. 다른 타입 포인터로 같은 메모리 접근 금지. char*만 예외! -fno-strict-aliasing.
compound literal
C99. (int[]){1,2,3}. 임시 배열/구조체를 인라인 생성! 함수 인자로 직접 전달 가능.
매개변수 완전 졸업! 값 전달→포인터→const→void*→에러 패턴. 함수 인터페이스의 달인!
[C 코드] 매개변수 — 함수형 파이프라인
1#include <stdio.h>
2
3typedef int (*IntFunc)(int);
4
5int pipeline(int val, IntFunc *funcs, int nfuncs) {
6 for (int i = 0; i < nfuncs; i++)
7 val = funcs[i](val);
8 return val;
9}
10
11int add_10(int x) { return x + 10; }
12int mul_2(int x) { return x * 2; }
13int sub_5(int x) { return x - 5; }
14int square(int x) { return x * x; }
15
16int main() {
17 IntFunc pipe1[] = {add_10, mul_2, sub_5};
18 printf("5 → +10 → ×2 → -5 = %d\n", pipeline(5, pipe1, 3));
19
20 IntFunc pipe2[] = {mul_2, add_10, square};
21 printf("3 → ×2 → +10 → ² = %d\n", pipeline(3, pipe2, 3));
22 return 0;
23}
실행 결과
5 → +10 → ×2 → -5 = 25 3 → ×2 → +10 → ² = 256
function composition
함수 합성. f∘g(x)=f(g(x)). 파이프라인=순차 합성. Unix 파이프 | 와 동일 개념!
designated initializer
C99. (Point){.x=1,.y=2}. 이름으로 초기화! 가독성 UP. C++20에서도 채택.
parameter pack
C11 _Generic: 타입별 함수 선택. #define print(x) _Generic((x),int:print_int,...)(x)
aliasing
두 포인터가 같은 메모리를 가리킴. 컴파일러 최적화 방해! restrict로 방지 선언.
매개변수 완전 졸업! 값→포인터→void*→에러패턴→파이프라인. 인터페이스 설계의 달인!
[심화 내용] ABI와 매개변수 전달
x86-64 System V ABI:
정수: RDI, RSI, RDX, RCX, R8, R9 (6개)
실수: XMM0~XMM7 (8개)
나머지: 스택으로
반환: RAX(정수), XMM0(실수)

ARM64 AAPCS:
X0~X7 (8개 정수), D0~D7 (8개 실수)

varargs의 비밀:
레지스터→스택 영역에 dump
va_arg가 순서대로 읽기
타입 정보 없음 → printf format이 필수!
red zone
레드 존. 스택 포인터 아래 128바이트. 커널에서 사용 안 함. 리프 함수가 스택 조정 없이 사용!
named parameters
이름 매개변수. C에는 없음! 구조체로 시뮬레이션: func((Options){.width=10,.height=20})
inout parameter
입출력 매개변수. 값을 읽고 수정. C: 포인터로. Swift: inout 키워드. Fortran: INTENT(INOUT).
array length idiom
sizeof(arr)/sizeof(arr[0]). 배열 크기 계산 관용구. 함수에 넘기면 sizeof가 포인터 크기!
register keyword
register int x; CPU 레지스터 배치 힌트. 현대 컴파일러는 무시! 주소 연산(&x) 불가.
volatile parameter
volatile. 컴파일러 최적화 방지. 하드웨어 레지스터, 시그널 핸들러 변수에 사용!
매개변수 설계 = API 설계! 좋은 함수 인터페이스가 좋은 프로그램의 시작!
transparent union
투명 유니언. GCC 확장. 여러 타입을 하나의 인자로 받기. 타입 안전한 오버로딩 시뮬레이션!
sentinel attribute
__attribute__((sentinel)). 가변 인자 끝에 NULL 필수. execl처럼 NULL 종단 보장!
format attribute
__attribute__((format(printf,1,2))). printf 형식 인자 검사! 타입 불일치 경고.
GCC attribute로 함수 안전성 강화! format, sentinel, warn_unused_result. 버그 예방!
callee-saved
피호출자 저장 레지스터. RBX,RBP,R12~R15. 함수가 보존해야 함! 호출자는 안심하고 사용.
caller-saved
호출자 저장 레지스터. RAX,RCX,RDX,RSI,RDI,R8~R11. 함수 호출 시 덮어쓸 수 있음!
stack alignment
스택 정렬. x86-64: 16바이트 정렬 필수! CALL 전에 RSP가 16의 배수. SSE 명령어 요구사항.
named params
명명 매개변수. C에는 없지만 구조체로 시뮬레이션! opts_t opts = {.width=100, .height=50};
parameter object
매개변수 객체. 인자가 많으면 구조체로 묶기! config_t cfg = {...}; init(&cfg);
인자 3개 넘으면 구조체로! 가독성 UP, 확장성 UP. 깔끔한 인터페이스의 비결!
builder pattern
빌더 패턴. 구조체를 단계적으로 설정. set_name→set_age→build. C에서는 함수 체인으로!
named params
이름 있는 매개변수. C에는 없지만 구조체로 시뮬레이션! Options opts = {.timeout=30, .retry=3};
variadic macro
가변 인자 매크로. #define LOG(fmt,...) printf(fmt, ##__VA_ARGS__). ##는 빈 인자 시 쉼표 제거!
매개변수 = 함수의 계약! 입력이 명확해야 함수가 신뢰됨. const, restrict로 의도 표현!
매개변수(parameter)
함수를 정의할 때 괄호 안에 선언하는 변수. 함수가 외부에서 값을 받는 창구.
인수(argument)
함수를 호출할 때 전달하는 실제 값. 매개변수에 복사되어 들어감.
값에 의한 전달(call by value)
함수에 변수를 넘기면 값이 복사되어 전달됨. 원본 변수는 바뀌지 않음.
이번 유닛 핵심 정리
1
매개변수(parameter): 함수 정의의 변수, 인수(argument): 호출 시 전달하는 값
2
매개변수는 쉼표로 구분하고 각각 자료형을 써줘야 한다
3
C언어는 call by value: 값을 복사해서 전달하므로 원본은 안 바뀜
4
호출할 때는 자료형을 쓰지 않고 값만 넘긴다
