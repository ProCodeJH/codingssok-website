구조체 크기가 예상과 다르다?

char(1바이트) + int(4바이트) = 5바이트일 것 같지? 그런데 sizeof를 찍어보면 8바이트라고 나와. 왜 그런지 알아보자!
struct_sizeof.c
1#include <stdio.h>
2
3struct Example {
4 char a; // 1 byte
5 int b; // 4 bytes
6 char c; // 1 byte
7};
8
9int main(void) {
10 printf("char: %zu bytes\n", sizeof(char));
11 printf("int: %zu bytes\n", sizeof(int));
12 printf("struct Example: %zu bytes\n", sizeof(struct Example));
13 return 0;
14}
실행 결과
char: 1 bytes int: 4 bytes struct Example: 12 bytes

char(1) + int(4) + char(1) = 6바이트여야 할 것 같지만, 실제로는 12바이트예요. 이 차이는 패딩(padding) 때문에 발생해요.

메모리 패딩이란

같은 멤버인데 순서만 바꿔도 메모리 사용량이 달라진다
CPU는 메모리를 읽을 때 4바이트(또는 8바이트) 단위로 읽어. 그래서 int가 4의 배수 주소에 오도록 빈 공간(패딩)을 넣는 거야. 이걸 메모리 정렬(alignment)이라고 해.
패딩 규칙 요약
1. 각 멤버는 자기 크기의 배수 주소에 배치된다 (int는 4의 배수, char은 1의 배수)
2. 구조체 전체 크기는 가장 큰 멤버의 크기의 배수가 된다
3. 멤버 순서를 바꾸면 패딩이 줄어들 수 있다

sizeof 연산자

기본 자료형과 구조체의 sizeof 비교

sizeof는 자료형이나 변수의 바이트 크기를 알려주는 연산자예요. 함수가 아니라 컴파일 시점에 계산되는 연산자라는 점이 중요해요.

sizeof(struct Example)와 sizeof(s1) 둘 다 사용할 수 있어. 타입 이름이든 변수 이름이든 상관없어!

memset과 memcpy

memset은 메모리를 특정 값(보통 0)으로 채우고, memcpy는 메모리를 통째로 복사해요. 둘 다 <string.h>에 선언되어 있어요.

memset은 전부 0으로 채우고, memcpy는 구조체를 통째로 복사한다
memset_memcpy.c
1#include <stdio.h>
2#include <string.h>
3
4typedef struct {
5 char name[20];
6 int score;
7 int id;
8} Student;
9
10int main(void) {
11 Student s1 = {"Kim", 95, 1001};
12 Student s2;
13
14 memset(&s2, 0, sizeof(Student));
15 printf("memset 후: %s, %d, %d\n", s2.name, s2.score, s2.id);
16
17 memcpy(&s2, &s1, sizeof(Student));
18 printf("memcpy 후: %s, %d, %d\n", s2.name, s2.score, s2.id);
19 return 0;
20}
실행 결과
memset 후: , 0, 0 memcpy 후: Kim, 95, 1001
14줄memset(&s2, 0, sizeof(Student)) -- s2의 모든 바이트를 0으로 채워요. 구조체 전체 초기화에 유용해요.
17줄memcpy(&s2, &s1, sizeof(Student)) -- s1의 내용을 s2에 바이트 단위로 복사해요.
[하드웨어] 패딩과 성능
메모리 정렬은 CPU가 데이터를 빠르게 읽기 위한 하드웨어 최적화예요. 정렬되지 않은 접근은 CPU에 따라 속도가 느려지거나, 일부 CPU에서는 오류가 발생하기도 해요.

문제 1 CPU가 메모리를 효율적으로 읽기 위해 각 멤버를 자기 크기의 배수 주소에 배치하는데, 이때 생기는 빈 공간이 패딩이다.
문제 2 sizeof는 컴파일 시점에 계산되는 연산자이다.
문제 3 memset은 메모리를 특정 값(주로 0)으로 채우고, memcpy는 한 메모리 영역의 내용을 다른 영역으로 복사한다.
문제 4 struct A: 8바이트 (char+char = 2, 패딩 2, int = 4). struct B: 12바이트 (char = 1, 패딩 3, int = 4, char = 1, 패딩 3).
문제 5 memset은 바이트 단위로 채우므로, int를 1로 memset하면 각 바이트가 0x01이 되어 0x01010101 = 16843009가 된다. 원하는 결과가 아니므로 int는 직접 대입해야 한다.
문제 6 16 -- double은 8바이트 정렬이 필요하므로, int(4) 뒤에 패딩 4바이트가 추가되어 8 + 8 = 16.

[심화 내용] 메모리 정렬의 이유
하드웨어 요구사항:
CPU는 자연 경계에서 데이터를 읽음
4바이트 int → 4의 배수 주소에 배치
비정렬 접근 → x86은 느림, ARM은 크래시!

캐시 라인:
64바이트 단위로 로드
구조체가 라인을 넘으면 2번 로드!
__attribute__((aligned(64)))로 라인 정렬

False Sharing:
다른 코어가 같은 캐시 라인의 다른 변수 수정
→ 캐시 무효화 → 심각한 성능 저하
→ 패딩으로 라인 분리!

offsetof 매크로:
#define offsetof(type,member) ((size_t)&((type*)0)->member)
멤버의 구조체 내 위치를 계산
offsetof
멤버의 바이트 오프셋. offsetof(struct S, field). . 직렬화, container_of에 필수!
bit field
비트필드. unsigned int flag:1; 비트 단위 멤버. 플래그, 프로토콜 헤더에 사용! 이식성 주의.
false sharing
거짓 공유. 다른 스레드가 같은 캐시 라인 접근. 성능 100배 저하! 패딩으로 분리!
구조체 멤버 순서만 바꿔도 8바이트 절약! 큰 멤버→작은 멤버 순서로 패딩 최소화!
비트필드로 4바이트에 active+admin+level+age를 모두 저장! 메모리 극한 절약!

4void hex_dump(const void *ptr, int size) {
5 const unsigned char *p = (const unsigned char*)ptr;
6 for (int i = 0; i < size; i++) {
7 if (i % 16 == 0) printf("%04x: ", i);
8 printf("%02x ", p[i]);
9 if (i % 16 == 15 || i == size-1) {
10 // 패딩
11 for (int j = i%16; j < 15; j++) printf(" ");
12 printf(" |");
13 int start = i - (i%16);
14 for (int j = start; j <= i; j++)
15 printf("%c", (p[j]>=32&&p[j]<127)?p[j]:'.');
16 printf("|\n");
17 }
18 }
19}

[C 코드] 구조체 메모리 — 직렬화/역직렬화
1#include <stdio.h>
2#include <string.h>
3
4typedef struct {
5 int version;
6 char name[20];
7 double score;
8 int flags;
9} SaveData;
10
11// 바이너리 직렬화
12int serialize(const SaveData *data, unsigned char *buf, int bufsize) {
13 if (bufsize < (int)sizeof(SaveData)) return -1;
14 memcpy(buf, data, sizeof(SaveData));
15 return sizeof(SaveData);
16}
17
18int deserialize(const unsigned char *buf, int size, SaveData *data) {
19 if (size < (int)sizeof(SaveData)) return -1;
20 memcpy(data, buf, sizeof(SaveData));
21 return 0;
22}
23
24// 텍스트 직렬화
25int to_text(const SaveData *data, char *buf, int bufsize) {
26 return snprintf(buf, bufsize, "version=%d\nname=%s\nscore=%.2f\nflags=%d\n",
27 data->version, data->name, data->score, data->flags);
28}
29
30int from_text(const char *buf, SaveData *data) {
31 return sscanf(buf, "version=%d\nname=%s\nscore=%lf\nflags=%d",
32 &data->version, data->name, &data->score, &data->flags);
33}
34
35void print_bytes(const void *ptr, int size) {
36 const unsigned char *p = ptr;
37 for (int i = 0; i < size; i++) {
38 printf("%02x ", p[i]);
39 if ((i+1) % 16 == 0) printf("\n");
40 }
41 printf("\n");
42}
43
44int main() {
45 SaveData save = {3, "Player1", 99.5, 0x0F};
46
47 // 바이너리
48 unsigned char bin[sizeof(SaveData)];

51 print_bytes(bin, bsize);
52
53 SaveData loaded;

5// C99 Flexible Array Member
6typedef struct {

8 int length;

12Message* msg_create(int type, const char *text) {
13 int len = strlen(text);
14 Message *m = (Message*)malloc(sizeof(Message) + len + 1);
15 m->type = type;
16 m->length = len;
17 memcpy(m->data, text, len + 1);
18 return m;
19}
20
21void msg_print(Message *m) {
22 printf(" [type=%d len=%d] \"%s\"\n", m->type, m->length, m->data);
23}

[심화 내용] 메모리 레이아웃 최적화
Flexible Array Member (C99):
struct { int n; char data[]; };
sizeof = 헤더만! data는 0바이트
malloc(sizeof + len)으로 가변 크기 할당

vs 포인터 멤버:
struct { int n; char *data; };
→ 2번 malloc (구조체 + 데이터)
→ 2번 free, 캐시 미스 증가
FAM: 1번 malloc, 연속 메모리!

SoA (Struct of Arrays):
struct { float *x, *y, *z; int n; };
vs AoS: struct V { float x,y,z; } arr[];
SoA: SIMD 벡터화에 최적!
게임/과학 계산에서 2~4배 빠름
FAM
Flexible Array Member. C99. struct 끝에 크기 없는 배열. 가변 길이 데이터를 한 번의 malloc으로!
SIMD
Single Instruction Multiple Data. 하나의 명령으로 여러 데이터 처리! SSE/AVX(x86), NEON(ARM).
FAM = 가변 길이의 정석! char data[]; 로 선언, malloc(sizeof+len)으로 할당. 한 덩어리!
SoA vs AoS! SIMD로 x좌표만 한 번에 처리하려면 SoA가 최적. 게임 엔진의 비밀!
💡 구조체 메모리 체크리스트
✅ sizeof로 실제 크기 확인 (패딩 포함!)
✅ 큰 멤버→작은 멤버 순서로 패딩 최소화
✅ offsetof로 멤버 오프셋 확인
✅ #pragma pack은 네트워크/파일에만 사용
✅ 비트필드는 이식성 주의 (비트 순서 비표준)
✅ 직렬화 시 엔디안 변환 고려
✅ 메모리 정렬: alignas(16) for SIMD
✅ 캐시 라인(64B) 크기 인식
alignas
C11. alignas(16) struct Vec4. 메모리 정렬 지정. SIMD 연산에 필수! _Alignas의 편의 매크로.
cache line
캐시 라인. 보통 64바이트. CPU가 메모리를 읽는 최소 단위. 구조체가 라인에 맞으면 성능↑!
data alignment
데이터 정렬. 4바이트 int→4의 배수 주소. 비정렬→x86 느림, ARM 크래시! 자동 패딩 이유.
메모리 레이아웃 마스터! sizeof, offsetof, hex_dump로 구조체의 진짜 모습을 보자!
struct hack
C89 구조체 해킹. char data[1]; 끝에 배열. malloc(sizeof+len-1). FAM의 전신!
memory mapped IO
메모리 매핑 I/O. 하드웨어 레지스터를 메모리 주소로 접근. volatile 필수! 임베디드 핵심.
volatile
volatile. 컴파일러 최적화 금지. 하드웨어 레지스터, 시그널 핸들러, 멀티스레드 변수에 필수!
메모리 마스터! 패딩·비트필드·FAM·직렬화·hex dump. 바이트 레벨의 이해!
memory fence
메모리 펜스. CPU 명령 재배치 방지. __sync_synchronize(). 멀티스레드에서 순서 보장!
numa
NUMA. Non-Uniform Memory Access. CPU마다 로컬 메모리. 로컬 접근이 원격보다 빠름! 서버 최적화.
page fault
페이지 폴트. 접근한 가상 주소가 물리 메모리에 없음. OS가 디스크에서 로드! 성능 영향.
메모리 완전 정복! 패딩→비트필드→FAM→직렬화→hexdump→캐시→NUMA. 깊이가 다르다!
flexible array member
유연 배열 멤버. struct S { int n; int data[]; }; malloc(sizeof(S)+n*4)로 가변 크기!
alignment requirement
정렬 요구. double은 8바이트 경계. _Alignof(double)==8. alignas()로 직접 지정!
패딩 = 빈 공간! CPU가 빠르게 읽으려고 넣는 거야. sizeof가 멤버 합보다 큰 이유!
struct hack
구조체 해킹. C89에서 char data[1]로 가변 크기 구현. C99의 flexible array member가 공식 대체!
sizeof(struct) ≠ 멤버 합! 패딩 때문. offsetof로 직접 확인해보면 눈에 보여!
패딩(padding)
메모리 정렬을 위해 구조체 멤버 사이에 삽입되는 빈 바이트.
메모리 정렬(alignment)
데이터를 자기 크기의 배수 주소에 배치하여 CPU 접근 효율을 높이는 것.
sizeof
자료형이나 변수의 바이트 크기를 반환하는 컴파일 시점 연산자.
memset
메모리 영역을 특정 바이트 값으로 채우는 함수. string.h에 선언.
memcpy
메모리 영역을 바이트 단위로 복사하는 함수. string.h에 선언.
