printf 디버깅의 한계

버그를 찾으려고 printf를 잔뜩 넣었다가, 다 찾고 나면 지우는 것도 일이지? 디버거를 쓰면 코드를 수정하지 않고도 프로그램을 한 줄씩 실행하면서 변수 값을 볼 수 있어!

디버거(debugger)는 프로그램의 실행을 제어하면서 버그를 찾는 도구예요. 원하는 위치에서 멈추고, 변수 값을 확인하고, 한 줄씩 실행할 수 있어요.

중단점(Breakpoint) 설정하기

중단점은 프로그램이 실행 중에 멈출 위치를 표시하는 것이에요. VS Code에서는 줄 번호 왼쪽을 클릭하면 빨간 점이 생기면서 중단점이 설정돼요.

debug_example.c
1#include <stdio.h>
2
3int sum(int arr[], int n) {
4 int total = 0;
5 for (int i = 0; i <= n; i++) { /* 버그: i <= n (범위 초과) */
6 total += arr[i];
7 }
8 return total;
9}
10
11int main(void) {
12 int numbers[] = {10, 20, 30, 40, 50};
13 int result = sum(numbers, 5);
14 printf("합계: %d\n", result);
15 return 0;
16}

위 코드에는 버그가 있어요. for 루프에서 i <= n이라고 써서 배열 범위를 벗어나요. 디버거로 이 버그를 찾아볼게요.

스텝 실행: Step Over, Step Into, Step Out

디버거에서 프로그램을 한 줄씩 실행할 때 3가지 방법이 있어요.

step_demo.c
1#include <stdio.h>
2
3int multiply(int a, int b) {
4 return a * b; /* Step Into로 여기까지 들어옴 */
5}
6
7int calculate(int x) {
8 int doubled = multiply(x, 2); /* Step Into: 함수 안으로 */
9 return doubled + 10; /* Step Out: 여기로 돌아옴 */
10}
11
12int main(void) {
13 int a = 5;
14 int b = calculate(a); /* Step Over: 함수 건너뜀 */
15 int c = b + 3; /* Step Over: 다음 줄로 */
16 printf("결과: %d\n", c);
17 return 0;
18}
Step Over(F10)는 함수를 통째로 실행하고 다음 줄로. Step Into(F11)는 함수 안으로 들어가서 한 줄씩. Step Out(Shift+F11)은 현재 함수에서 빠져나와!

변수 감시와 호출 스택

변수 감시(Watch)는 특정 변수나 식의 값을 계속 추적하는 기능이에요. 호출 스택(Call Stack)은 현재 어떤 함수가 어떤 함수를 호출해서 여기까지 왔는지 보여줘요.

watch_demo.c
1#include <stdio.h>
2
3int factorial(int n) {
4 if (n <= 1) return 1;
5 return n * factorial(n - 1); /* 재귀 호출 */
6}
7
8int main(void) {
9 int result = factorial(5);
10 printf("5! = %d\n", result);
11 return 0;
12}

factorial(5)를 디버거로 추적하면 호출 스택에 factorial이 5번 쌓이는 것을 볼 수 있어요. 각 단계에서 n의 값이 5, 4, 3, 2, 1로 바뀌는 것을 Watch 창에서 확인할 수 있어요.

연습문제

OX 퀴즈

디버거를 사용하면 코드를 수정하지 않고도 변수 값을 확인할 수 있다.

정답: ⭕ O
디버거는 프로그램 실행 중에 변수 값을 보여줘요. printf를 추가할 필요가 없어요.
객관식

함수 안으로 들어가서 한 줄씩 실행하는 디버거 명령은?

  1. Step Over (F10)
  2. Step Into (F11)
  3. Step Out (Shift+F11)
  4. Continue (F5)
정답: 2
Step Into(F11)는 함수 호출을 만났을 때 그 함수 안으로 들어가서 실행해요.
단답형

프로그램이 특정 줄에서 멈추도록 설정하는 것을 무엇이라 하나요?

정답: 중단점
중단점(breakpoint)은 프로그램 실행이 멈출 위치를 지정하는 것이에요.
객관식

현재 실행 중인 함수에서 빠져나와 호출한 곳으로 돌아가는 명령은?

  1. Step Over
  2. Step Into
  3. Step Out
  4. Restart
정답: 3
Step Out(Shift+F11)은 현재 함수의 나머지를 다 실행하고 호출한 곳으로 돌아가요.
OX 퀴즈

printf 디버깅은 멀티스레드 프로그램에서도 항상 정확한 실행 순서를 보여준다.

정답: ❌ X
멀티스레드 환경에서 printf 출력은 버퍼링과 스레드 스케줄링에 의해 실제 실행 순서와 다르게 나타날 수 있습니다.

[C 코드] 디버깅 — 메모리 검사기
1#include <stdio.h>
2#include <stdlib.h>
3#include <string.h>
4
5// === 메모리 검사기 (Sanitizer 원리) ===
6#define MAX_ALLOCS 100
7#define GUARD_BYTE 0xCD
8#define FREE_BYTE 0xDD
9
10typedef struct {
11 void *ptr;
12 size_t size;
13 const char *file;
14 int line;
15 int freed;
16} AllocInfo;
17
18static AllocInfo allocs[MAX_ALLOCS];
19static int nallocs = 0;
20
21void* checked_malloc(size_t size, const char *file, int line) {
22 // 가드 바이트 추가 (앞뒤 8바이트)
23 size_t total = size + 16;
24 unsigned char *raw = (unsigned char*)malloc(total);
25 if (!raw) return NULL;
26
27 memset(raw, GUARD_BYTE, 8); // 앞 가드
28 memset(raw+8+size, GUARD_BYTE, 8); // 뒤 가드
29 memset(raw+8, 0, size); // 사용 영역 초기화
30
31 allocs[nallocs++] = (AllocInfo){raw, size, file, line, 0};
32 printf(" [ALLOC] %zu bytes at %s:%d\n", size, file, line);
33 return raw + 8;
34}
35
36void checked_free(void *ptr, const char *file, int line) {
37 if (!ptr) return;
38 unsigned char *raw = (unsigned char*)ptr - 8;
39
40 // 할당 정보 찾기
41 int found = -1;
42 for (int i = 0; i < nallocs; i++)
43 if (allocs[i].ptr == raw) { found = i; break; }
44
45 if (found < 0) {
46 printf(" ⚠️ [FREE] Unknown pointer at %s:%d!\n", file, line);
47 return;
48 }
49 if (allocs[found].freed) {
50 printf(" ⚠️ [DOUBLE FREE] at %s:%d (originally at %s:%d)!\n",
51 file, line, allocs[found].file, allocs[found].line);
52 return;
53 }
54
55 // 가드 바이트 검사
56 size_t size = allocs[found].size;
57 int front_ok = 1, back_ok = 1;
58 for (int i = 0; i < 8; i++) {
59 if (raw[i] != GUARD_BYTE) front_ok = 0;
60 if (raw[8+size+i] != GUARD_BYTE) back_ok = 0;
61 }
62 if (!front_ok) printf(" ⚠️ [UNDERFLOW] Buffer underflow detected at %s:%d!\n", file, line);
63 if (!back_ok) printf(" ⚠️ [OVERFLOW] Buffer overflow detected at %s:%d!\n", file, line);
64
65 memset(raw+8, FREE_BYTE, size); // use-after-free 감지
66 allocs[found].freed = 1;
67 printf(" [FREE] %zu bytes at %s:%d\n", size, file, line);
68 free(raw);
69}

78 leaks++;

80 }

[심화 내용] 메모리 검사 원리
가드 바이트:
할당 앞뒤에 특수 패턴(0xCD) 배치
free 시 검사 → 변경됐으면 오버플로!
→ AddressSanitizer의 "red zone" 원리

해제 후 접근(Use-After-Free) 감지:
free 시 0xDD로 채움
→ 접근하면 0xDD 패턴 발견 = 버그!

더블 프리 감지:
freed 플래그로 추적
이미 freed면 경고!

실제 도구 비교:
Valgrind: 바이너리 인스트루멘테이션 (느림)
ASAN: 컴파일러 삽입 (2배 느림)
직접 구현: 교육용 (기능 제한)
red zone
레드 존. 할당 영역 앞뒤 여분 바이트. 접근하면 오류! AddressSanitizer의 핵심 메커니즘.
use-after-free
해제 후 사용. 보안 취약점 1위급. free 후에도 포인터 사용→정의되지 않은 동작! 해제 후 NULL 대입.
가드 바이트 0xCD! free할 때 확인. 바뀌었으면 경계 넘은 것! ASAN도 같은 원리!
core dump
코어 덤프. 크래시 시 메모리 스냅샷. gdb core로 분석. ulimit -c unlimited로 활성화!
breakpoint mechanism
중단점 메커니즘. INT3(0xCC) 삽입→트랩→디버거 활성화. 소프트/하드웨어 중단점!
DWARF debug info
DWARF 디버그 정보. -g 옵션. 변수 이름, 줄 번호, 타입 정보. ELF .debug_info 섹션!
sanitizer family
새너타이저 가족. ASAN(메모리), MSAN(초기화), TSAN(스레드), UBSAN(UB). 전부 -fsanitize=!
중단점 원리: 코드에 INT3(0xCC) 삽입! CPU가 트랩→OS→디버거 활성화. 하드웨어 지원!
[심화] 디버깅 체크리스트
버그 분류: 1. 컴파일 에러 → 문법 확인, 헤더 포함 2. 링크 에러 → 함수 정의 누락, 라이브러리 3. 런타임 크래시 → ASAN, GDB backtrace 4. 논리 버그 → printf/로그, 단위 테스트 5. 메모리 버그 → Valgrind, MSAN 6. 성능 문제 → perf, gprof 5 Whys 기법: 왜 크래시? → 널 포인터 왜 널? → 함수 반환값 미확인 왜 미확인? → 에러 처리 누락 왜 누락? → 코딩 표준 없음 왜 없음? → 도입 예정 → 근본 원인 발견! 고무 오리 디버깅: 오리에게 코드를 한 줄씩 설명 → 설명하다 보면 버그 발견! → 진짜 효과 있음 (코드 리뷰 원리)
rubber duck debugging
고무 오리 디버깅. 코드를 소리 내어 설명하면 버그 발견! 코드 리뷰의 셀프 버전. 진짜 효과적!
perf profiler
perf. Linux 성능 프로파일러. perf record+perf report. CPU 핫스팟 찾기! 캐시 미스 분석.
gprof
gprof. 함수별 실행 시간 측정. gcc -pg→실행→gprof. 콜 그래프 생성! 기본 프로파일러.
perf: CPU 핫스팟을 찾아라! perf record ./a.out → perf report. 어디서 시간을 쓰는지 한눈에!
ltrace
ltrace. 라이브러리 함수 호출 추적. printf, malloc, strlen 호출 확인! strace(시스템콜)와 쌍!
coredump analysis
코어 덤프 분석. ulimit -c unlimited→크래시→gdb core. bt로 콜 스택! 프로덕션 디버깅 필수.
conditional breakpoint
조건부 중단점. GDB: break main.c:42 if x>100. 특정 조건에서만 멈춤! 반복문 디버깅 필수.
reverse debugging
역방향 디버깅. GDB: reverse-step. 시간을 되돌려 실행! rr(Record and Replay)로 결정적 재현.
rr: 실행을 녹화→정확히 재현→역방향 스텝! 비결정적 버그(레이스 컨디션)도 잡는다!
debug symbol strip
디버그 심볼 제거. strip a.out. 릴리즈 바이너리 크기 50%↓. 심볼은 별도 .debug 파일로!
watchpoint
감시점. GDB: watch x. x 값 변경 시 자동 멈춤! 하드웨어 지원(DR0~DR3). 메모리 변조 추적!
printf debugging
printf 디버깅. 가장 원시적이지만 가장 많이 씀! __FILE__, __LINE__, __func__ 매크로 활용.
watch x: x가 바뀌면 자동 중단! 누가 변수를 건드리는지 추적. 하드웨어 감시점 최대 4개!

디버거(debugger)
프로그램의 실행을 제어하면서 버그를 찾는 도구. 중단점, 스텝 실행, 변수 감시 기능을 제공한다.
중단점(breakpoint)
프로그램 실행이 멈출 위치를 지정하는 표시. 해당 줄에 도달하면 실행이 일시 중지된다.
호출 스택(call stack)
현재 실행 중인 함수들의 호출 관계를 보여주는 목록. 어떤 함수가 어떤 함수를 호출했는지 추적할 수 있다.
변수 감시(watch)
특정 변수나 식의 값을 디버거에서 지속적으로 추적하는 기능.
이번 유닛 핵심 정리
1
디버거는 코드를 수정하지 않고 프로그램을 분석하는 도구이다
2
중단점을 설정하면 원하는 위치에서 실행을 멈출 수 있다
3
Step Over는 다음 줄, Step Into는 함수 안으로, Step Out은 함수 밖으로
4
Watch로 변수 값을 추적하고, Call Stack으로 호출 관계를 본다
5
디버거 실력은 연습할수록 느는 실전 기술이다
