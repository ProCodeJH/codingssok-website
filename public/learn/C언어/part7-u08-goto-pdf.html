goto문이란?

goto는 C언어에서 가장 논란이 많은 명령어야. 프로그램의 실행 흐름을 원하는 곳으로 점프시켜 버리거든. 왜 있는데 쓰지 말라는 건지, 그리고 언제는 써도 되는지 알아보자!
goto_basic.c
1#include <stdio.h>
2
3int main(void) {
4 printf("1번 줄\n");
5 goto skip; /* skip 라벨로 점프! */
6 printf("2번 줄\n"); /* 이건 실행 안 됨 */
7 printf("3번 줄\n"); /* 이것도 실행 안 됨 */
8skip: /* 라벨: 점프 목적지 */
9 printf("4번 줄\n"); /* 여기부터 실행 */
10 return 0;
11}
실행 결과
1번 줄 4번 줄
5줄goto skip; -- skip이라는 라벨이 있는 곳으로 즉시 점프해요.
8줄skip: -- 라벨이에요. goto의 목적지를 표시해요. 이름 뒤에 콜론(:)을 붙여요.

왜 goto를 쓰지 말라고 하는가?

goto를 남용하면 코드의 흐름이 이리저리 뛰어다녀서 스파게티 코드가 돼요. 읽기도 어렵고, 버그를 찾기도 매우 힘들어져요.

undefined
1968년 다이크스트라(Dijkstra)라는 유명한 컴퓨터 과학자가 'goto문은 해롭다'라는 편지를 썼어. 그 이후로 goto 대신 if, for, while 같은 구조적 프로그래밍이 표준이 됐지.

goto가 허용되는 경우 1: 중첩 루프 탈출

break는 가장 안쪽 반복문 하나만 빠져나와요. 중첩된 반복문을 한 번에 빠져나오려면 goto가 유용해요.

goto_nested_loop.c
1#include <stdio.h>
2
3int main(void) {
4 int target = 7;
5
6 for (int i = 0; i < 10; i++) {
7 for (int j = 0; j < 10; j++) {
8 if (i * 10 + j == target) {
9 printf("찾았다! i=%d, j=%d\n", i, j);
10 goto found; /* 중첩 루프 한번에 탈출 */
11 }
12 }
13 }
14 printf("못 찾았다\n");
15
16found:
17 printf("탐색 종료\n");
18 return 0;
19}
실행 결과
찾았다! i=0, j=7 탐색 종료

goto가 허용되는 경우 2: 리소스 정리 (cleanup 패턴)

메모리를 여러 번 할당할 때, 중간에 실패하면 이미 할당한 메모리를 다 해제해야 해요. goto cleanup 패턴을 쓰면 깔끔하게 처리할 수 있어요.

goto_cleanup.c
1#include <stdio.h>
2#include <stdlib.h>
3
4int processData(void) {
5 int *data1 = NULL;
6 int *data2 = NULL;
7 int *data3 = NULL;
8
9 data1 = (int *)malloc(100 * sizeof(int));
10 if (data1 == NULL) goto cleanup;
11
12 data2 = (int *)malloc(200 * sizeof(int));
13 if (data2 == NULL) goto cleanup;
14
15 data3 = (int *)malloc(300 * sizeof(int));
16 if (data3 == NULL) goto cleanup;
17
18 /* 정상 처리 */
19 printf("모든 메모리 할당 성공!\n");
20 data1[0] = 42;
21 printf("data1[0] = %d\n", data1[0]);
22
23cleanup:
24 free(data3); /* NULL이면 free가 아무 것도 안 함 */
25 free(data2);
26 free(data1);
27 return 0;
28}

Linux 커널 코드에서도 goto cleanup 패턴을 많이 써. 이 패턴만큼은 goto가 if-else 중첩보다 훨씬 깔끔해!
goto 사용 규칙
1. 금지 -- 일반적인 흐름 제어에 goto 사용 금지
2. 허용 1 -- 중첩 반복문 한 번에 빠져나올 때
3. 허용 2 -- 에러 발생 시 리소스 정리(cleanup 패턴)
4. 방향 -- goto는 항상 아래쪽(앞쪽)으로만 점프할 것
5. 범위 -- goto는 같은 함수 안에서만 사용 가능
[코딩기초] 코드 품질과 goto
좋은 코드의 조건은 읽기 쉽고, 유지보수가 쉬운 것이에요. goto를 쓰면 코드의 흐름을 한눈에 파악하기 어려워져요. goto 없이도 해결 가능하면 if/for/while을 쓰세요.

goto문은 함수 밖으로 점프할 수 있다.

정답: ❌ X
goto문은 같은 함수 내에서만 점프할 수 있습니다. 함수 간 점프는 setjmp/longjmp를 사용해야 합니다.

[C 코드] goto — 에러 처리 패턴
1#include <stdio.h>
2#include <stdlib.h>
3#include <string.h>
4
5// goto가 유용한 유일한 패턴: 에러 처리 정리
6typedef struct { char *name; int *data; FILE *log; } Resource;
7
8int process_file(const char *filename) {
9 Resource res = {NULL, NULL, NULL};
10 int result = -1;
11
12 // 자원 할당
13 res.name = (char*)malloc(256);
14 if (!res.name) { printf("malloc name failed\n"); goto cleanup; }
15
16 res.data = (int*)malloc(1000 * sizeof(int));
17 if (!res.data) { printf("malloc data failed\n"); goto cleanup; }
18
19 // 시뮬레이션: 파일 열기 실패
20 if (strcmp(filename, "bad.txt") == 0) {
21 printf("file open failed\n");
22 goto cleanup;
23 }
24
25 // 정상 처리

29 result = 0; // 성공!
30
31cleanup:
32 // 모든 자원 정리 (할당된 것만)
33 if (res.name) free(res.name);
34 if (res.data) free(res.data);
35 if (res.log) fclose(res.log);
36 return result;
37}

3가지 패턴:
1. 단순 cleanup: 자원 해제 (가장 흔함)
2. 에러 체인: 역순 정리 (err3→err2→err1)
3. 상태 머신: 라벨 = 상태 (드물게 사용)

goto 대안들:
1. do-while(0) + break → 매크로에서 흔함
2. 중첩 if → 간단할 때만
3. 별도 함수로 분리 → 복잡할 때
4. C++: RAII (소멸자 자동 호출)
RAII alternative

1#include <stdio.h>

6// === setjmp/longjmp로 예외 처리 ===
7static jmp_buf exception_buf;
8static const char *exception_msg = NULL;
9
10// TRY/CATCH 매크로
11#define TRY if (setjmp(exception_buf) == 0)
12#define CATCH else
13#define THROW(msg) do { exception_msg = msg; longjmp(exception_buf, 1); } while(0)
14
15void risky_function(int x) {
16 printf(" risky_function(%d)\n", x);
17 if (x == 0) THROW("Division by zero!");
18 if (x < 0) THROW("Negative number!");
19 printf(" Result: %d\n", 100 / x);
20}

22// === 에러 코드 체계 ===
23typedef enum {
24 ERR_OK = 0,
25 ERR_NULL_PTR = -1,
26 ERR_OUT_OF_MEM = -2,
27 ERR_FILE_NOT_FOUND = -3,
28 ERR_PERMISSION = -4,
29 ERR_INVALID_ARG = -5
30} ErrorCode;
31
32const char* err_str(ErrorCode e) {
33 switch(e) {
34 case ERR_OK: return "Success";
35 case ERR_NULL_PTR: return "Null pointer";
36 case ERR_OUT_OF_MEM: return "Out of memory";
37 case ERR_FILE_NOT_FOUND: return "File not found";
38 case ERR_PERMISSION: return "Permission denied";
39 case ERR_INVALID_ARG: return "Invalid argument";
40 default: return "Unknown error";
41 }
42}

44// 에러 체인 (errno 스타일)
45typedef struct {
46 ErrorCode code;
47 char msg[100];
48 const char *file;
49 int line;
50} Error;
51
52static Error last_error = {ERR_OK, "", NULL, 0};
53
54void set_error(ErrorCode code, const char *file, int line, const char *fmt, ...) {
55 last_error.code = code;
56 last_error.file = file;
57 last_error.line = line;
58 va_list args;
59 va_start(args, fmt);
60 vsnprintf(last_error.msg, 100, fmt, args);
61 va_end(args);
62}

64#define SET_ERR(code, fmt, ...) set_error(code, __FILE__, __LINE__, fmt, ##__VA_ARGS__)
65
66ErrorCode open_config(const char *path) {
67 if (!path) { SET_ERR(ERR_NULL_PTR, "path is NULL"); return ERR_NULL_PTR; }
68 if (strcmp(path, "/etc/config") != 0) {
69 SET_ERR(ERR_FILE_NOT_FOUND, "Cannot find %s", path);
70 return ERR_FILE_NOT_FOUND;
71 }
72 return ERR_OK;
73}

75void print_last_error() {
76 if (last_error.code != ERR_OK)
77 printf(" Error [%d] %s: %s at %s:%d\n",
78 last_error.code, err_str(last_error.code),
79 last_error.msg, last_error.file, last_error.line);
80}

[심화 내용] C의 에러 처리 비교
5가지 에러 처리 방법:

1. 반환 코드: return -1; (가장 일반적)
2. errno: 전역 변수 (POSIX 표준)
3. goto cleanup: 자원 정리 패턴
4. setjmp/longjmp: C의 의사 예외처리
5. 콜백: on_error(handler)

setjmp/longjmp 주의:
• 자원 정리가 안 됨! (malloc한 거 leak)
• 시그널 안전하지 않음
• 중첩 불가 (전역 jmp_buf 1개)
→ C++의 try/catch가 이 문제를 해결

실전 추천:
간단한 함수: 반환 코드
복잡한 초기화: goto cleanup
라이브러리: errno 스타일
setjmp/longjmp detail
setjmp: 현재 상태 저장(레지스터, SP). longjmp: 저장된 상태로 복귀. 스택 해제 없음! 자원 누수 주의.
errno
전역 에러 번호. . ENOENT(파일없음), ENOMEM(메모리부족). strerror(errno)로 메시지. 스레드별 저장!
TRY/CATCH를 C에서?! setjmp/longjmp 매크로로 가능. 하지만 자원 정리 안 됨 주의!
에러 구조체: code+msg+file+line. 어디서 무슨 에러인지 완벽 추적! 디버깅이 쉬워져!
structured programming
구조적 프로그래밍. Dijkstra "Go To Considered Harmful"(1968). if/while/for로 충분!
cleanup attribute
GCC cleanup 속성. __attribute__((cleanup(fn))). 스코프 종료 시 자동 호출. C의 RAII!
computed goto
계산된 goto. GCC 확장. void *labels[]={&&L1,&&L2}; goto *labels[i]; VM 디스패치에 사용!
error handling patterns
에러 처리 패턴 5가지. 반환코드/errno/goto cleanup/setjmp/콜백. 상황에 맞게 선택!
computed goto: goto *labels[opcode]. VM에서 switch보다 20% 빠름! CPython이 사용!
[심화] 에러 처리 패턴 비교표
방법 | 장점 | 단점 반환코드 | 간단, 빠름 | 무시 가능 errno | 표준, 호환 | 전역 상태 goto cleanup | 깔끔한 정리 | goto 알레르기 setjmp | try/catch 흉내 | 자원 누수 콜백 | 유연 | 복잡 Linux 커널 스타일: ret = do_something(); if (ret) goto err_cleanup; → 일관된 패턴으로 수천 개 함수 관리! 에러 처리 황금 규칙: 1. 에러는 가능한 빨리 처리 2. 자원은 할당 역순으로 해제 3. 에러 경로도 테스트 4. 에러 메시지는 구체적으로 5. 에러 전파는 일관되게
error propagation

longjmp stack
longjmp 스택 주의. longjmp는 스택을 되감지 않음! 지역 변수 소멸자 호출 안 됨. C++에서 위험!

error code convention
에러 코드 관례. 0=성공, 음수=에러. errno.h 표준. ENOENT=2(파일없음), ENOMEM=12(메모리부족).
Dijkstra goto harmful
Dijkstra 1968. "Go To Statement Considered Harmful". 구조적 프로그래밍 혁명! goto 논쟁의 시작.
scope and goto
goto와 스코프. goto로 변수 초기화를 건너뛰면 UB! C++에서는 컴파일 에러. C에서는 경고만.
Dijkstra 1968: "goto 해롭다!" 그런데 Linus: "cleanup goto는 좋다!" 둘 다 맞는 말!
tail call optimization
꼬리 호출 최적화. 재귀→goto(루프)로 변환. 스택 오버플로 방지! GCC -O2에서 자동.
Linux goto usage
Linux 커널 goto 사용량. 10만 개 이상! 전부 cleanup 패턴. goto는 도구일 뿐, 용도가 핵심!
RAII in C
C의 RAII. GCC __attribute__((cleanup)). 스코프 종료→자동 해제. C++의 소멸자를 C에서! 비표준.
Linux 커널: goto 10만 개! 전부 cleanup용. "goto는 나쁘다"가 아니라 "bad goto는 나쁘다"!
