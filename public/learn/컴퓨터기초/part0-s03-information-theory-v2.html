UNIT S-03 정보 이론

정보란 뭘까?

'내일 해가 뜬다'는 당연한 말이에요. 새로운 정보가 거의 없어요. 하지만 '내일 서울에 눈이 3미터 쌓인다'는 엄청 놀라운 소식이에요. 일어날 확률이 낮은 일일수록, 그 일이 실제로 일어났을 때 전달되는 정보의 양이 많아요.

[핵심] 정보량의 직관 확률이 낮은 사건 = 놀라운 소식 = 정보가 많다
확률이 높은 사건 = 당연한 소식 = 정보가 적다

예시:
• 동전 앞면 (확률 50%) → 약간 놀라움 → 정보 보통
• 주사위 1 (확률 약 17%) → 꽤 놀라움 → 정보 많음
• 로또 1등 (확률 0.000012%) → 엄청 놀라움 → 정보 아주 많음

정보의 단위 — 비트

정보의 양을 숫자로 표현할 수 있을까요? 네! 단위는 우리가 잘 아는 비트(bit)예요.

[핵심] 1비트 = 예/아니오 1번 1비트 = 두 가지 중 하나를 알려주는 정보량

동전 던지기: 앞 또는 뒤 → 결과를 알려면 1비트
4지선다 문제: 정답을 알려면 2비트 (00, 01, 10, 11)
8가지 중 하나: 3비트 (000 ~ 111)

공식: n가지 중 하나 → log₂(n) 비트

2가지 → log₂(2) = 1비트
4가지 → log₂(4) = 2비트
8가지 → log₂(8) = 3비트
256가지 → log₂(256) = 8비트 = 1바이트
[팁] log₂를 쉽게 계산하는 법 log₂(n) = '2를 몇 번 곱해야 n이 되나?'

2¹ = 2 → log₂(2) = 1
2² = 4 → log₂(4) = 2
2³ = 8 → log₂(8) = 3
2⁴ = 16 → log₂(16) = 4
2¹⁰ = 1024 → log₂(1024) = 10

2의 거듭제곱이 아닌 수:
log₂(6) ≈ 2.58 → 6가지를 구별하려면 약 2.58비트 필요
(실제로는 3비트를 써야 해요. 비트는 정수니까요)

스무고개로 이해하는 정보

스무고개에서 질문 하나의 답은 '예' 또는 '아니오' → 1비트예요. 가능한 답이 1024개면 최소 10번(10비트)이면 찾을 수 있어요. 매 질문으로 가능성을 절반씩 줄이면 되니까요.

[예시] 1~16 중 숫자 맞히기 상대가 1~16 중 하나를 골랐어요. 최소 몇 번 물어야 할까요?

질문 1: '8보다 크니?' → 절반 제거 (16→8개)
질문 2: '12보다 크니?' 또는 '4보다 크니?' → (8→4개)
질문 3: → (4→2개)
질문 4: → (2→1개) → 정답!

16가지 → 4번 = log₂(16) = 4비트

매번 가능성을 정확히 반으로 나누는 게 최적의 전략이에요.
이것이 이진 탐색(Binary Search)의 원리이기도 해요!

확률과 정보량

모든 경우가 동일한 확률이 아닐 때는 어떨까요? 확률에 따라 정보량이 달라져요.

[핵심] 정보량 공식 사건의 확률이 P일 때, 정보량 I는:

I = log₂(1/P) 비트

예시:
• 동전 앞면 (P=1/2): I = log₂(2) = 1비트
• 주사위 1 (P=1/6): I = log₂(6) ≈ 2.58비트
• 52장 카드에서 스페이드A (P=1/52): I = log₂(52) ≈ 5.7비트

확률이 낮을수록 → 1/P가 커지고 → 정보량이 커져요.

엔트로피 — 평균 정보량

여러 사건이 있을 때, 각 사건의 정보량에 확률을 곱해서 평균을 내면 엔트로피(Entropy)가 돼요. '이 시스템이 평균적으로 얼마나 많은 정보를 전달하는가'를 나타내요.

[핵심] 엔트로피란? 엔트로피 H = 모든 사건의 (확률 x 정보량)을 합한 것

예: 공정한 동전
앞면: P=0.5, 정보량=1비트
뒷면: P=0.5, 정보량=1비트
H = 0.5×1 + 0.5×1 = 1비트

예: 편향된 동전 (앞면 90%)
앞면: P=0.9, 정보량 = log₂(1/0.9) ≈ 0.15비트
뒷면: P=0.1, 정보량 = log₂(1/0.1) ≈ 3.32비트
H = 0.9×0.15 + 0.1×3.32 ≈ 0.47비트

편향된 동전은 결과를 거의 예측할 수 있어서 엔트로피가 낮아요.
[팁] 엔트로피의 의미 엔트로피가 높다 = 결과를 예측하기 어렵다 = 불확실하다
엔트로피가 낮다 = 결과를 예측하기 쉽다 = 거의 확실하다
엔트로피가 0 = 결과가 100% 확정 = 정보 없음

동전: H=1비트 (최대 불확실)
편향 동전: H≈0.47비트 (좀 예측 가능)
양면 같은 동전: H=0비트 (완전 확실, 던질 필요 없음)

엔트로피가 압축의 한계

엔트로피는 데이터 압축에서 아주 중요한 의미가 있어요. 엔트로피 = 데이터를 줄일 수 있는 이론적 한계거든요.

[핵심] 샤넌의 소스 코딩 정리 클로드 섀넌이 증명한 정리:

'데이터를 엔트로피보다 더 작게 압축할 수는 없다'

예: 편향 동전의 엔트로피 = 0.47비트
→ 동전을 100번 던진 결과를 저장하려면
→ 최소 47비트가 필요 (100비트가 아니라!)

90%가 앞면이니까 패턴이 있고, 패턴이 있으면 줄일 수 있어요.
하지만 47비트 이하로는 절대 줄일 수 없어요. 이게 물리적 한계예요.

1. 정보원 (Source)
메시지를 생성합니다. 예: 사람이 말을 하거나, 컴퓨터가 파일을 전송합니다.

고정 길이 코드 vs 가변 길이 코드

[핵심] 고정 길이 vs 가변 길이 고정 길이 코드: 모든 문자에 같은 비트 수
예: ASCII — 모든 문자가 7비트 (A=1000001, B=1000010)
장점: 단순함. 단점: 비효율적.

가변 길이 코드: 자주 나오는 문자는 짧게, 드문 문자는 길게
예: 모스 부호 — E는 · (짧음), Q는 --·- (김)
장점: 효율적. 단점: 복잡함.

일상에서도 같은 원리: 자주 쓰는 '가'는 1글자, 드문 '뷁'도 1글자지만,
자판 위치를 보면 자주 쓰는 키가 편한 곳에 있죠!
[예시] 고정 vs 가변의 차이 문자 4개: A(빈도 50%), B(25%), C(15%), D(10%)

고정 길이 (2비트씩):
A=00, B=01, C=10, D=11
평균: 2비트/문자

가변 길이:
A=0 (1비트), B=10 (2비트), C=110 (3비트), D=111 (3비트)
평균: 0.5×1 + 0.25×2 + 0.15×3 + 0.10×3
= 0.5 + 0.5 + 0.45 + 0.3 = 1.75비트/문자

가변 길이가 12.5% 절약! 데이터가 클수록 차이가 커져요.

접두어 코드 — 헷갈리지 않는 규칙

가변 길이 코드에는 중요한 규칙이 있어요. 어떤 코드가 다른 코드의 앞부분이면 안 돼요. 이걸 접두어 코드(Prefix Code)라고 해요.

[주의] 접두어 조건을 어기면? 나쁜 코드: A=0, B=01, C=1
'01'이 들어오면? A(0)+C(1)인지 B(01)인지 구별 불가!

좋은 코드 (접두어 코드): A=0, B=10, C=11
A의 코드(0)는 다른 코드의 앞부분이 아니에요.
B(10)도, C(11)도 마찬가지.

'01011' → 0 | 10 | 11 → A B C ← 유일하게 해석돼요!

규칙: 어떤 코드도 다른 코드의 시작 부분이 되면 안 돼요.

허프만 코딩 — 최적의 압축 코드 만들기

허프만 코딩(Huffman Coding)은 1952년 MIT 학생 데이비드 허프만이 과제로 만든 알고리즘이에요. 빈도가 높은 문자에 짧은 코드를, 빈도가 낮은 문자에 긴 코드를 자동으로 배정해요.

[핵심] 허프만 코딩 방법 단계:
1. 모든 문자를 빈도순으로 나열해요.
2. 빈도가 가장 낮은 2개를 골라요.
3. 두 개를 하나로 합쳐요 (빈도 합산).
4. 합친 노드에서 왼쪽=0, 오른쪽=1을 붙여요.
5. 노드가 1개 남을 때까지 2~4를 반복해요.
6. 루트에서 각 문자까지의 경로가 코드예요.

2단계: 가장 작은 2개 합치기

3단계: 다시 정렬 후 합치기

4단계: 계속 합치기

RLE — 가장 간단한 압축

허프만 코딩 말고도 간단한 압축 방법이 있어요. RLE(Run-Length Encoding)는 같은 값이 연속되면 '값+반복횟수'로 바꿔요.

[핵심] RLE 압축 원본: AAAAABBBCCDDDDDDDD
RLE: A5B3C2D8

C허프만 코딩 시뮬레이션
1import heapq
2from collections import Counter
3
4def huffman_coding(text):
5 # 1. 빈도 세기
6 freq = Counter(text)
7 # 2. 힙(우선순위 큐)에 넣기
8 heap = [[count, [char, ""]] for char, count in freq.items()]
9 heapq.heapify(heap)
10 # 3. 트리 만들기
11 while len(heap) > 1:
12 lo = heapq.heappop(heap)
13 hi = heapq.heappop(heap)
14 for pair in lo[1:]:
15 pair[1] = '0' + pair[1]
16 for pair in hi[1:]:
17 pair[1] = '1' + pair[1]
18 heapq.heappush(heap, [lo[0] + hi[0]] + lo[1:] + hi[1:])
19 # 4. 결과 출력
20 result = sorted(heap[0][1:], key=lambda p: (len(p[1]), p))
21 print(f"원본: {len(text) * 8}비트 (ASCII)")
22 total = 0
23 for char, code in result:
24 count = freq[char]
25 total += count * len(code)
26 print(f" '{char}': {code} ({len(code)}비트, {count}회)")
27 print(f"압축: {total}비트")
28 print(f"절약: {100 - total * 100 // (len(text) * 8)}%")
29
30huffman_coding("AAAAABBBCCDDDDDDDD")

역사 — 정보 이론의 탄생

[역사] 클로드 섀넌 — 정보 이론의 아버지 클로드 섀넌(1916~2001)은 1948년에 '통신의 수학적 이론'이라는 논문을 발표했어요. 이 논문에서 '정보를 수학적으로 측정할 수 있다'는 혁명적인 아이디어를 제시했어요. '비트'라는 단위도 이 논문에서 처음 사용됐어요. 전화, 인터넷, Wi-Fi, 블루투스 등 모든 통신 기술의 이론적 기반이 이 논문 한 편이에요.
[역사] 데이비드 허프만 — 과제가 세상을 바꾸다 데이비드 허프만(1925~1999)은 MIT 대학원생일 때, 교수가 낸 과제를 풀다가 허프만 코딩을 발명했어요(1952). 교수는 '최적 코드를 찾으라'는 과제를 냈는데, 허프만은 수학적으로 증명된 최적 알고리즘을 만들어 버렸어요. ZIP, PNG, JPEG, MP3 등 거의 모든 압축 형식의 내부에 허프만 코딩이 들어 있어요.

용어 정리

용어설명
정보량 (Information)사건이 전달하는 놀라움의 정도. 확률이 낮을수록 정보량이 커요. 단위는 비트.
엔트로피 (Entropy)평균 정보량. 시스템의 불확실성을 나타내요. 엔트로피가 높을수록 예측하기 어려워요.
허프만 코딩 (Huffman Coding)빈도에 따라 가변 길이 코드를 배정하는 최적 압축 알고리즘.
접두어 코드 (Prefix Code)어떤 코드도 다른 코드의 앞부분이 되지 않는 코드. 구분 기호 없이도 해석 가능해요.
RLE (Run-Length Encoding)연속 반복을 '값+횟수'로 바꾸는 간단한 압축 방법.
무손실 압축원본을 100% 복원할 수 있는 압축. ZIP, PNG 등.
손실 압축사람이 인지하기 어려운 부분을 삭제하는 압축. JPEG, MP3 등.
샤넌의 소스 코딩 정리데이터를 엔트로피보다 더 작게 압축할 수 없다는 이론적 한계.
이번 유닛 핵심 정리
1. 정보량 = 놀라움의 정도. 확률이 낮을수록 정보가 많다
2. 1비트 = '예/아니오' 하나. n가지 중 하나 → log₂(n) 비트
3. 엔트로피 = 평균 정보량 = 압축의 이론적 한계
4. 허프만 코딩: 자주 나오는 건 짧게, 드문 건 길게 → 최적 압축
5. 접두어 코드: 코드끼리 헷갈리지 않는 규칙
6. 무손실(ZIP/PNG) vs 손실(JPEG/MP3) 압축
이 유닛에서 연결된 축 [수학연결] 로그(log₂), 확률, 기댓값 — 정보 이론의 수학적 기초. 중학교 수학과 직접 연결돼요.
