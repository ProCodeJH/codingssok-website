확률이란 뭘까?

확률은 어떤 일이 일어날 가능성을 숫자로 나타낸 거예요. 0이면 절대 안 일어나고, 1이면 반드시 일어나요.

[핵심] 확률의 기본 확률 = 원하는 경우의 수 ÷ 전체 경우의 수

• 동전 앞면: 1/2 = 0.5 = 50%
• 주사위 6: 1/6 ≈ 0.167 = 약 16.7%
• 52장 카드에서 하트: 13/52 = 1/4 = 25%

확률의 범위: 0 ≤ P ≤ 1
모든 경우의 확률을 더하면 항상 1이에요.

직관은 왜 틀릴까?

우리 뇌는 확률을 직관적으로 판단하는 데 약해요. 진화 과정에서 정확한 확률 계산보다는 빠른 판단이 생존에 유리했거든요. 그래서 확률 문제에서 '느낌'으로 답하면 틀리는 경우가 많아요.

몬티 홀 문제 — 문을 바꿀까, 말까?

미국의 TV 퀴즈쇼 'Let's Make a Deal'에서 유래한 문제예요. 사회자 이름이 몬티 홀이라서 몬티 홀 문제라고 불러요.

[핵심] 몬티 홀 문제 문이 3개 있어요. 하나 뒤에는 자동차, 나머지 둘 뒤에는 염소가 있어요.

1. 당신이 문 하나를 골라요 (예: 1번 문)
2. 사회자가 나머지 중 염소가 있는 문을 하나 열어요 (예: 3번 문)
3. 사회자가 물어요: '선택을 바꾸시겠습니까?'

문을 바꾸는 게 유리할까요, 안 바꾸는 게 유리할까요?

대부분의 사람들은 '바꾸나 안 바꾸나 50:50 아냐?'라고 생각해요. 남은 문이 2개니까요. 하지만 정답은 다릅니다!

[정답] 바꾸는 게 2배 유리해요! 안 바꾸면: 당첨 확률 = 1/3 (약 33%)
바꾸면: 당첨 확률 = 2/3 (약 67%)

바꾸는 게 2배나 유리해요!

몬티 홀 — 왜 바꾸는 게 유리할까?

이해가 안 되는 게 정상이에요. 천천히 경우의 수를 따져 볼게요.

[풀이] 경우의 수로 따져보기 자동차가 1번 문 뒤에 있다고 가정하고, 내가 1번을 골랐을 때:

경우 1: 자동차 = 1번 (내 선택 = 1번)
사회자: 2번 또는 3번 열기 → 바꾸면 꽝, 안 바꾸면 당첨

경우 2: 자동차 = 2번 (내 선택 = 1번)
사회자: 반드시 3번 열기 → 바꾸면 2번 = 당첨!

경우 3: 자동차 = 3번 (내 선택 = 1번)
사회자: 반드시 2번 열기 → 바꾸면 3번 = 당첨!

3가지 경우 중:
• 안 바꾸면 당첨: 1번만 (1/3)
• 바꾸면 당첨: 2번, 3번 (2/3)

C 코드 — 몬티 홀 시뮬레이션

정말 바꾸는 게 유리한지, C언어로 10만 번 실험해 볼게요!

C몬티 홀 10만 번 시뮬레이션
1#include <stdio.h>
2#include <stdlib.h>
3#include <time.h>
4
5int main(void) {
6 srand(time(NULL));
7 int trials = 100000;
8 int stay_wins = 0;
9 int switch_wins = 0;
10
11 for (int i = 0; i < trials; i++) {
12 int car = rand() % 3; // 자동차 위치 (0, 1, 2)
13 int choice = rand() % 3; // 내 선택
14
15 // 사회자: 내가 안 고르고, 자동차도 아닌 문을 열기
16 int opened;
17 do {
18 opened = rand() % 3;
19 } while (opened == choice || opened == car);
20
21 // 안 바꾸면: 처음 선택 유지
22 if (choice == car) stay_wins++;
23
24 // 바꾸면: 남은 문으로 변경
25 int switched = 3 - choice - opened; // 0+1+2=3
26 if (switched == car) switch_wins++;
27 }
28
29 printf("=== 몬티 홀 시뮬레이션 (%d회) ===\n", trials);
30 printf("안 바꿈: %d승 (%.1f%%)\n",
31 stay_wins, 100.0 * stay_wins / trials);
32 printf("바꿈: %d승 (%.1f%%)\n",
33 switch_wins, 100.0 * switch_wins / trials);
34
35 return 0;
36}
실행 결과
=== 몬티 홀 시뮬레이션 (100000회) === 안 바꿈: 33421승 (33.4%) 바꿈: 66579승 (66.6%)
[코드 해설] • srand(time(NULL)): 매번 다른 난수를 생성하기 위한 씨앗 설정
• rand() % 3: 0, 1, 2 중 랜덤 선택
• 3 - choice - opened: 세 문 번호(0+1+2=3)에서 내 선택과 열린 문을 빼면 나머지 문!
• 10만 번 실험하면 이론값(33.3% vs 66.7%)에 아주 가까워져요

수학으로 증명한 걸 코드로 실험적으로 검증하는 거예요. 이것을 몬테카를로 시뮬레이션이라고 해요!

생일 역설 — 왜 23명일까?

'생일이 같은 쌍이 있을 확률'을 직접 구하기는 어려워요. 대신 모두 다를 확률을 구해서 1에서 빼면 돼요.

[풀이] 모두 생일이 다를 확률 365일 중에서 n명의 생일이 모두 다를 확률:

1번째 사람: 365/365 (아무 날이나 OK)
2번째 사람: 364/365 (1명의 생일을 피해야)
3번째 사람: 363/365 (2명의 생일을 피해야)
...
n번째 사람: (365-n+1)/365

P(모두 다름) = 365/365 × 364/365 × 363/365 × ... × (365-n+1)/365

P(겹치는 쌍 있음) = 1 - P(모두 다름)

생일 역설 — 비밀은 '쌍'의 개수

왜 이렇게 빠르게 확률이 올라갈까요? 비밀은 비교해야 할 쌍의 개수에 있어요.

[핵심] 쌍의 개수가 핵심! n명에서 2명을 뽑는 쌍의 수 = n × (n-1) ÷ 2

• 5명: 5×4÷2 = 10쌍
• 10명: 10×9÷2 = 45쌍
• 23명: 23×22÷2 = 253쌍
• 30명: 30×29÷2 = 435쌍
• 50명: 50×49÷2 = 1225쌍

23명이면 253번이나 비교해요!
각 쌍이 겹칠 확률은 1/365로 낮지만,
253번 비교하면 전체적으로 확률이 50%를 넘게 돼요.
[팁] 사람 수는 선형, 쌍의 수는 제곱! 사람이 2배 늘면 → 쌍은 약 4배 늘어요
사람이 3배 늘면 → 쌍은 약 9배 늘어요

이런 걸 조합적 폭발이라고 해요.
프로그래밍에서 O(n²) 알고리즘이 느린 이유와 같은 원리예요!

생일 역설과 프로그래밍

생일 역설은 단순한 수학 퀴즈가 아니에요. 프로그래밍에서 아주 중요한 문제와 직결돼요.

[핵심] 해시 충돌과 생일 역설 해시 함수: 데이터를 고정 크기 숫자로 변환하는 함수

해시 값이 겹치는 것을 해시 충돌이라고 해요.
이것은 생일 역설과 완전히 같은 문제예요!

해시 값의 범위가 N이면:
약 √N개의 데이터만으로 충돌이 50% 확률로 발생해요.

• 해시가 32비트(약 43억 가지): √43억 ≈ 65,536개면 충돌
• 해시가 64비트: √(2⁶⁴) = 2³² ≈ 43억 개면 충돌
• 해시가 256비트(SHA-256): 충돌에 2¹²⁸개 필요 → 사실상 불가능

이것을 생일 공격(birthday attack)이라고 하고,
암호 해시의 길이를 정할 때 핵심 기준이에요.

C 코드 — 생일 역설 시뮬레이션

C생일 역설 10만 번 시뮬레이션
1#include <stdio.h>
2#include <stdlib.h>
3#include <time.h>
4
5int simulate(int people) {
6 int birthdays[70];
7 for (int i = 0; i < people; i++) {
8 birthdays[i] = rand() % 365;
9 }
10
11 // 겹치는 쌍이 있는지 확인
12 for (int i = 0; i < people; i++) {
13 for (int j = i + 1; j < people; j++) {
14 if (birthdays[i] == birthdays[j]) {
15 return 1; // 겹침 발견!
16 }
17 }
18 }
19 return 0; // 모두 다름
20}
21
22int main(void) {
23 srand(time(NULL));
24 int trials = 100000;

27

34

36 matches += simulate(people);
37 }

41 }

44}

조건부 확률

몬티 홀 문제의 핵심이었던 조건부 확률을 좀 더 자세히 알아볼게요.

[핵심] 조건부 확률이란? 어떤 일(B)이 이미 일어났다는 걸 알 때,
다른 일(A)이 일어날 확률을 조건부 확률이라고 해요.

기호: P(A|B) = 'B가 일어났을 때 A의 확률'

공식: P(A|B) = P(A와 B 동시) ÷ P(B)

예시: 주사위를 던져서 짝수가 나왔어요. 이때 6일 확률은?
P(6|짝수) = P(6이면서 짝수) ÷ P(짝수) = (1/6) ÷ (3/6) = 1/3

전체에서는 1/6이지만, 짝수라는 조건이 붙으면 1/3으로 바뀌어요!
[예시] 조건부 확률 — 카드 뽑기 52장에서 카드 1장을 뽑았는데, 빨간색이라는 정보를 알았어요.
이 카드가 하트일 확률은?

빨간 카드: 하트 13장 + 다이아 13장 = 26장
P(하트|빨간) = 13/26 = 1/2

원래 P(하트) = 13/52 = 1/4이었는데,
'빨간색'이라는 조건 덕분에 1/2로 올라갔어요!

조건(정보)이 주어지면 확률이 바뀌어요.
조건 (B)사건 (A)P(A and B)P(B)P(A|B) = P(A and B)/P(B)
짝수 나옴6이다1/63/61/3
짝수 나옴2 이하이다1/63/61/3
3 이상 나옴짝수이다2/64/62/4 = 1/2
3 이상 나옴5 이상이다2/64/62/4 = 1/2
홀수 나옴소수이다2/63/62/3
4 이하 나옴1이다1/64/61/4

베이즈 정리 맛보기

조건부 확률을 뒤집으면? 그게 바로 베이즈 정리예요. AI의 핵심 원리이기도 해요.

[핵심] 베이즈 정리 — 역방향 추론 'A가 주어졌을 때 B의 확률'을 알면,
'B가 주어졌을 때 A의 확률'을 구할 수 있어요!

P(A|B) = P(B|A) × P(A) ÷ P(B)

쉽게 말하면: 결과를 보고 원인을 추정하는 거예요.
[예시] 비가 올 때 우산 가져온 확률 • 비 올 확률: P(비) = 30%
• 비 올 때 우산 가져올 확률: P(우산|비) = 90%
• 비 안 올 때 우산 가져올 확률: P(우산|맑음) = 20%

Q: 오늘 우산을 가져온 친구가 있어요. 오늘 비가 올 확률은?

P(우산) = 0.3×0.9 + 0.7×0.2 = 0.27 + 0.14 = 0.41

P(비|우산) = P(우산|비) × P(비) ÷ P(우산)
= 0.9 × 0.3 ÷ 0.41 = 0.27 ÷ 0.41 ≈ 65.9%

원래 30%였던 비 확률이, 우산이라는 관측 덕분에 66%로 올라갔어요!
[팁] 베이즈 정리와 AI 스팸 메일 필터는 베이즈 정리를 써요!

• '무료', '당첨' 같은 단어가 있으면 → 스팸일 확률 UP
• '회의', '일정' 같은 단어가 있으면 → 스팸일 확률 DOWN

이메일의 단어들을 보고, 스팸일 확률을 계산하는 거예요.
이것이 나이브 베이즈 분류기의 원리예요.
단계항목계산결과
1P(병)유병률0.001 (0.1%)
2P(건강)1 - P(병)0.999
3P(양성|병)검사 정확도 (민감도)0.99
4P(양성|건강)오진율 (위양성률)0.01
5P(양성,병)P(양성|병) x P(병)0.99 x 0.001 = 0.00099
6P(양성,건강)P(양성|건강) x P(건강)0.01 x 0.999 = 0.00999
7P(양성)P(양성,병) + P(양성,건강)0.00099 + 0.00999 = 0.01098
8P(병|양성)P(양성,병) / P(양성)0.00099 / 0.01098 = 9.0%

기댓값 — 평균적으로 얼마나?

확률과 함께 꼭 알아야 할 개념이 기댓값이에요.

[핵심] 기댓값이란? 기댓값 = 각 결과 × 그 확률의 합

예: 주사위의 기댓값
= 1×(1/6) + 2×(1/6) + 3×(1/6) + 4×(1/6) + 5×(1/6) + 6×(1/6)
= 21/6 = 3.5

주사위를 아주 많이 던지면, 평균이 3.5에 가까워져요.
기댓값은 '오랫동안 반복했을 때의 평균 결과'예요.
[예시] 가챠 게임의 기댓값 모바일 게임에서 뽑기를 해요:
• SSR (확률 1%): 가치 10,000원
• SR (확률 9%): 가치 1,000원
• R (확률 90%): 가치 100원

뽑기 1회 비용: 1,000원

기댓값 = 10000×0.01 + 1000×0.09 + 100×0.90
= 100 + 90 + 90 = 280원

1,000원을 내고 평균 280원어치를 받는 거예요.
장기적으로 손해라는 걸 수학적으로 알 수 있어요!

디버깅 — 확률 코드의 흔한 실수

[디버깅] 실수 1: srand를 반복문 안에 넣기 잘못된 코드:
for (int i = 0; i < 100; i++) {
    srand(time(NULL));  // 매번 같은 씨앗!
    int r = rand() % 6;
}

올바른 코드:
srand(time(NULL));  // 한 번만!
for (int i = 0; i < 100; i++) {
    int r = rand() % 6;
}

srand는 프로그램 시작 시 딱 한 번만 호출해요. 반복문 안에 넣으면 같은 초(second)에서 같은 난수가 나와요.
[디버깅] 실수 2: rand() % N의 편향 문제: RAND_MAX가 32767이고 N=10이면
32768 ÷ 10 = 3276 나머지 8
→ 0~7은 3277번, 8~9는 3276번 나올 수 있어요 (미세한 편향)

간단한 해결:
교육용 시뮬레이션에서는 이 정도 편향은 무시해도 괜찮아요.
정밀한 결과가 필요하면 rejection sampling 사용:
int fair_rand(int n) {
    int limit = RAND_MAX - (RAND_MAX % n);
    int r;
    do { r = rand(); } while (r >= limit);
    return r % n;
}
[디버깅] 실수 3: 정수 나눗셈으로 확률 계산 잘못된 코드:
int wins = 667;
int trials = 1000;
printf("확률: %d%%\n", wins / trials * 100);
// 출력: 확률: 0%  (정수÷정수=0!)

올바른 코드:
printf("확률: %.1f%%\n", 100.0 * wins / trials);
// 출력: 확률: 66.7%

C에서 정수÷정수는 정수예요! 100.0을 먼저 곱해서 실수 연산으로 바꿔야 해요.

입력: 5
출력: 4/5 (또는 0.800)

풀이: 바꿀 때 당첨 확률 = (N-1)/N
처음에 틀릴 확률이 (N-1)/N이고, 사회자가 나머지를 모두 제거해 주니까!

용어 정리

용어설명
확률 (Probability)어떤 일이 일어날 가능성을 0~1 사이 수로 나타낸 것

독립 시행 (Independent Trial)이전 결과가 다음 결과에 영향을 주지 않는 실험
