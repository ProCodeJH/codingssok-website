CPU, 컴퓨터의 두뇌

컴퓨터에서 가장 중요한 부품이 뭐냐고 물으면, 대부분 CPU라고 대답해요. CPU는 Central Processing Unit, 우리말로 중앙처리장치예요. 이름 그대로 컴퓨터의 '중앙'에서 모든 '처리'를 담당하는 장치예요.

사람의 두뇌가 생각하고, 판단하고, 몸에 명령을 내리듯이, CPU도 계산하고, 판단하고, 다른 부품에 명령을 내려요. 그래서 CPU를 '컴퓨터의 두뇌'라고 부르는 거예요.

[핵심] CPU란? CPU (Central Processing Unit, 중앙처리장치)

컴퓨터에서 프로그램의 명령어를 읽고, 해석하고, 실행하는 핵심 부품이에요.

CPU가 하는 일 세 가지:
1. 연산 — 덧셈, 뺄셈, 비교 등의 계산
2. 제어 — 다음에 어떤 명령어를 실행할지 결정
3. 데이터 이동 — 메모리에서 데이터를 가져오고 결과를 돌려보냄

CPU가 하는 세 가지 일

CPU 안에는 크게 두 개의 핵심 부서가 있어요. ALU와 CU예요. 이 둘이 역할을 나눠서 일해요.

[핵심] ALU — 연산 담당 ALU (Arithmetic Logic Unit, 산술논리장치)

이름에 힌트가 있어요:
• Arithmetic(산술): 덧셈, 뺄셈, 곱셈 같은 수학 계산
• Logic(논리): AND, OR, NOT 같은 논리 연산, 크기 비교

S-01에서 배운 논리 게이트들이 모여서 ALU를 만드는 거예요!
반가산기, 전가산기를 연결하면 덧셈 회로가 되고, 그런 회로들이 모인 게 ALU예요.
[핵심] CU — 제어 담당 CU (Control Unit, 제어장치)

오케스트라의 지휘자 같은 역할이에요.
• 다음에 실행할 명령어를 메모리에서 가져와요
• 명령어가 무슨 뜻인지 해석해요
• ALU에게 '덧셈 해!' 같은 지시를 내려요
• 메모리에게 '3번 주소의 데이터 줘!' 같은 신호를 보내요

CU가 없으면 ALU는 뭘 해야 할지 모르고, 메모리는 언제 데이터를 보내야 할지 몰라요.
[비유] CPU = 학교 CPU를 학교에 비유하면:

• CU(제어장치) = 교장 선생님 — 시간표를 보고 '지금은 수학 시간!' 하고 지시
• ALU(연산장치) = 수학 선생님 — 실제로 계산 문제를 풀어주는 분
• 레지스터 = 칠판 — 지금 사용 중인 숫자를 잠깐 적어두는 곳
• 메모리(RAM) = 사물함 — 데이터를 보관하는 곳

교장 선생님(CU)이 사물함(메모리)에서 문제를 가져와서 칠판(레지스터)에 쓰고, 수학 선생님(ALU)이 풀어요!

데이터 이동 — 버스(Bus)

CPU, 메모리, 입출력 장치 사이에 데이터가 오가려면 길이 필요해요. 이 길을 버스(Bus)라고 불러요. 사람을 태우는 버스가 아니라, 데이터를 실어 나르는 통로예요.

[핵심] 세 종류의 버스 1. 데이터 버스(Data Bus) — 실제 데이터(숫자, 명령어)가 오가는 길
2. 주소 버스(Address Bus) — '메모리 몇 번지?' 주소를 보내는 길
3. 제어 버스(Control Bus) — '읽어!', '써!' 같은 제어 신호가 오가는 길

택배에 비유하면:
• 데이터 버스 = 택배 물건
• 주소 버스 = 배송 주소
• 제어 버스 = '배달해 주세요' / '수거해 주세요' 같은 지시

명령어란 무엇일까?

CPU는 우리가 쓴 C 코드를 직접 읽지 못해요. CPU가 이해하는 건 오직 기계어(Machine Language)뿐이에요. 기계어는 0과 1로만 이루어진 명령이에요.

기계어 명령 하나하나를 명령어(Instruction)라고 해요. 모든 명령어는 두 부분으로 나뉘어요.

[핵심] 명령어의 구조 명령어 = Opcode + Operand

• Opcode (연산 코드): '무엇을 할지' — 더해라, 저장해라, 비교해라...
• Operand (피연산자): '무엇을 대상으로' — 어떤 숫자, 어떤 주소...

예시:
ADD 3, 5 → Opcode: ADD(더해라), Operand: 3과 5
LOAD 100 → Opcode: LOAD(가져와라), Operand: 100번지
STORE 200 → Opcode: STORE(저장해라), Operand: 200번지

국어 문장에 비유하면:
'사과를 먹어라' → 동사(Opcode): 먹어라, 목적어(Operand): 사과
[예시] 기계어는 이렇게 생겼어요 사람이 보기엔 이런 모습이에요:

0001 0000 0000 0011 → LOAD 3번지 (3번지에서 데이터 가져와)
0010 0000 0000 0101 → ADD 5번지 (5번지 데이터와 더해)
0011 0000 0000 0111 → STORE 7번지 (결과를 7번지에 저장해)

앞 4비트가 Opcode(0001=LOAD, 0010=ADD, 0011=STORE)
뒤 12비트가 Operand(메모리 주소)

사람이 이걸 직접 쓰면 너무 힘드니까 C 같은 프로그래밍 언어를 쓰는 거예요!

폰 노이만 구조

1945년, 헝가리 출신 수학자 존 폰 노이만(John von Neumann)이 획기적인 아이디어를 냈어요. '프로그램(명령어)도 데이터처럼 메모리에 저장하자!'는 거예요. 이것을 저장 프로그램 개념(Stored-Program Concept)이라고 해요.

그 전에는 프로그램을 바꾸려면 전선을 뽑았다 꽂았다 해야 했어요. 폰 노이만 구조에서는 프로그램을 메모리에 저장하니까, 다른 프로그램을 실행하고 싶으면 메모리에 새 프로그램을 올리기만 하면 돼요.

[핵심] 폰 노이만 구조의 구성 요소 폰 노이만 구조는 다섯 가지로 이루어져 있어요:

1. CPU — 명령어를 실행하는 중앙처리장치
- ALU (연산장치)
- CU (제어장치)
- 레지스터 (임시 저장소)

2. 메모리 (주기억장치) — 프로그램과 데이터를 저장

3. 입력 장치 — 키보드, 마우스 등

4. 출력 장치 — 모니터, 프린터 등

5. 버스 — 위 장치들을 연결하는 데이터 통로

핵심: 프로그램도 데이터도 같은 메모리에 저장해요!
[역사] 폰 노이만 (1903-1957) 존 폰 노이만은 헝가리 부다페스트에서 태어난 천재 수학자예요.

• 6살에 8자리 나눗셈을 암산으로 풀었대요
• 게임 이론, 양자역학, 컴퓨터 과학 등 수많은 분야에 기여했어요
• EDVAC 보고서(1945)에서 저장 프로그램 개념을 제안했어요
• 이 아이디어가 너무 영향력이 커서, 오늘날 대부분의 컴퓨터를 '폰 노이만 구조'라고 불러요

지금 여러분이 쓰는 컴퓨터, 스마트폰, 게임기 모두 폰 노이만 구조예요!

레지스터 — CPU 안의 초고속 메모장

CPU가 일을 하려면, 지금 처리하는 데이터와 명령어를 아주 빠르게 꺼내 볼 수 있어야 해요. 메모리(RAM)도 빠르지만, CPU 입장에서는 좀 멀어요. 그래서 CPU 안에 아주 작지만 엄청나게 빠른 저장소가 있어요. 그게 레지스터(Register)예요.

[비유] 레지스터 = 손 위의 메모지 여러분이 수학 문제를 풀 때를 생각해 보세요.

• 레지스터 = 손에 들고 있는 메모지 (바로 볼 수 있어요)
• RAM (메모리) = 책상 위 공책 (손을 뻗어야 해요)
• 하드디스크/SSD = 사물함 (걸어가야 해요)

레지스터는 용량이 아주 작지만 (보통 32비트 또는 64비트) 속도가 가장 빨라요.
CPU는 계산할 때 항상 레지스터에 있는 데이터를 사용해요.
[핵심] 꼭 알아야 할 레지스터 5가지 1. PC (Program Counter, 프로그램 카운터)
→ '다음에 실행할 명령어의 주소'를 저장해요
→ 명령어를 하나 가져올 때마다 자동으로 1 증가해요

2. IR (Instruction Register, 명령어 레지스터)
→ '지금 실행 중인 명령어'를 저장해요
→ 메모리에서 가져온 명령어가 여기에 들어와요

3. ACC (Accumulator, 누산기)
→ 계산 결과를 잠깐 저장하는 레지스터
→ 덧셈, 뺄셈 결과가 여기에 쌓여요 (누적된다 → 누산기)

4. MAR (Memory Address Register, 메모리 주소 레지스터)
→ '메모리 몇 번지에 접근할지' 주소를 저장해요
→ 주소 버스로 이 값이 나가요

5. MDR (Memory Data Register, 메모리 데이터 레지스터)
→ 메모리에서 읽어온 데이터, 또는 메모리에 쓸 데이터를 저장해요
→ 데이터 버스로 이 값이 오가요
[예시] 레지스터의 역할 — 택배 비유 CPU가 메모리에서 데이터를 가져오는 과정을 택배에 비유하면:

1. PC: '다음 택배 주문서 번호는 5번이야' (다음 명령어 주소)
2. MAR: '5번 주소로 가!' (주소를 주소 버스에 올림)
3. MDR: '5번 주소에서 가져온 물건은 이거야' (데이터 수신)
4. IR: '주문서를 읽어보니, ADD 연산이네' (명령어 해석)
5. ACC: '계산 결과는 42야' (결과 저장)

이 다섯 레지스터가 팀워크를 이뤄서 명령어 하나를 처리해요!

명령어 사이클 — Fetch, Decode, Execute

CPU는 전원이 켜진 순간부터 꺼질 때까지 같은 세 단계를 반복해요. 이것을 명령어 사이클(Instruction Cycle) 또는 머신 사이클(Machine Cycle)이라고 해요.

[비유] 명령어 사이클 = 택배 시스템 택배 기사가 일하는 방식과 똑같아요!

Fetch (인출) = 물류센터에서 택배 상자를 꺼내요
→ PC(프로그램 카운터)가 '다음 상자는 5번 선반'이라고 알려줘요
→ 상자를 꺼낸 뒤 PC는 '다음은 6번'으로 업데이트

Decode (해석) = 송장(운송장)을 읽어요
→ '이건 서울 강남구로 보내는 거구나' 파악
→ 어떤 트럭에 실을지 결정

Execute (실행) = 실제로 배달해요
→ 트럭에 싣고 목적지까지 배달 완료

이걸 끊임없이 반복! 택배가 없을 때까지 (프로그램이 끝날 때까지)

명령어 사이클 상세 — 내부에서 벌어지는 일

Fetch-Decode-Execute 각 단계를 레지스터 수준에서 더 자세히 살펴볼게요.

[핵심] Fetch 단계 상세 ① MAR ← PC : PC의 값을 MAR에 복사해요 (몇 번지를 읽을지 알려줌)
② MDR ← Memory[MAR] : MAR이 가리키는 메모리 주소에서 명령어를 읽어서 MDR에 저장
③ IR ← MDR : 읽어온 명령어를 IR에 복사해요
④ PC ← PC + 1 : PC를 1 증가시켜요 (다음 명령어 준비)

이 네 단계가 Fetch 하나를 구성해요.
[핵심] Decode 단계 상세 ⑤ CU가 IR의 Opcode 부분을 읽어요
⑥ '이 명령어는 ADD구나' 하고 연산 종류를 파악해요
⑦ Operand 부분을 보고, 필요한 데이터의 주소를 확인해요
⑧ 필요하면 해당 주소에서 데이터를 가져와요 (MAR, MDR 사용)
[핵심] Execute 단계 상세 ⑨ CU가 ALU에게 연산 신호를 보내요 ('더해!', '비교해!')
⑩ ALU가 레지스터의 데이터로 연산을 수행해요
⑪ 결과를 ACC(누산기)에 저장해요
⑫ 필요하면 결과를 메모리에 다시 저장해요 (STORE 명령어일 때)

프로그램 실행 추적 — 덧셈 프로그램

이제 간단한 프로그램이 CPU에서 실행되는 과정을 한 단계씩 추적해 볼게요. '메모리 10번지의 값과 11번지의 값을 더해서 12번지에 저장하라'는 프로그램이에요.

[예시] 추적할 프로그램 메모리 상태:
주소 0: LOAD 10 (10번지 데이터를 ACC에 넣어라)
주소 1: ADD 11 (11번지 데이터를 ACC에 더해라)
주소 2: STORE 12 (ACC 값을 12번지에 저장해라)

주소 10: 25 (데이터)
주소 11: 17 (데이터)
주소 12: 0 (결과가 저장될 곳)
단계PCMARMDRIRACC설명
초기0---0프로그램 시작

Fetch 221ADD 11ADD 11251번지에서 명령어 가져옴, PC+1

결과적으로 메모리 12번지에 42(=25+17)가 저장되었어요. CPU가 세 번의 Fetch-Decode-Execute 사이클을 돌면서 덧셈 프로그램을 완성한 거예요.

[생각해 보기] 뺄셈 프로그램은? 만약 25에서 17을 빼고 싶다면 프로그램을 어떻게 바꿔야 할까요?

힌트: 두 번째 명령어만 바꾸면 돼요.
주소 1: SUB 11 (11번지 데이터를 ACC에서 빼라)

결과는 12번지에 8(=25-17)이 저장될 거예요. 명령어(소프트웨어)만 바꾸면 하드웨어는 그대로인데 다른 계산을 할 수 있어요. 이게 바로 폰 노이만 구조의 힘이에요!

클럭 — CPU의 심장 박동

CPU 안의 모든 동작은 아무 때나 일어나는 게 아니에요. 클럭(Clock)이라는 신호에 맞춰서 '째깍, 째깍' 일정한 박자로 움직여요.

[핵심] 클럭이란? 클럭(Clock): CPU에 일정한 간격으로 전기 신호를 보내주는 장치

클럭이 '째깍' 한 번 울릴 때마다 CPU는 한 동작을 해요.
이 '째깍' 한 번을 1 클럭 사이클이라고 해요.

• 클럭이 빠르면 → CPU가 더 빨리 일해요
• 클럭이 느리면 → CPU가 더 느리게 일해요
[비유] 클럭 = 음악의 박자 행진할 때 '하나, 둘, 하나, 둘' 구령에 맞춰 걷는 거랑 같아요.

• 구령이 빠르면 → 빨리 행진
• 구령이 느리면 → 천천히 행진

CPU도 클럭 신호에 맞춰서 일해요. 클럭이 없으면 CPU 안의 부품들이 제각각 다른 타이밍에 움직여서 엉망이 될 거예요.
[핵심] 주파수 단위: Hz, MHz, GHz 주파수 = 1초에 클럭이 몇 번 째깍이는지

• 1 Hz (헤르츠) = 1초에 1번
• 1 MHz (메가헤르츠) = 1초에 1,000,000번 (백만 번)
• 1 GHz (기가헤르츠) = 1초에 1,000,000,000번 (십억 번)

요즘 CPU는 보통 3~5 GHz 정도예요.
3 GHz = 1초에 30억 번 째깍!

1초에 30억 번이면, 클럭 한 번의 시간은?
1 / 3,000,000,000 = 약 0.33 나노초 (0.00000000033초)
[주의] 클럭 속도 = CPU 성능? 클럭 속도가 높다고 무조건 빠른 건 아니에요!

• 클럭 한 번에 처리하는 일의 양이 CPU마다 달라요
• 어떤 CPU는 클럭 1번에 명령어 1개, 어떤 CPU는 클럭 1번에 명령어 2개를 처리할 수도 있어요
• 캐시 메모리 크기, 파이프라인 단계 수 등도 영향을 줘요

같은 회사의 같은 시리즈 CPU끼리는 클럭이 높을수록 빠르지만, 다른 종류의 CPU끼리는 단순 비교가 어려워요.

C 코드가 기계어가 되기까지

우리가 작성한 C 코드는 어떻게 CPU가 실행할 수 있는 기계어로 바뀌는 걸까요? 이 과정을 단계별로 살펴볼게요.

[예시] a + b 한 줄이 거치는 여정 C 코드: c = a + b;

→ 컴파일러가 어셈블리어로 변환:
MOV R0, a ; a의 값을 레지스터 R0에 넣어
MOV R1, b ; b의 값을 레지스터 R1에 넣어
ADD R0, R1 ; R0 = R0 + R1
MOV c, R0 ; 결과를 c에 저장해

→ 어셈블러가 기계어로 변환:
0001 0000 ... (MOV R0, a)
0001 0001 ... (MOV R1, b)
0010 0000 ... (ADD R0, R1)
0001 0010 ... (MOV c, R0)

C 코드 한 줄이 기계어 여러 줄이 돼요!

C 코드 1: 간단한 덧셈과 CPU 실행

아주 간단한 C 프로그램을 보면서, 이 코드가 CPU에서 어떻게 실행되는지 생각해 볼게요.

Cadd.c — 두 수의 합 구하기
1#include <stdio.h>
2
3int main(void) {
4 int a = 25;
5 int b = 17;
6 int c = a + b; /* CPU의 ALU가 덧셈 수행 */
7
8 printf("a + b = %d\n", c);
9 return 0;
10}
실행 결과
a + b = 42
[설명] 이 코드가 CPU에서 실행되는 과정 int a = 25; 가 실행될 때 CPU 내부에서 벌어지는 일:

1. CU가 '25를 메모리에 저장하라'는 명령어를 Fetch
2. CU가 이 명령어를 Decode — STORE 명령이구나!
3. 25라는 값을 a의 메모리 주소에 Execute (저장)

int c = a + b; 가 실행될 때:
1. a의 값(25)을 메모리에서 레지스터로 LOAD
2. b의 값(17)을 메모리에서 레지스터로 LOAD
3. ALU가 두 레지스터를 ADD → 결과 42가 ACC에
4. ACC의 값(42)을 c의 메모리 주소에 STORE

C 코드 한 줄이 CPU 명령어 여러 개로 실행되는 거예요!

C 코드 2: 클럭 속도 계산기

CPU의 클럭 속도와 명령어 수를 입력하면 실행 시간을 계산하는 프로그램을 만들어 볼게요.

Cclock_calc.c — 실행 시간 계산기
1#include <stdio.h>
2
3int main(void) {
4 double clock_ghz; /* 클럭 속도 (GHz) */
5 long long num_inst; /* 명령어 수 */
6 int cpi; /* CPI: 명령어당 클럭 사이클 수 */
7
8 printf("클럭 속도 (GHz): ");
9 scanf("%lf", &clock_ghz);
10
11 printf("명령어 수: ");
12 scanf("%lld", &num_inst);
13
14 printf("CPI (명령어당 사이클 수): ");
15 scanf("%d", &cpi);
16
17 /* 실행 시간 = 명령어 수 * CPI / 클럭 속도 */
18 double clock_hz = clock_ghz * 1000000000.0;
19 double exec_time = (double)num_inst * cpi / clock_hz;
20
21 printf("\n--- 결과 ---\n");
22 printf("총 클럭 사이클: %lld\n", num_inst * cpi);
23 printf("실행 시간: %.9f 초\n", exec_time);
24
25 if (exec_time < 0.000001) {
26 printf(" = %.3f 나노초\n", exec_time * 1000000000);
27 } else if (exec_time < 0.001) {
28 printf(" = %.3f 마이크로초\n", exec_time * 1000000);
29 } else if (exec_time < 1.0) {
30 printf(" = %.3f 밀리초\n", exec_time * 1000);
31 }
32
33 return 0;
34}
실행 결과
클럭 속도 (GHz): 3.0 명령어 수: 1000000 CPI (명령어당 사이클 수): 2 --- 결과 --- 총 클럭 사이클: 2000000 실행 시간: 0.000000667 초 = 0.667 밀리초
[핵심] 실행 시간 공식 실행 시간 = 명령어 수 x CPI / 클럭 속도

• 명령어 수: 프로그램에 포함된 기계어 명령어의 총 개수
• CPI (Cycles Per Instruction): 명령어 하나를 실행하는 데 필요한 클럭 사이클 수
- 간단한 명령어(ADD)는 CPI가 1일 수 있어요
- 복잡한 명령어(MUL, DIV)는 CPI가 3~10일 수도 있어요
• 클럭 속도: 1초에 몇 번 째깍이는지 (Hz 단위)

CPU를 빠르게 만들려면?
① 클럭 속도를 높이거나
② CPI를 줄이거나
③ 명령어 수를 줄이거나 (더 좋은 알고리즘!)

C 코드 3: 명령어 시뮬레이터

Fetch-Decode-Execute 사이클을 C 코드로 직접 흉내내 볼게요. 아주 간단한 가상의 CPU를 만드는 거예요!

Ccpu_sim.c — 간단한 CPU 시뮬레이터
1#include <stdio.h>
2
3/* 명령어 종류 (Opcode) */
4#define LOAD 1 /* 메모리 → ACC */
5#define ADD 2 /* ACC + 메모리 → ACC */
6#define STORE 3 /* ACC → 메모리 */
7#define HALT 0 /* 정지 */
8
9int main(void) {
10 /* 메모리: 각 칸에 명령어 또는 데이터 저장 */
11 /* 명령어 형식: opcode * 100 + operand(주소) */
12 int memory[20] = {
13 110, /* 주소 0: LOAD 10 (10번지 값을 ACC에) */
14 211, /* 주소 1: ADD 11 (11번지 값을 ACC에 더해) */
15 312, /* 주소 2: STORE 12 (ACC를 12번지에 저장) */
16 0, /* 주소 3: HALT (정지) */
17 0, 0, 0, 0, 0, 0,
18 25, /* 주소 10: 데이터 (25) */
19 17, /* 주소 11: 데이터 (17) */
20 0, /* 주소 12: 결과 저장 공간 */
21 };
22
23 /* 레지스터 */
24 int PC = 0; /* 프로그램 카운터 */
25 int IR = 0; /* 명령어 레지스터 */
26 int ACC = 0; /* 누산기 */
27 int MAR = 0; /* 메모리 주소 레지스터 */
28 int MDR = 0; /* 메모리 데이터 레지스터 */
29
30 int running = 1;
31 int cycle = 0;
32
33 printf("=== CPU 시뮬레이터 시작 ===\n\n");
34
35 while (running) {
36 cycle++;
37 printf("--- 사이클 %d ---\n", cycle);
38
39 /* 1단계: Fetch */
40 MAR = PC;
41 MDR = memory[MAR];
42 IR = MDR;
43 PC = PC + 1;
44 printf("[Fetch] PC=%d, IR=%d\n", PC, IR);
45
46 /* 2단계: Decode */
47 int opcode = IR / 100; /* 앞자리 = 연산 종류 */
48 int operand = IR % 100; /* 뒷자리 = 주소 */
49 printf("[Decode] opcode=%d, operand=%d\n", opcode, operand);
50
51 /* 3단계: Execute */
52 switch (opcode) {
53 case LOAD:
54 MAR = operand;
55 MDR = memory[MAR];
56 ACC = MDR;
57 printf("[Execute] LOAD: ACC = memory[%d] = %d\n",
58 operand, ACC);
59 break;
60 case ADD:
61 MAR = operand;
62 MDR = memory[MAR];
63 ACC = ACC + MDR;
64 printf("[Execute] ADD: ACC = ACC + memory[%d] = %d\n",
65 operand, ACC);
66 break;
67 case STORE:
68 MAR = operand;
69 memory[MAR] = ACC;
70 printf("[Execute] STORE: memory[%d] = ACC = %d\n",
71 operand, ACC);
72 break;
73 case HALT:
74 printf("[Execute] HALT: 프로그램 종료\n");
75 running = 0;
76 break;
77 }
78 printf("\n");
79 }
80
81 printf("=== 실행 결과 ===\n");
82 printf("memory[12] = %d (25 + 17 = 42)\n", memory[12]);
83 printf("총 %d 사이클 실행\n", cycle);
84
85 return 0;
86}
실행 결과
=== CPU 시뮬레이터 시작 === --- 사이클 1 --- [Fetch] PC=1, IR=110 [Decode] opcode=1, operand=10 [Execute] LOAD: ACC = memory[10] = 25 --- 사이클 2 --- [Fetch] PC=2, IR=211 [Decode] opcode=2, operand=11 [Execute] ADD: ACC = ACC + memory[11] = 42 --- 사이클 3 --- [Fetch] PC=3, IR=312 [Decode] opcode=3, operand=12 [Execute] STORE: memory[12] = ACC = 42 --- 사이클 4 --- [Fetch] PC=4, IR=0 [Decode] opcode=0, operand=0 [Execute] HALT: 프로그램 종료 === 실행 결과 === memory[12] = 42 (25 + 17 = 42) 총 4 사이클 실행
[코드 해설] CPU 시뮬레이터 핵심 포인트 이 프로그램은 진짜 CPU의 동작을 C 코드로 흉내낸 거예요:

1. memory 배열 = 컴퓨터의 메모리 역할
2. PC, IR, ACC, MAR, MDR 변수 = 레지스터 역할
3. while 루프 = 명령어 사이클의 반복
4. IR / 100 = Opcode 추출 (Decode)
5. switch문 = 명령어 종류에 따라 다른 실행 (Execute)

실제 CPU는 이것보다 훨씬 복잡하지만, 기본 원리는 똑같아요!

파이프라인 — 빨래 공장의 비밀

CPU를 더 빠르게 만들고 싶으면 어떻게 해야 할까요? 클럭 속도를 높이는 방법도 있지만, 한계가 있어요. 그래서 엔지니어들이 생각한 똑똑한 방법이 파이프라인(Pipeline)이에요.

[비유] 세탁소에서 배우는 파이프라인 세탁소에서 빨래 4묶음을 처리한다고 해 볼게요.
각 단계는 30분씩 걸려요: 세탁 → 건조 → 접기 → 정리

파이프라인 없이 (한 묶음씩 완료)
빨래1: [세탁][건조][접기][정리]
빨래2: [세탁][건조][접기][정리]
빨래3: [세탁][건조][접기][정리]
빨래4: [세탁][건조][접기][정리]
총 시간: 30분 x 4단계 x 4묶음 = 480분 (8시간)

파이프라인 사용 (동시에 진행)
빨래1: [세탁][건조][접기][정리]
빨래2: [세탁][건조][접기][정리]
빨래3: [세탁][건조][접기][정리]
빨래4: [세탁][건조][접기][정리]
총 시간: 30분 x 7 = 210분 (3시간 30분)

핵심: 세탁기가 빨래2를 돌리는 동안, 건조기는 빨래1을 말리고 있어요!
[핵심] CPU 파이프라인 CPU도 세탁소와 같은 원리를 적용해요!

명령어1: [Fetch][Decode][Execute]
명령어2: [Fetch][Decode][Execute]
명령어3: [Fetch][Decode][Execute]
명령어4: [Fetch][Decode][Execute]

• 명령어1을 Decode하는 동안, 명령어2를 Fetch해요
• 명령어1을 Execute하는 동안, 명령어2를 Decode하고, 명령어3을 Fetch해요

파이프라인이 가득 차면, 매 클럭마다 명령어 하나가 완료돼요!
실제 CPU는 5단계, 10단계, 20단계 이상의 파이프라인을 사용하기도 해요.

파이프라인은 명령어 하나를 빨리 처리하는 게 아니에요. 명령어 여러 개를 동시에 조금씩 처리해서, 전체적으로 빨라지는 거예요. 이것을 처리량(Throughput)이 높아진다고 해요.

C 코드 4: 파이프라인 vs 비파이프라인 비교

Cpipeline_sim.c — 파이프라인 시뮬레이션
1#include <stdio.h>
2
3int main(void) {
4 int num_inst = 10; /* 명령어 수 */
5 int stages = 3; /* 파이프라인 단계 수 (F, D, E) */
6 int cycle_time = 1; /* 한 단계에 걸리는 시간 (클럭) */
7
8 /* 비파이프라인: 각 명령어가 모든 단계를 마쳐야 다음 시작 */
9 int no_pipe = num_inst * stages * cycle_time;
10
11 /* 파이프라인: 첫 명령어가 채우는 시간 + 나머지 명령어 */
12 int with_pipe = stages * cycle_time + (num_inst - 1) * cycle_time;
13
14 printf("=== 파이프라인 비교 ===");
15 printf("\n명령어 수: %d개\n", num_inst);
16 printf("파이프라인 단계: %d단계\n\n", stages);
17
18 printf("비파이프라인: %d 클럭\n", no_pipe);
19 printf("파이프라인: %d 클럭\n", with_pipe);
20 printf("\n속도 향상: %.1f배 빨라짐\n",
21 (double)no_pipe / with_pipe);
22
23 /* 시각화: 파이프라인 동작 (처음 5개만) */
24 printf("\n=== 파이프라인 시각화 (처음 5개) ===\n");
25 printf("클럭: ");
26 for (int t = 1; t <= stages + 4; t++) {
27 printf(" %2d ", t);
28 }
29 printf("\n");
30
31 for (int i = 0; i < 5; i++) {
32 printf("명령%d: ", i + 1);
33 for (int t = 1; t <= stages + 4; t++) {
34 if (t == i + 1) printf(" F ");
35 else if (t == i + 2) printf(" D ");
36 else if (t == i + 3) printf(" E ");
37 else printf(" . ");
38 }
39 printf("\n");
40 }
41
42 return 0;
43}
실행 결과
=== 파이프라인 비교 === 명령어 수: 10개 파이프라인 단계: 3단계 비파이프라인: 30 클럭 파이프라인: 12 클럭 속도 향상: 2.5배 빨라짐 === 파이프라인 시각화 (처음 5개) === 클럭: 1 2 3 4 5 6 7 명령1: F D E . . . . 명령2: . F D E . . . 명령3: . . F D E . . 명령4: . . . F D E . 명령5: . . . . F D E
[핵심] 파이프라인 공식 비파이프라인 시간 = 명령어 수 x 단계 수
파이프라인 시간 = 단계 수 + (명령어 수 - 1)

명령어가 많아질수록 파이프라인의 효과가 커져요!
• 100개 명령어, 3단계: 비파이프라인 300 vs 파이프라인 102 → 약 3배
• 1000개 명령어, 3단계: 비파이프라인 3000 vs 파이프라인 1002 → 약 3배

이론적으로 파이프라인의 최대 속도 향상은 단계 수와 같아요!
3단계 파이프라인 → 최대 3배, 5단계 → 최대 5배

파이프라인 해저드 — 파이프라인이 막힐 때

파이프라인이 항상 잘 돌아가면 좋겠지만, 가끔 문제가 생겨요. 이런 문제를 파이프라인 해저드(Pipeline Hazard)라고 해요.

[핵심] 데이터 해저드 데이터 해저드(Data Hazard): 앞 명령어의 결과가 필요한데 아직 안 나왔을 때

예시:
명령어1: ADD R1, R2, R3 (R1 = R2 + R3, 결과를 R1에 저장)
명령어2: SUB R4, R1, R5 (R4 = R1 - R5, R1의 값이 필요!)

명령어2가 R1을 읽으려는데, 명령어1이 아직 R1에 결과를 안 써넣었어요!

세탁소 비유: 빨래1의 깨끗한 수건이 필요한데, 빨래1이 아직 건조 중이에요.
기다려야 해요! (이것을 'stall' 또는 'bubble'이라고 해요)
[핵심] 제어 해저드 제어 해저드(Control Hazard): if문이나 반복문 때문에 다음에 어떤 명령어를 실행할지 모를 때

예시:
명령어1: CMP R1, 0 (R1이 0인지 비교)
명령어2: BEQ label (같으면 label로 점프!)
명령어3: ADD ... (점프 안 하면 이거 실행)
...
label: SUB ... (점프하면 이거 실행)

명령어2의 결과가 나올 때까지, 명령어3을 실행해야 할지 label을 실행해야 할지 몰라요!

세탁소 비유: 다음 빨래가 흰옷인지 색깔옷인지 모르는 상태에서 세제를 미리 넣을 수 없는 거예요.
[팁] 해저드 해결 방법 CPU 설계자들이 생각한 해결책들:

1. 대기(Stall): 결과가 나올 때까지 기다려요 (느려지지만 확실해요)

2. 전방 전달(Forwarding): 결과가 레지스터에 기록되기 전에 바로 다음 명령어에 전달해요

3. 분기 예측(Branch Prediction): if문의 결과를 '예측'해서 미리 실행해요
- 맞으면: 시간 절약!
- 틀리면: 실행한 걸 취소하고 다시 해야 해요 (flush)
- 요즘 CPU는 90% 이상 정확하게 예측해요!

멀티코어 — 요리사를 더 고용하자!

파이프라인으로 빨라졌지만, 클럭 속도를 계속 높이는 데는 한계가 있어요. 너무 빨리 돌리면 열이 너무 많이 나거든요. 그래서 생각한 방법이 멀티코어(Multi-core)예요.

[비유] 요리사와 멀티코어 식당에서 요리를 더 빨리 내놓고 싶으면?

방법1: 요리사 한 명을 더 빠르게 일하게 한다 → 한계가 있어요
방법2: 요리사를 여러 명 고용한다!

• 싱글코어 = 요리사 1명 → 한 번에 요리 1개
• 듀얼코어 = 요리사 2명 → 한 번에 요리 2개
• 쿼드코어 = 요리사 4명 → 한 번에 요리 4개
• 옥타코어 = 요리사 8명 → 한 번에 요리 8개

각 요리사(코어)가 독립적으로 일하면서 전체 작업을 나눠서 처리해요!
[핵심] 멀티코어란? 코어(Core): CPU 안에서 실제로 명령어를 실행하는 처리 장치
각 코어에는 ALU, CU, 레지스터가 모두 있어요.

멀티코어: 하나의 CPU 칩 안에 코어가 여러 개 있는 것

• 듀얼코어(2코어): 2개의 프로그램을 동시에 실행 가능
• 쿼드코어(4코어): 4개의 프로그램을 동시에 실행 가능
• 옥타코어(8코어): 8개의 프로그램을 동시에 실행 가능

요즘 스마트폰 CPU도 보통 8코어 이상이에요!
데스크탑 CPU는 6코어~24코어까지 있어요.
[주의] 코어가 2배면 속도도 2배? 아쉽게도 아니에요! 이유가 있어요.

1. 모든 작업을 나눌 수 있는 건 아니에요
- 순서대로 해야 하는 일은 나눌 수 없어요
- 예: '달걀 삶기 → 껍질 벗기기 → 자르기'를 세 요리사가 동시에 못 해요

2. 암달의 법칙(Amdahl's Law)
- 프로그램에서 병렬화할 수 없는 부분이 전체 속도를 제한해요
- 프로그램의 50%만 병렬화 가능하면, 코어가 아무리 많아도 최대 2배까지만 빨라져요
- 90%를 병렬화 가능하면 최대 10배

3. 코어 사이의 소통 비용
- 요리사들이 서로 '소금 줘', '다 됐어?' 하고 대화하는 데도 시간이 걸려요

CISC 방식 (1줄):

STORE [200], R3 ← R3을 200번지에

CISC는 명령어가 적지만, RISC는 각 명령어가 단순해서 파이프라인에 유리해요!

지금 CPU와 비교하면?

• 4004보다 약 5,000배 빠른 클럭, 수백만 배 많은 트랜지스터

50년 만에 이만큼 발전한 거예요!
[역사] 무어의 법칙 (1965) 인텔의 공동 창업자 고든 무어(Gordon Moore)가 1965년에 예측한 법칙이에요.

'반도체 칩에 들어가는 트랜지스터 수가 약 2년마다 2배로 늘어난다'

실제로 50년 넘게 대략 맞았어요!
• 1971년 Intel 4004: 2,300개
• 1989년 Intel 486: 120만 개
• 2006년 Core 2 Duo: 2.9억 개
• 2020년대 최신 CPU: 수백억 개

하지만 트랜지스터가 원자 크기에 가까워지면서 물리적 한계에 다가가고 있어요. 무어의 법칙이 언제까지 유효할지는 아무도 몰라요.
[역사] ARM의 탄생 ARM (Advanced RISC Machines)

• 1983년 영국의 에이콘(Acorn) 컴퓨터 회사에서 시작
• 처음 이름은 Acorn RISC Machine이었어요
• RISC 철학을 따라 간단하고 전력 소비가 적은 CPU를 설계
• ARM은 직접 칩을 만들지 않고, 설계도를 다른 회사에 판매(라이선스)해요

지금 세계에서 가장 많이 사용되는 CPU 구조예요:
• 스마트폰의 거의 100%가 ARM 기반
• 전 세계에서 매년 수백억 개의 ARM 칩이 생산돼요
• 여러분의 스마트폰 안에도 ARM CPU가 들어 있어요!

디버깅 훈련 2: 잘못된 명령어 순서

Cdebug2.c — 명령어 순서가 틀렸어요!
1#include <stdio.h>
2
3int main(void) {
4 /* 시뮬레이터: a + b를 계산하려는 프로그램 */
5 int memory[10] = {0, 0, 0, 0, 0, 25, 17, 0};
6 /* 주소: 0 1 2 3 4 5 6 7 */
7
8 int ACC = 0;
9
10 /* 잘못된 순서! */
11 ACC = ACC + memory[6]; /* ADD 6 (먼저 더하기?) */
12 ACC = memory[5]; /* LOAD 5 (나중에 불러오기?) */
13 memory[7] = ACC; /* STORE 7 */
14
15 printf("memory[7] = %d\n", memory[7]);
16 /* 기대: 42 (= 25 + 17) */
17 /* 실제: ??? */
18
19 return 0;
20}
[디버깅] 무엇이 문제일까? 문제: 명령어 순서가 잘못되었어요!

잘못된 순서:
1. ADD 6 → ACC = 0 + 17 = 17
2. LOAD 5 → ACC = 25 (이전 결과를 덮어씀!)
3. STORE 7 → memory[7] = 25

실제 출력: memory[7] = 25 (42가 아니라 25!)

올바른 순서:
1. LOAD 5 → ACC = 25 (먼저 값을 불러와야 해요)
2. ADD 6 → ACC = 25 + 17 = 42 (그다음 더해야 해요)
3. STORE 7 → memory[7] = 42

교훈: CPU 명령어의 순서가 매우 중요해요! LOAD를 먼저 하고, 연산을 하고, STORE를 해야 해요. 프로그래밍에서도 변수를 초기화한 후에 사용해야 하는 것과 같은 원리예요.

3int main(void) {

명령어1이 아직 Execute를 마치지 않은 상태에서 명령어2가 Decode 단계에서 R1을 읽으면?

• 올바른 R4: 30 (R1이 15로 업데이트된 후 사용)
• 잘못된 R4: 0 (R1의 옛날 값 0을 사용)

이것이 바로 데이터 해저드예요!

해결: CPU는 이런 상황을 감지하고,
① 명령어2를 1~2 클럭 지연시키거나 (stall)
② 명령어1의 결과를 바로 전달해요 (forwarding)

교훈: 파이프라인은 '동시에 실행'이 아니라 '겹쳐서 실행'이에요. 데이터 의존성이 있으면 주의해야 해요.

KOI 도전 문제

[KOI 연습 1] 명령어 사이클 계산 문제: 어떤 CPU의 클럭 속도가 2 GHz이고, 프로그램이 총 500만 개의 명령어로 이루어져 있어요. 명령어당 평균 CPI가 3이라면, 이 프로그램의 실행 시간은 몇 밀리초일까요?

풀이 과정:
클럭 속도 = 2 GHz = 2,000,000,000 Hz
총 클럭 사이클 = 명령어 수 x CPI = 5,000,000 x 3 = 15,000,000
실행 시간 = 15,000,000 / 2,000,000,000 = 0.0075초 = 7.5 밀리초

정답: 7.5 밀리초
[KOI 연습 2] 파이프라인 효율 문제: 5단계 파이프라인 CPU에서 100개의 명령어를 실행해요.
(a) 파이프라인 없이 실행하면 총 몇 클럭 사이클이 필요할까요?
(b) 파이프라인으로 실행하면 몇 클럭 사이클이 필요할까요?
(c) 속도가 몇 배 향상되었나요?

풀이:
(a) 비파이프라인: 100 x 5 = 500 클럭
(b) 파이프라인: 5 + (100 - 1) = 104 클럭
(c) 속도 향상: 500 / 104 = 약 4.8배

5단계 파이프라인의 이론적 최대 속도 향상은 5배인데, 104/100 = 1.04이므로 거의 5배에 가까워요! 명령어가 많을수록 이론적 최대치에 가까워져요.
[KOI 연습 3] 레지스터 추적 문제: 다음 명령어들이 순서대로 실행될 때, 실행 후 ACC와 메모리 상태를 구하세요.

초기 상태: ACC = 0, memory[10] = 8, memory[11] = 3, memory[12] = 0

명령어 1: LOAD 10 (memory[10]을 ACC에)
명령어 2: ADD 11 (memory[11]을 ACC에 더해)
명령어 3: ADD 11 (memory[11]을 ACC에 또 더해)
명령어 4: STORE 12 (ACC를 memory[12]에 저장)

풀이:
명령어 1 후: ACC = 8
명령어 2 후: ACC = 8 + 3 = 11
명령어 3 후: ACC = 11 + 3 = 14
명령어 4 후: ACC = 14, memory[12] = 14

정답: ACC = 14, memory[12] = 14
(이것은 8 + 3 + 3 = 8 + 3 x 2 = 14 와 같아요)
