UNIT S-11 메모리 계층 구조 -- 빠른 것과 큰 것 사이의 선택

왜 기억장치가 여러 종류일까?

S-10에서 CPU를 배웠어요. CPU는 엄청나게 빠르게 계산하지만, 계산할 데이터를 어딘가에서 가져와야 해요. 이 '어딘가'가 바로 메모리(기억장치)예요.

그런데 문제가 있어요. 세상에는 '빠르면서 용량도 크고 가격도 싼' 완벽한 메모리가 없어요. 빠른 메모리는 비싸고 작고, 큰 메모리는 느리고 싸요. 그래서 컴퓨터는 여러 종류의 메모리를 층층이 쌓아서 사용해요. 이것을 메모리 계층 구조(Memory Hierarchy)라고 불러요.

[핵심] 메모리 계층 구조란? 메모리 계층 구조 (Memory Hierarchy)

빠르지만 작은 메모리부터, 느리지만 큰 메모리까지 여러 층으로 쌓아 올린 구조예요.

위로 갈수록: 빠르고, 비싸고, 용량이 작아요
아래로 갈수록: 느리고, 싸고, 용량이 커요

레지스터 > 캐시(L1/L2/L3) > RAM > SSD > HDD

왜 이런 구조가 필요할까요? 만약 모든 데이터를 레지스터처럼 빠른 메모리에 저장하면 컴퓨터 한 대 가격이 집 한 채 값이 될 거예요. 반대로 모든 데이터를 하드디스크에만 저장하면 너무 느려서 프로그램 하나 여는 데 몇 분이 걸릴 거예요.

[비유] 공부하는 학생의 자리 여러분이 공부하는 모습을 떠올려 보세요.

- 손에 든 연필 = 레지스터 (지금 바로 쓰는 것, 아주 적은 양)
- 책상 위 = 캐시 (자주 보는 교과서 1~2권을 올려놓음)
- 책상 서랍 = RAM (오늘 쓸 책 여러 권을 넣어 둠)
- 책장 = SSD (방에 있는 모든 책)
- 창고 = HDD (집 밖 창고에 있는 오래된 책들)

연필은 바로 쓸 수 있지만 한 자루뿐이고, 창고는 엄청 많이 넣을 수 있지만 가져오려면 한참 걸려요.

기억장치 종류 한눈에 보기

컴퓨터의 기억장치를 위에서 아래로, 빠른 순서대로 살펴볼게요. 각각의 특징을 표로 정리하면 이해하기 쉬워요.

계층속도용량가격(GB당)휘발성
레지스터~0.3ns수백 바이트측정 불가(극비쌈)O
L1 캐시~1ns수십 KB매우 비쌈O
L2 캐시~3ns수백 KB매우 비쌈O
L3 캐시~10ns수~수십 MB비쌈O
RAM (DRAM)~100ns수~수십 GB약 3~5천원O
SSD~100us수백 GB~수 TB약 100~200원X
HDD~10ms수 TB약 20~40원X

표에서 ns는 나노초(10억분의 1초), us는 마이크로초(100만분의 1초), ms는 밀리초(1000분의 1초)예요. 레지스터와 HDD의 속도 차이는 무려 수천만 배나 돼요!

[비유] 속도 차이를 시간으로 느껴보기 만약 레지스터 접근 시간이 1초라고 치면:

- 레지스터: 1초
- L1 캐시: 3초
- L2 캐시: 10초
- L3 캐시: 30초
- RAM: 약 5분
- SSD: 약 4일
- HDD: 약 1년

레지스터에서 꺼내면 1초인 걸, HDD에서 가져오면 1년을 기다리는 셈이에요! 그래서 자주 쓰는 데이터를 빠른 메모리에 미리 올려놓는 게 중요해요.

속도 vs 용량 vs 가격 트레이드오프

트레이드오프(trade-off)란 하나를 얻으면 다른 하나를 포기해야 하는 관계를 말해요. 메모리에서는 세 가지가 트레이드오프 관계에 있어요.

1. 속도 -- 데이터를 읽고 쓰는 빠르기
2. 용량 -- 얼마나 많은 데이터를 저장할 수 있는지
3. 가격 -- 같은 용량을 만드는 데 드는 비용

빠르게 만들려면 특수한 회로가 필요해서 비싸지고, 비싸니까 많이 넣을 수 없어요. 반대로 싸게 만들면 많이 넣을 수 있지만 느려져요. 이 세 가지를 동시에 다 좋게 만드는 건 불가능해요.

[핵심] 트레이드오프 법칙 메모리의 황금 법칙:

빠르면 -> 비싸고 -> 작아요
크면 -> 싸고 -> 느려요

완벽한 메모리는 없어요! 그래서 여러 층을 조합해서 '빠르면서도 큰 것처럼' 보이게 만드는 거예요.

메모리 계층 구조의 핵심 아이디어는 이거예요: 자주 쓰는 데이터는 빠른 메모리에, 가끔 쓰는 데이터는 느린 메모리에 넣자. 이렇게 하면 전체적으로 빠르면서도 많은 데이터를 저장할 수 있어요.

레지스터 -- CPU 속의 초고속 메모리

S-10에서 CPU를 배울 때 레지스터를 잠깐 봤어요. 레지스터는 CPU 칩 안에 들어 있는, 가장 빠른 기억장치예요.

[핵심] 레지스터란? 레지스터 (Register)

- CPU 내부에 있는 초고속 저장 공간
- 크기: 보통 32비트(4바이트) 또는 64비트(8바이트)짜리가 수십~수백 개
- 속도: 약 0.3ns (1초에 30억 번 접근 가능)
- 용도: 현재 계산 중인 값, 주소, 명령어 등을 임시 저장
- 종류: PC(프로그램 카운터), ACC(누산기), MAR(메모리 주소), MDR(메모리 데이터) 등

레지스터는 CPU 바로 옆이 아니라, CPU 안에 있어요. 그래서 전기 신호가 이동하는 거리가 극도로 짧아서 매우 빨라요. 하지만 트랜지스터를 많이 사용하기 때문에 많이 만들면 칩이 너무 커지고 비싸져요.

[비유] 레지스터 = 손에 든 물건 레지스터는 여러분이 손에 들고 있는 물건과 같아요.

손에 들고 있으면 바로 쓸 수 있지만, 손은 두 개뿐이라 많은 걸 들 수 없어요. 더 필요하면 책상(캐시)이나 서랍(RAM)에서 가져와야 해요.

C 언어에서 변수를 선언하면, 컴파일러가 자주 쓰는 변수를 자동으로 레지스터에 넣어줘요. 예전에는 register int x;처럼 직접 지정할 수도 있었지만, 요즘 컴파일러가 더 똑똑해서 거의 안 써요.

캐시 메모리 -- CPU와 RAM 사이의 중간 저장소

CPU는 매우 빠르고, RAM은 상대적으로 느려요. 이 속도 차이를 줄여주는 것이 바로 캐시 메모리(Cache Memory)예요.

[핵심] 캐시 메모리란? 캐시 메모리 (Cache Memory)

- CPU와 RAM 사이에 있는 고속 메모리
- CPU가 자주 사용하는 데이터를 미리 복사해 둠
- SRAM으로 만들어져서 DRAM(RAM)보다 훨씬 빠름
- L1, L2, L3 세 단계로 나뉨

L1: 가장 빠르고 가장 작음 (보통 32~64KB, CPU 코어 내부)
L2: L1보다 크고 약간 느림 (보통 256KB~1MB, 코어마다 하나)
L3: 가장 크고 가장 느림 (보통 수~수십 MB, 모든 코어가 공유)

캐시는 왜 세 단계일까요? 한 단계면 안 될까요? 역시 트레이드오프 때문이에요. L1은 극도로 빠르지만 너무 작고, L3은 크지만 상대적으로 느려요. 세 단계로 나눠서 상황에 맞게 데이터를 찾아요.

[비유] 캐시 = 책상 위의 교과서 공부할 때 모든 책을 서랍에서 꺼내 읽으면 너무 귀찮아요. 그래서 자주 보는 교과서 1~2권은 책상 위에 올려놓죠.

- L1 캐시 = 펼쳐놓은 교과서 (바로 눈앞에 있어서 가장 빠름)
- L2 캐시 = 책상 한쪽에 쌓아둔 참고서 (손 뻗으면 닿음)
- L3 캐시 = 책상 옆 작은 선반 (의자를 돌려서 가져옴)
- RAM = 서랍 (열어서 뒤져야 함)

지역성 원리 -- 캐시가 잘 작동하는 이유

캐시는 RAM의 데이터 중 '자주 쓸 것 같은 것'을 미리 가져다 놓아요. 그런데 컴퓨터가 어떻게 앞으로 뭘 쓸지 예측할 수 있을까요? 바로 지역성 원리(Locality) 때문이에요.

[핵심] 지역성 원리 지역성 원리 (Principle of Locality)

프로그램은 데이터를 사용할 때 일정한 패턴이 있어요:

1. 시간적 지역성 (Temporal Locality)
최근에 사용한 데이터를 곧 다시 사용할 가능성이 높아요.
예: for 루프에서 같은 변수 i를 반복해서 읽고 씀

2. 공간적 지역성 (Spatial Locality)
사용한 데이터 근처의 데이터도 곧 사용할 가능성이 높아요.
예: 배열 a[0]을 읽은 다음 a[1], a[2]를 순서대로 읽음

이 원리 덕분에 캐시는 아주 효율적으로 동작해요. 프로그램이 데이터를 사용하는 패턴이 대부분 '최근 것을 또 쓰거나, 옆에 있는 것을 쓰는' 패턴이거든요.

[비유] 교과서 읽기와 지역성 시간적 지역성: 수학 공식집의 피타고라스 정리 페이지를 열었다면, 그 문제를 풀 동안 같은 페이지를 여러 번 다시 볼 거예요. -> 최근에 본 것을 또 봄!

공간적 지역성: 교과서 42쪽을 읽고 있다면, 다음에 43쪽을 읽을 가능성이 높아요. 갑자기 200쪽으로 건너뛸 확률은 낮죠. -> 가까이 있는 것을 읽음!

C 언어에서 배열을 순서대로 접근하면 공간적 지역성이 좋아서 캐시가 잘 동작하고 빨라요. 반대로 배열을 이리저리 뛰어다니면서 접근하면 캐시가 잘 안 맞아서 느려져요. 이건 뒤에서 코드로 직접 확인해 볼 거예요.

캐시 히트와 미스

CPU가 데이터를 필요로 할 때, 먼저 캐시에 있는지 확인해요. 이때 두 가지 경우가 생겨요.

[핵심] 캐시 히트와 미스 캐시 히트 (Cache Hit)
필요한 데이터가 캐시에 있는 경우. 바로 가져올 수 있어서 빨라요!

캐시 미스 (Cache Miss)
필요한 데이터가 캐시에 없는 경우. RAM까지 가서 가져와야 해서 느려요.

히트율 (Hit Rate) = 히트 횟수 / 전체 접근 횟수
좋은 프로그램은 히트율이 90% 이상이에요!
[비유] 캐시 히트와 미스 = 책상 위 교과서 수학 문제를 풀고 있는데...

캐시 히트: '피타고라스 공식이 필요한데... 아, 책상 위에 있네!' -> 바로 봄 (빠름!)

캐시 미스: '사인 코사인 공식이 필요한데... 책상에 없네. 서랍에서 찾아야겠다.' -> 서랍 열고, 뒤지고, 꺼냄 (느림!)

미스가 나면 RAM에서 데이터를 가져오면서, 그 주변 데이터도 함께 캐시에 올려요. (공간적 지역성 활용!)

캐시 미스가 나면 CPU는 RAM에서 데이터를 가져오는 동안 기다려야 해요. 이 대기 시간을 스톨(stall)이라고 해요. 스톨이 자주 일어나면 프로그램이 느려지겠죠?

RAM -- 프로그램이 실행되는 무대

우리가 흔히 '메모리'라고 부르는 것이 바로 RAM(Random Access Memory)이에요. 컴퓨터를 살 때 '메모리 8GB', '메모리 16GB' 하는 것이 RAM의 용량이에요.

[핵심] RAM이란? RAM (Random Access Memory, 주기억장치)

- 프로그램이 실행되는 동안 데이터와 코드를 저장하는 메모리
- 휘발성: 전원을 끄면 데이터가 사라져요
- 용량: 보통 8GB ~ 64GB (2024년 기준)
- 속도: 약 100ns (레지스터보다 수백 배 느림)
- 랜덤 접근: 어떤 위치든 같은 시간에 접근 가능 (순서대로 찾을 필요 없음)

RAM은 왜 '랜덤 접근'이라는 이름일까요? 카세트 테이프는 처음부터 감아야 원하는 노래를 들을 수 있어요(순차 접근). 하지만 RAM은 어떤 위치든 바로 접근할 수 있어요. 1번지든 100만 번지든 접근 시간이 같아요. 이것이 '랜덤 접근'의 의미예요.

C 프로그램에서 변수를 선언하면 그 변수는 RAM 어딘가에 자리를 잡아요. int a = 5;라고 쓰면 RAM에서 4바이트 공간을 찾아서 거기에 5를 저장하는 거예요.

RAM은 휘발성(volatile)이에요. 전기가 끊기면 저장된 모든 데이터가 사라져요. 그래서 문서를 작성할 때 수시로 저장(Ctrl+S)을 해야 해요. 저장하면 데이터가 RAM에서 SSD나 HDD로 옮겨져서 전원이 꺼져도 남아요.

DRAM vs SRAM

RAM에도 종류가 있어요. 크게 DRAM과 SRAM으로 나뉘어요.

[핵심] DRAM과 SRAM의 차이 DRAM (Dynamic RAM)
- 축전기(콘덴서)에 전하를 저장하는 방식
- 시간이 지나면 전하가 빠져나가서 주기적으로 다시 채워야 함 (리프레시)
- 구조가 단순해서 저렴하고 대용량으로 만들 수 있음
- 우리가 쓰는 RAM(주기억장치)이 바로 DRAM

SRAM (Static RAM)
- 플립플롭 회로로 데이터를 저장하는 방식
- 전원이 켜져 있는 한 데이터가 유지됨 (리프레시 불필요)
- 구조가 복잡해서 비싸고 칩 면적을 많이 차지함
- 하지만 DRAM보다 훨씬 빠름
- 캐시 메모리에 사용됨
[비유] DRAM vs SRAM = 모래성 vs 레고 DRAM = 모래성
만들기 쉽고 재료(모래)가 싸지만, 시간이 지나면 무너져요. 계속 다시 다져줘야 해요(리프레시).

SRAM = 레고 블록
단단하게 만들어지면 그대로 유지되지만, 블록이 비싸고 같은 크기를 만들려면 재료가 더 많이 들어요.

정리하면, DRAM은 싸고 크게 만들 수 있어서 주기억장치(RAM)에 쓰이고, SRAM은 비싸지만 빨라서 캐시 메모리에 쓰여요. 각각의 장점을 살려서 적재적소에 배치한 거예요.

ROM -- 꺼져도 남는 기억

RAM은 전원이 꺼지면 데이터가 사라져요. 하지만 컴퓨터를 켤 때는 뭔가 읽어야 부팅이 되잖아요? 그 역할을 하는 것이 ROM(Read-Only Memory)이에요.

[핵심] ROM이란? ROM (Read-Only Memory)

- 비휘발성: 전원이 꺼져도 데이터가 유지됨
- 이름 그대로 '읽기 전용' -- 보통 쓰기가 어렵거나 불가능
- 용도: BIOS(컴퓨터 켤 때 실행되는 프로그램), 펌웨어(가전제품의 기본 프로그램)
- 종류: PROM(한 번만 기록), EPROM(자외선으로 지움), EEPROM(전기로 지움), 플래시 메모리

요즘은 순수한 ROM보다 플래시 메모리가 더 많이 쓰여요. 플래시 메모리는 읽기도 되고 쓰기도 되면서, 전원이 꺼져도 데이터가 남아요. USB 메모리, SD 카드, SSD가 모두 플래시 메모리 기반이에요.

SSD vs HDD -- 보조기억장치 대결

프로그램, 사진, 동영상 등을 영구적으로 저장하는 장치를 보조기억장치라고 해요. 대표적인 것이 SSD와 HDD예요.

[핵심] SSD와 HDD HDD (Hard Disk Drive, 하드디스크)
- 자기(磁気)를 이용해 금속 원판(플래터)에 데이터를 기록
- 원판이 초고속으로 회전하고, 헤드가 움직여서 읽고 씀
- 장점: GB당 가격이 매우 저렴, 대용량
- 단점: 기계적 동작이라 느리고, 충격에 약함

SSD (Solid State Drive)
- 플래시 메모리(반도체)에 데이터를 기록
- 움직이는 부품이 없음
- 장점: HDD보다 수십~수백 배 빠름, 충격에 강함, 소음 없음
- 단점: GB당 가격이 HDD보다 비쌈
항목SSDHDD
저장 방식반도체(플래시 메모리)자기 디스크(플래터)
읽기 속도500MB/s ~ 7000MB/s100~200MB/s
소음없음회전/이동 소리 있음
충격 내구성강함약함 (떨어뜨리면 고장)
GB당 가격약 100~200원약 20~40원
수명쓰기 횟수 제한 있음기계 마모로 제한
무게가벼움무거움

요즘 대부분의 노트북과 데스크탑은 SSD를 기본으로 사용해요. 윈도우 부팅 시간만 봐도 HDD는 1~2분 걸리던 것이 SSD로 바꾸면 10~20초로 줄어들어요. 그만큼 체감 속도 차이가 커요.

가상 메모리 -- RAM이 부족할 때

RAM이 8GB인 컴퓨터에서 크롬 브라우저(3GB), 게임(4GB), 음악 앱(1GB), 기타 프로그램(2GB)을 동시에 실행하면 총 10GB가 필요해요. RAM은 8GB밖에 없는데 어떻게 할까요?

이때 운영체제가 사용하는 기술이 가상 메모리(Virtual Memory)예요.

[핵심] 가상 메모리란? 가상 메모리 (Virtual Memory)

RAM이 부족할 때, SSD나 HDD의 일부 공간을 RAM인 척 사용하는 기술이에요.

- 당장 안 쓰는 데이터는 SSD/HDD로 보내고 (스왑 아웃)
- 필요해지면 다시 RAM으로 가져와요 (스왑 인)
- 이 과정을 스왑(Swap)이라고 해요

덕분에 RAM 용량보다 더 많은 프로그램을 동시에 실행할 수 있어요!
[비유] 가상 메모리 = 임시 책상 책상(RAM)이 꽉 찼어요. 그런데 새 책을 올려야 해요.

1. 지금 안 보는 책을 서랍(SSD)에 넣어요 (스왑 아웃)
2. 빈 자리에 새 책을 올려요
3. 아까 넣은 책이 다시 필요하면 서랍에서 꺼내요 (스왑 인)

책상이 작아도 서랍을 활용하면 많은 책을 '번갈아가며' 읽을 수 있어요. 다만 서랍에서 꺼내는 시간이 추가로 걸리니까, 책상이 큰 것(RAM이 많은 것)보다는 느려요.

가상 메모리 덕분에 프로그램은 자기가 컴퓨터의 메모리를 혼자 다 쓰는 것처럼 느낄 수 있어요. 실제로는 운영체제가 뒤에서 열심히 데이터를 RAM과 SSD 사이에서 왔다 갔다 시키고 있는 거예요.

페이지와 페이지 폴트

가상 메모리에서 데이터를 주고받는 단위를 페이지(Page)라고 해요. 보통 한 페이지는 4KB(4096바이트)예요.

[핵심] 페이지와 페이지 폴트 페이지 (Page)
가상 메모리에서 데이터를 나누는 고정 크기 단위 (보통 4KB)

페이지 폴트 (Page Fault)
CPU가 필요한 페이지가 RAM에 없을 때 발생하는 이벤트

페이지 폴트가 나면:
1. 운영체제가 SSD/HDD에서 해당 페이지를 찾음
2. RAM에 빈 공간이 있으면 거기에 넣음
3. 빈 공간이 없으면 안 쓰는 페이지를 SSD/HDD로 보내고(스왑 아웃) 그 자리에 넣음
4. CPU가 다시 실행을 이어감

페이지 폴트가 자주 일어나면 SSD/HDD 접근이 잦아져서 컴퓨터가 느려져요!

페이지 폴트가 심하게 자주 일어나는 상황을 스래싱(Thrashing)이라고 해요. 데이터를 RAM에 올렸다가 내렸다가를 쉴 새 없이 반복하면서 정작 계산은 거의 못 하는 상태예요. 컴퓨터가 갑자기 엄청 느려졌다면 스래싱이 원인일 수 있어요.

[비유] 페이지 폴트 = 책이 서랍에 없다! 책상 위에서 책을 찾았는데 없어요(페이지 폴트!). 서랍을 열어봐야 해요.

서랍에서 바로 찾으면 다행이지만, 서랍도 꽉 차있으면 지금 안 읽는 책을 하나 빼고(스왑 아웃) 넣어야 해요.

이 과정이 1초에 수백 번 일어나면? 공부는 안 하고 책만 왔다 갔다 하는 꼴이에요. 이게 스래싱!

전체 구조 정리 -- 메모리 피라미드

지금까지 배운 내용을 하나의 피라미드로 정리해 볼게요. 위로 갈수록 빠르고 작고 비싸요.

[정리] 메모리 계층 피라미드 [레지스터] <- 가장 빠름, 가장 작음
[ L1 캐시 ]
[ L2 캐시 ]
[ L3 캐시 ]
[ RAM ]
[ SSD ]
[ HDD ] <- 가장 느림, 가장 큼

위: 빠르고, 비싸고, 작음
아래: 느리고, 싸고, 큼

CPU는 위에서부터 아래로 내려가며 데이터를 찾아요. 위에서 찾으면 히트, 못 찾으면 아래 단계로 내려가요.

C 언어와 메모리

C 언어는 다른 언어보다 메모리를 직접 다룰 수 있는 강력한 언어예요. 변수가 메모리 어디에 저장되는지, 크기가 얼마나 되는지, 주소가 무엇인지 직접 확인할 수 있어요.

[핵심] C 언어의 메모리 관련 특징 1. 변수 = RAM의 특정 위치에 저장된 데이터
2. 배열 = RAM에서 연속된 공간을 차지하는 변수 모음
3. 포인터 = 메모리 주소를 저장하는 변수
4. sizeof = 변수나 타입의 크기(바이트)를 알려주는 연산자
5. &(주소 연산자) = 변수의 메모리 주소를 알려줌

이제 C 코드로 메모리를 탐험해 볼게요!

코드 2: 순차 접근 vs 랜덤 접근 (캐시 효과 체감)

배열을 순서대로(0, 1, 2, 3, ...) 접근하면 캐시 히트가 잘 일어나요. 하지만 이리저리 뛰어다니며 접근하면 캐시 미스가 많아져서 느려져요. 직접 확인해 볼까요?

Ccache_effect.c
1#include <stdio.h>
2#include <stdlib.h>
3#include <time.h>
4
5#define SIZE 10000000 /* 1천만 개 */
6
7int main(void) {
8 int *arr = (int *)malloc(sizeof(int) * SIZE);
9 int i, sum = 0;
10 clock_t start, end;
11
12 /* 배열 초기화 */
13 for (i = 0; i < SIZE; i++) {
14 arr[i] = i;
15 }
16
17 /* 방법 1: 순차 접근 (0, 1, 2, 3, ...) */
18 sum = 0;
19 start = clock();
20 for (i = 0; i < SIZE; i++) {
21 sum += arr[i]; /* 순서대로 접근 */
22 }
23 end = clock();
24 printf("순차 접근: %.3f초\n",
25 (double)(end - start) / CLOCKS_PER_SEC);
26
27 /* 방법 2: 큰 간격 접근 (0, 128, 256, ...) */
28 sum = 0;
29 start = clock();
30 for (i = 0; i < SIZE; i++) {
31 sum += arr[(i * 128) % SIZE]; /* 128칸씩 건너뛰며 접근 */
32 }
33 end = clock();
34 printf("건너뛰기 접근: %.3f초\n",
35 (double)(end - start) / CLOCKS_PER_SEC);
36
37 free(arr);
38 return 0;
39}
실행 결과
순차 접근: 0.025초 건너뛰기 접근: 0.098초

결과를 보면 건너뛰기 접근이 순차 접근보다 약 3~4배 느려요! 같은 데이터를 같은 횟수만큼 읽었는데도요. 이유는 순차 접근은 공간적 지역성이 좋아서 캐시 히트가 잘 나고, 건너뛰기 접근은 캐시 미스가 많이 나기 때문이에요.

[팁] 배열은 순서대로 접근하자 C에서 배열을 다룰 때는 가능하면 순서대로 접근하는 것이 빨라요.

좋은 예: for (i = 0; i < n; i++) arr[i] ...
나쁜 예: for (i = 0; i < n; i++) arr[rand() % n] ...

이것은 KOI나 프로그래밍 대회에서도 중요해요. 시간 제한이 빡빡한 문제에서 메모리 접근 순서가 실행 시간에 영향을 줄 수 있거든요.

18 size = sizes[t];
19 printf("%s개 int 배열 (%ldMB)\n",
20 labels[t], size * 4 / 1024 / 1024);
21
22 arr = (int *)malloc(sizeof(int) * size);
23 if (arr == NULL) {
24 printf(" -> 메모리 할당 실패!\n\n");
25 continue;
26 }
27
28 /* 전체 배열에 쓰기 (페이지 폴트 유발) */
29 start = clock();
30 for (i = 0; i < size; i++) {
31 arr[i] = (int)i;
32 }
33 end = clock();
34 printf(" 쓰기: %.3f초\n",
35 (double)(end - start) / CLOCKS_PER_SEC);
36
37 free(arr);
38 printf("\n");
39 }
40
41 return 0;
42}
실행 결과
100만개 int 배열 (3MB) 쓰기: 0.003초 1000만개 int 배열 (38MB) 쓰기: 0.028초 1억개 int 배열 (381MB) 쓰기: 0.285초

배열 크기가 10배씩 커질 때마다 시간도 거의 10배씩 늘어요. 이것은 순차 접근이라 캐시가 잘 동작하기 때문에 선형적으로 증가하는 거예요. 만약 RAM 용량을 넘어가는 크기를 시도하면 스왑이 발생해서 시간이 급격히 느려질 수 있어요.

[주의] 메모리 할당 실패 malloc()은 메모리 할당에 실패하면 NULL을 반환해요.

반드시 반환값이 NULL인지 확인하고 사용해야 해요! NULL인 상태에서 배열처럼 접근하면 프로그램이 비정상 종료(세그먼트 폴트)돼요.

if (arr == NULL) { printf("할당 실패!\n"); return 1; }

디버깅 연습 -- 메모리 관련 흔한 실수

디버깅 1: 배열 범위를 벗어난 접근

Cdebug1_out_of_bounds.c (버그 있음)
1#include <stdio.h>
2
3int main(void) {
4 int scores[5] = {90, 85, 78, 92, 88};
5 int i, sum = 0;
6
7 for (i = 0; i <= 5; i++) { /* 버그! */
8 sum += scores[i];
9 }
10
11 printf("평균: %d\n", sum / 5);
12 return 0;
13}
[디버깅] 어디가 잘못되었을까요? 버그: i <= 5이면 i가 0, 1, 2, 3, 4, 5까지 가요. scores[5]는 존재하지 않아요! (인덱스는 0~4)

문제: scores[5]는 배열 바깥의 메모리를 읽어요. 어떤 쓰레기 값이 들어있을지 모르고, 프로그램이 죽을 수도 있어요.

수정: i <= 5를 i < 5로 바꿔야 해요.

이것을 버퍼 오버런(buffer overrun) 또는 배열 범위 초과라고 해요. C에서 가장 흔한 메모리 버그 중 하나예요!

디버깅 2: malloc 후 free를 안 한 경우

Cdebug2_memory_leak.c (버그 있음)
1#include <stdio.h>
2#include <stdlib.h>
3
4void process_data(int n) {
5 int *data = (int *)malloc(sizeof(int) * n);
6 int i;
7
8 for (i = 0; i < n; i++) {
9 data[i] = i * 2;
10 }
11
12 printf("합계: %d\n", data[0] + data[n - 1]);
13 /* free(data)가 없다! */
14}
15
16int main(void) {
17 int i;
18 for (i = 0; i < 1000; i++) {
19 process_data(10000);
20 }
21 printf("완료\n");
22 return 0;
23}
[디버깅] 어디가 잘못되었을까요? 버그: malloc()으로 메모리를 할당했는데 free()로 반환하지 않았어요.

문제: 함수가 호출될 때마다 40KB(10000 x 4바이트)가 계속 쌓여요. 1000번 호출하면 약 40MB가 낭비돼요! 이것을 메모리 누수(memory leak)라고 해요.

수정: 함수 끝에 free(data);를 추가해야 해요.

메모리 누수가 계속되면 결국 RAM이 부족해져서 프로그램이 느려지거나 죽을 수 있어요.

디버깅 3: NULL 포인터 접근

Cdebug3_null_pointer.c (버그 있음)

3
4int main(void) {

6 /* 매우 큰 메모리를 요청 -- 실패할 수 있음 */

[디버깅] 어디가 잘못되었을까요? 버그: malloc()이 실패하면 NULL을 반환하는데, NULL인지 확인하지 않고 바로 ptr[0] = 42를 실행했어요.

메모리의 역사

[알고 보면] 자기 코어 메모리 (1950년대) 최초의 RAM은 자기 코어 메모리(Magnetic Core Memory)였어요. 작은 도넛 모양의 자성체 고리를 격자 형태로 엮어서 만들었어요. 한 고리가 1비트를 저장했죠.

1955년경 MIT에서 개발한 자기 코어 메모리는 수천 개의 고리를 손으로 하나하나 꿰어서 만들었어요. 가격이 비트당 약 1달러였으니, 오늘날의 1GB 메모리를 만들려면 80억 달러가 필요했을 거예요!

지금도 메모리를 'core'라고 부르는 관습이 남아 있어요. 프로그램이 죽을 때 생기는 'core dump'라는 용어도 여기서 왔어요.
[알고 보면] DRAM의 발명 (1966년, Dennard) 1966년, IBM의 로버트 데나드(Robert Dennard)가 DRAM을 발명했어요. 하나의 트랜지스터와 하나의 축전기만으로 1비트를 저장하는 혁신적인 구조였어요.

자기 코어 메모리에 비해 훨씬 작고, 싸고, 대량생산이 가능했어요. 1970년 인텔이 1Kbit(128바이트) DRAM 칩 '1103'을 출시하면서 메모리 혁명이 시작됐어요.

지금 여러분 컴퓨터에 들어있는 16GB RAM은 약 1370억 비트예요. 1970년의 1103 칩으로 만들려면 약 1억 3천만 개가 필요하겠네요!
[알고 보면] 플래시 메모리 (1980년, Masuoka) 1980년, 일본 도시바의 후지오 마수오카(Fujio Masuoka)가 플래시 메모리를 발명했어요. 전원 없이도 데이터를 유지하면서 전기적으로 지우고 다시 쓸 수 있는 메모리였어요.

'플래시'라는 이름은 데이터를 지우는 과정이 카메라 플래시처럼 한 번에 확 일어난다고 해서 붙였어요.

이 발명이 오늘날의 USB 메모리, SD 카드, SSD의 기반이 됐어요. 스마트폰에도 플래시 메모리가 들어있어요. 마수오카의 발명이 없었다면 우리가 아는 스마트폰도 없었을 거예요!
[알고 보면] 무어의 법칙과 메모리 무어의 법칙: 반도체 집적도가 약 2년마다 2배가 된다.

이 법칙 덕분에 메모리 용량은 계속 늘어났어요:
- 1970년: 1Kbit DRAM
- 1980년: 64Kbit DRAM
- 1990년: 16Mbit DRAM
- 2000년: 256Mbit DRAM
- 2010년: 2Gbit DRAM
- 2020년: 16Gbit DRAM

50년 동안 용량이 약 1600만 배 늘어났어요! 동시에 가격은 비트당 수 달러에서 0.000000001달러 이하로 떨어졌어요.

KOI 도전 -- 메모리 관련 계산 문제

KOI 1: 메모리 용량 계산

[KOI 유형] 메모리 용량 계산 문제: 한 컴퓨터의 RAM이 주소를 24비트로 표현하고, 한 주소에 1바이트를 저장한다. 이 컴퓨터의 최대 RAM 용량은 몇 MB인가?

풀이:
24비트 주소 -> 2^24 = 16,777,216개의 주소
한 주소에 1바이트 -> 16,777,216바이트
= 16,777,216 / 1024 = 16,384 KB
= 16,384 / 1024 = 16 MB

핵심 공식: n비트 주소 -> 최대 메모리 = 2^n 바이트

KOI 2: 캐시 히트율 계산

[KOI 유형] 캐시 성능 계산 문제: 캐시 접근 시간이 2ns이고 RAM 접근 시간이 100ns이다. 캐시 히트율이 90%일 때, 평균 메모리 접근 시간은?

풀이:
평균 접근 시간 = 히트율 x 캐시 시간 + (1 - 히트율) x RAM 시간
= 0.9 x 2ns + 0.1 x 100ns
= 1.8ns + 10ns
= 11.8ns

만약 히트율이 95%라면?
= 0.95 x 2 + 0.05 x 100 = 1.9 + 5 = 6.9ns

히트율이 5% 올라갔을 뿐인데 속도가 거의 2배 빨라졌어요!

KOI 3: 배열 메모리 크기 계산

[KOI 유형] 배열과 메모리 문제: C 프로그램에서 다음과 같이 2차원 배열을 선언했다. 이 배열이 차지하는 메모리는 몇 KB인가? (int = 4바이트)

int map[1000][1000];

풀이:
원소 개수: 1000 x 1000 = 1,000,000개
한 원소: 4바이트
전체: 1,000,000 x 4 = 4,000,000바이트
= 4,000,000 / 1024 = 약 3906 KB (약 3.8 MB)

KOI 문제에서 배열 크기를 잡을 때, RAM 제한(보통 256MB)을 넘지 않도록 주의해야 해요!

용어 정리

용어설명
메모리 계층 구조 (Memory Hierarchy)빠르지만 작은 메모리부터 느리지만 큰 메모리까지 여러 층으로 구성한 구조. 레지스터 > 캐시 > RAM > SSD > HDD 순서.
캐시 메모리 (Cache Memory)CPU와 RAM 사이에서 자주 사용하는 데이터를 미리 저장해 두는 고속 메모리. L1, L2, L3 세 단계가 있음.
지역성 원리 (Locality)프로그램이 데이터를 사용할 때의 패턴. 최근 사용한 것을 또 쓰는 시간적 지역성과, 근처에 있는 것을 쓰는 공간적 지역성이 있음.
캐시 히트 / 캐시 미스필요한 데이터가 캐시에 있으면 히트(빠름), 없으면 미스(느림). 히트율이 높을수록 성능이 좋음.
RAM (Random Access Memory)프로그램 실행 중 데이터를 저장하는 주기억장치. 휘발성이라 전원이 꺼지면 데이터가 사라짐.
가상 메모리 (Virtual Memory)RAM이 부족할 때 SSD/HDD 일부를 RAM처럼 사용하는 기술. 스왑을 통해 데이터를 주고받음.
페이지 폴트 (Page Fault)CPU가 필요한 데이터 페이지가 RAM에 없을 때 발생하는 이벤트. SSD/HDD에서 가져와야 하므로 느려짐.
트레이드오프 (Trade-off)하나를 얻으면 다른 하나를 포기해야 하는 관계. 메모리에서는 속도, 용량, 가격이 트레이드오프 관계에 있음.
