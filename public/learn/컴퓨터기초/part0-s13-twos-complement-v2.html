UNIT S-13 음수 표현

컴퓨터에게 마이너스란?

지금까지 우리가 배운 이진수는 전부 양수뿐이었어요. 0000은 0, 0101은 5, 1111은 15... 이렇게 0 이상의 수만 다뤘어요.

그런데 현실 세계에서는 음수가 꼭 필요해요. 온도가 영하로 내려갈 때, 은행 잔고가 마이너스일 때, 게임 캐릭터가 뒤로 이동할 때... 음수가 없으면 이런 것들을 표현할 수 없어요.

[생각해 보기] 음수가 필요한 상황들
1. 오늘 기온: -5도 (영하 5도)
2. 엘리베이터 층수: 지하 2층 = -2층
3. 게임에서 캐릭터 이동: 왼쪽 = -3칸
4. 시험 점수 변화: 지난 시험보다 -10점
5. 은행 잔액: -5000원 (빚)

컴퓨터가 이런 값들을 다루려면 음수를 표현할 방법이 있어야 해요!

사람은 숫자 앞에 '-' 기호를 쓰면 끝이지만, 컴퓨터는 0과 1만 사용할 수 있어요. 그래서 0과 1만으로 '이 숫자는 음수야!'라고 알려 주는 영리한 방법이 필요해요. 역사적으로 세 가지 방법이 발명되었고, 오늘 그 세 가지를 모두 배워 볼 거예요.

부호 있는 수 vs 부호 없는 수

음수 표현 방법을 배우기 전에, 먼저 중요한 구분 하나를 알아야 해요. 컴퓨터에서 정수는 크게 두 종류로 나뉘어요.

[핵심] 부호 없는 수 vs 부호 있는 수
부호 없는 수 (Unsigned)
- 양수와 0만 표현해요.
- 4비트 기준: 0 ~ 15 (0000 ~ 1111)
- 모든 비트를 '크기'에 사용해요.

부호 있는 수 (Signed)
- 양수, 0, 음수를 모두 표현해요.
- 4비트 기준: 음수 ~ 양수 범위
- 일부 비트를 '부호 정보'에 사용해요.

방법 1: 부호-크기 (Sign-Magnitude)

가장 직관적인 방법이에요. 우리 사람이 음수를 쓸 때 맨 앞에 '-' 기호를 붙이듯이, 컴퓨터도 맨 앞 비트를 부호 표시용으로 쓰는 거예요.

[핵심] 부호-크기 방식이란?
가장 왼쪽 비트(MSB, 최상위 비트)를 부호 비트로 사용해요.

- MSB = 0이면 양수(+)
- MSB = 1이면 음수(-)

나머지 비트는 숫자의 절댓값(크기)을 나타내요.

예) 4비트에서
0101 = +5 (부호: 0=양수, 크기: 101=5)
1101 = -5 (부호: 1=음수, 크기: 101=5)
[비유] 신호등 카드
수학 시간에 선생님이 O/X 카드를 들어 올리는 것과 비슷해요.

- 첫 번째 칸: 초록색(양수) 또는 빨간색(음수) 카드
- 나머지 칸: 숫자 크기 적힌 카드

예) [빨강][1][0][1] = '음수 5' = -5
예) [초록][1][0][1] = '양수 5' = +5

첫 번째 카드만 바뀌면 양수가 음수로 변해요!

[주의] 부호-크기 방식의 문제점
문제 1: 0이 두 개!
0000 = +0, 1000 = -0 두 가지가 있어요. 수학에서 +0과 -0은 같은 수인데, 컴퓨터가 두 개로 저장하면 비교할 때 혼란스러워요. 0인지 확인하려면 두 가지를 모두 검사해야 해요.

문제 2: 덧셈이 복잡해요!
부호 비트를 따로 비교하고, 크기도 따로 계산해야 해요. 양수+양수, 양수+음수, 음수+양수, 음수+음수... 경우가 너무 많아서 회로가 복잡해져요.

문제 3: 표현 범위가 아까워요!
4비트로 16가지를 표현할 수 있는데, 0이 두 개니까 실제로는 15가지 수만 표현해요. -7 ~ +7 (0 포함 15개)
단계설명값
1+3의 부호-크기 표현0011
2-3의 부호-크기 표현1011
3그냥 이진수 덧셈을 하면?0011 + 1011 = 1110
41110을 부호-크기로 해석하면-6 (틀린 답!)
5올바른 답은0 (0000 또는 1000)

보이시나요? 부호-크기 방식에서는 그냥 이진수 덧셈을 하면 엉뚱한 답이 나와요. 부호를 따로 확인하고 크기를 비교하는 복잡한 과정이 필요해요. 이건 하드웨어 회로를 매우 복잡하게 만들어요. 그래서 더 나은 방법을 찾게 되었어요.

방법 2: 1의 보수 (One's Complement)

두 번째 방법은 '1의 보수'예요. 이 방법은 음수를 만들 때 모든 비트를 뒤집어요. 0은 1로, 1은 0으로 바꾸는 거예요.

[핵심] 1의 보수란?
어떤 수의 1의 보수는 모든 비트를 반전(NOT)시킨 값이에요.

0 -> 1, 1 -> 0

양수의 1의 보수 = 해당 양수의 음수 표현

예) +5 = 0101
1의 보수(비트 반전) = 1010 = -5

'보수(complement)'란 '보충해서 채우는 수'라는 뜻이에요. 1의 보수에서 원래 수와 보수를 더하면 1111이 되거든요!
0101 + 1010 = 1111
[비유] 거울 속 반대 세상
거울에 비친 모습은 좌우가 뒤집히죠? 1의 보수도 비슷해요.

원래: 0 1 0 1 (=+5)
거울: 1 0 1 0 (=-5)

0과 1이 거울에 비친 것처럼 전부 뒤집혀요. 이것이 1의 보수예요!

부호-크기 방식보다 덧셈이 쉬워졌지만(비트 덧셈 후 올림을 다시 더하면 됨), 0이 두 개인 문제는 해결되지 않았어요. 그래서 세 번째 방법이 등장했어요!

방법 3: 2의 보수 (Two's Complement) -- 최종 승자!

[핵심] 2의 보수란?
어떤 수의 2의 보수 = 1의 보수 + 1

즉, 모든 비트를 뒤집고(1의 보수), 거기에 1을 더해요.

예) +5 = 0101
1의 보수: 1010
2의 보수: 1010 + 1 = 1011 = -5

'2의 보수'라고 부르는 이유: 원래 수와 2의 보수를 더하면 2^n(비트 수만큼 거듭제곱)이 되거든요!
0101 + 1011 = 10000 (= 2^4 = 16)

2의 보수 표에서 중요한 패턴을 발견할 수 있어요.

1. MSB(맨 왼쪽 비트)가 0이면 양수, 1이면 음수예요. (부호-크기와 동일!)
2. 0은 딱 하나 (0000)뿐이에요. +0, -0 문제가 사라졌어요!
3. 양수 범위: 0 ~ +7 (8개), 음수 범위: -8 ~ -1 (8개), 총 16개
4. 음수가 하나 더 많아요: -8은 있지만 +8은 없어요.

[예제] -8은 어떻게 만들어질까?
4비트에서 -8 = 1000인데, +8을 2의 보수로 변환하면 확인할 수 있어요.

+8 = 1000 (4비트에서는 넘치지만 계산해 보면)
비트 반전: 0111
+1: 0111 + 0001 = 1000

그런데 1000의 2의 보수를 구하면?
비트 반전: 0111
+1: 0111 + 0001 = 1000 (자기 자신!)

-8은 대응하는 +8이 4비트에 없는 특별한 수예요. 그래서 2의 보수에서 음수가 양수보다 하나 더 많아요.
[꿀팁] 2의 보수 빠르게 구하는 방법
오른쪽에서부터 첫 번째 1을 찾아요. 그 1과 그 오른쪽은 그대로 두고, 왼쪽 비트들만 뒤집으면 돼요!

예) +6 = 0110
오른쪽에서 첫 번째 1: 두 번째 자리의 1
그 1과 오른쪽(10)은 그대로, 왼쪽(01)만 뒤집기 -> 10
결과: 1010 = -6

예) +5 = 0101
오른쪽에서 첫 번째 1: 첫 번째 자리의 1
그 1(1)은 그대로, 왼쪽(010)만 뒤집기 -> 101
결과: 1011 = -5

2의 보수의 마법 -- 뺄셈이 덧셈이 된다!

[핵심] 2의 보수에서 뺄셈 = 덧셈
A - B = A + (-B)

B를 2의 보수로 변환하면 -B가 돼요. 그러면 뺄셈이 덧셈으로 바뀌어요!

이게 왜 대단하냐면, CPU에 '덧셈기(가산기)' 하나만 있으면 뺄셈도 할 수 있다는 뜻이거든요. 하드웨어가 훨씬 단순해져요!

실제로 계산해 봐요. 4비트에서 7 - 3 = 4를 2의 보수 덧셈으로 풀어 보겠어요.

자리3번(MSB)2번1번0번(LSB)
A = +70111
-B = -3 (2의 보수)1101
올림(carry)1110
합계(1)0100
결과 (캐리 버림)0100 = +4

이번에는 결과가 음수인 경우를 살펴봐요. 3 - 7 = -4를 계산해 보겠어요.

자리3번(MSB)2번1번0번(LSB)
A = +30011
-B = -7 (2의 보수)1001
올림(carry)0001
합계1100
결과1100 = -4

두 경우 모두 그냥 이진수 덧셈만 했을 뿐인데 정확한 답이 나왔어요! 이것이 2의 보수의 마법이에요. CPU 설계자들이 2의 보수를 사랑하는 이유가 바로 이거예요.

[예제] 2의 보수 덧셈 더 연습해 보기
(1) -3 + (-2) = ?
-3 = 1101, -2 = 1110
1101 + 1110 = 11011 -> 캐리 버림 -> 1011 = -5 (정답!)

(2) -1 + 1 = ?
-1 = 1111, +1 = 0001
1111 + 0001 = 10000 -> 캐리 버림 -> 0000 = 0 (정답!)

(3) 5 + (-3) = ?
+5 = 0101, -3 = 1101
0101 + 1101 = 10010 -> 캐리 버림 -> 0010 = +2 (정답!)

오버플로우 -- 범위를 넘어서면?

4비트 2의 보수로는 -8 ~ +7만 표현할 수 있어요. 그런데 만약 계산 결과가 이 범위를 넘어서면 어떻게 될까요? 이것을 오버플로우(overflow)라고 해요.

[핵심] 오버플로우란?
오버플로우(Overflow): 계산 결과가 표현 가능한 범위를 넘어서는 현상

4비트 2의 보수에서:
- 양수 오버플로우: 결과가 +7보다 커질 때
- 음수 오버플로우: 결과가 -8보다 작아질 때

오버플로우가 발생하면 결과가 완전히 엉뚱한 값이 돼요!

7 + 1 = 8이 되어야 하는데, 4비트에서는 +8을 표현할 수 없어서 -8이 되어 버렸어요! 양수 두 개를 더했는데 음수가 나온 거예요. 이것이 바로 양수 오버플로우예요.

음수 두 개를 더했는데 양수가 나왔어요! -5 + (-4) = -9인데, 4비트에서는 -9를 표현할 수 없어서 +7이 되어 버렸어요.

[핵심] 오버플로우 감지 규칙
2의 보수 덧셈에서 오버플로우를 감지하는 방법:

양수 + 양수 = 음수 -> 양수 오버플로우!
음수 + 음수 = 양수 -> 음수 오버플로우!

반대로 이런 경우는 오버플로우가 아니에요:
양수 + 음수 -> 절대 오버플로우 안 일어남!
음수 + 양수 -> 절대 오버플로우 안 일어남!

왜냐하면 양수와 음수를 더하면 결과가 둘 사이 어딘가에 있으므로 범위를 벗어날 수 없어요.
[비유] 주행거리계 오버플로우
자동차 주행거리계를 생각해 보세요. 숫자가 999,999 km에서 1 km를 더 가면 000,000으로 돌아가요. 가장 큰 수 다음에 가장 작은 수가 되는 거예요!

4비트 2의 보수도 같아요:
...+5 -> +6 -> +7 -> -8 -> -7 -> -6...

+7 다음이 -8로 돌아가요! 마치 원형 시계처럼 빙글빙글 돌아요.

원형 수직선 -- 오버플로우를 그림으로 이해하기

2의 보수를 원형 수직선(원형 넘버 라인)으로 그리면 오버플로우가 왜 일어나는지 직관적으로 이해할 수 있어요.

[그림] 4비트 2의 보수 원형 수직선
           0 (0000)

    (0001)/       \(1111)
    +2  /         \  -2

원형 수직선에서 보면, 수들이 동그랗게 이어져 있어요. +7 다음이 -8이고, -8 이전이 +7이에요. 1을 계속 더하면 0 -> 1 -> 2 -> ... -> 7 -> -8 -> -7 -> ... -> -1 -> 0으로 한 바퀴 돌아와요.

비트 수별 표현 범위

지금까지 4비트로 설명했지만, 실제 컴퓨터는 8비트, 16비트, 32비트, 64비트를 사용해요. 비트 수가 많아질수록 표현할 수 있는 범위가 기하급수적으로 늘어나요.

[핵심] N비트 2의 보수 표현 범위 공식

최솟값: -2^(N-1)
최댓값: +2^(N-1) - 1
총 개수: 2^N개

C 언어에서의 음수

이제 C 언어에서 음수를 어떻게 다루는지 알아봐요. C는 정수 타입마다 부호 있는(signed) 버전과 부호 없는(unsigned) 버전이 있어요.

[핵심] C 언어의 정수 타입과 부호
C에서 int는 기본적으로 signed예요. 즉 int와 signed int는 같아요.

음수를 쓸 필요가 없으면 unsigned를 붙여서 양수 범위를 두 배로 늘릴 수 있어요.

- char: 1바이트(8비트). signed: -128~+127, unsigned: 0~255
- short: 2바이트(16비트). signed: -32768~+32767
- int: 보통 4바이트(32비트). signed: 약 -21억~+21억
- long long: 8바이트(64비트). 아주 큰 범위

부호 확장 (Sign Extension)

4비트 수를 8비트로 확장하거나, 8비트를 32비트로 확장할 때는 어떻게 할까요? 양수는 앞에 0을 채우고, 음수는 앞에 1을 채워요. 이것을 부호 확장이라고 해요.

4비트10진수8비트 (부호 확장)검증
0101+500000101MSB=0이므로 0으로 채움
0011+300000011MSB=0이므로 0으로 채움
1011-511111011MSB=1이므로 1로 채움
1101-311111101MSB=1이므로 1로 채움
1000-811111000MSB=1이므로 1로 채움
0000000000000MSB=0이므로 0으로 채움

부호 확장의 핵심은 간단해요. MSB(부호 비트)를 복사해서 왼쪽 빈 자리를 채우면 돼요! 양수면 0으로, 음수면 1로 채워요. 이렇게 하면 수의 값은 변하지 않아요.

디버깅 훈련 -- 흔한 음수 관련 실수

[디버깅 1] unsigned 변수에 음수 대입
잘못된 코드:
unsigned int count = -1;
printf("%u\n", count);

출력: 4294967295

왜 이런 일이?
-1의 비트 패턴은 11111111 11111111 11111111 11111111이에요. unsigned로 해석하면 2^32 - 1 = 4294967295가 돼요.

교훈: unsigned 변수에 음수를 넣으면 컴파일러가 경고를 줘요. 경고를 무시하지 마세요!
[디버깅 2] 오버플로우를 무시한 계산
잘못된 코드:
short a = 30000;
short b = 30000;
short sum = a + b;
printf("%d\n", sum);

출력: -5536 (기대: 60000)

왜 이런 일이?
short는 16비트로 최대 +32767이에요. 60000은 범위를 넘어서 오버플로우가 발생해요.
60000 = 1110 1010 0110 0000 -> 2의 보수 해석 = -5536

교훈: 큰 값을 다룰 때는 int나 long long을 사용하세요!
[디버깅 3] signed/unsigned 비교 함정
잘못된 코드:
int x = -1;
unsigned int y = 0;
if (x < y) {
    printf("-1은 0보다 작다\n");
} else {
    printf("-1은 0보다 크거나 같다(?)\n");
}

출력: -1은 0보다 크거나 같다(?)

왜 이런 일이?
signed와 unsigned를 비교하면, C 언어는 signed를 unsigned로 변환해요. -1의 비트 패턴 = 4294967295(unsigned). 그래서 4294967295 > 0 이 되어 else 쪽으로 가요!

교훈: signed와 unsigned를 섞어서 비교하지 마세요! 컴파일러 경고를 잘 확인하세요.

역사 이야기 -- 2의 보수는 어떻게 표준이 되었을까?

[역사] 초기 컴퓨터와 음수 표현의 고민
1940년대 초기 컴퓨터 ENIAC(1945)은 10진수 기반이었어요. 음수는 10의 보수를 사용했는데, 이것은 2의 보수의 10진수 버전이에요.

1950~60년대에는 세 가지 방식이 모두 실제 컴퓨터에서 사용되었어요.
- 부호-크기: IBM 7090 (1959)
- 1의 보수: CDC 6600 (1964), UNIVAC 1100 시리즈
- 2의 보수: IBM System/360 (1964)

1964년 IBM System/360이 2의 보수를 채택하면서 대세가 기울었고, 이후 거의 모든 컴퓨터가 2의 보수를 사용하게 되었어요.
[역사] 폰 노이만과 2의 보수
존 폰 노이만(John von Neumann)은 1945년 EDVAC 보고서에서 2진수 체계와 2의 보수 사용을 강력히 추천했어요.

그의 주요 논점:
1. 2진수는 전자 회로(켜짐/꺼짐)와 자연스럽게 맞아요.
2. 2의 보수를 쓰면 덧셈기 하나로 뺄셈도 할 수 있어요.
3. 하드웨어가 단순해지면 컴퓨터를 더 싸고 안정적으로 만들 수 있어요.

이 결정이 오늘날 모든 컴퓨터의 기초가 되었어요!
[역사] 2의 보수가 이긴 진짜 이유
2의 보수가 다른 방식을 이긴 결정적인 이유는 하드웨어 비용이었어요.

1. 덧셈기 하나면 충분: 뺄셈기를 따로 만들지 않아도 돼요. 비트 반전 회로(NOT 게이트)와 +1만 추가하면 뺄셈이 가능해요.
2. 0이 하나: +0, -0을 구분하는 추가 회로가 필요 없어요.
3. 비교가 쉬움: 뺄셈한 결과의 MSB만 보면 대소 비교가 돼요.

1960년대에는 트랜지스터 하나가 매우 비쌌기 때문에, 회로를 줄이는 것이 곧 돈을 아끼는 것이었어요!

KOI 실전 문제

[KOI 문제 1] 정수 범위 판단
문제: 두 정수 A, B가 주어질 때, A + B의 결과를 int(32비트 signed)에 안전하게 저장할 수 있는지 판단하세요. 안전하면 합을, 아니면 "OVERFLOW"를 출력하세요.
- 입력 범위: -2,147,483,648 <= A, B <= 2,147,483,647

핵심 아이디어:
- A와 B를 long long으로 받아서 더한 뒤, INT_MAX보다 크거나 INT_MIN보다 작으면 오버플로우예요.

[KOI 문제 3] unsigned 주의 문제
문제: 배열의 크기 N이 주어지고, N개의 원소를 역순으로 출력하세요. 단, N은 unsigned int로 선언되어 있어요.

흔한 실수:
unsigned int i;
for (i = N - 1; i >= 0; i--) {  /* 무한 루프! */
    printf("%d ", arr[i]);
}

unsigned int는 항상 >= 0이므로 i >= 0 조건이 항상 참이에요! i가 0에서 1을 빼면 4294967295가 되어 무한 루프에 빠져요.

올바른 방법:
unsigned int i;
for (i = N; i > 0; i--) {
    printf("%d ", arr[i - 1]);
}

용어 정리

용어설명
2의 보수 (Two's Complement)이진수의 모든 비트를 반전(1의 보수)한 뒤 1을 더해서 음수를 표현하는 방식. 현대 컴퓨터의 표준 음수 표현법이에요.
1의 보수 (One's Complement)이진수의 모든 비트를 반전(0->1, 1->0)해서 음수를 표현하는 방식. 0이 두 개인 단점이 있어요.
부호-크기 (Sign-Magnitude)MSB를 부호 비트로 사용하고 나머지를 크기로 해석하는 방식. 직관적이지만 덧셈이 복잡해요.
MSB (Most Significant Bit)가장 왼쪽의 비트(최상위 비트). 2의 보수에서는 부호를 나타내요. 0이면 양수, 1이면 음수.
오버플로우 (Overflow)계산 결과가 표현 가능한 범위를 넘어서는 현상. 양수+양수=음수, 음수+음수=양수가 되면 오버플로우예요.
부호 확장 (Sign Extension)작은 비트 수의 수를 큰 비트 수로 확장할 때, MSB(부호 비트)로 빈 자리를 채우는 것.
signed / unsignedsigned는 부호 있는 정수(양수+음수), unsigned는 부호 없는 정수(양수만). C 언어에서 int는 기본적으로 signed예요.
보수 (Complement)'보충해서 채우는 수'라는 뜻. 어떤 수와 그 보수를 더하면 특정 기준값(1의 보수: 모두 1, 2의 보수: 2^N)이 돼요.
