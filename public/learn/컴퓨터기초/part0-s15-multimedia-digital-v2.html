사진, 음악, 영상 -- 전부 0과 1이라고?

여러분이 매일 보는 사진, 듣는 음악, 시청하는 영상 -- 이 모든 것은 컴퓨터 안에서 0과 1의 나열로 저장되어 있어요. 숫자나 문자를 0과 1로 바꾸는 것은 S-13, S-14에서 배웠죠? 그런데 형형색색의 그림, 아름다운 선율, 움직이는 영상까지 0과 1로 바꿀 수 있다니, 정말 신기하지 않나요?

[생각해 보기] 0과 1로 바뀌는 것들
1. 사진 -- 풍경 사진 한 장이 수백만 개의 숫자로 이루어져 있어요.
2. 음악 -- 3분짜리 노래 하나에 약 3천만 개의 숫자가 들어 있어요.
3. 영상 -- 1분 영상에는 수십억 개의 숫자가 필요해요!
4. 전화 통화 -- 목소리도 실시간으로 0과 1로 변환돼요.

어떻게 이런 일이 가능한 걸까요? 지금부터 하나씩 알아봐요!

이미지 디지털화 -- 그림을 숫자로!

디지털 사진은 어떻게 만들어질까요? 핵심은 픽셀(Pixel)이에요. Pixel은 Picture Element(그림 요소)의 줄임말이에요. 이미지를 아주 작은 네모칸으로 쪼개면, 각 칸이 하나의 픽셀이에요.

[핵심] 픽셀 (Pixel) = Picture + Element
디지털 이미지는 아주 작은 네모칸(픽셀)으로 이루어져 있어요. 각 픽셀은 하나의 색을 가지고 있어요.

모자이크 타일을 떠올려 보세요! 멀리서 보면 하나의 그림이지만, 가까이 가면 작은 타일 조각들이 모여 있는 거예요. 디지털 이미지도 같은 원리예요.
열1열2열3열4
행11111
행21001
행31001
행41111

위 표는 4x4 = 16픽셀짜리 아주 작은 이미지예요. 가운데가 검정(0)이고 테두리가 흰색(1)인 네모 상자 모양이에요! 이렇게 각 픽셀에 숫자를 부여하면 그림을 데이터로 저장할 수 있어요.

해상도 -- 픽셀이 많을수록 선명해!

해상도(Resolution)란 이미지를 이루는 픽셀의 수를 말해요. 가로 픽셀 수 x 세로 픽셀 수로 표현하죠. 픽셀이 많을수록 더 세밀하고 선명한 이미지를 만들 수 있어요.

[비유] 해상도 = 모자이크 타일의 수
같은 크기의 벽에 모자이크를 만든다고 생각해 보세요.

- 타일 100개로 만든 그림 -> 투박하고 거친 느낌
- 타일 10,000개로 만든 그림 -> 훨씬 정밀하고 세밀
- 타일 1,000,000개로 만든 그림 -> 거의 사진 같아!

해상도가 높을수록 타일(픽셀)이 작아져서 더 세밀한 그림을 그릴 수 있어요.

색상 표현 -- RGB 모델

흑백 이미지는 0(검정)과 1(흰색)만 있으면 돼요. 하지만 컬러 이미지는 어떻게 표현할까요? 바로 RGB 모델을 사용해요!

[핵심] RGB = Red + Green + Blue
화면의 모든 색은 빨강(Red), 초록(Green), 파랑(Blue) 세 가지 빛을 섞어서 만들어요.

각 색상은 0~255 사이의 값을 가져요.
- 0 = 그 색의 빛이 전혀 없음
- 255 = 그 색의 빛이 최대

세 색을 모두 최대로 켜면 흰색, 모두 끄면 검정이에요.

색 깊이(Color Depth)

색 깊이란 하나의 픽셀을 표현하는 데 사용하는 비트 수예요. 비트 수가 많을수록 더 많은 색을 표현할 수 있어요.

색 깊이비트 수표현 가능 색상 수설명
1비트12 (흑백)검정 또는 흰색만
2비트24검정, 진한 회색, 연한 회색, 흰색
4비트416기본 16색 (옛날 컴퓨터)
8비트8256256색 (GIF 이미지)
16비트1665,536High Color
24비트24 (R:8+G:8+B:8)16,777,216True Color (약 1,677만)
32비트32 (R:8+G:8+B:8+A:8)16,777,216 + 투명도True Color + 알파 채널
[예제] 24비트 컬러는 왜 RGB 각 8비트일까?
24비트 = R에 8비트 + G에 8비트 + B에 8비트

8비트로 표현할 수 있는 범위: 0 ~ 2^8-1 = 0 ~ 255
-> 빨강 256단계, 초록 256단계, 파랑 256단계
-> 총 조합: 256 x 256 x 256 = 16,777,216가지

사람의 눈이 구분할 수 있는 색은 약 1,000만 가지라고 해요. 24비트 컬러면 사람이 구분할 수 있는 것 이상이에요!

이미지가 저장되는 과정

[예제] 한 픽셀의 주황색 저장 과정
주황색 픽셀: (R=255, G=165, B=0)

R: 255 -> 11111111 (8비트)
G: 165 -> 10100101 (8비트)
B: 0 -> 00000000 (8비트)

저장되는 데이터: 11111111 10100101 00000000 (총 24비트 = 3바이트)

이런 식으로 모든 픽셀이 각각 3바이트로 저장돼요!

이미지 압축 -- 파일 크기를 줄이는 마법

무압축 이미지는 너무 커서 실제로는 거의 사용하지 않아요. 이미지 파일을 작게 만드는 두 가지 방법이 있어요.

[핵심] 비손실 vs 손실 압축
비손실 압축 (Lossless): 압축 후 풀면 원본과 100% 동일. PNG, GIF, BMP(무압축)
- 같은 색 반복, 패턴을 찾아 줄여요
- 한 픽셀도 변하지 않아요!

손실 압축 (Lossy): 원본과 약간 달라지지만 파일이 훨씬 작아요. JPEG, WebP
- 사람 눈이 잘 느끼지 못하는 미세한 차이를 제거해요
- 압축률을 높이면 화질이 떨어져요

소리 디지털화 -- 파동을 숫자로!

소리는 공기의 진동이에요. 기타 줄을 튕기면 줄이 떨리고, 그 떨림이 공기를 통해 여러분의 귀까지 전달되죠. 이 진동은 파동(wave)의 형태를 갖고 있어요.

[비유] 소리 = 물결 파동
연못에 돌을 던지면 동그란 물결이 퍼져나가죠? 소리도 이와 비슷해요. 스피커의 진동이 공기를 통해 물결처럼 퍼져나가는 거예요.

- 파동의 높이(진폭) = 소리의 크기 (클수록 시끄러움)
- 파동의 빈도(주파수) = 소리의 높낮이 (높을수록 높은 음)

소리 파동은 연속적인 아날로그 신호예요. 이것을 컴퓨터에 저장하려면 디지털로 바꿔야 해요. 여기서 핵심 개념 두 가지가 등장해요: 샘플링(Sampling)과 양자화(Quantization).

샘플링 -- 일정 간격으로 높이 측정

[핵심] 샘플링(Sampling) = 일정 간격으로 측정
연속적인 소리 파동을 일정한 시간 간격으로 잘라서 그 순간의 높이(세기)를 기록하는 것이에요.

1초에 몇 번 측정하는지를 샘플링 레이트(Sampling Rate)라고 해요.
단위: Hz (헤르츠) = 1초에 몇 번

양자화 -- 높이를 단계로 나누기

샘플링으로 시간 간격을 정했으면, 다음은 각 측정값의 높이를 몇 단계로 나눌지 정해야 해요. 이것을 양자화(Quantization)라고 해요. 단계를 결정하는 것이 바로 비트 깊이(Bit Depth)예요.

비트 깊이양자화 단계 수설명
1비트2단계큰 소리 / 작은 소리만 구분
4비트16단계매우 거친 소리
8비트256단계전화 품질, 옛날 게임 효과음
16비트65,536단계CD 품질 -- 표준
24비트16,777,216단계스튜디오 녹음 품질
32비트4,294,967,296단계전문 오디오 작업용
[핵심] 나이퀴스트 정리 (Nyquist Theorem)
원래 소리를 충실하게 재현하려면, 원래 소리의 최대 주파수의 2배 이상으로 샘플링해야 해요.

사람이 들을 수 있는 최대 주파수: 약 20,000Hz
필요한 최소 샘플링 레이트: 20,000 x 2 = 40,000Hz
CD 표준: 44,100Hz (여유를 두고 설정)

만약 이보다 낮은 레이트로 샘플링하면, 원래 없던 이상한 소리(에일리어싱)가 생겨요!

[핵심] 오디오 파일 크기 공식

- 채널 수: 모노=1, 스테레오=2
- 시간: 초 단위
- 나누기 8: 비트를 바이트로 변환

영상 디지털화 -- 이미지 + 소리 + 시간

영상(비디오)은 무엇일까요? 간단히 말하면, 이미지를 빠르게 연속으로 보여주면서 소리를 함께 재생하는 거예요. 옛날 만화영화를 떠올려 보세요 -- 종이에 조금씩 다른 그림을 그려서 빠르게 넘기면 움직이는 것처럼 보이죠!

[핵심] 영상의 구성 요소
영상 = 이미지(프레임) + 소리(오디오) + 시간

- 프레임(Frame): 영상을 이루는 한 장 한 장의 이미지
- FPS (Frames Per Second): 1초에 보여주는 프레임 수
- FPS가 높을수록 움직임이 부드러워요
FPS1초당 프레임 수용도특징
15fps15장초기 인터넷 영상, 감시 카메라약간 끊기는 느낌
24fps24장영화관 영화영화 특유의 감성적 느낌
30fps30장TV 방송, 유튜브자연스러운 움직임
60fps60장게임, 스포츠 중계매우 부드러운 움직임
120fps120장슬로모션 촬영, 고급 게임초부드러운 화면
240fps240장초슬로모션아주 느린 동작 재현
[비유] 플립북(넘겨 보는 만화)
공책 모서리에 조금씩 다른 그림을 그리고 빠르게 넘겨 본 적 있나요?

- 한 페이지 = 한 프레임
- 넘기는 속도 = FPS
- 천천히 넘기면(낮은 FPS) -> 끊겨 보임
- 빠르게 넘기면(높은 FPS) -> 부드럽게 움직이는 것처럼 보여요!

사람의 눈은 약 24fps 이상이면 자연스러운 움직임으로 인식해요.

영상 파일 크기 계산

[핵심] 영상 파일 크기 공식 (무압축)
영상 크기 = (가로 x 세로 x 색깊이 / 8) x FPS x 시간(초) + 오디오 크기

즉, 한 프레임(이미지)의 크기 x 초당 프레임 수 x 총 시간에 오디오를 더하면 돼요.

영상 압축 -- 코덱의 마법

영상 압축 알고리즘을 코덱(Codec)이라고 해요. Codec은 Coder(압축) + Decoder(해제)의 합성어예요. 코덱은 두 가지 핵심 기법을 사용해요.

[핵심] 영상 압축의 두 가지 기법
1. 프레임 내 압축 (공간적 압축):
한 프레임 안에서 JPEG처럼 비슷한 색 영역을 줄여요.

2. 프레임 간 압축 (시간적 압축):
연속된 프레임 사이에서 변하지 않는 부분은 다시 저장하지 않아요!
예: 배경이 같으면 변한 부분(움직이는 사람)만 저장.

영상은 대부분 연속 프레임이 비슷하기 때문에 이 기법이 매우 효과적이에요.

C 코드 1: RGB 밝기(그레이스케일) 계산

컬러 이미지를 흑백으로 바꾸려면, RGB 값을 하나의 밝기 값으로 합쳐야 해요. 가장 간단한 방법은 세 값의 평균을 내는 것이고, 좀 더 정확한 방법은 사람의 눈이 초록에 더 민감한 점을 반영하는 거예요.

CRGB 값을 입력받아 그레이스케일(밝기)로 변환
1#include <stdio.h>
2
3int main(void) {
4 int r, g, b;
5
6 printf("RGB 값을 입력하세요 (0~255)\n");
7 printf("R: ");
8 scanf("%d", &r);
9 printf("G: ");
10 scanf("%d", &g);
11 printf("B: ");
12 scanf("%d", &b);
13
14 /* 범위 확인 */
15 if (r < 0 || r > 255 || g < 0 || g > 255 || b < 0 || b > 255) {
16 printf("오류: RGB 값은 0~255 범위여야 합니다!\n");
17 return 1;
18 }
19
20 /* 방법 1: 단순 평균 */
21 int gray_avg = (r + g + b) / 3;
22
23 /* 방법 2: 가중 평균 (사람 눈의 민감도 반영) */
24 /* R: 0.299, G: 0.587, B: 0.114 */
25 int gray_weighted = (int)(0.299 * r + 0.587 * g + 0.114 * b);
26
27 printf("\n--- 결과 ---\n");
28 printf("입력 RGB: (%d, %d, %d)\n", r, g, b);
29 printf("단순 평균 밝기: %d\n", gray_avg);
30 printf("가중 평균 밝기: %d\n", gray_weighted);
31
32 if (gray_weighted < 64) {
33 printf("-> 어두운 색이에요.\n");
34 } else if (gray_weighted < 192) {
35 printf("-> 중간 밝기 색이에요.\n");
36 } else {
37 printf("-> 밝은 색이에요.\n");
38 }
39
40 return 0;
41}
실행 결과
RGB 값을 입력하세요 (0~255) R: 255 G: 165 B: 0 --- 결과 --- 입력 RGB: (255, 165, 0) 단순 평균 밝기: 140 가중 평균 밝기: 173 -> 중간 밝기 색이에요.
[왜 가중 평균?]
사람의 눈은 초록에 가장 민감하고, 파랑에 가장 둔해요.
그래서 밝기를 계산할 때 R:G:B = 0.299:0.587:0.114 비율로 가중치를 줘요.

주황색 (255,165,0)의 경우:
- 단순 평균: (255+165+0)/3 = 140
- 가중 평균: 0.299x255 + 0.587x165 + 0.114x0 = 173

초록 성분(165)에 가중치가 더 크니까 가중 평균이 더 높게 나와요.

원인: 1920 x 1080 x 24 = 49,766,400. 이 값을 8로 나누기 전에 이미 int 범위를 넘어 정수 오버플로우가 발생해요! int는 약 21억까지만 저장 가능한데, 1920 x 1080 x 24 = 49,766,400은 괜찮지만 더 큰 해상도에서는 위험해요. 안전하게 long long을 사용하세요.

수정: long long file_size = (long long)width * height * color_depth / 8;
[디버깅 2] RGB 범위 초과
문제: 밝기를 높이려고 RGB에 50을 더했는데 이상한 색이 나와요.

int r = 230, g = 200, b = 180;
r = r + 50; /* 280! */
g = g + 50; /* 250 */
b = b + 50; /* 230 */
printf("밝게: (%d, %d, %d)\n", r, g, b);

출력: 밝게: (280, 250, 230)

원인: R이 280이 되었어요! RGB는 0~255 범위인데 이를 넘어버렸어요. 실제 이미지 처리에서는 255보다 큰 값은 255로 잘라야 해요(클램핑).

수정: r = r + 50;
if (r > 255) r = 255; /* 클램핑 */
[디버깅 3] 영상 크기 계산에서 정수 오버플로우
문제: 4K 무압축 영상 1분 크기를 계산하는데 음수가 나와요!

int frame_size = 3840 * 2160 * 24 / 8; /* 한 프레임 */
int one_min = frame_size * 30 * 60; /* 30fps x 60초 */
printf("1분 크기: %d바이트\n", one_min);

원인: frame_size = 24,883,200 (약 2,400만). 여기에 30 x 60 = 1,800을 곱하면 약 447억이 되어 int 범위(약 21억)를 훨씬 넘어요!

수정: long long frame_size = (long long)3840 * 2160 * 24 / 8;
long long one_min = frame_size * 30 * 60;
printf("1분 크기: %lld바이트\n", one_min);

역사 속 멀티미디어 디지털화

[역사] 최초의 디지털 이미지 (1957년)
1957년, 미국 국립표준기술연구소(NIST)의 러셀 커쉬(Russell Kirsch)가 자신의 아들 사진을 세계 최초로 디지털 이미지로 스캔했어요. 해상도는 겨우 176 x 176 픽셀, 흑백 이미지였어요.

176 x 176 = 30,976 픽셀! 요즘 스마트폰 사진의 수천만 픽셀에 비하면 아주 작지만, 이것이 디지털 이미지의 역사적인 첫걸음이었어요.
[역사] CD의 발명 (1982년, 소니 + 필립스)
1982년, 일본의 소니(Sony)와 네덜란드의 필립스(Philips)가 함께 CD(Compact Disc)를 발명했어요. 44,100Hz 샘플링, 16비트, 스테레오 -- 이 표준은 40년이 지난 지금도 사용되고 있어요!

CD 한 장의 용량: 약 700MB -> 약 80분의 음악 저장 가능.
LP(비닐 레코드)와 달리 복사해도 음질 저하가 없어서 음악 산업을 완전히 바꿔 놓았어요.
[역사] JPEG 표준 (1992년)
JPEG는 Joint Photographic Experts Group의 약자로, 1992년에 국제 표준으로 제정되었어요. 사진을 효율적으로 압축할 수 있게 되면서 인터넷에서 이미지 공유가 폭발적으로 늘어났어요.

JPEG가 없었다면 인터넷에서 사진 한 장을 다운받는 데 몇 분씩 걸렸을 거예요!
[역사] MP3의 탄생 (1993년)
1993년, 독일 프라운호퍼 연구소에서 MP3(MPEG-1 Audio Layer 3) 기술을 완성했어요. 30MB짜리 WAV 파일을 3MB로 줄이면서도 대부분의 사람이 음질 차이를 느끼지 못할 정도였어요.

이 기술은 1990년대 후반 인터넷 음악 공유 혁명을 일으켰고, 아이팟(iPod) 같은 휴대용 음악 플레이어의 탄생으로 이어졌어요.

KOI 도전 문제

[KOI 문제 1] 이미지 저장 용량
디지털 카메라로 24비트 컬러 사진을 찍는다. 카메라의 해상도가 4000 x 3000 픽셀일 때, 무압축 BMP 파일 하나의 크기는 몇 MB인가? (1MB = 1,000,000바이트로 계산)

풀이:
파일 크기 = 4000 x 3000 x 24 / 8
= 4000 x 3000 x 3
= 36,000,000바이트
= 36MB

답: 36MB
[KOI 문제 2] 오디오 저장 용량
CD 품질(44,100Hz, 16비트, 스테레오)로 5분 동안 녹음한 무압축 WAV 파일의 크기를 MB 단위로 구하시오. (1MB = 1,000,000바이트로 계산, 소수 첫째 자리까지)

풀이:
파일 크기 = 44,100 x 16 x 2 x 300 / 8
= 44,100 x 16 x 2 x 300 / 8
= 44,100 x 1,200
= 52,920,000바이트
= 52.9MB

답: 52.9MB
[KOI 문제 3] 영상 전송 시간
FHD(1920x1080) 24비트 컬러 영상을 30fps로 10초간 촬영했다. 이 무압축 영상 파일을 100Mbps 인터넷으로 전송하려면 몇 초가 걸리는가? (소수 첫째 자리까지)

풀이:
1) 한 프레임 크기 = 1920 x 1080 x 24 / 8 = 6,220,800바이트
2) 총 프레임 수 = 30 x 10 = 300프레임
3) 영상 크기 = 6,220,800 x 300 = 1,866,240,000바이트
4) 비트로 변환 = 1,866,240,000 x 8 = 14,929,920,000비트
5) 전송 시간 = 14,929,920,000 / 100,000,000 = 149.3초

답: 149.3초 (약 2분 29초)

용어 정리

용어설명
픽셀 (Pixel)Picture Element의 줄임말. 디지털 이미지를 이루는 가장 작은 단위의 네모칸이에요. 각 픽셀은 하나의 색 정보를 가지고 있어요.
해상도 (Resolution)이미지를 이루는 픽셀의 수. 가로 x 세로로 표현해요. 해상도가 높을수록 이미지가 선명해요.
RGBRed(빨강), Green(초록), Blue(파랑)의 약자. 빛의 3원색을 섞어 모든 색을 표현하는 방식이에요. 각 색은 0~255의 값을 가져요.
색 깊이 (Color Depth)한 픽셀의 색을 표현하는 데 사용하는 비트 수. 24비트면 약 1,677만 색을 표현할 수 있어요.
샘플링 (Sampling)연속적인 아날로그 신호를 일정 간격으로 측정해서 디지털 값으로 바꾸는 과정이에요. 소리 디지털화의 첫 단계예요.
샘플링 레이트 (Sampling Rate)1초에 몇 번 샘플링하는지를 나타내는 값. 단위는 Hz(헤르츠). CD 표준은 44,100Hz예요.
코덱 (Codec)Coder + Decoder의 합성어. 영상이나 오디오를 압축(인코딩)하고 재생(디코딩)하는 알고리즘이에요. H.264, H.265가 대표적이에요.
FPS (Frames Per Second)1초에 보여주는 이미지(프레임)의 수. 영화는 24fps, 게임은 보통 60fps를 사용해요. 높을수록 부드러워요.

한눈에 보기

멀티미디어 디지털화 -- 한눈에 보기
1. 아날로그(연속)를 디지털(불연속)로 바꾸는 것이 디지털화예요. 경사로를 계단으로 바꾸는 것과 같아요.
2. 이미지: 픽셀(작은 네모칸)로 나누고, 각 픽셀의 색을 RGB(빨강, 초록, 파랑) 숫자로 저장해요.
3. 이미지 크기 = 가로 x 세로 x 색깊이 / 8. FHD 24비트 사진 한 장은 무압축으로 약 5.9MB예요.
4. 이미지 압축: PNG(비손실, 원본 보존) vs JPEG(손실, 크기 매우 작음). 용도에 맞게 선택하세요!
5. 소리: 파동을 샘플링(일정 간격 측정) + 양자화(단계화)해서 숫자로 바꿔요.
6. 오디오 크기 = 샘플링레이트 x 비트깊이 x 채널 x 시간 / 8. CD 품질 3분은 약 30MB예요.
7. 오디오 압축: WAV(무압축) vs MP3(손실, 1/10) vs FLAC(비손실, 1/2). MP3가 가장 널리 쓰여요.
8. 영상 = 이미지(프레임) + 소리 + 시간. FPS가 높을수록 부드럽지만 파일도 커져요.
9. 영상 압축(코덱): H.264, H.265 등이 프레임 간 중복을 제거해서 1/50~1/100로 줄여요.
10. 멀티미디어 파일 크기 계산에서 큰 수를 다룰 때는 long long을 사용하세요!

10축 평가

[10축 매핑] 이 단원에서 기른 능력
1. 수리/논리력: 파일 크기 공식을 유도하고 다양한 조건에서 직접 계산했어요.
2. 추상화: 연속적인 아날로그 신호를 불연속적인 디지털 데이터로 모델링하는 과정을 이해했어요.
3. 패턴 인식: 이미지, 소리, 영상 디지털화에서 공통 패턴(샘플링+양자화+이진수 변환)을 발견했어요.
4. 분해: 영상을 이미지+소리+시간으로, 이미지를 픽셀+색상으로, 색상을 RGB로 분해했어요.
5. 알고리즘: 비손실/손실 압축 알고리즘의 원리와 차이를 이해했어요.
6. 데이터 표현: RGB, 샘플링, 양자화 등 다양한 멀티미디어 데이터 표현 방식을 배웠어요.
7. 자료 구조: 2D 배열로 이미지(픽셀 그리드)를 표현하는 방법을 실습했어요.
8. 디버깅: 정수 오버플로우, RGB 범위 초과 등 멀티미디어 프로그래밍의 흔한 버그를 잡았어요.
9. 컴퓨팅 사고: 일상의 사진, 음악, 영상이 어떻게 0과 1로 변환되는지 체계적으로 분석했어요.
10. 실생활 연결: 스마트폰 사진, 유튜브 영상, 음악 스트리밍 등 매일 사용하는 기술의 원리를 이해했어요.
