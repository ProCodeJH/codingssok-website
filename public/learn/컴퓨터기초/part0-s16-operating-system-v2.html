UNIT S-16 운영체제 기초

운영체제가 없으면 어떻게 될까?

여러분은 컴퓨터를 켜면 당연히 바탕화면이 나오고, 마우스로 클릭하면 프로그램이 실행되고, 파일을 저장하면 나중에 다시 열 수 있다고 생각하죠? 하지만 이 모든 것은 운영체제(Operating System, OS)가 뒤에서 열심히 일하고 있기 때문에 가능한 거예요.

[생각해 보기] 운영체제가 없는 컴퓨터
운영체제가 없다면 이런 일이 벌어져요:

1. 화면에 아무것도 나타나지 않아요 -- 그래픽을 관리할 프로그램이 없거든요.
2. 키보드, 마우스가 작동하지 않아요 -- 입력 장치를 관리할 프로그램이 없거든요.
3. 프로그램을 실행할 수 없어요 -- 프로그램을 메모리에 올려줄 관리자가 없거든요.
4. 파일을 저장하거나 열 수 없어요 -- 파일 시스템이 없거든요.
5. 여러 프로그램을 동시에 사용할 수 없어요 -- CPU 시간을 나눠줄 관리자가 없거든요.

운영체제는 우리가 의식하지 못하는 사이에 이 모든 것을 해주고 있어요!

운영체제란 무엇일까?

운영체제(Operating System, OS)는 컴퓨터의 하드웨어와 소프트웨어를 관리하는 시스템 소프트웨어예요. 쉽게 말하면, 컴퓨터의 모든 부품(CPU, 메모리, 하드디스크, 키보드 등)과 프로그램 사이에서 중재자 역할을 하는 거예요.

[비유] 학교의 교장 선생님
운영체제를 학교에 비유하면 교장 선생님(또는 교무 매니저)과 같아요.

- 교실 배정 = 메모리 관리 (어떤 프로그램이 어디 앉을지 정해줘요)
- 시간표 관리 = CPU 스케줄링 (어떤 프로그램이 언제 실행될지 정해줘요)
- 서류함 관리 = 파일 시스템 (파일을 어디에 저장할지 관리해요)
- 방송 시스템 = 입출력 관리 (키보드, 모니터 등을 관리해요)
- 교무실 창구 = 사용자 인터페이스 (학생들이 교장 선생님께 요청하는 창구예요)

교장 선생님이 없으면 학교가 엉망이 되겠죠? OS가 없으면 컴퓨터도 마찬가지예요!

운영체제의 5가지 역할

운영체제는 크게 5가지 중요한 역할을 해요. 이 역할들이 모두 잘 이루어져야 컴퓨터가 정상적으로 동작해요.

역할비유관리 대상없으면 어떻게 되나?
프로세스 관리시간표 관리 선생님CPU, 프로세스프로그램이 하나만 실행 가능
메모리 관리교실 배정 담당RAM프로그램끼리 메모리 충돌
파일 시스템도서관 사서HDD/SSD파일 저장/검색 불가능
입출력 관리교무실 안내 데스크키보드, 모니터 등장치 사용 불가
사용자 인터페이스교무실 민원 창구화면, 명령어컴퓨터에 명령할 방법 없음
[핵심] 운영체제 = 자원 관리자
운영체제의 가장 중요한 역할은 자원(Resource) 관리예요.

컴퓨터의 자원은 한정되어 있어요:
- CPU는 하나(또는 몇 개)인데 프로그램은 수십 개가 실행돼요.
- RAM은 16GB인데 모든 프로그램이 메모리를 원해요.
- 하드디스크는 공간이 한정되어 있어요.

운영체제는 이 한정된 자원을 여러 프로그램에게 공평하고 효율적으로 나눠주는 관리자예요!

운영체제의 종류

운영체제는 하나만 있는 게 아니에요! 컴퓨터 종류와 용도에 따라 다양한 운영체제가 있어요. 가장 많이 쓰이는 데스크톱 운영체제 3가지를 먼저 비교해 볼게요.

프로세스와 프로그램의 차이

운영체제를 이해하려면 프로그램(Program)과 프로세스(Process)의 차이를 꼭 알아야 해요. 이 둘은 비슷해 보이지만 전혀 달라요!

[핵심] 프로그램 vs 프로세스
프로그램(Program) = 하드디스크에 저장된 파일. 아직 실행되지 않은 코드 덩어리.
비유: 요리 레시피(종이에 적힌 것). 가만히 있으면 아무 일도 안 일어나요.

프로세스(Process) = 메모리에 올라와서 실행 중인 프로그램. 살아 움직이는 것.
비유: 실제로 요리하는 행위. 재료를 꺼내고, 불을 켜고, 볶고 있는 상태예요.

같은 레시피로 여러 번 요리할 수 있듯이, 같은 프로그램을 여러 번 실행하면 여러 개의 프로세스가 만들어져요!

멀티태스킹 -- 동시에 여러 프로그램 실행하기

여러분은 음악을 들으면서 인터넷을 하고, 동시에 카카오톡도 하죠? 이렇게 여러 프로그램을 동시에 실행하는 것을 멀티태스킹(Multitasking)이라고 해요. 그런데 정말 '동시에' 실행되는 걸까요?

[핵심] 사실은 번갈아 실행! (시분할, Time Sharing)
CPU 코어 하나는 한 순간에 하나의 프로세스만 실행할 수 있어요.

그런데 어떻게 여러 프로그램이 '동시에' 돌아가는 것처럼 보일까요?

비밀은 시분할(Time Sharing)이에요! OS가 CPU 시간을 아주 짧게 쪼개서(예: 0.01초) 프로세스들에게 번갈아 나눠주는 거예요. 너무 빨리 바뀌어서 사람 눈에는 동시에 돌아가는 것처럼 보여요!
시간0~10ms10~20ms20~30ms30~40ms40~50ms50~60ms
CPU 실행프로세스 A프로세스 B프로세스 C프로세스 A프로세스 B프로세스 C
A 상태실행!준비준비실행!준비준비
B 상태준비실행!준비준비실행!준비
C 상태준비준비실행!준비준비실행!

위 표를 보세요. 0~10ms 동안은 A가 실행되고, 10~20ms 동안은 B가 실행되고, 20~30ms 동안은 C가 실행돼요. 30ms 만에 A, B, C가 한 번씩 실행되었어요. 이 과정이 1초에 수십~수백 번 반복되니까, 사람 눈에는 세 프로그램이 동시에 돌아가는 것처럼 보이는 거예요!

CPU 스케줄링 알고리즘

CPU는 하나인데 프로세스는 여러 개예요. 누구를 먼저 실행할까요? 이 순서를 정하는 규칙을 CPU 스케줄링 알고리즘이라고 해요. 대표적인 3가지를 알아볼게요.

알고리즘영어 이름방식비유장점단점
선입선출FCFS (First Come First Served)먼저 온 순서대로 실행은행 번호표 시스템단순하고 공평긴 작업이 앞에 있으면 뒤가 오래 기다림
최단작업 우선SJF (Shortest Job First)실행 시간이 짧은 것부터마트 소량 계산대평균 대기 시간 최소실행 시간을 미리 알기 어려움
돌아가며(라운드 로빈)Round Robin정해진 시간만큼만 실행하고 교대릴레이 경주/놀이기구 줄모든 프로세스가 골고루 실행교대 비용(오버헤드) 발생

[비유] 타임 퀀텀 = 놀이기구 탑승 시간
Round Robin에서 타임 퀀텀(Time Quantum)은 한 프로세스가 CPU를 사용할 수 있는 최대 시간이에요.

놀이기구에 비유하면, 한 사람이 탈 수 있는 시간이 정해져 있는 거예요.
- 타임 퀀텀이 너무 짧으면: 자리 바꾸는 시간(오버헤드)이 너무 커요.
- 타임 퀀텀이 너무 길면: FCFS와 다를 바 없어요.
- 적당한 타임 퀀텀(보통 10~100ms)이 중요해요!

메모리 관리

여러 프로그램이 동시에 실행되려면 모두 RAM에 올라가야 해요. 하지만 RAM 공간은 한정되어 있죠. 운영체제는 이 한정된 메모리를 여러 프로세스에게 나눠주고 관리하는 역할을 해요.

커널(Kernel) -- OS의 핵심

커널(Kernel)은 운영체제의 가장 핵심적인 부분이에요. '커널'은 영어로 '핵심, 알맹이'라는 뜻이에요. 커널이 하드웨어를 직접 제어하고, 프로세스와 메모리를 관리해요.

[핵심] 커널 모드 vs 사용자 모드
CPU는 두 가지 모드에서 동작해요:

커널 모드 (Kernel Mode)
- OS 커널이 실행되는 모드
- 하드웨어에 직접 접근할 수 있어요
- 모든 메모리 영역에 접근 가능
- 비유: 학교 교장 선생님 -- 모든 곳에 출입 가능

사용자 모드 (User Mode)
- 일반 프로그램이 실행되는 모드
- 하드웨어에 직접 접근할 수 없어요
- 자기 메모리 영역만 사용 가능
- 비유: 학생 -- 교무실에는 허락 없이 못 들어감

왜 이렇게 나눌까요? 일반 프로그램이 하드웨어를 마음대로 건드리면 시스템이 망가질 수 있으니까요!

그렇다면 일반 프로그램이 파일을 읽거나, 화면에 글자를 출력하려면 어떻게 해야 할까요? 바로 시스템 콜(System Call)을 통해 OS에게 부탁하는 거예요!

[비유] 시스템 콜 = 교무실 창구
학생(프로그램)이 교장 선생님(OS)에게 뭔가를 부탁하려면 교무실 창구(시스템 콜)를 통해야 해요.

학생: '선생님, 교실(메모리) 좀 빌려주세요!' -> malloc() -> 시스템 콜
학생: '선생님, 이 서류(파일)를 서류함에 넣어 주세요!' -> fwrite() -> 시스템 콜
학생: '선생님, 방송실(모니터)로 글자 좀 내보내 주세요!' -> printf() -> 시스템 콜

학생이 직접 방송실에 들어가면 안 되는 것처럼, 프로그램도 직접 하드웨어를 건드리면 안 돼요!

컴퓨터의 전원 버튼을 누르면 화면이 나오기까지 무슨 일이 일어날까요? 이 과정을 부팅(Booting)이라고 해요. 부팅은 컴퓨터를 사용할 수 있는 상태로 준비하는 과정이에요.

[주의] system() 함수의 위험성
system() 함수는 강력하지만 조심해야 해요!

1. 보안 위험: 사용자 입력을 그대로 system()에 넣으면 악의적인 명령어가 실행될 수 있어요.

디버깅: OS 관련 흔한 버그 3가지

운영체제와 관련된 프로그래밍에서 자주 발생하는 버그 3가지를 알아볼게요. 이런 버그들은 프로그램이 당장은 잘 돌아가는 것처럼 보이지만, 시간이 지나면서 시스템을 망가뜨릴 수 있어요.

[디버깅 1] 좀비 프로세스 (Zombie Process)
문제: 자식 프로세스가 종료되었는데, 부모 프로세스가 종료 상태를 확인하지 않으면 '좀비 프로세스'가 돼요.

비유: 학생이 하교했는데 출석부에서 지워지지 않은 것처럼, 프로세스 테이블에 쓸모없는 항목이 남아 있어요.

증상: 시스템 자원(프로세스 테이블 슬롯)이 낭비되고, 쌓이면 새 프로세스를 만들 수 없어요.

해결: 부모 프로세스에서 wait() 또는 waitpid()를 호출해서 자식의 종료 상태를 수거해야 해요.
[디버깅 2] 메모리 누수 (Memory Leak)
문제 코드:
void process_data() {
  int *data = malloc(1000 * sizeof(int));
  /* data를 사용... */
  return; /* free(data)를 안 했어요! */
}

문제: malloc()으로 할당한 메모리를 free()로 해제하지 않으면, 그 메모리는 프로그램이 끝날 때까지 낭비돼요.

비유: 도서관에서 책을 빌려놓고 반납하지 않는 것과 같아요. 다른 사람이 그 책을 쓸 수 없어요!

증상: 프로그램이 오래 실행될수록 메모리 사용량이 계속 늘어나서, 결국 시스템이 느려지거나 멈춰요.

해결: malloc()을 쓸 때마다 반드시 짝이 되는 free()를 작성하세요!
[디버깅 3] 파일 핸들 누수 (File Handle Leak)
문제 코드:
void read_file() {
  FILE *fp = fopen("data.txt", "r");
  /* 파일을 읽음... */
  return; /* fclose(fp)를 안 했어요! */
}

문제: fopen()으로 연 파일을 fclose()로 닫지 않으면, 파일 핸들(파일 디스크립터)이 누수돼요.

비유: 도서관 대출 카드를 쓰고 반납하지 않으면, 결국 대출 카드가 바닥나서 아무도 책을 빌릴 수 없는 것과 같아요.

증상: OS가 허용하는 동시 열기 파일 수(보통 수천 개)를 초과하면 더 이상 파일을 열 수 없어요.

해결: fopen() 뒤에는 반드시 fclose()를 쓰세요! 에러가 발생해도 파일을 닫도록 처리하세요.
버그 유형원인비유해결 방법관련 함수
좀비 프로세스자식 종료 상태 미수거출석부에 안 지운 학생wait()/waitpid() 호출fork(), wait()

운영체제의 역사

[역사] 운영체제의 탄생과 발전
1956년 -- 최초의 운영체제 GM-NAA I/O
GM(제너럴 모터스)과 NAA(북미항공)가 IBM 704 컴퓨터를 위해 만들었어요. 사실 '운영체제'라기보다는 한 프로그램이 끝나면 자동으로 다음 프로그램을 실행해주는 간단한 시스템이었어요.

1969년 -- UNIX 탄생
벨 연구소의 켄 톰슨(Ken Thompson)과 데니스 리치(Dennis Ritchie)가 만들었어요. 처음에는 어셈블리어로, 이후 C 언어로 다시 작성되었어요. 현대 운영체제의 조상이에요!

1985년 -- Windows 1.0
마이크로소프트가 첫 Windows를 발표했어요. MS-DOS 위에서 돌아가는 그래픽 환경이었어요.

1991년 -- Linux 커널 탄생
핀란드의 21살 대학생 리누스 토르발스(Linus Torvalds)가 취미로 만들기 시작했어요. 오픈소스로 공개되어 전 세계 개발자가 함께 발전시켰고, 지금은 서버와 스마트폰(Android)의 기반이 되었어요.

2007년 -- iOS / 2008년 -- Android
스마트폰 시대가 열리면서 모바일 운영체제가 등장했어요. 지금은 사람들이 데스크톱보다 모바일을 더 많이 사용해요.

[KOI 1] 프로세스 스케줄링 시뮬레이션
문제: N개의 프로세스가 있고, 각 프로세스의 도착 시간과 실행 시간이 주어진다. FCFS(선입선출) 방식으로 스케줄링할 때, 각 프로세스의 완료 시간과 대기 시간을 구하시오.

입력 예시:
3
0 5 (프로세스 1: 0초 도착, 5초 실행)
1 3 (프로세스 2: 1초 도착, 3초 실행)
2 2 (프로세스 3: 2초 도착, 2초 실행)

풀이:
프로세스 1: 0초 시작, 5초 완료. 대기시간 = 0 - 0 = 0초
프로세스 2: 5초 시작(1이 끝나야), 8초 완료. 대기시간 = 5 - 1 = 4초
프로세스 3: 8초 시작(2가 끝나야), 10초 완료. 대기시간 = 8 - 2 = 6초

평균 대기 시간 = (0 + 4 + 6) / 3 = 약 3.3초

용어 사전

용어설명
운영체제 (Operating System, OS)컴퓨터의 하드웨어와 소프트웨어를 관리하는 시스템 소프트웨어예요. Windows, macOS, Linux 등이 대표적이에요.
프로세스 (Process)메모리에 올라와서 실행 중인 프로그램이에요. 프로그램이 '파일(레시피)'이라면, 프로세스는 '실행 중인 행위(요리)'예요.
커널 (Kernel)운영체제의 핵심 부분으로, 하드웨어를 직접 제어하고 프로세스와 메모리를 관리해요. '알맹이'라는 뜻이에요.
시스템 콜 (System Call)일반 프로그램이 OS 커널에게 서비스를 요청하는 방법이에요. 파일 열기, 메모리 할당 등을 할 때 사용해요.
스케줄링 (Scheduling)여러 프로세스 중 어떤 것을 먼저 실행할지 순서를 정하는 것이에요. FCFS, SJF, Round Robin 등의 방식이 있어요.
타임 퀀텀 (Time Quantum)Round Robin 스케줄링에서 한 프로세스가 CPU를 사용할 수 있는 최대 시간이에요. 보통 10~100ms예요.
파일 시스템 (File System)데이터를 파일과 폴더로 체계적으로 저장하고 관리하는 방법이에요. NTFS, ext4, FAT32 등이 있어요.
부팅 (Booting)컴퓨터의 전원을 켜서 운영체제가 실행되기까지의 과정이에요. BIOS -> 부트로더 -> 커널 순서로 진행돼요.

한눈에 보기

운영체제 기초 -- 한눈에 보기
1. 운영체제(OS)는 하드웨어와 소프트웨어 사이의 중재자예요. 학교의 교장 선생님처럼 모든 것을 관리해요.
2. OS의 5가지 역할: 프로세스 관리, 메모리 관리, 파일 시스템, 입출력 관리, 사용자 인터페이스 제공.
3. 프로그램 = 파일(레시피), 프로세스 = 실행 중인 프로그램(요리). 같은 프로그램으로 여러 프로세스를 만들 수 있어요.
4. 프로세스 상태: 생성 -> 준비 -> 실행 -> 대기 -> 종료. 실행과 준비/대기 사이를 반복해요.
5. 멀티태스킹은 사실 시분할! CPU가 아주 빠르게 프로세스를 번갈아 실행해서 동시에 보이는 거예요.
6. CPU 스케줄링: FCFS(선입선출), SJF(최단작업우선), Round Robin(돌아가며). Round Robin이 가장 공평해요.
7. 파일 시스템: 파일과 폴더의 트리 구조. 절대경로(루트부터)와 상대경로(현재 위치 기준)가 있어요.
8. 커널은 OS의 핵심. 커널 모드(하드웨어 직접 접근)와 사용자 모드(제한된 접근)로 나뉘어요.
9. 시스템 콜은 프로그램이 OS에게 서비스를 요청하는 방법. printf()도 내부적으로 시스템 콜을 사용해요.
10. C 언어는 OS를 만드는 데 가장 많이 사용되는 언어. Linux, Windows 커널 등이 모두 C로 작성되었어요.
