UNIT S-18 소프트웨어 개발 과정

게임, 앱, 웹사이트는 어떻게 만들어질까?

여러분이 매일 사용하는 스마트폰 앱, 컴퓨터 게임, 웹사이트는 모두 소프트웨어(Software)예요. 이런 소프트웨어는 갑자기 뚝딱 나타나는 게 아니라, 정해진 과정을 거쳐서 만들어져요.

집을 지을 때 설계도를 그리고, 공사를 하고, 검사를 받고, 입주하는 것처럼, 소프트웨어도 비슷한 단계를 거쳐요. 이번 단원에서는 소프트웨어가 어떻게 태어나서 사용자에게 전달되는지, 그 전체 과정을 배울 거예요.

[생각해 보기] 프로그램을 만들려면?
여러분이 친구들과 함께 학교 시간표 앱을 만든다고 상상해 보세요.

1. 어떤 기능이 필요할까요? (시간표 보기? 알림?)
2. 화면은 어떻게 생겼으면 좋을까요?
3. 코드를 어떻게 나눠서 작성할까요?
4. 만든 후에 제대로 작동하는지 어떻게 확인할까요?
5. 완성된 앱을 친구들에게 어떻게 나눠줄까요?

이 질문들이 바로 소프트웨어 개발의 각 단계와 연결돼요!

소프트웨어 개발 생명주기 (SDLC)

소프트웨어를 만드는 과정에는 정해진 순서가 있어요. 이것을 소프트웨어 개발 생명주기(SDLC, Software Development Life Cycle)라고 해요. 마치 사람이 태어나서 자라고, 일하고, 은퇴하는 것처럼, 소프트웨어도 탄생부터 은퇴까지 생명주기가 있어요.

[비유] 집 짓기와 소프트웨어 개발
소프트웨어 개발은 집을 짓는 과정과 아주 비슷해요.

1. 요구분석 = 고객이 "방 3개, 화장실 2개인 집을 만들어 주세요" 하는 단계
2. 설계 = 건축사가 설계도(도면)를 그리는 단계
3. 구현 = 공사팀이 실제로 벽돌을 쌓고 집을 짓는 단계
4. 테스트 = 수도, 전기, 가스가 잘 되는지 검사하는 단계
5. 배포 = 열쇠를 건네주고 입주하는 단계
6. 유지보수 = 수도꼭지가 고장 나면 수리하는 단계

1단계: 요구분석

요구분석(Requirements Analysis)은 "무엇을 만들 것인가?"를 정하는 단계예요. 누가 이 프로그램을 사용할 것인지, 어떤 기능이 필요한지, 얼마나 빨라야 하는지를 정리해요.

요구사항은 크게 두 가지로 나뉘어요. 기능 요구사항은 프로그램이 "무엇을 해야 하는가"이고, 비기능 요구사항은 프로그램이 "어떤 조건을 만족해야 하는가"예요.

- 1학년부터 3학년까지 모든 반을 지원해야 한다

2단계: 설계

설계(Design)는 "어떻게 만들 것인가?"를 계획하는 단계예요. 요구분석에서 정한 기능들을 실제로 어떻게 구현할지, 프로그램의 구조를 설계해요. 건축가가 집을 짓기 전에 도면을 그리는 것과 같아요.

설계 단계에서는 순서도(Flowchart)와 의사코드(Pseudocode)를 많이 사용해요. Part 0 U0-27, U0-28에서 배운 순서도와 의사코드가 바로 이 단계에서 쓰이는 거예요!

순서도알고리즘의 흐름을 그림으로 표현도형과 화살표

1. 분해(Decomposition): 큰 문제를 작은 문제로 쪼개요. (U0-23에서 배웠어요!)
2. 추상화(Abstraction): 중요한 것만 남기고 세부 사항은 숨겨요. (U0-25에서 배웠어요!)
3. 모듈화(Modularity): 독립적인 부품(모듈)으로 나눠요. 레고 블록처럼!
4. 재사용성(Reusability): 한 번 만든 모듈을 다른 곳에서도 쓸 수 있게 해요.

3단계: 구현 (코딩)

구현(Implementation)은 설계를 바탕으로 실제 코드를 작성하는 단계예요. 프로그래머가 C, Python, Java 같은 프로그래밍 언어로 코드를 작성하죠. 우리가 배우는 C 언어도 바로 이 단계에서 사용돼요!

코드를 잘 작성하려면 몇 가지 규칙을 지켜야 해요. 변수 이름을 알아보기 쉽게 짓고, 들여쓰기를 깔끔하게 하고, 주석을 달아서 코드를 설명해야 해요.

[주의] 코딩 단계에서 흔한 실수
1. 설계 없이 바로 코딩을 시작하면 나중에 구조를 바꿔야 해서 시간이 더 걸려요.
2. 변수 이름을 a, b, c로 지으면 나중에 자기가 쓴 코드도 이해할 수 없어요.
3. 주석을 안 달면 3개월 후에 "이 코드가 뭐 하는 거지?" 하게 돼요.
4. 한 함수에 너무 많은 기능을 넣으면 수정할 때 다른 기능이 고장 나요.

4단계: 테스트

테스트(Testing)는 만든 프로그램이 제대로 작동하는지 확인하는 단계예요. 아무리 뛰어난 프로그래머라도 실수를 할 수 있기 때문에, 테스트는 반드시 필요해요.

프로그램에서 발견되는 오류를 버그(Bug)라고 해요. 버그는 일찍 발견할수록 고치는 비용이 적게 들어요. 요구분석 단계에서 발견한 오류는 고치기 쉽지만, 배포 후에 발견한 오류는 고치는 데 100배 이상의 비용이 들 수 있어요!

6단계: 유지보수

유지보수(Maintenance)는 배포 후에도 계속 프로그램을 관리하고 개선하는 단계예요. 놀랍게도, 소프트웨어 전체 비용의 60~80%가 유지보수에 들어가요! 만드는 것보다 관리하는 게 훨씬 더 많은 노력이 필요한 거예요.

[핵심] 유지보수 비용이 큰 이유
1. 사용자가 많을수록 다양한 환경에서 버그가 발견돼요.
2. 운영체제, 하드웨어가 계속 바뀌니까 프로그램도 따라 바꿔야 해요.
3. 사용자가 새로운 기능을 계속 요청해요.
4. 시간이 지나면 코드를 작성한 사람이 바뀌어서 이해하기 어려워요.

그래서 처음부터 읽기 쉽고 관리하기 쉬운 코드를 작성하는 것이 매우 중요해요!

개발 방법론: 폭포수 vs 애자일

SDLC의 6단계를 어떤 순서로, 어떤 방식으로 진행할지를 정한 것이 개발 방법론이에요. 대표적인 두 가지를 비교해 볼게요.

폭포수 모델(Waterfall Model)은 각 단계를 순서대로 한 번에 진행하는 방법이에요. 물이 위에서 아래로 떨어지는 폭포처럼, 이전 단계가 끝나야 다음 단계로 넘어가요. 뒤로 돌아가기 어렵다는 단점이 있어요.

애자일(Agile)은 작은 기능을 빠르게 만들고 반복하는 방법이에요. 2~4주 단위의 스프린트(Sprint)를 반복하면서 조금씩 완성해 나가요. 변화에 유연하게 대응할 수 있다는 장점이 있어요.

버전 관리: Git

코드를 작성하다 보면 수정이 많아지고, 여러 사람이 함께 작업하기도 해요. 이때 필요한 것이 버전 관리 시스템(VCS, Version Control System)이에요. 가장 널리 사용되는 버전 관리 도구가 바로 Git이에요.

[비유] Git은 게임 세이브 포인트
Git을 이해하는 가장 쉬운 비유는 게임의 세이브 포인트예요.

- 커밋(commit) = 세이브하기: 현재 상태를 저장해요.
- 체크아웃(checkout) = 이전 세이브 불러오기: 과거 상태로 돌아가요.
- 브랜치(branch) = 세이브 슬롯 나누기: 여러 갈래로 다른 시도를 해볼 수 있어요.
- 머지(merge) = 세이브 합치기: 다른 슬롯의 결과를 하나로 합쳐요.
- 리포지토리(repository) = 세이브 파일 전체: 모든 세이브 기록을 담은 저장소예요.

[C 코드 1] assert를 사용한 간단한 단위 테스트

C 언어에는 assert()라는 매크로가 있어요. 조건이 거짓이면 프로그램을 멈추고 오류 메시지를 출력해요. 이것을 이용하면 간단한 단위 테스트를 할 수 있어요.

Cassert를 사용한 단위 테스트
1#include <stdio.h>
2#include <assert.h>
3
4/* 두 정수의 최댓값을 반환하는 함수 */
5int max(int a, int b) {
6 if (a > b) {
7 return a;
8 }
9 return b;
10}
11
12/* 절댓값을 반환하는 함수 */
13int abs_val(int x) {
14 if (x < 0) {
15 return -x;
16 }
17 return x;
18}
19
20int main(void) {
21 /* max 함수 테스트 */

23 assert(max(10, 2) == 10); /* 첫 번째가 더 큰 경우 */
24 assert(max(7, 7) == 7); /* 같은 경우 */
25 assert(max(-3, -1) == -1); /* 음수끼리 비교 */
26 assert(max(0, -5) == 0); /* 0과 음수 */
27
28 /* abs_val 함수 테스트 */
29 assert(abs_val(5) == 5); /* 양수 */
30 assert(abs_val(-5) == 5); /* 음수 */
31 assert(abs_val(0) == 0); /* 0 */
32
33 printf("모든 테스트 통과!\n");
34 return 0;
35}

[디버깅 1] 테스트 누락으로 인한 버그

다음 코드에서 버그를 찾아보세요. 0으로 나누는 경우를 테스트하지 않아서 발생하는 문제예요.

C버그가 있는 코드 -- 평균 계산
1#include <stdio.h>
2
3int average(int scores[], int count) {
4 int sum = 0;
5 int i;
6 for (i = 0; i < count; i++) {
7 sum += scores[i];
8 }
9 return sum / count; /* 버그! count가 0이면? */
10}
11
12int main(void) {
13 int data[] = {80, 90, 70};
14 int empty[] = {};
15
16 printf("평균: %d\n", average(data, 3));
17 printf("빈 배열 평균: %d\n", average(empty, 0));
18
19 return 0;
20}
[버그 분석]
문제: count가 0일 때 sum / count에서 0으로 나누기(Division by Zero) 오류가 발생해요. 프로그램이 비정상 종료돼요.

원인: 학생이 0명인 경우(빈 배열)를 테스트하지 않았어요. 경계값 테스트가 빠진 거예요.

수정: count가 0인지 먼저 확인해야 해요.
if (count == 0) return 0;
이 한 줄을 sum / count 앞에 추가하면 해결돼요.

[디버깅 2] 경계값 오류

C버그가 있는 코드 -- 등급 판정 경계
1#include <stdio.h>
2
3char get_grade(int score) {
4 if (score > 90) return 'A'; /* 버그! */
5 if (score > 80) return 'B'; /* 버그! */
6 if (score > 70) return 'C';
7 if (score > 60) return 'D';
8 return 'F';
9}
10
11int main(void) {
12 printf("90점: %c\n", get_grade(90));
13 printf("80점: %c\n", get_grade(80));
14 printf("91점: %c\n", get_grade(91));
15
16 return 0;
17}
실행 결과
90점: B 80점: C 91점: A
[버그 분석]
문제: 90점인 학생이 A가 아니라 B를 받아요! 80점인 학생은 C를 받아요.

원인: score > 90이 아니라 score >= 90이어야 해요. > 는 '초과'이고, >= 는 '이상'이에요. 90점은 90 초과가 아니라 90 이상이니까 >= 를 써야 해요.

수정: 모든 비교에서 >를 >=로 바꿔요.
if (score >= 90) return 'A';
if (score >= 80) return 'B';

교훈: 경계값(90, 80, 70, 60)에서 > 와 >= 의 차이는 매우 중요해요. 경계값 테스트가 이런 버그를 잡아줘요!

[디버깅 3] 버전 충돌 문제

C버그가 있는 코드 -- 버전 체크
1#include <stdio.h>
2
3#define VERSION_MAJOR 2
4#define VERSION_MINOR 0
5
6/* 이 함수는 v1.5에서 추가된 기능 */
7int new_feature(int x) {
8 return x * x + 1;
9}
10
11int main(void) {
12 int result;
13
14 /* 버전이 1.5 이상인지 확인 */
15 if (VERSION_MAJOR >= 1 && VERSION_MINOR >= 5) { /* 버그! */
16 result = new_feature(3);
17 printf("결과: %d\n", result);
18 } else {
19 printf("이 기능은 v1.5 이상에서 사용 가능합니다.\n");
20 }
21
22 return 0;
23}
실행 결과
이 기능은 v1.5 이상에서 사용 가능합니다.
[버그 분석]
문제: 현재 버전은 v2.0인데 "v1.5 이상에서 사용 가능"이라는 메시지가 나와요. v2.0은 v1.5보다 높은데!

원인: VERSION_MAJOR >= 1 && VERSION_MINOR >= 5에서, VERSION_MAJOR가 2이고 VERSION_MINOR가 0이면, 0 >= 5가 거짓이라서 전체 조건이 거짓이 돼요.

하지만 v2.0은 v1.5보다 높아요! 메이저 버전이 더 크면 마이너 버전은 상관없어야 해요.

수정:
if (VERSION_MAJOR > 1 || (VERSION_MAJOR == 1 && VERSION_MINOR >= 5))

메이저 버전이 1보다 크면 무조건 통과, 메이저 버전이 1이면 마이너 버전을 확인하는 거예요.

소프트웨어 개발의 역사

[역사] 최초의 버그 (1947년)
1947년 9월 9일, 하버드 대학교의 Mark II 컴퓨터가 오작동했어요. 원인을 조사하니 컴퓨터 안에 진짜 나방(moth)이 끼어 있었어요! 그레이스 호퍼(Grace Hopper) 중위가 이 나방을 꺼내면서 "첫 번째 실제 버그를 발견했다(First actual case of bug being found)"라고 기록했어요. 이후로 프로그램 오류를 버그(bug), 버그를 찾아 고치는 것을 디버깅(debugging)이라고 부르게 되었어요.
[역사] 소프트웨어 위기 (1968년)
1960년대, 컴퓨터가 발전하면서 소프트웨어 프로젝트가 점점 커졌어요. 하지만 체계적인 개발 방법이 없었기 때문에, 프로젝트가 기한을 넘기고, 예산을 초과하고, 버그투성이인 프로그램이 나오는 일이 많았어요. 1968년 NATO 회의에서 이 문제를 '소프트웨어 위기(Software Crisis)'라고 부르며, '소프트웨어 공학(Software Engineering)'이라는 분야가 탄생했어요. 체계적으로 소프트웨어를 만들어야 한다는 인식이 시작된 거예요.
[역사] 애자일 선언문 (2001년)
2001년 2월, 미국 유타주 스노우버드 리조트에 17명의 소프트웨어 개발자들이 모였어요. 이들은 기존의 무거운 개발 방법론(폭포수 등)에 불만을 품고 있었어요. 이 모임에서 '애자일 선언문(Agile Manifesto)'이 탄생했어요.

핵심 가치 4가지:
1. 프로세스와 도구보다 개인과 상호작용을
2. 방대한 문서보다 작동하는 소프트웨어를
3. 계약 협상보다 고객과의 협력을
4. 계획을 따르기보다 변화에 대응하기를 중요하게 여긴다
[역사] Git의 탄생 (2005년)
리눅스 커널을 만든 리누스 토르발스(Linus Torvalds)는 기존 버전 관리 도구에 불만이 있었어요. 그래서 2005년에 직접 새로운 버전 관리 시스템을 만들었는데, 그것이 바로 Git이에요. 리누스는 단 10일 만에 Git의 첫 번째 버전을 완성했다고 해요! Git은 빠르고, 분산형이고, 브랜치 관리가 뛰어나서 지금은 전 세계 개발자의 90% 이상이 Git을 사용해요. GitHub, GitLab 같은 서비스도 Git을 기반으로 만들어졌어요.
