UNIT S-20 컴파일러 원리

printf("Hello")를 치면 컴퓨터 안에서 무슨 일이?

여러분이 C 언어로 작성한 소스코드는 사람이 읽을 수 있는 글자예요. 하지만 CPU는 오직 0과 1로 된 기계어(Machine Code)만 이해할 수 있어요. 그래서 사람이 쓴 코드를 컴퓨터가 이해할 수 있는 기계어로 번역해 주는 프로그램이 필요해요.

이 번역 프로그램이 바로 컴파일러(Compiler)예요. 컴파일러는 여러분이 작성한 C 코드를 읽고, 여러 단계를 거쳐서 CPU가 바로 실행할 수 있는 실행 파일(.exe)로 만들어 줘요.

[생각해 보기] 번역이 필요한 이유
여러분이 영어를 모르는 친구에게 영어 편지를 전달해야 한다고 상상해 보세요.

1. 편지를 그대로 전달하면 친구가 이해할 수 있을까요?
2. 누군가가 한국어로 번역해 줘야 하지 않을까요?
3. 번역할 때 문법이 틀린 부분이 있으면 어떻게 할까요?

컴파일러가 하는 일이 바로 이런 '번역'이에요. C 언어라는 '외국어'를 기계어라는 'CPU의 모국어'로 바꿔 주는 거예요!

프로그래밍 언어의 번역 -- 사람의 언어에서 기계어로

프로그래밍 언어를 기계어로 번역하는 방법은 크게 두 가지가 있어요. 하나는 컴파일러(Compiler)이고, 다른 하나는 인터프리터(Interpreter)예요.

컴파일러는 소스코드 전체를 한꺼번에 읽어서, 처음부터 끝까지 번역한 다음에 실행 파일을 만들어요. 마치 책 전체를 번역한 다음에 번역본을 건네주는 것과 같아요. C, C++, Rust 같은 언어가 컴파일러를 사용해요.

인터프리터는 소스코드를 한 줄씩 읽으면서 바로바로 실행해요. 마치 통역사가 말을 한 문장씩 듣고 바로 통역해 주는 것과 같아요. Python, JavaScript 같은 언어가 인터프리터를 사용해요.

2. 컴파일 (조리법 번역): 한국어 레시피를 요리사가 이해하는 작업 순서로 바꿔요. '감자를 볶다'를 '프라이팬 - 기름 - 감자 - 중불 5분'으로 변환하는 거예요.

3. 어셈블 (실제 조리): 작업 순서대로 실제로 음식을 만들어요. 글자가 실제 동작으로 바뀌는 거예요.

4. 링크 (담아서 완성): 밥, 카레, 반찬을 한 그릇에 담아 완성해요. 여러 부품을 합쳐서 완제품을 만드는 거예요.

1단계: 전처리 (Preprocessor)

전처리(Preprocessing)는 컴파일의 가장 첫 번째 단계예요. 전처리기(Preprocessor)가 소스코드에서 # 으로 시작하는 지시자들을 처리해요. 전처리기는 코드의 의미를 이해하지 않고, 단순히 텍스트를 변환하는 일만 해요.

전처리기가 하는 일을 쉽게 말하면, 본격적인 번역(컴파일) 전에 '준비 작업'을 하는 거예요. 마치 시험 전에 교과서의 중요 부분을 복사해서 노트에 붙여넣고, 약어를 풀어쓰는 것과 같아요.

#include -- 파일 복사 붙여넣기

#include <stdio.h>는 "stdio.h 파일의 내용을 여기에 통째로 복사해서 붙여넣어 줘"라는 뜻이에요. 실제로 전처리기는 stdio.h 파일을 찾아서, 그 안의 수천 줄의 코드를 여러분의 소스코드에 삽입해요.

[비유] #include는 복사-붙여넣기
수학 숙제를 하는데, 공식집이 따로 있다고 해 보세요.

#include <공식집.h> = "공식집 전체를 숙제 앞에 복사해서 붙여넣어!"

이렇게 하면 숙제 안에서 공식을 바로 쓸 수 있게 돼요. printf() 함수를 쓸 수 있는 것도, stdio.h에 printf의 '설명서'가 있기 때문이에요.

#define -- 텍스트 치환

#define은 특정 이름을 다른 텍스트로 바꿔치기하는 지시자예요. 전처리기는 코드에서 그 이름이 나오면, 전부 지정된 텍스트로 교체해요.

#define 정의원본 코드전처리 후 결과
#define PI 3.14area = PI * r * r;area = 3.14 * r * r;
#define MAX 100int arr[MAX];int arr[100];
#define HELLO "안녕"printf(HELLO);printf("안녕");
#define SQUARE(x) ((x)*(x))y = SQUARE(5);y = ((5)*(5));
#define SQUARE(x) ((x)*(x))y = SQUARE(a+1);y = ((a+1)*(a+1));
[주의] 매크로의 함정
만약 #define SQUARE(x) x*x 로 정의했다면 어떻게 될까요?

y = SQUARE(3+2);

우리가 기대하는 결과: (3+2) * (3+2) = 5 * 5 = 25
실제 결과: 3+2 * 3+2 = 3+6+2 = 11 (곱셈이 먼저!)

그래서 매크로를 정의할 때는 반드시 괄호로 감싸야 해요!
#define SQUARE(x) ((x)*(x)) 이렇게요.

#ifdef는 "만약 이 이름이 정의되어 있으면, 아래 코드를 포함해"라는 뜻이에요. 반대로 #ifndef는 "정의되어 있지 않으면 포함해"라는 뜻이에요. 하나의 소스코드로 여러 상황에 맞는 프로그램을 만들 수 있어요.

2단계: 컴파일 (소스코드에서 어셈블리로)

전처리가 끝난 소스코드(.i 파일)는 이제 본격적으로 컴파일러에게 전달돼요. 컴파일러는 C 코드를 읽고, CPU가 이해하기 한 단계 전인 어셈블리 코드(.s 파일)로 번역해요. 이 과정은 여러 세부 단계로 나뉘어요.

어휘 분석 (Lexical Analysis) -- 코드를 단어로 쪼개기

어휘 분석(Lexical Analysis)은 소스코드를 토큰(Token)이라는 의미 있는 최소 단위로 쪼개는 과정이에요. 마치 국어 시간에 문장을 단어 단위로 나누는 것과 같아요.

"나는 학교에 간다"를 [나는] [학교에] [간다]로 나누듯이, 컴파일러도 코드를 [int] [x] [=] [5] [+] [3] [;]으로 나눠요. 이때 빈칸이나 줄바꿈 같은 공백 문자는 버려요.

구문 분석 (Syntax Analysis) -- 문법 체크와 구문 트리

구문 분석(Syntax Analysis)은 토큰들이 C 언어의 문법 규칙에 맞는지 확인하는 과정이에요. 그리고 문법에 맞으면 구문 트리(AST, Abstract Syntax Tree)라는 나무 구조를 만들어요.

국어 시간에 문장을 분석할 때 "주어 + 목적어 + 서술어" 구조로 나누잖아요? 구문 분석도 비슷해요. "int x = 5 + 3;"이 C 언어 문법에 맞는 올바른 문장인지 확인하는 거예요.

노드종류왼쪽 자식오른쪽 자식설명
=대입 연산x+(5,3)최상위: x에 값을 대입
x변수--대입의 왼쪽: 값을 저장할 변수
+덧셈 연산53대입의 오른쪽: 5+3을 계산
5정수 리터럴--덧셈의 왼쪽 피연산자
3정수 리터럴--덧셈의 오른쪽 피연산자

구문 트리(AST)는 코드의 구조를 나무 형태로 표현한 것이에요. 연산의 우선순위도 트리 구조에 자연스럽게 반영돼요. 깊이 있는 노드일수록 먼저 계산되거든요.

[예시] 구문 오류(Syntax Error)가 발생하는 경우
구문 분석 단계에서 문법 오류를 발견하면 컴파일 에러가 발생해요.

int x = 5 +; --> 오류! + 뒤에 피연산자가 없어요
int = 5; --> 오류! 변수 이름이 없어요
int x = 5 --> 오류! 세미콜론(;)이 빠졌어요
if x > 5 {} --> 오류! 조건식에 괄호가 없어요 -- if (x > 5) {}가 맞아요

의미 분석 (Semantic Analysis) -- 뜻이 맞는지 확인

의미 분석(Semantic Analysis)은 문법적으로는 맞지만 의미적으로 틀린 것을 잡아내는 과정이에요. 비유하면, "나는 사과를 마셨다"는 문법은 맞지만 뜻이 이상하잖아요? 의미 분석이 이런 걸 잡아내요.

코드문법의미오류 내용
int x = "hello";맞음틀림정수 변수에 문자열을 넣으려고 함 (타입 불일치)
y = 10;맞음틀림y가 선언되지 않은 변수 (미선언 변수 사용)
int x; int x;맞음틀림같은 이름의 변수를 두 번 선언 (중복 선언)
int arr[5]; arr[10] = 1;맞음경고배열 범위를 벗어난 접근 (일부 컴파일러만 감지)

코드 최적화 -- 불필요한 계산 제거

[핵심] 목적 파일의 구성
목적 파일(.o)에는 다음 내용이 들어 있어요:

1. 코드 섹션 (.text): 기계어로 번역된 프로그램 코드
2. 데이터 섹션 (.data): 초기값이 있는 전역 변수
3. BSS 섹션 (.bss): 초기값이 없는 전역 변수
4. 심볼 테이블: 함수 이름, 변수 이름과 주소 목록
5. 재배치 정보: 아직 주소를 모르는 부분의 위치 정보 (링커가 나중에 채움)

4단계: 링크 (여러 목적 파일 합치기)

링크(Linking)는 컴파일의 마지막 단계예요. 링커(Linker)가 여러 개의 목적 파일(.o)과 라이브러리 파일을 합쳐서 하나의 실행 파일(.exe)을 만들어요.

여러분이 printf를 사용할 때, printf의 코드는 여러분의 .c 파일에 없어요. printf의 실제 기계어 코드는 C 표준 라이브러리(libc)에 들어 있어요. 링커가 여러분의 코드와 이 라이브러리를 연결해 줘요.

정적 링크 vs 동적 링크

라이브러리를 연결하는 방법에는 두 가지가 있어요. 정적 링크(Static Linking)는 라이브러리 코드를 실행 파일 안에 복사해 넣는 방식이고, 동적 링크(Dynamic Linking)는 실행할 때 라이브러리를 불러오는 방식이에요.

[Debugging 3] 매크로 부작용

C[버그 찾기] 매크로의 부작용
1#include <stdio.h>
2
3#define MAX(a, b) (a > b ? a : b)
4
5int main(void) {
6 int x = 3;
7 int y = 5;
8 int result = MAX(x++, y++);
9 printf("result = %d\n", result);
10 printf("x = %d, y = %d\n", x, y);
11 return 0;
12}

에러 원인: MAX(x++, y++)는 전처리 후 (x++ > y++ ? x++ : y++)로 바뀌어요. y++가 두 번 실행돼요! (한 번은 비교할 때, 한 번은 결과를 결정할 때) 그래서 y가 5에서 7로 2가 증가해요.

해결 방법:
1. 매크로 대신 함수를 사용해요. 함수는 인자를 한 번만 평가하니까요.
2. 매크로를 쓸 때는 ++, -- 같은 부작용이 있는 표현식을 인자로 넣지 않아요.

[역사] 그레이스 호퍼 -- 컴파일러의 어머니
그레이스 호퍼(Grace Hopper, 1906~1992)는 미국 해군 제독이자 컴퓨터 과학자예요. 1952년에 세계 최초의 컴파일러 A-0를 만들었어요.

당시 사람들은 '컴퓨터가 코드를 번역할 수 있다'는 생각 자체를 믿지 않았어요. 호퍼가 컴파일러를 발표했을 때, 동료들은 '컴퓨터는 산술만 할 수 있고, 프로그램을 작성할 수는 없다'고 반박했어요.

하지만 호퍼의 컴파일러는 정말 작동했고, 이후 COBOL 언어 개발에도 큰 역할을 했어요. '버그(Bug)'라는 용어를 대중화시킨 것도 호퍼예요. 컴퓨터에서 진짜 나방(bug)을 발견한 일화로 유명하죠!
[역사] GCC와 자유 소프트웨어 운동
GCC(GNU Compiler Collection)는 1987년 리처드 스톨만이 시작한 GNU 프로젝트의 핵심 도구예요.

스톨만은 '모든 소프트웨어는 자유로워야 한다'고 믿었어요. 당시 상용 컴파일러는 비싸서 학생이나 개인이 사용하기 어려웠죠. GCC는 무료이면서 소스코드가 공개된 컴파일러예요.

GCC 덕분에 전 세계 누구나 C 프로그램을 컴파일할 수 있게 되었어요. 리눅스(Linux)도 GCC로 컴파일된 거예요! 지금 여러분이 사용하는 gcc도 이 프로젝트의 후손이에요.
[역사] LLVM/Clang -- 현대 컴파일러의 혁신
LLVM은 2003년 크리스 래트너가 대학원생 시절에 시작한 프로젝트예요. 컴파일러를 레고 블록처럼 모듈화해서, 여러 언어와 여러 CPU를 지원할 수 있게 설계했어요.

Clang은 LLVM을 기반으로 만든 C/C++ 컴파일러예요. GCC보다 컴파일 속도가 빠르고, 에러 메시지가 친절해요. Apple이 적극 지원해서 macOS와 iOS 개발의 기본 컴파일러가 되었어요.

래트너는 이후 Apple에서 Swift 언어도 만들었어요!

컴파일러 원리 -- 한눈에 보기
1. 컴파일러는 C 소스코드 전체를 한꺼번에 기계어로 번역해서 실행 파일을 만들어요. 인터프리터는 한 줄씩 바로 실행해요.
2. C 컴파일 4단계: 전처리(.c->.i) -> 컴파일(.i->.s) -> 어셈블(.s->.o) -> 링크(.o->실행파일).
3. 전처리: #include(파일 복사), #define(텍스트 치환), #ifdef(조건부 포함). 텍스트 변환만 해요.
4. 컴파일 세부 과정: 어휘 분석(토큰 분리) -> 구문 분석(문법 체크, AST) -> 의미 분석(타입 체크) -> 최적화 -> 어셈블리 생성.
5. 어셈블: 어셈블리 명령어를 1:1로 기계어 바이너리로 변환해서 목적 파일(.o)을 만들어요.
6. 링크: 여러 목적 파일과 라이브러리를 합쳐서 실행 파일을 만들어요. 정적/동적 링크 방식이 있어요.
7. 에러 3종류: 컴파일 에러(문법 틀림, 발견 쉬움) > 런타임 에러(실행 중 멈춤) > 논리 에러(결과 틀림, 발견 어려움).
8. gcc 옵션: -E(전처리만), -S(어셈블리까지), -c(목적파일까지), -o(출력이름), -Wall(경고 표시), -O2(최적화).
9. 매크로 사용 시 괄호가 매우 중요해요! #define SQUARE(x) ((x)*(x))처럼 꼭 괄호로 감싸세요.
10. Grace Hopper(1952, 최초 컴파일러), GCC(1987, 자유 소프트웨어), LLVM/Clang(2003, 모듈화 컴파일러).
