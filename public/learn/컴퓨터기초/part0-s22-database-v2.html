UNIT S-22 데이터베이스 기초

학교 성적표도 데이터베이스?

여러분의 학교에는 수백, 수천 명의 학생이 있어요. 각 학생마다 이름, 학번, 학년, 반, 여러 과목의 점수, 출석 기록 등 엄청나게 많은 데이터가 있죠. 이 데이터를 어떻게 정리하고 관리할까요?

만약 이 모든 정보를 종이에 적는다면, 한 학생의 점수를 찾으려면 종이 더미를 뒤져야 하고, 점수를 수정하려면 지우고 다시 써야 해요. 전교생의 수학 평균을 구하려면 계산기를 두드리며 한참을 계산해야 하겠죠.

바로 이런 문제를 해결하기 위해 데이터베이스(Database)가 탄생했어요. 데이터베이스는 데이터를 체계적으로 저장하고, 빠르게 찾고, 안전하게 관리하는 시스템이에요.

[생각해 보기] 데이터를 정리하는 방법
여러분이 반 친구 30명의 연락처를 관리한다고 생각해 보세요.

1. 노트에 적으면 어떤 점이 불편할까요?
2. 엑셀(스프레드시트)에 정리하면 어떤 점이 편할까요?
3. 만약 친구가 1만 명이라면 엑셀로도 충분할까요?
4. 여러 명이 동시에 같은 데이터를 수정해야 한다면?

이런 질문에 대한 답이 바로 데이터베이스가 필요한 이유예요!

데이터베이스란 무엇일까?

데이터베이스(Database, DB)란 여러 사람이 공유해서 사용할 목적으로 체계적으로 관리되는 데이터의 모음이에요. 그냥 데이터를 모아 놓은 것이 아니라, 규칙에 따라 정리하고, 빠르게 검색하고, 안전하게 보관하는 시스템이에요.

일상생활에서 데이터베이스의 예를 찾아볼까요? 도서관의 도서 목록을 생각해 보세요. 책 제목, 저자, 출판사, 위치 등이 체계적으로 정리되어 있어서 원하는 책을 빠르게 찾을 수 있죠. 전화번호부도 마찬가지예요. 이름 순서로 정리되어 있어서 원하는 사람의 번호를 쉽게 찾을 수 있어요.

데이터베이스를 관리하는 소프트웨어를 데이터베이스 관리 시스템(DBMS, Database Management System)이라고 해요. MySQL, Oracle, SQLite 같은 프로그램이 DBMS예요. DBMS가 데이터의 저장, 검색, 수정, 삭제를 담당해요.

관계형 데이터베이스 (RDBMS)

데이터베이스에도 여러 종류가 있는데, 가장 많이 사용되는 것이 관계형 데이터베이스(RDBMS, Relational Database Management System)예요. 관계형 데이터베이스는 데이터를 표(테이블) 형태로 저장해요.

성적 테이블에서 학번과 과목코드는 외래키(FK, Foreign Key)예요. 학생 테이블과 과목 테이블의 기본키를 참조하고 있죠. 이렇게 테이블끼리 연결하는 것이 '관계형'이라는 이름의 이유예요.

[비유] 기본키는 주민등록번호
기본키(Primary Key)는 각 행을 유일하게 구별하는 값이에요.

- 학생 테이블에서 '학번'이 기본키 -- 같은 학번을 가진 학생은 없어요
- 과목 테이블에서 '과목코드'가 기본키 -- 같은 코드를 가진 과목은 없어요

기본키의 규칙:
1. 유일성: 같은 값이 두 번 나오면 안 돼요
2. NOT NULL: 비어 있으면 안 돼요 (값이 반드시 있어야 해요)
3. 변경 불가: 한 번 정하면 바꾸지 않는 게 좋아요

SQL 기초 -- 데이터베이스에게 말하는 언어

데이터베이스에 데이터를 저장하고, 찾고, 수정하고, 삭제하려면 데이터베이스에게 명령을 내려야 해요. 이때 사용하는 언어가 바로 SQL(Structured Query Language, 구조화 질의 언어)이에요.

SQL은 영어 문장처럼 읽히도록 설계되었어요. 예를 들어 "학생 테이블에서 이름을 가져와" 라고 말하고 싶으면, SQL로 SELECT name FROM students라고 쓰면 돼요. 꽤 직관적이죠?

테이블 관계와 외래키 (Foreign Key)

관계형 데이터베이스의 핵심은 테이블끼리 관계(Relationship)를 맺을 수 있다는 거예요. 이 관계를 만들어 주는 것이 외래키(Foreign Key, FK)예요.

테이블 간의 관계에는 세 가지 유형이 있어요.

관계 유형의미예시설명
1:1 (일대일)하나가 하나와 연결학생 - 사물함한 학생은 사물함 하나, 사물함 하나는 학생 하나
1:N (일대다)하나가 여러 개와 연결반 - 학생한 반에 학생 여러 명, 학생 하나는 반 하나에 소속
N:M (다대다)여러 개가 여러 개와 연결학생 - 과목한 학생이 여러 과목 수강, 한 과목에 여러 학생
학번(PK)이름-->성적번호(PK)학번(FK)과목점수
2024001김민수-->12024001국어85
-->22024001수학92
2024002이서연-->32024002국어78
-->42024002수학88

김민수(2024001) 한 명이 국어와 수학, 두 개의 성적을 가지고 있어요. 이것이 1:N(일대다) 관계예요. 학생 한 명(1)이 여러 성적(N)과 연결되는 거죠.

JOIN -- 두 테이블 합치기

JOIN은 두 개의 테이블을 합쳐서 하나의 결과로 보여주는 명령이에요. 학생 테이블에는 이름이 있고, 성적 테이블에는 점수가 있는데, '이름과 점수를 함께' 보고 싶을 때 JOIN을 사용해요.

CJOIN 사용 예시
1-- 학생 이름과 점수를 함께 보기
2SELECT students.name, subjects.subject_name, grades.score
3FROM grades
4JOIN students ON grades.student_id = students.student_id
5JOIN subjects ON grades.subject_code = subjects.subject_code;

정규화 기초 -- 데이터 중복 줄이기

정규화(Normalization)란 데이터의 중복을 최소화하도록 테이블을 설계하는 방법이에요. 같은 정보를 여러 곳에 적어 놓으면, 수정할 때 모두 다 고쳐야 하니까 불편하고 실수할 수 있거든요.

[비유] 정규화는 정리 정돈
여러분의 방에 같은 교과서가 3권씩 있다고 생각해 보세요.

- 교과서가 개정되면 3권 모두 바꿔야 해요 (수정 번거로움)
- 2권만 바꾸고 1권을 잊으면 서로 다른 내용이 돼요 (데이터 불일치)
- 같은 책이 3권이나 있으니 공간 낭비예요 (저장 공간 낭비)

교과서를 1권만 두면 이런 문제가 다 해결되죠? 이것이 정규화의 원리예요!

트랜잭션 -- 은행 이체는 반드시 둘 다 성공해야!

트랜잭션(Transaction)이란 여러 개의 작업을 하나의 묶음으로 처리하는 것을 말해요. 묶음 안의 작업이 모두 성공하면 완료(Commit)하고, 하나라도 실패하면 전부 취소(Rollback)해요.

[비유] 은행 계좌 이체
여러분이 친구에게 10,000원을 이체한다고 해 보세요. 이 작업은 두 단계예요.

1단계: 내 계좌에서 10,000원을 빼기
2단계: 친구 계좌에 10,000원을 넣기

만약 1단계만 성공하고 2단계가 실패하면? 내 돈은 빠졌는데 친구는 못 받은 거예요! 10,000원이 사라진 거죠!

트랜잭션은 이런 일이 생기지 않게, 두 단계를 하나로 묶어서 '둘 다 성공 아니면 둘 다 취소'로 처리해요.

트랜잭션은 ACID라는 4가지 속성을 만족해야 해요.

데이터베이스의 역사

[역사] 최초의 데이터베이스 IMS (1966년)
1960년대, NASA의 아폴로 우주 프로그램에서 엄청난 양의 부품 데이터를 관리해야 했어요. IBM은 이를 위해 IMS(Information Management System)라는 최초의 상용 데이터베이스를 만들었어요 (1966년). IMS는 데이터를 나무(트리) 구조로 저장하는 계층형 데이터베이스였어요. 달에 사람을 보내기 위해 데이터베이스 기술이 발전한 거예요!
[역사] 관계형 모델의 탄생 -- 에드거 코드 (1970년)
1970년, IBM 연구원 에드거 F. 코드(Edgar F. Codd)가 논문 한 편을 발표했어요. '대규모 공유 데이터 뱅크를 위한 관계형 모델'이라는 제목의 이 논문에서, 데이터를 표(테이블) 형태로 저장하고 수학적 관계로 연결하는 방법을 제안했어요. 이것이 오늘날 우리가 사용하는 관계형 데이터베이스(RDBMS)의 시작이에요. 코드 박사는 이 공로로 1981년 튜링상을 받았어요.
[역사] SQL의 탄생 (1974년)
코드의 관계형 모델을 실제로 사용하려면 명령어가 필요했어요. 1974년, IBM의 레이 보이스(Ray Boyce)와 도널드 챔벌린(Donald Chamberlin)이 SEQUEL이라는 언어를 만들었어요. 이것이 나중에 이름이 바뀌어서 SQL(Structured Query Language)이 되었어요. SQL은 영어 문장처럼 읽히도록 설계되어서 프로그래머가 아닌 사람도 쉽게 배울 수 있어요.
[역사] MySQL과 오픈소스 DB (1995년)
1995년, 스웨덴의 미카엘 비데니우스(Michael Widenius)와 데이비드 액스마크(David Axmark)가 MySQL을 만들었어요. MySQL의 'My'는 비데니우스의 딸 이름이에요. MySQL은 무료 오픈소스여서 폭발적으로 인기를 얻었고, 지금도 전 세계에서 가장 많이 사용되는 데이터베이스 중 하나예요. 구글, 페이스북, 유튜브 등이 MySQL을 사용했어요.
[역사] NoSQL 시대의 개막 (2009년)
2000년대 후반, 페이스북, 트위터, 구글 같은 서비스에서 하루에 수십억 건의 데이터가 쏟아졌어요. 기존 관계형 데이터베이스로는 이 엄청난 양을 처리하기 어려웠죠. 2009년, NoSQL이라는 용어가 공식적으로 사용되기 시작했고, MongoDB, Cassandra, Redis 같은 비관계형 데이터베이스가 등장했어요. 지금은 SQL과 NoSQL을 용도에 맞게 함께 사용하는 시대예요.

용어설명
데이터베이스 (Database, DB)여러 사람이 공유하여 사용할 목적으로 체계적으로 관리되는 데이터의 모음이에요.
DBMS (Database Management System)데이터베이스를 관리하는 소프트웨어예요. MySQL, Oracle, SQLite 등이 있어요.
테이블 (Table)관계형 데이터베이스에서 데이터를 저장하는 표 형태의 구조예요. 행(Row)과 열(Column)로 구성돼요.
기본키 (Primary Key, PK)테이블에서 각 행을 유일하게 구별하는 값이에요. 중복되지 않고 비어 있을 수 없어요.
외래키 (Foreign Key, FK)다른 테이블의 기본키를 참조하는 열이에요. 테이블 간의 관계를 만들어요.
SQL (Structured Query Language)데이터베이스에 명령을 내리는 언어예요. SELECT, INSERT, UPDATE, DELETE 등의 명령이 있어요.
트랜잭션 (Transaction)여러 작업을 하나의 묶음으로 처리하는 것이에요. 모두 성공하면 COMMIT, 하나라도 실패하면 ROLLBACK해요.
정규화 (Normalization)데이터의 중복을 줄이기 위해 테이블을 나누는 설계 방법이에요. 데이터 불일치를 방지해요.

한눈에 보기

데이터베이스 기초 -- 한눈에 보기
1. 데이터베이스는 데이터를 체계적으로 저장하고, 빠르게 검색하고, 안전하게 관리하는 시스템이에요.
2. 관계형 데이터베이스(RDBMS)는 데이터를 표(테이블) 형태로 저장해요. 행(Row)=레코드, 열(Column)=필드예요.
3. 기본키(PK)는 각 행을 유일하게 구별하는 값이에요. 외래키(FK)는 테이블을 연결하는 다리예요.
4. SQL은 데이터베이스에 명령을 내리는 언어예요. CRUD: Create(INSERT), Read(SELECT), Update(UPDATE), Delete(DELETE).
5. WHERE로 조건 필터링, ORDER BY로 정렬, JOIN으로 테이블 합치기를 해요.
6. 정규화는 데이터 중복을 줄이기 위해 테이블을 나누는 방법이에요. 같은 정보는 한 곳에만 저장해요.
7. 트랜잭션은 여러 작업을 하나로 묶어요. ACID(원자성, 일관성, 격리성, 지속성)를 만족해야 해요.
8. NoSQL은 테이블이 아닌 다양한 형태(키-값, 문서, 그래프)로 데이터를 저장하는 데이터베이스예요.
9. C 구조체 배열은 메모리 속 간이 테이블이에요. 파일 I/O를 추가하면 간이 데이터베이스가 돼요.
10. 관계형 모델은 1970년 코드(Codd)가 제안했고, SQL은 1974년에 탄생했어요. 1995년 MySQL, 2009년 NoSQL 시대가 열렸어요.
