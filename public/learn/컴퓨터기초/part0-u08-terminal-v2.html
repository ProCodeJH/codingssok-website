UNIT 08 터미널과 명령어

두 가지 방법으로 컴퓨터 다루기

컴퓨터를 다루는 방법은 크게 두 가지예요.

첫 번째는 GUI(Graphical User Interface). 우리가 평소에 쓰는 방식이에요. 아이콘을 마우스로 클릭하고, 창을 끌어다 놓고, 메뉴에서 골라요. 탐색기, 웹 브라우저가 전부 GUI예요.

두 번째는 CLI(Command Line Interface). 까만 화면에 글자로 명령어를 직접 입력하는 방식이에요. 이 까만 화면을 터미널 또는 명령 프롬프트라고 불러요.

왜 터미널을 배울까?

터미널을 쓰는 이유는 간단해요. C언어 코드를 실행하려면 컴파일이라는 과정이 필요한데, 이 과정이 터미널에서 이루어져요. 나중에 이런 명령어를 칠 거예요:

[핵심] 터미널이 필요한 이유 C 프로그래밍의 기본 흐름:
1. 코드 파일(hello.c)을 만든다
2. 터미널에서 gcc hello.c -o hello 입력 (컴파일)
3. 터미널에서 ./hello 입력 (실행)
4. 결과가 터미널 화면에 나타난다!

이 흐름을 알아야 Part 1에서 바로 코딩을 시작할 수 있어요.

활동: 터미널로 폴더 만들기

[활동] 터미널 실습 터미널을 열고 아래 명령어를 순서대로 입력해요:

1. mkdir coding (코딩 폴더 만들기)
2. cd coding (코딩 폴더로 이동)
3. dir (안에 뭐가 있나 확인 → 아직 비어있어요)
4. mkdir part0 (part0 폴더 만들기)
5. mkdir part1 (part1 폴더 만들기)
6. dir (폴더 2개가 보이면 성공!)
7. cd .. (한 단계 위로 나가기)

명령어를 치고 Enter를 눌러야 실행돼요!
[주의] 자주 하는 실수 1. 오타 — mkdir을 mkdri로 치면 안 돼요. 철자를 정확히!
2. 띄어쓰기 — 'cd coding'에서 cd와 coding 사이에 공백 하나!
3. Enter 누르기 — 명령어를 다 치고 Enter를 눌러야 실행돼요
4. 현재 위치 확인 — 프롬프트(C:\Users\...)가 지금 어디에 있는지 알려줘요

터미널의 역사

[역사] 터미널은 진짜 기계였다! 1960~70년대에 '터미널'은 진짜 물리적인 기계였어요. 텔레타이프(Teletype, TTY)라는 기계로 키보드를 치면 종이에 글자가 찍혀 나왔어요. 나중에 모니터가 달린 VT100 같은 단말기가 등장했고, 지금의 소프트웨어 터미널은 이 기계들을 흉내 내는 거예요.

터미널에서 쓰는 tty라는 명령어의 이름이 바로 Teletype에서 왔어요!
[역사] Unix와 셸의 탄생 1969년: Ken Thompson과 Dennis Ritchie가 AT&T 벨 연구소에서 Unix를 만들었어요.
1971년: 최초의 Unix 셸 — Thompson shell
1979년: Bourne shell(sh) — 본격적인 스크립팅 지원
1989년: bash(Bourne Again Shell) — 지금까지도 가장 많이 쓰이는 셸
2006년: Microsoft가 PowerShell을 발표

C언어를 만든 Dennis Ritchie가 Unix도 만들었어요. C와 터미널은 태생부터 하나예요!

[심화 내용] 셸의 내부 동작: 시스템 콜

셸에서 gcc hello.c -o hello를 입력하면, 내부적으로 다음 과정이 일어나요.

C
1
[심화 내용] 핵심 시스템 콜 3가지 fork() — 현재 프로세스를 복제해서 자식 프로세스 생성
exec() — 현재 프로세스의 메모리를 새 프로그램으로 교체
wait() — 자식 프로세스가 끝날 때까지 대기

셸은 이 세 가지 시스템 콜의 무한 반복이에요!
이것이 Unix 철학의 핵심이며, 모든 프로세스 생성의 기반이에요.

fork + exec 모델은 Unix/Linux의 근간이에요. Windows는 CreateProcess()라는 단일 시스템 콜로 프로세스를 생성하는데, fork와 exec를 합친 것과 비슷해요.

ls | grep txt를 실행하면 셸 내부에서 어떤 일이 일어나는지 봐요.

[심화 내용] 파이프 = 커널 버퍼 pipe() 시스템 콜은 커널 내부에 버퍼(기본 64KB)를 만들어요.
dup2()로 프로세스의 표준 입출력(stdin/stdout)을 이 버퍼에 연결하면, 프로세스는 자신의 출력이 파이프로 가는지 화면으로 가는지 모르고 그냥 출력해요.

이것이 Unix의 핵심 설계 철학인 '작은 도구를 파이프로 연결'을 가능하게 해요.

[심화 내용] 터미널 에뮬레이터와 PTY

현대의 '터미널'은 실제로는 터미널 에뮬레이터예요. 1970년대 VT100 같은 하드웨어 단말기를 소프트웨어로 흉내 내는 거예요.

내부적으로는 PTY(Pseudo Terminal)라는 가상 장치를 사용해요. PTY는 두 부분으로 나뉘어요:

줄코드역할
1PTY Master터미널 에뮬레이터(예: Windows Terminal, iTerm2)가 이 쪽을 잡고 있음. 화면 출력과 키보드 입력을 중계
2PTY Slave셸(bash, zsh 등)이 이 쪽에 연결됨. 프로그램은 이것이 진짜 터미널인 줄 알고 동작
3라인 디시플린Ctrl+C(SIGINT), Ctrl+Z(SIGTSTP) 같은 제어 문자를 신호로 변환하는 커널 모듈
[심화 내용] SSH도 PTY를 쓴다 원격으로 서버에 접속하는 SSH도 원격 서버에 PTY를 만들어서 동작해요.
그래서 SSH로 접속해도 마치 그 컴퓨터 앞에 앉아있는 것처럼 터미널을 쓸 수 있는 거예요.

용어 정리

용어설명
GUIGraphical User Interface. 그래픽(아이콘, 창, 메뉴)을 마우스로 조작하는 방식.
CLICommand Line Interface. 텍스트 명령어를 키보드로 입력하는 방식.
터미널 (terminal)CLI를 사용하는 프로그램. 까만 화면에 명령어를 입력해요. Windows에서는 '명령 프롬프트'라고도 해요.
명령 프롬프트Windows의 기본 터미널 프로그램. cmd라고도 불러요.
프롬프트 (prompt)터미널에서 명령어를 입력할 수 있는 위치를 알려주는 표시. C:\Users\나> 같은 거예요.
셸 (shell)명령어를 해석하고 실행하는 프로그램. bash, zsh, cmd, PowerShell 등이 있어요.
와일드카드 (wildcard)파일 이름 패턴 매칭에 쓰이는 특수 문자. *는 아무 문자열, ?는 한 글자를 대신해요.
리다이렉션 (redirection)명령어의 출력을 화면 대신 파일로 보내는 기능. > (덮어쓰기), >> (이어쓰기).
파이프 (pipe)앞 명령어의 출력을 다음 명령어의 입력으로 연결하는 기능. | 기호를 사용.
환경 변수 (environment variable)운영체제가 참조하는 설정값. PATH는 실행 파일을 찾는 경로 목록.
셸 스크립트 (shell script)여러 명령어를 파일에 모아 놓고 한 번에 실행하는 자동화 도구. .sh(Linux), .bat(Windows).
fork()[심화 내용] 현재 프로세스를 복제하여 자식 프로세스를 생성하는 Unix 시스템 콜.
exec()[심화 내용] 현재 프로세스의 메모리를 새 프로그램으로 교체하는 시스템 콜.
PTY (Pseudo Terminal)[심화 내용] 하드웨어 터미널을 소프트웨어로 에뮬레이션하는 가상 장치.
시그널 (signal)[심화 내용] 프로세스에 보내는 비동기 알림. SIGINT(Ctrl+C), SIGKILL(강제종료) 등.
이번 유닛 핵심 정리
1. GUI는 마우스로, CLI는 키보드 명령어로 컴퓨터를 조작해요
2. 터미널 기본 명령어: dir(목록), cd(이동), mkdir(폴더 만들기), cls(화면 지우기)
3. 파일 명령어: type/cat(보기), copy/cp(복사), del/rm(삭제)
4. 와일드카드(*,?)로 여러 파일을 한 번에 지정
5. 리다이렉션(>)은 결과를 파일로, 파이프(|)는 결과를 다른 명령어로 연결
6. PATH 환경 변수가 명령어를 찾는 경로를 결정
7. 셸 스크립트로 반복 작업을 자동화할 수 있어요
8. C언어 컴파일은 터미널에서 해요 — 그래서 터미널을 알아야 해요
9. [심화 내용] 셸 = fork + exec + wait의 무한 반복
