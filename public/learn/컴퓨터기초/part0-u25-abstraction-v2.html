중요한 것만 남기고 나머지는 무시! 추상화는 복잡한 현실을 단순한 모델로 바꾸는 핵심 기술이에요.

정보가 너무 많아! 뭘 골라야 하지?

현실 세계에는 정보가 너무 많아요. 프로그래밍할 때 이 모든 걸 다 담을 수는 없어요. 그래서 목적에 맞는 중요한 정보만 골라내고 나머지는 과감하게 무시하는 것이 필요해요. 이것을 추상화(Abstraction)라고 해요.

[핵심] 추상화란? 추상화(Abstraction) = 중요한 것만 남기고 나머지는 무시하는 것

핵심 원칙:
1. 목적에 따라 필요한 정보가 달라요
2. 복잡한 것을 단순하게 만들어요
3. 핵심만 남기면 이해하기 쉬워요
4. 불필요한 세부사항을 숨겨요

예시: 지도 = 세계의 추상화!

지도에는 건물의 벽돌 색, 나무의 나이, 지나가는 사람 같은 건 없어요. 오직 위치와 이름만 있어요. 왜? 길 찾기라는 목적에 그것만 필요하니까요!

[C 연결] 추상화의 3가지 도구 1. 변수 = 현실의 추상화

2. 함수 = 행동의 추상화

추상화는 한 단계가 아니에요. 여러 층(레벨)으로 나뉘어요. 높은 레벨일수록 간단하고, 낮은 레벨일수록 자세해요.

[심화] 블랙박스 — 안은 몰라도 써!

[심화] 블랙박스 추상화 블랙박스 = 내부 동작은 숨기고, 입력과 출력만 알면 쓸 수 있게 하는 것

예시:
• printf() → 내부에서 시스템콜, 버퍼링 등 복잡한 일을 하지만 우리는 그냥 호출

• 자동차 → 엔진 원리 몰라도 핸들, 페달로 운전 가능
블랙박스입력 (우리가 하는 것)출력 (결과)숨겨진 내부 동작
TV 리모컨채널 버튼 누르기채널이 바뀜적외선 발사, 신호 디코딩, 채널 튜닝

[심화 내용] ADT — 추상 자료형

[심화 내용] ADT (Abstract Data Type) ADT = 데이터 + 연산을 하나로 묶되, 구현을 숨기는 것

예: 스택 ADT
• 데이터: 정수들의 모음
• 연산: push(넣기), pop(빼기), peek(보기), isEmpty(비었나?)
• 구현? 배열이든 연결 리스트든 사용자는 몰라도 됨!

ADT = 인터페이스만 정의, 구현은 자유
ADT 이름주요 연산실생활 비유가능한 구현 방식
스택 (Stack)push, pop, peek접시 쌓기 (위에서만 넣고 빼기)배열 또는 연결 리스트
큐 (Queue)enqueue, dequeue, front줄 서기 (앞에서 빠지고 뒤에서 들어옴)배열 또는 연결 리스트
리스트 (List)insert, delete, get출석부 (순서대로 이름 나열)배열 또는 연결 리스트
맵 (Map)put, get, remove사전 (단어로 뜻 찾기)해시 테이블 또는 트리

1#include <stdio.h>
2#define MAX 100
3
4// ===== 스택 ADT =====
5typedef struct {
6 int data[MAX]; // 구현 세부사항 (배열 사용)
7 int top; // 구현 세부사항
8} Stack;
9
10// --- 인터페이스 (사용자가 쓰는 함수들) ---
11void init(Stack *s) { s->top = -1; }
12int isEmpty(Stack *s) { return s->top == -1; }
13void push(Stack *s, int v) { s->data[++(s->top)] = v; }
14int pop(Stack *s) { return s->data[(s->top)--]; }
15int peek(Stack *s) { return s->data[s->top]; }
16
17int main(void) {
18 Stack s;
19 init(&s);
20 push(&s, 10);
21 push(&s, 20);
22 push(&s, 30);
23 printf("top: %d\n", peek(&s)); // 30
24 printf("pop: %d\n", pop(&s)); // 30
25 printf("pop: %d\n", pop(&s)); // 20
26 return 0;
27}

[심화 내용] 정보 은닉 (Information Hiding)

[심화 내용] 정보 은닉 원칙 정보 은닉 = 내부 구현을 외부에서 접근하지 못하게 숨기는 것

왜 숨길까?
1. 사용자가 내부를 직접 건드리면 버그 발생
2. 구현을 바꿀 때 사용자 코드에 영향 없음
3. 모듈 간 독립성 보장

C에서의 정보 은닉:
• static 키워드: 파일 내부에서만 접근 가능
• 불완전 타입: 구조체 선언만 노출, 정의는 숨김
• 헤더 파일: 인터페이스만 공개

[심화 내용] 컴퓨터 과학의 추상화 계층

[심화 내용] 추상화 계층 스택 컴퓨터의 추상화 계층 (아래→위):

1. 물리 계층: 전자, 트랜지스터, 전압
2. 논리 게이트: AND, OR, NOT (전압→0/1)
3. 디지털 회로: 가산기, MUX, 레지스터
4. 마이크로아키텍처: 파이프라인, 캐시
5. ISA(명령어 집합): MOV, ADD, CMP (x86, ARM)
6. 어셈블리어: 사람이 읽을 수 있는 기계어
7. 운영체제: 프로세스, 파일, 메모리 관리
8. 프로그래밍 언어: C, Python, Java
9. 라이브러리/프레임워크: React, TensorFlow
10. 애플리케이션: 웹 브라우저, 게임

각 계층은 아래 계층의 복잡성을 숨기고
위 계층에 단순한 인터페이스를 제공!

[심화 내용] 누수되는 추상화

[심화 내용] Leaky Abstraction (Joel Spolsky, 2002) "모든 비자명 추상화는 어느 정도 누수된다"

추상화가 내부 세부사항을 완벽히 숨기지 못하는 현상

예시:
1. TCP는 "신뢰할 수 있는 전송"을 추상화하지만
네트워크가 끊기면 타임아웃 에러!

2. SQL은 "데이터 쿼리"를 추상화하지만
인덱스가 없으면 느려짐 → 내부 구조 알아야 함

3. 가비지 컬렉션은 "메모리 관리"를 추상화하지만
GC 일시정지가 성능에 영향 → 메모리 패턴 알아야 함

4. ORM은 "데이터베이스 접근"을 추상화하지만
N+1 쿼리 문제 → SQL 알아야 해결

결론: 추상화를 사용하되, 한 단계 아래도 이해하라!

[심화 내용] API — 추상화의 경계

API(Application Programming Interface)는 추상화의 경계를 정의하는 계약이에요.

[심화 내용] 좋은 추상화 설계 좋은 API의 원칙:

1. 최소 노출 원칙: 필요한 것만 공개
printf()만 알면 됨. 내부 버퍼 관리는 숨김

2. 일관성: 비슷한 것은 비슷하게
fopen/fclose, malloc/free (쌍으로)

3. 직교성: 독립적인 기능은 독립적으로
입출력과 계산은 분리

4. 계층적: 단계별로 추상화
하드웨어 → 드라이버 → 파일시스템 → fopen

David Parnas (1972):
"정보 은닉(Information Hiding)" 원칙 제안
→ 모듈은 내부 결정을 숨기고 인터페이스만 공개
C[C 코드] 추상화 수준 비교
1// 추상화 레벨 1: 기계어 수준
2// MOV EAX, [0x1000]
3// ADD EAX, [0x1004]
4// MOV [0x1008], EAX
5
6// 추상화 레벨 2: C 코드
7int sum = a + b;
8
9// 추상화 레벨 3: 함수
10int add(int a, int b) { return a + b; }
11
12// 추상화 레벨 4: 제네릭 (C는 없지만 개념적으로)
13// template<T> T add(T a, T b) { return a + b; }
14
15// 추상화 레벨 5: 고수준 라이브러리
16// result = numpy.add(array1, array2) // 수백만 원소 병렬 덧셈
17
18// 같은 "더하기"인데 추상화 수준에 따라

용어설명
추상화 (Abstraction)복잡한 현실에서 목적에 맞는 중요한 정보만 골라내고 나머지를 무시하는 것.
모델 (Model)현실을 추상화해서 만든 단순화된 표현. 지도, 설계도 등.
블랙박스 (Black Box)내부 구현은 숨기고 입력/출력만 보여주는 것.
인터페이스외부에서 보이는 사용 방법. 함수 선언, 메뉴판.
구현내부에서 실제로 동작하는 방식. 함수 정의, 주방.
ADTAbstract Data Type. 데이터+연산을 정의하되 구현을 숨기는 추상화.
정보 은닉내부 구현을 외부에서 접근 못하게 숨기는 원칙. static, 불완전 타입.
구조체 (Struct)관련 변수를 묶어 새 타입을 만드는 C의 추상화 도구.
추상화 레벨추상화의 층(단계). 높을수록 간단, 낮을수록 자세.
OSI 7 Layer네트워크 통신의 7단계 추상화 모델.
추상화 계층복잡한 시스템을 여러 단계로 나누어 각 단계가 아래의 복잡성을 숨기는 구조.
Leaky Abstraction추상화가 내부 세부사항을 완벽히 숨기지 못하고 "누수"되는 현상.
APIApplication Programming Interface. 추상화의 경계를 정의하는 계약.
정보 은닉모듈이 내부 결정을 숨기고 인터페이스만 공개하는 원칙. Parnas, 1972.
ISAInstruction Set Architecture. CPU가 이해하는 명령어 집합. 하드웨어/소프트웨어 경계.
